Account, hierarchy & billing
OrganizationINFRA
The root node of GCP's resource hierarchy, tied to a verified domain (e.g. hawaii.edu). One per institution. Owns all folders, projects, and resources beneath it.
See also: Folder, ProjectIntroduced in M1L1
FolderINFRA
A grouping container under the organization, used for divisions, centers, or labs (e.g. uhcc → shepherd-research-lab). Policies set on a folder cascade to every project inside.
See also: Organization, Org policyM1L1
ProjectINFRA
The unit of work in GCP. APIs, billing, IAM, quotas, and VPCs are all project-scoped. The recommended default: one project per study or environment (e.g. hipimr-prod, makawalu-dev).
See also: Billing account, APIM1L1
ResourceINFRA
Any concrete GCP service instance — a Cloud Run service, a GCS bucket, a BigQuery dataset. Every resource lives inside exactly one project.
See also: ProjectM0L2
Billing accountINFRA
The payment entity that funds resources. A billing account can fund many projects; a project must be linked to exactly one. Used for grant accounting — link halo-prod to the HALO R01 billing account, not your discretionary card.
See also: Budget alert, BAAM0L1
Budget alertINFRA
A configured threshold (e.g. 50%, 90%, 100% of monthly budget) that emails or notifies when spend crosses it. Does not stop charges — pair with a kill-switch Cloud Function if you need a hard cap.
See also: Billing accountM0L1
QuotaINFRA
A per-project, per-region ceiling on how much of a resource you can use (GPUs, CPU cores, API requests per minute). Most quotas start low; you raise them via a console request.
See also: RegionM0L2
Org policyHIPAA
A constraint applied at the organization or folder level — e.g. "no external IPs," "only US regions," "require CMEK." Inherited by every project beneath. The primary tool for institutional guardrails.
See also: CMEK, VPC SCM3
Identity & access (IAM)
IAMINFRA
Identity and Access Management. GCP's access control system: who (principal) can do what (role) on which resource. Every action runs as some IAM identity.
See also: Principal, Role, Service accountM1L1
PrincipalINFRA
An identity that IAM can grant a role to: a Google user, a Google Group, a service account, or a federated identity. Formerly called a "member."
See also: Role, Service accountM0L2
IAM roleINFRA
A bundle of permissions you grant to a principal. Three flavors: basic (Owner/Editor/Viewer — too broad, avoid), predefined (per-service, e.g. roles/run.invoker), custom (you author).
See also: Permission, Least privilegeM0L2
PermissionINFRA
A single allowed action, like storage.buckets.get. You don't grant permissions directly — you bundle them into a role and grant the role.
See also: IAM roleM0L2
Service accountINFRA
A non-human identity for code. Cloud Run services, GKE pods, and Vertex training jobs all run as service accounts. Never put personal credentials in code — always use a service account scoped to only what that workload needs.
See also: Workload Identity, ADCM1L1
Workload IdentityINFRA
The mechanism that lets a GKE pod or external workload assume a GCP service account without long-lived key files. The right way to give pods access to GCP APIs.
See also: Service account, GKEM5
ADC (Application Default Credentials)INFRA
A credential lookup chain Google SDKs use automatically: env vars → gcloud auth application-default login file → attached service account. Set up once locally with the gcloud command; running on GCP, ADC just works.
See also: Service account, gcloud CLIM0L1
Least privilegeHIPAA
The principle that every identity should have only the permissions it actually needs — nothing more. The default posture for every service account you create.
See also: Service account, IAM roleM1L1
Secret ManagerHIPAA
GCP's managed store for API keys, DB passwords, and other secrets. Versioned, IAM-controlled, audit-logged. Cloud Run and GKE can mount secrets directly without baking them into images.
See also: CMEK, Service accountM3
Tooling & developer environment
Cloud ConsolePREREQ
The web UI for GCP at console.cloud.google.com. Good for exploring, viewing logs, and one-off changes. Anything you do repeatedly should move to gcloud or Terraform.
See also: gcloud CLI, Cloud ShellM0L1
Cloud ShellPREREQ
A free browser-based Linux VM with gcloud, kubectl, Docker, and an editor pre-installed. Pre-authenticated to your Google identity. Best place to start before installing anything locally.
See also: Cloud Console, gcloud CLIM0L1
gcloud CLIPREREQ
The primary command-line tool for GCP, part of the Google Cloud SDK. Manages projects, IAM, deploys to Cloud Run, drives GKE — every service has gcloud commands.
See also: gsutil, bq, kubectlM0L1
gcloud configurationsPREREQ
Named sets of gcloud settings (account + project + region). Switch between studies with one command: gcloud config configurations activate hipimr-prod. Use this instead of editing properties one at a time.
See also: gcloud CLIM0L1
gsutil / gcloud storagePREREQ
CLI for Cloud Storage. gcloud storage is the newer, faster replacement for gsutil and is what we use throughout the course.
See also: GCS, gcloud CLIM0L1
bqOMOP
The BigQuery CLI. Used for queries, loads, dataset management. Heavy ATLAS / OMOP work runs through it.
See also: BigQueryM9
kubectlINFRA
The Kubernetes CLI. Manages clusters, pods, deployments, services. gcloud container clusters get-credentials wires up kubectl for a GKE cluster.
See also: GKE, PodM5
Cloud CodePREREQ
Google's VS Code (and JetBrains) extension. Adds GKE/Cloud Run dev loops, YAML schemas, log viewers, and Secret Manager browser inside the editor. Optional but very useful.
See also: Docker DesktopM0L1
Docker DesktopPREREQ
The local container runtime you need to build images on your laptop (Mac, Windows). Rancher Desktop and OrbStack are licence-friendly alternatives. Cloud Shell already has Docker.
See also: Container, ImageM0L1
TerraformINFRA
HashiCorp's infrastructure-as-code tool. Declare resources in .tf files; terraform apply reconciles GCP to match. Anything you build by hand more than twice should move into Terraform.
See also: Cloud BuildM11
Compute
Compute Engine (GCE)INFRA
Raw VMs. Maximum control, maximum ops burden. Useful for legacy software or specific hardware needs. Rare in this course — we default to managed and serverless.
See also: Cloud Run, GKEM0L2
Cloud RunINFRA
Fully managed serverless container platform. You give it a container image; it serves HTTP and gRPC, scales from zero to thousands automatically. The default place to put a containerised API.
See also: Cloud Run Jobs, Scale to zero, ConcurrencyM4
Cloud Run JobsINFRA
Run-to-completion container workloads on the same platform as Cloud Run. Use for batch processing, scheduled training warmups, ETL — anything that runs, finishes, and stops.
See also: Cloud Run, Cloud SchedulerM4
Cloud FunctionsINFRA
Function-as-a-service: write a single function in Python/Node/Go, GCP wraps it in a container and runs it on events (HTTP, Pub/Sub, GCS). Cloud Functions v2 is built on Cloud Run.
See also: Cloud Run, EventarcM4
GKE (Google Kubernetes Engine)INFRA
Managed Kubernetes. The right answer when you need long-running services, multi-container workloads, GPU pools, or fine-grained networking that Cloud Run can't do.
See also: GKE Autopilot, Pod, kubectlM5
GKE AutopilotINFRA
GKE mode where Google manages nodes for you. You declare pods; Autopilot provisions compute, handles patching, upgrades, and right-sizes. The default GKE mode in this course.
See also: GKE, PodM5
PodINFRA
Kubernetes' smallest deployable unit: one or more containers that share a network namespace and storage. Pods are ephemeral — controllers (Deployment, Job, CronJob) manage them.
See also: GKE, ContainerM5
NamespaceINFRA
A logical partition inside a Kubernetes cluster. Used for multi-tenancy — Makawalu in one namespace, ShepBERT in another, each with its own RBAC and quotas.
See also: GKE, RBACM5
RBACINFRA
Role-Based Access Control — Kubernetes' in-cluster permissions system. Separate from (but bridged to) GCP IAM via Workload Identity.
See also: Workload Identity, NamespaceM5
Vertex AIML
GCP's managed ML platform. Umbrella for Training (run any container as a training job), Pipelines (DAGs of ML steps), Endpoints (model serving), Model Garden (curated open and Google models), and Workbench (managed Jupyter).
See also: Vertex AI Pipelines, Vertex EndpointM6
Vertex AI PipelinesML
Managed Kubeflow Pipelines on GCP. Define an ML workflow as a Python DAG; each step runs in its own container with its own IAM. The right home for repeatable training and inference jobs.
See also: Vertex AI, Cloud BuildM11
Vertex AI EndpointML
Managed online model serving — upload a model, get a low-latency HTTPS endpoint with autoscaling, traffic splitting, and explainability hooks. Alternative to serving a model yourself from Cloud Run.
See also: Vertex AI, Cloud RunM6
ContainerINFRA
A running instance of an image — your code plus its dependencies, isolated from the host. The unit of deployment for Cloud Run, GKE, Cloud Functions v2, and Vertex Training.
See also: Image, Docker DesktopM2
ImageINFRA
An immutable, layered tarball of your code, dependencies, and a starting command. Built from a Dockerfile, pushed to a registry, pulled and run as containers.
See also: Container, Artifact RegistryM2
Artifact RegistryINFRA
GCP's managed registry for container images, Python wheels, Maven jars, etc. Successor to Container Registry. Cloud Run and GKE pull images from here.
See also: Image, Cloud BuildM2
Cold startINFRA
The latency added when a serverless platform has to pull your image, start a new container, and load your app before serving the first request. Hits scale-from-zero. Mitigations: minimum instances, smaller images, lazy imports.
See also: Scale to zero, Cloud RunM4
Scale to zeroINFRA
When idle, the platform runs zero instances of your service and charges you nothing. The core economic lever of serverless. Trade-off: cold starts on the next request.
See also: Cold start, Cloud RunM0L2
ConcurrencyINFRA
The maximum number of in-flight requests a single Cloud Run container handles. Default 80. Lower for heavy ML inference; higher for cheap I/O-bound APIs.
See also: Cloud RunM4
Ephemeral computeINFRA
Compute that exists only for the duration of the work. The course's core paradigm: containers spin up beside the data, do the job, and terminate. State lives in storage; compute is disposable.
See also: Scale to zero, Data gravityM0L2
Storage & data
GCS (Cloud Storage)INFRA
Object storage. Buckets hold objects (files). The default home for DICOM, model weights, training data, exports. Strongly consistent, regional or multi-regional, lifecycle-managed.
See also: Bucket, Object, DICOMM0L2
BucketINFRA
A globally-named container for GCS objects. Naming is global across all of GCP — choose names like uhcc-hipimr-dicom-prod, not data.
See also: GCS, ObjectM0L2
ObjectINFRA
A file stored in GCS. Identified by gs://bucket/path/to/file. Immutable on write (overwrites create a new version if versioning is on).
See also: BucketM0L2
BigQueryOMOP
Serverless data warehouse. Petabyte-scale SQL with no infrastructure to manage. The home for OMOP CDM in HALO, for analytics across the federated network, and for any structured research data.
See also: Dataset, Table, OMOP CDMM9
Dataset (BigQuery)OMOP
A BigQuery container for tables and views. IAM is applied at the dataset level. One dataset per study or per OMOP vocabulary version is a clean pattern.
See also: BigQuery, TableM9
Table (BigQuery)OMOP
A columnar table inside a BigQuery dataset. Partition by date, cluster by patient_id — the difference between a $1 query and a $400 query.
See also: BigQuery, DatasetM9
Cloud SQLINFRA
Managed Postgres, MySQL, or SQL Server. Use when you need a real relational DB (ATLAS metadata, app state). For analytical workloads, use BigQuery instead.
See also: BigQueryM0L2
FirestoreINFRA
Managed NoSQL document store. Useful for metadata, audit trails, session state. Cheap, fast, integrates cleanly with Cloud Run.
See also: Cloud SQLM0L2
Cloud Healthcare APIHIPAA
HIPAA-eligible managed store for DICOM, FHIR, and HL7v2. Handles ingest, de-identification, and FHIR-native querying. The right home for HIPIMR DICOM and HALO FHIR ingests.
See also: DICOM store, FHIR store, DLPM8
DICOM storeHIPAA
A DICOMweb-compatible managed store inside the Healthcare API. Holds studies, series, instances. Pixels can be exported for ML; metadata is queryable directly.
See also: Cloud Healthcare APIM8
FHIR storeHIPAA
An HL7 FHIR-native managed store inside the Healthcare API. Stream FHIR resources, query them, export to BigQuery for analytics.
See also: Cloud Healthcare API, BigQueryM8
Data gravityHIPAA
The principle that data is hard to move and easy to leak — so you move compute to the data, not the other way around. PHI lives in the VPC perimeter; Cloud Run, Vertex, and GKE jobs run beside it and terminate.
See also: VPC SC, Ephemeral computeM1L1
Networking
VPC (Virtual Private Cloud)INFRA
Your project's private network. Defines subnets, firewall rules, routes. GCP VPCs are global by default — one VPC can span every region.
See also: Subnet, VPC SCM0L2
SubnetINFRA
A regional IP range inside a VPC. Resources live in subnets; firewall rules and routes target them.
See also: VPC, RegionM0L2
VPC Service Controls (VPC SC)HIPAA
A perimeter you draw around projects so their resources can't be reached from outside it — even with valid credentials. The single most important HIPAA control for PHI buckets and BigQuery datasets.
See also: VPC, Data gravityM3
Private Service ConnectHIPAA
Lets services in your VPC reach Google APIs (and other tenants' services) over private IPs, never traversing the public internet. Pair with VPC SC.
See also: VPC SCM3
Cloud NATINFRA
Managed outbound NAT for VMs and pods without public IPs. Required if your private pods need to reach external package mirrors, model registries, etc.
See also: VPCM3
Cloud ArmorHIPAA
Edge WAF and DDoS protection. Apply rate limits, geo restrictions, and OWASP rule sets to externally-exposed services.
See also: Cloud RunM3
RegionINFRA
A geographic location with multiple isolated zones (e.g. us-west1, us-central1). Most resources are regional. Pick close to your data — and to where your IRB lets you put it.
See also: ZoneM0L2
ZoneINFRA
An isolated failure domain inside a region (e.g. us-west1-a). HA-aware services spread across zones automatically.
See also: RegionM0L2
Security & compliance
HIPAAHIPAA
The US health-information privacy law that governs PHI. Compliance on GCP requires a signed BAA, HIPAA-eligible services only, plus your own configuration discipline (VPC SC, CMEK, audit logs, least privilege).
See also: BAA, PHI, VPC SCM3
BAA (Business Associate Agreement)HIPAA
The contract that allows GCP to process PHI on your behalf. Must be signed (typically by UH ITS, not you) before anything HIPAA touches your projects. Without a BAA, no PHI in GCP — period.
See also: HIPAA, PHIM0L1
PHI (Protected Health Information)HIPAA
Health information that identifies an individual, combined with the 18 HIPAA identifiers. The thing the entire HIPAA track is designed to keep contained, audited, and (when possible) de-identified before analysis.
See also: HIPAA, De-identification, DLPM8
CMEKHIPAA
Customer-Managed Encryption Keys. Encryption keys you create and control in Cloud KMS, used to encrypt GCS, BigQuery, Persistent Disks, etc. Required by many institutional HIPAA policies on top of Google's default encryption.
See also: HIPAA, Org policyM3
Cloud DLP / Sensitive Data ProtectionHIPAA
A managed service that inspects text, images, and structured data for sensitive identifiers (SSNs, MRNs, names) and can redact, mask, or tokenize them. The backbone of de-identification pipelines.
See also: De-identification, PHIM8
De-identificationHIPAA
Removing or obscuring identifiers so data is no longer PHI under HIPAA (Safe Harbor or Expert Determination). The boundary between "needs the perimeter" and "can move to a research project."
See also: DLP, PHIM8
Cloud Audit LogsHIPAA
Immutable logs of who did what, where, when. Admin Activity logs are always on; Data Access logs (every read of a PHI object) must be turned on explicitly for HIPAA.
See also: HIPAAM3
Eventing, scheduling & glue
Pub/SubINFRA
Managed message bus. Publishers send messages to a topic; subscribers pull or are pushed to. The plumbing under most event-driven serverless patterns on GCP.
See also: Topic, EventarcM4
Topic / SubscriptionINFRA
Pub/Sub primitives. A topic is a named channel; a subscription is a consumer's view of it. Multiple subscriptions on one topic each get their own copy of every message.
See also: Pub/SubM4
EventarcINFRA
A managed event router. Subscribes to ~100 GCP event sources (GCS object created, BigQuery job done, Pub/Sub message) and delivers them to Cloud Run, Cloud Functions, or Workflows in CloudEvents format.
See also: Pub/Sub, Cloud RunM4
Cloud SchedulerINFRA
Managed cron. Fires HTTP requests, Pub/Sub messages, or App Engine tasks on a schedule. Pair with Cloud Run Jobs for nightly batch work.
See also: Cloud Run JobsM4
Cloud WorkflowsINFRA
A serverless orchestrator for chaining GCP APIs and HTTP services with retries, parallelism, and conditional logic. Lightweight alternative to a full Airflow.
See also: Eventarc, Vertex AI PipelinesM4
Cloud BuildINFRA
Managed CI/CD. Builds container images, runs tests, pushes to Artifact Registry, deploys to Cloud Run / GKE. Triggered by git pushes or PRs.
See also: Artifact Registry, TerraformM2
DataflowOMOP
Managed Apache Beam — for batch and streaming ETL. Used in HIPAA pipelines to scan, transform, and de-identify large datasets without standing up infrastructure.
See also: DLP, BigQueryM8
ML, analytics & research data
OMOP CDMOMOP
OHDSI's Observational Medical Outcomes Partnership Common Data Model. A canonical schema for clinical observations that lets standard analytic packages (ATLAS, ACHILLES) work across institutions. HALO sits on this.
See also: ATLAS, ACHILLES, BigQueryM9
ATLASOMOP
OHDSI's web UI for defining cohorts, characterizations, and incidence analyses over OMOP data. Deployed via Broadsea or as a container on Cloud Run/GKE.
See also: OMOP CDM, BroadseaM9
ACHILLESOMOP
OHDSI's data characterization and quality-assessment tool for OMOP datasets. Generates per-domain summary statistics that ATLAS surfaces.
See also: OMOP CDM, ATLASM9
BroadseaOMOP
A containerised OHDSI stack (ATLAS + WebAPI + tooling) that drops onto Docker / Cloud Run / GKE so you can stand up an OHDSI environment in an afternoon.
See also: ATLAS, OMOP CDMM9
Federated learningML
A training pattern where the model travels to each site's data instead of the data moving. Sites compute gradient updates locally; only updates leave. The architecture for PacificFed.
See also: DP-SGDM10
DP-SGDML
Differentially-Private Stochastic Gradient Descent. Clips and noises per-example gradients during training so the resulting model carries a formal privacy budget (ε, δ). Pair with federated learning for the strongest guarantees.
See also: Federated learningM10
LoRAML
Low-Rank Adaptation — a fine-tuning method that trains small rank-decomposition matrices on top of a frozen base model. Cheap, GPU-friendly, the default for ShepBERT-style clinical fine-tuning.
See also: ClinicalBERTM7
ClinicalBERTML
A family of BERT-derived language models pretrained on clinical notes (MIMIC). Starting point for ShepBERT and most domain-specific clinical NLP.
See also: LoRAM7
Concept bottleneck model (CBM)ML
An interpretable model architecture that predicts a set of human-readable concepts as an intermediate layer, then predicts the target from those concepts. The core of Makawalu's XAI design.
See also: Vertex AIM6
API (enabling)PREREQ
Every GCP service is gated by a per-project API toggle. Before you can use Cloud Run in a project, you gcloud services enable run.googleapis.com. Catches a lot of "but I have permission" confusion.
See also: ProjectM0L1
