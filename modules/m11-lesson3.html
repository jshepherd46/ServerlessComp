From grant to infrastructure

"Launch a new study" — the script

With M11L1\'s modules + M11L2\'s CI pipelines in place, standing up a new study is a sequence of commands that runs in hours, not weeks. The choreography is the same every time; codify it as a script in the lab repo.

0:00
Decide. Confirm study name, PI, grant, expected data types, IRB protocol number, billing account. The Terraform variables you\'ll fill in. Allow ~30 min for the conversation; document outcomes in the runbook.
0:30
Create the study directory. scripts/new-study.sh halo-v2 scaffolds studies/halo-v2/ from a template — main.tf with the standard module calls + a blank terraform.tfvars.
1:00
Fill in variables. Edit terraform.tfvars with study name, billing account, IRB number, IAM groups, perimeter to join. PR review — at least one other engineer signs off.
2:00
Plan + apply. CI runs terraform plan on the PR; reviewers see exactly what will be created. After merge: terraform apply in the studies/halo-v2 directory. ~30 minutes of provisioning; outputs: project IDs, SA emails, bucket URIs, perimeter membership confirmation.
2:30
BAA enrollment ticket (if HIPAA-tier). Open a ticket with UH ITS referencing the new project IDs. While waiting (1-5 business days per M3L1), continue with non-PHI setup.
3:00
Wire CI/CD. Add a Cloud Build trigger for the study\'s training-pipelines repo (M11L2 pattern). Push a smoke-test commit; watch the pipeline land in the new dev project.
4:00
Document. Copy the study-runbook template from the lab handbook into handbook/studies/halo-v2.md. Fill in the sections that are known so far (data sources, IRB, PI, key contacts). Commit.
later
BAA confirmed; load real data. Per M3L1\'s seven-step enrollment workflow + M8L3\'s cohort-instantiation pattern. Now the study is fully operational.

The scaffolding script

#!/usr/bin/env bash
# scripts/new-study.sh — scaffold a study directory from a template
set -euo pipefail

STUDY=$1
DIR="studies/$STUDY"

if [ -d "$DIR" ]; then
  echo "$DIR already exists"; exit 1
fi

cp -r templates/study-template $DIR
sed -i "" "s/STUDY_NAME/$STUDY/g" $DIR/*.tf $DIR/*.tfvars

echo "Scaffolded $DIR. Edit terraform.tfvars and open a PR."
The economic shift. Without this scaffolding, a new study takes 1-3 weeks of cloud engineering (most of which is rote: create projects, set up perimeter, wire IAM, configure CI, write runbook stub). With it: hours of editing variables + waiting for BAA. The lab engineer\'s time is freed for the genuinely new work — the study-specific ETL, the cohort definition, the model design.
The single canonical document

The lab handbook — pulling every runbook section together

M1L4 introduced the lab runbook with nine sections. M3L3 added incident response. M6L3 added per-model lifecycle. M8L3 added data-pipeline metrics. M10L3 added the FL study runbook. The handbook is the single Markdown document — in the lab\'s private git repo — that holds all of these in one place, with stable cross-references.

The SRL lab handbook — canonical table of contents

  1. Lab overview — PI, members + roles, current studies, contact info.
  2. Topology — the M1L2 diagram updated; org / folder / projects / shared-services / perimeter membership.
  3. IAM matrix — the M1L3 table; groups, personas, role assignments at each scope.
  4. Per-study runbooks — one subsection per active study, with the per-study runbook template (see Section 3 below).
  5. Per-model lifecycle — for each served model (Makawalu CBM, HIPIMR density, ShepBERT NER, etc.) the M6L3 lifecycle runbook section.
  6. Data pipeline operations — the M8L3 + M9 patterns; ingest freshness, de-id completeness, cohort-build SLO targets.
  7. Federated learning protocols — for each active FL study, the M10L3 ten-section template.
  8. Onboarding — what a new postdoc / RA / collaborator reads, in what order; the access request workflow.
  9. Offboarding — what happens when a member leaves; group removals, audit confirmation, asset reassignment.
  10. Launching a new study — the workflow from Section 1; references to scripts and modules.
  11. Closing a study — final cohort snapshot, project deletion (with grace period), archive bucket lifecycle, IAM cleanup, runbook archival.
  12. Incident response — the M3L3 four-hour playbook; UH ITS contact info; the audit-query templates.
  13. Monthly hygiene — the billing-export query, IAM Recommender sweep, SA key audit, vocabulary-version freshness, ACHILLES re-run.
  14. External references — links to UH ITS portal, IRB office, GCP docs, OHDSI vocabulary, the course site itself.

The handbook IS the documentation. Every link in the table of contents resolves to a section in the same Markdown file (or to a Markdown sibling). Anyone with read access to the lab repo can read everything; nothing is in someone\'s head, in a Slack thread, or in a notebook on a laptop.

The reusable unit

The per-study runbook template

Each active study gets its own runbook section, instantiated from a template at study launch. The template:

SRL per-study runbook template

  1. Study identity. Name, PI, grant + funding source, IRB protocol + expiry, key collaborators, primary contact engineer.
  2. Data sources. Where data comes from (EHR extracts, PACS, partner site uploads); ingest cadence; current data volume.
  3. Project topology. Per-environment project IDs (dev, staging, prod), perimeter membership, billing account.
  4. Service accounts + groups. The IAM matrix instantiated for this study: workload SAs, member groups + their roles.
  5. Data pipeline. Ingest path (M8 patterns), de-id config (which DLP template), cohort-instantiation workflow (M8L3), operational metrics dashboard URL.
  6. Models served. Each model\'s M6L3 lifecycle section. Where in the registry, current production version, last retrain date, next scheduled retrain.
  7. Operations. How to deploy a new model version (canary script), how to trigger a re-ingest, how to add a new researcher to the cohort access group.
  8. Incident response. Study-specific PHI types + recovery contacts. Same shape as M3L3\'s generic playbook, instantiated with this study\'s specifics.
  9. Closure plan. Conditions for closure (grant end, deliverable shipped); the archive bucket location; what gets retained vs deleted; IRB-required retention period.

Saved as handbook/studies/<study-name>.md. Updated as the study evolves. Read on day-one onboarding for that study.

The discipline that makes the handbook actually used. Every section has explicit cross-references to the modules in this course (M3L1\'s VPC SC, M6L3\'s model lifecycle, M8L3\'s cohort pattern). The handbook is the index; the course modules are the explanations. Together: anyone on the lab can find both "what we do" (handbook) and "why we do it" (course modules).
Hands-on exercise 11.3

Draft your lab\'s handbook outline

📓 Make the canonical document real
  1. Create a private repo for the lab handbook (or use a directory in an existing lab repo). Name it srl-handbook/. Add a README pointing at the table of contents.
  2. Generate the top-level handbook with the 14 sections from Section 2 as Markdown headings. For each: a one-line description and a stub. Commit.
  3. Fill in Section 1 + 2 for real — lab overview and topology — using what you actually have. The diagram from M1L2 can be a Mermaid or PlantUML block in the Markdown.
  4. Pick one of your real studies (HIPIMR, Makawalu, HALO, ShepBERT, MAPS, PAC-AID). Create studies/<name>.md from the per-study template in Section 3. Fill in every section that has a known answer; leave others as TODO.
  5. Write the launch-a-study script from Section 1\'s shape. Even a 30-line bash script that cps a template and seds the study name is useful. Commit.
  6. Note the gaps. Walking through the handbook for one study probably surfaces several "we do not have this in place" issues. Make those into issues in the lab repo; address them on a cadence.
  7. Add a "monthly hygiene" checklist as a Markdown file with check-able items: billing query, IAM recommender, key audit, vocabulary version, ACHILLES re-run. Set a recurring calendar reminder; mark items done in PRs against the checklist file.
  8. Bonus — automate the launch script into a real Make target or Python CLI that calls cookiecutter against a study template. Even one cycle of "scaffold + edit + apply" with the automation in place builds the habit.

What this exercise produces

Closing M11
