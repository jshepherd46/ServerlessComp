The privacy gap

Why FL alone is not a privacy guarantee

M10L1 noted: vanilla FL ships model updates in the clear. Those updates contain information about local training data. Two classes of attack matter:

Gradient inversion

Reconstructing training samples from gradient updates

A motivated adversary with access to gradient updates from a small-batch local training step can sometimes reconstruct individual training examples — especially when batches are small and the model is over-parameterized. Demonstrated for image classifiers; clinical NLP and tabular outcomes are also susceptible.

Membership inference

Determining whether a specific record was in training

Given access to the global model and a candidate record, infer whether that record was used to train. For a patient who was in the cohort vs not, membership inference can confirm participation — itself a privacy leak in a small site like Marshall Islands where the cohort might be 200 patients.

Two defences, stacked:

  • → DP-SGD — adds calibrated noise to per-example gradients during local training. Provides a formal (ε, δ) differential privacy guarantee. Even with full access to model updates, an attacker cannot reliably distinguish whether any single example was in training.
  • → Secure aggregation — uses cryptographic protocols so the aggregator can compute the SUM of site updates without seeing any individual site\'s contribution. Even if the aggregator is curious or compromised, individual site updates are protected.
DP-SGD in practice

DP-SGD — clip, noise, account

DP-SGD is the standard recipe for differentially-private deep learning. Three modifications to vanilla SGD:

  1. Compute per-example gradients, not per-batch.
  2. Clip each per-example gradient to a max L2 norm C (bounds the maximum influence of any single example).
  3. Sum the clipped gradients + add Gaussian noise with scale proportional to C × σ, then average.

The result is a noisy gradient update that\'s still useful for training but no longer reflects any single training example with reliability.

The privacy budget

DP-SGD comes with a privacy accountant. After N training steps with noise scale σ and clipping C, the accountant produces an (ε, δ) guarantee:

  • → ε (epsilon) — the privacy budget. Lower = more private; common health-domain targets are ε ≤ 8 (loose) to ε ≤ 1 (very strict).
  • → δ (delta) — the probability the guarantee fails entirely; typically set to 1 / (10 × dataset_size) or smaller.

For FL: each round of local training spends some privacy budget. The total study budget is the composition of per-round budgets across all rounds. A typical FL study aims for ε ≤ 8 across 100 rounds — implying tight per-round budgets that DP-SGD\'s accountant tracks.

DP-SGD in your training loop (PyTorch + Opacus)

from opacus import PrivacyEngine

privacy_engine = PrivacyEngine()
model, optimizer, dataloader = privacy_engine.make_private(
    module=model,
    optimizer=optimizer,
    data_loader=dataloader,
    noise_multiplier=1.0,        # sigma — higher = more privacy, less utility
    max_grad_norm=1.0,            # C — per-example clipping
)

# Normal training loop, no other changes
for x, y in dataloader:
    optimizer.zero_grad()
    loss = criterion(model(x), y)
    loss.backward()
    optimizer.step()

# After training, query the accountant
epsilon = privacy_engine.get_epsilon(delta=1e-5)
print(f"After {N} steps, (epsilon, delta) = ({epsilon:.2f}, 1e-5)")
The honest trade. DP-SGD costs accuracy. The noise added at each step degrades the model; how much depends on σ, C, and your dataset size. For a small-cohort site (Marshall Islands with 200 cohort patients), the privacy budget for a single round may produce too-noisy gradients to train usefully. Mitigations: train fewer rounds, larger batches, lower target ε at the cost of more rounds with less noise per round.
Keeping the aggregator honest

Secure aggregation

DP-SGD protects against an attacker who sees gradients but doesn\'t know which site they came from. Secure aggregation goes further: the aggregator computes the SUM of site updates without seeing any individual site\'s contribution. Even a fully compromised aggregator learns only the sum.

The cryptographic shape (intuition)

Each pair of sites agrees on a random mask. Site A adds the mask, site B subtracts it, before sending updates to the aggregator. When the aggregator sums all sites\' (now-masked) updates, the masks cancel and only the legitimate sum remains. Individual site updates, viewed alone, look like random noise.

Real protocols (Bonawitz et al.) use Diffie-Hellman key exchange + shamir secret sharing for robustness against dropouts. Flower\'s SecAgg+ implementation handles this; you opt into it with a strategy parameter.

When to use secure aggregation. Always for clinical FL across institutions. The aggregator at srl-pacificfed-infra is trusted today, but the assumption "trust the aggregator" is a single point of compliance failure. Secure aggregation removes the assumption; even if srl-pacificfed-infra is breached, individual sites\' updates are unrecoverable. Cost: 10-20% protocol overhead per round; well worth it for clinical-grade FL.
Running it in production

Monitoring + cross-site observability

A 100-round FL run across 7 sites generates dozens of failure modes that don\'t exist in single-site training. The minimum observability for production FL:

Per-round dashboard (aggregator side)

Round timing + participation + metrics

For each round: number of sites that checked in, sites that timed out, round duration, global loss / metric, per-site loss spread. Cloud Monitoring dashboard fed by the aggregator\'s structured logs.

Per-site dashboard (each site\'s view)

Local participation + local resource use

For each site\'s perspective: rounds participated in vs total rounds, local GPU utilisation, local data dimensionality, local model fit metrics. Each site\'s ops team monitors this.

Privacy budget tracker

How much ε has each site spent

Per the DP-SGD accountant, each site is spending budget every round. A site that started with ε=8 budget is at ε=5 after 50 rounds. The dashboard surfaces remaining budget; when a site\'s budget exhausts, it stops participating.

Alert policies

Page when things go meaningfully wrong

Aggregator: round duration > 4× baseline → page coordinator. Site: local training failure → page site engineer. Privacy: any site\'s residual ε falls below 1.0 → notify protocol owner.

The runbook section

Per M1L4 — every FL study\'s runbook section answers the operational questions before the FL run begins.

PacificFed FL study runbook — minimum table of contents

  1. Protocol overview. Hypothesis, primary outcome, secondary outcomes, cohort definition (link to ATLAS / Capr cohort), exclusion criteria.
  2. IRB and DUA status. Per-site IRB approval dates, the data-use agreement that authorises the federation, signed-by-whom, expiry.
  3. Privacy budget. Target (ε, δ); how budget is consumed per round; the per-site budget tracker location; what happens when a site\'s budget exhausts.
  4. Participating sites. Site IDs, contact engineers, last-known GKE cluster health, last ACHILLES run date.
  5. Topology. Aggregator project + cluster + LB IP; per-site project + cluster + VPC peering status. Diagram included.
  6. Training container. Image URL + version; reproducible build instructions; LoRA / DP-SGD / SecAgg configuration.
  7. Operations. How to start a run, how to pause / resume, how to abort. Dashboard URLs. Alert-channel routing.
  8. Incident response. Specific to FL: "a site drops out for 3 rounds in a row," "the aggregator pod restarts mid-round," "DP accountant says budget exhausted unexpectedly."
  9. Outputs. Where the final model lands (Vertex Model Registry). Where per-round checkpoints land. Where audit logs land. Retention policy.
  10. End-of-study. Tear-down checklist; archive snapshot location; how the protocol\'s data is retained per IRB.
Hands-on exercise 10.3

Add DP-SGD to the M10L1 simulation

🔐 Privacy without losing the model
  1. Install Opacus:
    pip install opacus
  2. Take the fl_demo.py from M10L1 and add DP-SGD to the client\'s fit method: 
    from opacus import PrivacyEngine
from torch.utils.data import DataLoader

class SiteClient(fl.client.NumPyClient):
    def __init__(self, dataset):
        self.model = make_model()
        self.dataset = dataset
        self.privacy_engine = PrivacyEngine()

    def fit(self, params, config):
        self.set_parameters(params)
        loader = DataLoader(self.dataset, batch_size=32, shuffle=True)
        opt = torch.optim.SGD(self.model.parameters(), lr=0.01)
        model, opt, loader = self.privacy_engine.make_private(
            module=self.model, optimizer=opt, data_loader=loader,
            noise_multiplier=1.0, max_grad_norm=1.0,
        )
        for _ in range(2):
            for x, y in loader:
                opt.zero_grad()
                loss = nn.functional.binary_cross_entropy_with_logits(model(x).squeeze(), y)
                loss.backward(); opt.step()
        eps = self.privacy_engine.get_epsilon(delta=1e-5)
        return self.get_parameters({}), len(loader.dataset), {"epsilon": eps}
  3. Run the FL simulation as before. Watch the log — each round now reports the privacy ε spent at each site. Add up across rounds for the total spent.
  4. Tune the noise. Try noise_multiplier=2.0 — more private, less accurate. Try 0.5 — less private, more accurate. Notice the trade.
  5. Track residual budget. Set a target ε of 8.0; print "remaining budget" each round. After how many rounds is the budget exhausted at each noise setting?
  6. Add a SecAgg simulator (Flower has documentation for SecAgg+ workflows). Watch the per-round timing — secure aggregation adds latency. Quantify the overhead.
  7. Document the protocol: write a one-paragraph summary of the chosen privacy + secure-aggregation configuration as it would appear in the PacificFed runbook. Include: target ε, noise multiplier, clipping norm, dataset size, number of rounds, SecAgg enabled.
