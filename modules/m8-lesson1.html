Cloud Healthcare API
in depth

M3 used the Healthcare API as "the place PHI lives." M6 and M7 used it as a data source. This lesson is the API itself — the three modalities, what each is for, the operations that move data between them, and the de-identification operation that makes the rest of the course possible.

⏱ ~50 min reading + exercise 🔧 Cloud Healthcare API, DICOMweb, FHIR R4, HL7v2 📍 Module 8 of 12
The three doors

DICOM, FHIR, and HL7v2 — which lives where

The Cloud Healthcare API wraps three healthcare data modalities under a single project + dataset hierarchy. Each modality is a different store type, optimized for a different kind of data. Knowing which is which is the difference between "I can ingest this" and "I can't even tell what the data is."

Imaging

DICOM store

For medical imaging. Studies → Series → Instances hierarchy. Pixels + extensive metadata in DICOM SOP classes. Queried via DICOMweb (QIDO-RS, WADO-RS, STOW-RS).

Used by: HIPIMR mammography, Makawalu POCUS, any imaging-bearing study.
Structured clinical

FHIR store

For clinical data — patients, encounters, observations, conditions, medications, documents. FHIR R4 (or R5) resource model. Native REST API plus search, bulk export, and BigQuery streaming.

Used by: HALO (clinical data + cohorts), ShepBERT (notes via DocumentReference), Queen's interop.
Legacy interop

HL7v2 store

For HL7 v2.x messages — the format most older EHR systems still emit. Pipe-and-hat encoded message segments. Mostly used as an inbound shim; the canonical path is HL7v2 → FHIR transformation.

Used when: ingesting from systems that won't emit FHIR directly. Less common in greenfield SRL workflows.
The relationship. One Healthcare dataset contains many stores; stores are independent and typed (DICOM, FHIR, HL7v2). A study's data may use one, two, or all three. They reference each other by ID, not by import — a FHIR ImagingStudy resource can reference a DICOM study UID without copying pixels.
DICOM in depth

DICOMweb operations + bulk paths

DICOM stores support the DICOMweb spec — REST endpoints for query (QIDO-RS), retrieve (WADO-RS), and store (STOW-RS) — plus Google-specific bulk operations for import/export.

STOW-RS
Store one or more instances. POST /studies with multipart/related content. Used by ingest pipelines streaming new DICOMs in real time. From M3L2: every STOW request is audit-logged and perimeter-protected.
QIDO-RS
Query by DICOM tag. GET /studies?PatientID=...&ModalitiesInStudy=MG returns matching study metadata. The query language is the DICOM-spec subset; useful for "find all mammography studies on this patient." Returns metadata only, not pixels.
WADO-RS
Retrieve a study, series, or instance. GET /studies/{uid} for the whole study, or down to /instances/{uid}/frames/1 for one frame. Returns multipart/related DICOM content (or JSON metadata-only via /metadata).
import gcs
Bulk ingest from a GCS bucket. Faster than per-instance STOW for backfills. gcloud healthcare dicom-stores import gcs ....
export gcs / bq
Bulk export. Pixels to GCS or metadata to BigQuery. The path that materialises a training-mirror snapshot for M6L1's data plane.
deidentify
Produce a de-identified sibling store. Runs Google's de-id rule set (or a custom config) against every instance; pixel-burn-in redaction for known modalities; outputs to a new store. The operation that creates the training-tier store for HIPIMR/Makawalu.

A typical DICOM STOW + export workflow

# Ingest path: site → GCS landing → Cloud Run Job → STOW into DICOM store
TOKEN=$(gcloud auth print-access-token)
curl -X POST \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/dicom" \
  --data-binary @study.dcm \
  "https://healthcare.googleapis.com/v1/projects/$PROJECT/locations/us-west1/datasets/hipimr/dicomStores/mammo-prod/dicomWeb/studies"

# Export DICOM metadata to BigQuery for cohort queries
gcloud healthcare dicom-stores export bq mammo-prod \
  --dataset=hipimr --location=us-west1 \
  --bq-table=bq://srl-hipimr-prod.imaging.dicom_meta \
  --overwrite-table

# Create a de-id'd sibling store using Google\'s default rules
gcloud healthcare dicom-stores deidentify mammo-prod \
  --dataset=hipimr --location=us-west1 \
  --destination-store=mammo-deid \
  --default-config
FHIR in depth

Resources, references, and streaming to BigQuery

A FHIR store holds resources. Each resource is a JSON document with a defined schema — Patient, Encounter, Observation, Condition, MedicationRequest, DocumentReference, ImagingStudy, etc. Resources reference each other by ID, forming a graph.

A typical resource graph for a clinical encounter

Patient/srl-patient-123
  └─ subject of:
     ├── Encounter/srl-enc-789 (inpatient admission)
     │   ├── encounter of: Observation/lab-glucose-2026-05-01
     │   ├── encounter of: Condition/diabetes-type-2
     │   ├── encounter of: MedicationRequest/metformin
     │   └── encounter of: DocumentReference/discharge-summary
     └── subject of: ImagingStudy/study-uid-...

FHIR operations you'll use

POST /fhir/Patient
Create a new resource. The server assigns an ID and returns the persisted version.
GET /fhir/Patient/{id}
Read a resource. Versioned read available via /_history/{vid} for audit.
GET /fhir/Observation?patient={id}&code=...
Search. FHIR's main querying interface. Filterable by resource attributes; supports chained references for graph queries.
POST /fhir (Bundle)
Atomic batch. Multiple resources in one transaction. Common ingest pattern: package an encounter + all child resources as a Bundle.
$export
Bulk export to GCS. Emit a snapshot of the store (or a cohort subset) as NDJSON files. The path to research-tier datasets.
streaming to BigQuery
Configure a streaming config on the store and every change writes a row to BigQuery in near-real-time. Same FHIR-native schema, queryable via SQL. The standard analytic path for HALO cohorts.

Enabling BigQuery streaming

gcloud healthcare fhir-stores update halo-fhir \
  --dataset=halo --location=us-west1 \
  --stream-config-pubsub-topic=projects/$PROJECT/topics/fhir-events \
  --stream-config-bq-destination=bq://$PROJECT.fhir_analytics.resources \
  --stream-config-schema-type=ANALYTICS_V2

After this, every FHIR write streams into BigQuery within seconds. The ANALYTICS_V2 schema is denormalised for cohort queries. Pair with Pub/Sub notifications for downstream pipelines (M4L2 patterns).

The pattern this enables. Live FHIR ingest → real-time BigQuery analytics. A clinician researcher can write a SQL query against the FHIR Patient + Condition + Observation tables and answer "how many T2DM patients with HbA1c > 8.5 in the last 30 days" without any ETL pipeline. The Healthcare API does the heavy lifting.
The most important operation

The de-identify operation

The Healthcare API's deidentify operation is what makes everything downstream possible. It applies a configurable rule set to PHI and produces a sibling store (DICOM or FHIR) that is no longer subject to the same HIPAA constraints — research-grade, training-grade, exportable to research projects.

What the default rule set does (FHIR, briefly)

Direct identifiers — replaced or redacted

Patient names, MRNs, account numbers, full addresses, phone numbers, email addresses, contact relationships. Mapped to consistent surrogates (Patient X always maps to the same surrogate "Patient-XYZ123") for cohort coherence.

Quasi-identifiers — date-shifted

Dates of service, birth dates, admission dates. Shifted by a per-patient random offset, preserving relative intervals (a 30-day follow-up is still 30 days) while removing absolute calendar alignment.

Geographic identifiers — truncated

ZIP codes truncated to first three digits (per HIPAA Safe Harbor for non-rare regions). Full addresses removed.

Free text — DLP integration

For DocumentReference and Observation notes, the de-id config invokes Cloud DLP to detect and redact PII patterns. M8L2 covers DLP in depth.

DICOM de-id specifics

Tag-level redaction

DICOM has well-defined PHI tags (PatientName, PatientID, PatientBirthDate, AccessionNumber, etc.). The de-id operation removes or anonymises these per the configured profile.

Burned-in PHI in pixels

Some imaging modalities (especially ultrasound, screening mammography) burn patient identifiers into the pixel data. The de-id operation can run OCR + redaction on the pixels themselves for known modality types.

Modality-specific profiles

DICOM Supplement 142 defines tag profiles for different research uses. The Healthcare API supports custom profiles plus several predefined ones (basic, clinical-trials, etc.).

Running a custom de-id config

# A custom config that uses DLP + applies surrogates
cat > deid-config.json <<EOF
{
  "fhir": {
    "fieldMetadataList": [
      {"paths": ["Patient.name"], "action": "TRANSFORM"},
      {"paths": ["Patient.birthDate"], "action": "DATE_SHIFT"}
    ]
  },
  "text": {
    "transformations": [
      {"infoTypes": ["PERSON_NAME"], "replaceWithInfoTypeConfig": {}},
      {"infoTypes": ["PHONE_NUMBER"], "redactConfig": {}}
    ]
  }
}
EOF

gcloud healthcare fhir-stores deidentify halo-fhir \
  --dataset=halo --location=us-west1 \
  --destination-store=halo-deid \
  --config-file=deid-config.json
The output is a sibling de-id'd store. Different store name, different IAM, different perimeter posture allowed. Research projects can be granted read access to the de-id'd store without ever touching the PHI-bearing original. This is the operational boundary that makes research data flow work.
Hands-on exercise 8.1

FHIR ingest + BigQuery streaming + de-identify

🏥 The Healthcare API end-to-end
  1. Enable APIs:
    gcloud services enable healthcare.googleapis.com bigquery.googleapis.com pubsub.googleapis.com
  2. Create a dataset + FHIR store:
    gcloud healthcare datasets create demo-hc --location=us-west1
    gcloud healthcare fhir-stores create demo-fhir --dataset=demo-hc --location=us-west1 --version=R4 --enable-update-create
  3. Create a BigQuery dataset for the FHIR stream + a Pub/Sub topic:
    bq mk --location=us-west1 demo_fhir
    gcloud pubsub topics create fhir-events
  4. Wire FHIR streaming to BigQuery:
    gcloud healthcare fhir-stores update demo-fhir \
  --dataset=demo-hc --location=us-west1 \
  --stream-config-pubsub-topic=projects/$PROJECT/topics/fhir-events \
  --stream-config-bq-destination=bq://$PROJECT.demo_fhir.resources \
  --stream-config-schema-type=ANALYTICS_V2
  5. Create a synthetic Patient (no real PHI):
    TOKEN=$(gcloud auth print-access-token)
curl -X POST \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/fhir+json" \
  -d '{
    "resourceType": "Patient",
    "name": [{"family": "TestPatient", "given": ["Alpha"]}],
    "gender": "female",
    "birthDate": "1985-06-15",
    "address": [{"city": "Honolulu", "state": "HI", "postalCode": "96813"}]
  }' \
  "https://healthcare.googleapis.com/v1/projects/$PROJECT/locations/us-west1/datasets/demo-hc/fhirStores/demo-fhir/fhir/Patient"
  6. Wait ~30 seconds, then query the BigQuery table:
    bq query --use_legacy_sql=false 'SELECT * FROM `demo_fhir.resources` WHERE resourceType="Patient"'
    You should see the Patient streamed through.
  7. Run a de-identification to a sibling store:
    gcloud healthcare fhir-stores create demo-fhir-deid --dataset=demo-hc --location=us-west1 --version=R4
    gcloud healthcare fhir-stores deidentify demo-fhir \
  --dataset=demo-hc --location=us-west1 \
  --destination-store=projects/$PROJECT/locations/us-west1/datasets/demo-hc/fhirStores/demo-fhir-deid
  8. Fetch the de-id'd Patient and compare to the original — the names should be replaced, birth date shifted, address truncated:
    curl -H "Authorization: Bearer $TOKEN" \
  "https://healthcare.googleapis.com/v1/projects/$PROJECT/locations/us-west1/datasets/demo-hc/fhirStores/demo-fhir-deid/fhir/Patient/$PATIENT_ID"
  9. Tear down:
    gcloud healthcare fhir-stores delete demo-fhir-deid --dataset=demo-hc --location=us-west1
    gcloud healthcare fhir-stores delete demo-fhir --dataset=demo-hc --location=us-west1
    gcloud healthcare datasets delete demo-hc --location=us-west1
    bq rm -r -d demo_fhir
    gcloud pubsub topics delete fhir-events
