Running containers:
Cloud Run, Jobs, and when to escalate

You can build a clean image. Now you have to run it somewhere. Cloud Run is the default for 80% of research workloads. Cloud Run Jobs handle the batch 15%. The remaining 5% is the GKE escalation — and knowing when you've hit it.

⏱ ~50 min reading + exercise 🔧 Cloud Run, Cloud Run Jobs, Cloud Scheduler, Eventarc 📍 Module 2 of 12
The runtime contract

What every serverless container platform expects

Cloud Run, Cloud Run Jobs, and GKE all run your container — but they each demand the same handful of behaviours from your image. Get the contract right once and the same image runs everywhere.

The container contract

  • Listen on the port given to you, not a hard-coded one. Read $PORT from the environment; default to 8080 if absent. Cloud Run sets it; Cloud Run Jobs ignore it; GKE leaves it to your Service spec.
  • Handle SIGTERM cleanly. When the platform scales you down, it sends SIGTERM and waits ~10 seconds before SIGKILL. Drain in-flight requests; finish writes; flush logs. Use CMD in exec form (from L1) so signals reach your process.
  • Be stateless across requests / runs. Anything that must persist goes to GCS, BigQuery, or Firestore — not the container filesystem, which disappears between revisions.
  • Start fast. Time from container start to "ready to serve" should be measured in seconds, not minutes. Lazy-load big model weights; do heavy initialisation on first request, not at module import.
  • Log to stdout / stderr. Cloud Logging picks up everything your container writes. No log files, no log rotation, no log shipping — just print.
  • Run as a non-root user. The USER app line from your L1 Dockerfile.

Your hello-flask image from M2L1+L2 already meets all six (Flask reads $PORT via gunicorn, gunicorn handles signals, the app is stateless, startup is <2s, logs go to stdout, the Dockerfile drops privileges). That's the floor.

The default — Cloud Run services

Deploying, configuring, scaling

The minimum deploy is one command. The rest of this section is the knobs you turn after.

IMG=us-west1-docker.pkg.dev/srl-shared-services/research/hello-flask:0.2.0

gcloud run deploy hello-flask \
  --image=$IMG \
  --region=us-west1 \
  --service-account=hello-flask-sa@srl-makawalu-prod.iam.gserviceaccount.com \
  --no-allow-unauthenticated

That deploys an authenticated HTTPS endpoint running as a sharply scoped service account. Default knobs: 80 concurrent requests per container, 512 MB memory, scales 0 → 100, 5-minute request timeout. Now the levers worth knowing.

--memory
Default 512 MB. ML inference needs 2Gi–16Gi. Pick the smallest that holds your model + a request buffer. Memory drives CPU allocation — more memory = more vCPU.
--cpu
Default 1 vCPU. Bump to 2 or 4 for CPU-bound inference (CBM models, scikit-learn). Per-instance, not per-request.
--concurrency
Default 80. Drop to 1–4 for heavy ML serving so each container handles few requests at a time. Raise to 200+ for cheap I/O APIs to keep instance counts (and cost) low.
--min-instances
Default 0 (full scale-to-zero). Set to 1 for low-latency paths — eliminates the first-request cold start but you pay for an idle container 24/7.
--max-instances
Default 100. A safety cap, not a target. Set to your wallet's ceiling so a traffic spike or runaway loop can't run up a bill.
--timeout
Default 5 min, max 60 min. Anything that needs longer is a Cloud Run Job, not a service.
--ingress
Default all. Use internal for services other GCP workloads call but no human should reach. internal-and-cloud-load-balancing for fronting via Cloud Armor.
--vpc-connector
Wires the service into your VPC so it can reach private resources (Cloud SQL on private IP, internal services). Required for any PHI-adjacent service that should never see the public internet.

Mounting secrets at runtime

From L1: secrets never go in the image. From M1L3: workload service accounts get roles/secretmanager.secretAccessor on specific secrets. Cloud Run mounts the secret value either as an env var or as a file:

gcloud run deploy hello-flask \
  --image=$IMG \
  --region=us-west1 \
  --service-account=hello-flask-sa@$PROJECT.iam.gserviceaccount.com \
  --update-secrets=API_KEY=projects/$PROJECT/secrets/hello-api-key:latest \
  --update-secrets=/etc/creds/db.json=projects/$PROJECT/secrets/db-creds:latest

First binding exposes the secret as an env var. Second mounts it as a file at /etc/creds/db.json. Both are read at container start; rotating the secret requires a redeploy (or use :latest and the new value picks up on the next cold start).

For batch — Cloud Run Jobs

Run-to-completion containers

A Cloud Run Job is the same runtime as a service, but the container doesn't serve HTTP — it runs, finishes, exits. The platform tracks completion, retries on failure, and emits events when done.

When to pick a Job over a Service

🌙
Scheduled work
Nightly HIPIMR DICOM ingest, weekly OMOP refresh, monthly cohort export. Triggered by Cloud Scheduler on a cron.
⏳
Long-running passes
Work that exceeds Cloud Run's 60-minute request timeout. A 4-hour DLP scan, a 90-minute Dataflow trigger sequence.
📦
Batch fan-out
Process 1000 patient studies in parallel. A Job with --tasks=1000 --parallelism=20 runs 20 at a time and tracks all 1000.
🔁
Idempotent retries
If a task fails, the platform retries it automatically. Your code must be idempotent — which is good discipline regardless.

Create and run a Job

# Create the Job spec (does not run yet)
gcloud run jobs create nightly-ingest \
  --image=$IMG \
  --region=us-west1 \
  --service-account=ingest-job-sa@$PROJECT.iam.gserviceaccount.com \
  --tasks=1 \
  --max-retries=2 \
  --task-timeout=30m \
  --memory=2Gi \
  --set-env-vars=INGEST_DATE="latest"

# Trigger an execution right now
gcloud run jobs execute nightly-ingest --region=us-west1 --wait

# Schedule it nightly at 02:00 HST via Cloud Scheduler
gcloud scheduler jobs create http nightly-ingest-trigger \
  --location=us-west1 \
  --schedule="0 2 * * *" \
  --time-zone="Pacific/Honolulu" \
  --uri="https://us-west1-run.googleapis.com/apis/run.googleapis.com/v1/namespaces/$PROJECT/jobs/nightly-ingest:run" \
  --http-method=POST \
  --oauth-service-account-email=scheduler-invoker-sa@$PROJECT.iam.gserviceaccount.com

The scheduler runs as its own service account with roles/run.invoker on the Job — separating "who decides when to run" from "what runs". Job logs land in Cloud Logging keyed by execution ID.

The patterns you'll reach for

Three patterns that cover most research workloads

Pattern 1 — Authenticated HTTPS inference API
Use for: Makawalu XAI serving, ShepBERT inference, any model-as-a-service.
[Caller (auth'd via IAP or service account)] ─▶ [Cloud Run service] ─▶ [GCS model weights + Secret Manager API keys]
Cloud Run service with --no-allow-unauthenticated, mounted secrets, model weights pulled from GCS on first request and cached in memory. --min-instances=1 if cold starts matter; otherwise scale-to-zero. --concurrency=2 for heavy models.
Pattern 2 — Scheduled ingest / ETL
Use for: nightly HIPIMR DICOM pulls, weekly OMOP staging refresh.
[Cloud Scheduler (cron)] ─▶ [Cloud Run Job] ─▶ [reads source, writes GCS + Pub/Sub completion event]
Job runs to completion, exits, emits an event. Downstream consumers (de-id pipeline, BigQuery loader) subscribe to that event via Eventarc and react. Decoupled, idempotent, traceable.
Pattern 3 — Object-triggered processing
Use for: every-new-DICOM de-identification, every-uploaded-image classification.
[GCS object created] ─▶ [Eventarc trigger] ─▶ [Cloud Run service handles one CloudEvent per request]
Eventarc wraps GCS Audit Log events as CloudEvents and POSTs them to your Cloud Run service. Your container processes one event per request, idempotently. Scales automatically with arrival rate.
The common thread: stateless containers + GCS/BigQuery for state + Pub/Sub or Eventarc as the wiring. You compose these three patterns to build everything from a one-shot demo to PacificFed's multi-site coordinator.
When Cloud Run is the wrong answer

Five conditions that mean it's time for GKE

Cloud Run covers most of what a research lab needs — but it isn't everything. GKE (Module 5) is what you escalate to when one of these conditions is true. Recognise them so you don't bend Cloud Run into shapes it wasn't built for.

1. The workload runs longer than 60 minutes per request
Cloud Run's max request timeout is 60 min. Cloud Run Jobs go to 24 h, which covers most batch — but if you have a continuous training loop or a long-lived stream consumer, you need GKE pods that run until you stop them.
2. You need a GPU pool that Cloud Run can't give you
Cloud Run added GPU support recently but with narrow choices. Production CBM training, PacificFed FL aggregator on A100s, multi-GPU jobs — these need GKE with a GPU node pool and explicit scheduling.
3. The workload is multi-container (sidecars, init containers)
A pod with a main container plus an Envoy sidecar, a log forwarder, an init container that warms model weights into a shared volume — that's a Kubernetes pod, not a Cloud Run service. Cloud Run supports sidecars now too, but limited to a small set of approved patterns.
4. You need in-cluster service discovery / mesh
Multiple services that talk to each other by name, route via Istio / Cloud Service Mesh, share a per-namespace identity. PacificFed's seven-site aggregator topology is GKE-native because of this.
5. You need persistent state inside the runtime
A stateful container that requires a Persistent Disk — say, a SQLite-backed cache that must survive restarts, or an ATLAS WebAPI instance with local files. Cloud Run filesystems are ephemeral; GKE has PersistentVolumeClaim.
What is not a reason to escalate: "we're using Kubernetes elsewhere," "Cloud Run feels too magic," "we want more control over networking." None of these justify the operational cost. Stay on Cloud Run until one of the five above forces your hand. M5 covers GKE properly when you're ready.
Hands-on exercise 2.3

Deploy a real service and a real Job

🚀 From registry to running
  1. You should have hello-flask:0.2.0 pushed to your practice registry from M2L2. Confirm:
    gcloud artifacts docker images list us-west1-docker.pkg.dev/$PROJECT/practice
  2. Create a dedicated service account for the workload (per M1L3 — never use the default compute SA):
    gcloud iam service-accounts create hello-flask-sa --display-name="hello-flask runtime"
  3. Grant it nothing yet (it doesn't need any roles to run a stateless app). When we add Secret Manager below, we'll add one binding.
  4. Deploy. Authenticated, scoped, small:
    IMG=us-west1-docker.pkg.dev/$PROJECT/practice/hello-flask:0.2.0
    gcloud run deploy hello-flask --image=$IMG --region=us-west1 --service-account=hello-flask-sa@$PROJECT.iam.gserviceaccount.com --no-allow-unauthenticated --memory=256Mi --concurrency=80 --max-instances=10
  5. Call it (auth required — the gcloud token works):
    URL=$(gcloud run services describe hello-flask --region=us-west1 --format='value(status.url)')
    curl -H "Authorization: Bearer $(gcloud auth print-identity-token)" $URL/
  6. Watch logs in real time from another shell:
    gcloud beta logging tail "resource.type=cloud_run_revision AND resource.labels.service_name=hello-flask"
  7. Add a secret, mount it. Create a Secret Manager secret, grant the SA accessor on it, redeploy with the binding:
    echo -n "shhh-test-value" | gcloud secrets create hello-api-key --data-file=-
    gcloud secrets add-iam-policy-binding hello-api-key --member="serviceAccount:hello-flask-sa@$PROJECT.iam.gserviceaccount.com" --role="roles/secretmanager.secretAccessor"
    gcloud run services update hello-flask --region=us-west1 --update-secrets=API_KEY=hello-api-key:latest
  8. Create a Cloud Run Job from the same image (it will just run Flask and exit when the run completes — fine for the exercise):
    gcloud run jobs create hello-once --image=$IMG --region=us-west1 --service-account=hello-flask-sa@$PROJECT.iam.gserviceaccount.com --max-retries=0 --task-timeout=2m
  9. Execute it once and watch:
    gcloud run jobs execute hello-once --region=us-west1 --wait
  10. Tear it all down when done:
    gcloud run services delete hello-flask --region=us-west1
    gcloud run jobs delete hello-once --region=us-west1
    gcloud secrets delete hello-api-key
    gcloud iam service-accounts delete hello-flask-sa@$PROJECT.iam.gserviceaccount.com

Knowledge check

Questions to sit with

  • Your XAI service's first request after an hour of idle takes 18 seconds. Subsequent requests take 200 ms. Walk through what's happening and three things you could change. Hint
  • Why is mounting a secret via Secret Manager strictly better than passing an API key via --set-env-vars? Hint
  • You're tempted to put your nightly HIPIMR ingest in a Cloud Run service that polls. Why is a Cloud Run Job + Cloud Scheduler the right call? Hint
  • Your collaborator argues you should "just move everything to GKE so it's consistent." Give them two concrete reasons why Cloud Run for the API + a Job for the nightly batch is the better default. Hint
Closing Module 2

You can now ship containers like a serverless engineer

The arc of M2: L1 — what a container is and how to build a working one. L2 — how to build one you'd actually ship: small, pinned, scanned, in Artifact Registry, built by Cloud Build on every push. L3 — how to run it: Cloud Run for services, Cloud Run Jobs for batch, GKE when (and only when) you outgrow them. The image is the unit; the runtime is the variable.
