Lesson 1 of 3

Container fundamentals
and your first Dockerfile

Module 1 told you where things live and who can touch them. From here on, "thing" means container. This lesson is the model — what a container actually is, how you build one, and the five mistakes that cost researchers a week.

⏱ ~50 min reading + exercise 🔧 Docker Desktop or Cloud Shell 📍 Module 2 of 12
The model

Image, container, registry

Three concepts. Learn them in order; everything else in the module follows.

📦
Image
An immutable, layered bundle of your code, its dependencies, and a starting command. Built once, run anywhere. Think "a frozen filesystem with metadata."
⚡
Container
A running instance of an image. Isolated from the host, ephemeral, disposable. Stop it and the container disappears — the image stays.
🗂
Registry
A network-addressable storage system for images. You push images to it after building; Cloud Run and GKE pull from it to run. Artifact Registry is GCP's. Docker Hub is the public one.
🧱
Layer
An image is a stack of layers — each one a filesystem diff. Layers are cached and shared between images. The single most important property to understand for fast, small builds.
Why containers exist: "it works on my machine" stops being a sentence anyone has to say. The image bundles the OS userland, the language runtime, the library versions, and your code into one artifact. The same image runs on your laptop, in Cloud Shell, on Cloud Run, in a GKE pod, and on a Vertex AI Training node — bit-for-bit identical.
How images are built

Layers, cached and stacked

Every line in a Dockerfile that changes the filesystem creates a new layer. Layers stack top-down. Crucially, layers are cached by their content hash — if a layer's inputs haven't changed, Docker reuses the cached version instead of rebuilding it.

CMD ["python", "app.py"] metadata, 0 B
↑ cached unless the line above changes
COPY . /app ~50 KB (your code)
↑ invalidates whenever any source file changes
RUN pip install -r requirements.txt ~120 MB
↑ invalidates only when requirements.txt changes
COPY requirements.txt /tmp/ ~2 KB
↑ invalidates only when requirements.txt changes
FROM python:3.12-slim ~50 MB (base image)

Read the stack bottom-up — that's the order Docker builds in. Read it top-down to see what runs first when a container starts.

The single most important Dockerfile principle: put slow-changing things (the base image, system packages, Python deps) early. Put fast-changing things (your code) late. Then a code-only edit rebuilds in seconds, not minutes — Docker reuses every layer up to the line you changed.
The Dockerfile

A complete file, line by line

Here is a real Dockerfile for a tiny Flask service — the kind of thing you'd build to serve a simple inference endpoint or a webhook. Every line earns its place. We'll walk through it after.

# Dockerfile
FROM python:3.12-slim

# 1. Set a working directory inside the image
WORKDIR /app

# 2. System packages (rare for pure-Python; here as an example)
RUN apt-get update && apt-get install -y --no-install-recommends \
        curl \
    && rm -rf /var/lib/apt/lists/*

# 3. Python deps — copy lockfile FIRST, install, THEN copy code
#    This is the layer-caching pattern from Section 2.
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 4. Now the application code
COPY . .

# 5. Run as a non-root user (security baseline)
RUN useradd --create-home --shell /bin/bash app && chown -R app:app /app
USER app

# 6. Document the port the app listens on (informational)
EXPOSE 8080

# 7. The command that starts the container
CMD ["python", "app.py"]

The directives that matter

python:3.12-slim = Debian slim + CPython 3.12, around 50 MB. python:3.12 = full Debian, ~1 GB. For pure-Python services, always start with -slim. For ML work with CUDA, start with an NVIDIA-published image like nvidia/cuda:12.4-runtime-ubuntu22.04.

Pick /app and stick with it. Don't cd inside RUN — that doesn't persist between lines because every RUN is its own shell.

Chain related commands with && into one RUN so they share a single layer. apt-get update followed by a separate apt-get install in two RUNs is a classic bug — the cached update layer goes stale and you install ancient packages.

Prefer COPY over ADD (ADD has surprising URL-fetching and tar-extraction behaviour). Order COPYs by change frequency — lockfile first, code last — to maximise caching.

Create an unprivileged user and switch to it. Cloud Run and GKE will run your container as root if you let them — and image-scanning tools and HIPAA reviewers will both flag it.

CMD ["python", "app.py"] — exec form, no shell, signals propagate correctly so Cloud Run can stop the container cleanly. Avoid CMD python app.py (shell form) — it wraps your command in sh -c and breaks signal handling. ENTRYPOINT is for when you want the image to behave like a CLI binary; rare in research services.

The local loop

Build, tag, run, inspect

Three commands take you from a Dockerfile to a running container on your laptop.

# 1. Build the image. The . is the build context (current directory).
docker build -t hello-flask:0.1 .

# 2. Run it, mapping container port 8080 to your laptop's 8080.
docker run --rm -p 8080:8080 hello-flask:0.1

# 3. From another terminal, hit it
curl http://localhost:8080/

Tags, IDs, and what's where

An image has a content-addressed ID (a sha256 hash) that never changes. Tags are mutable, human-friendly pointers at IDs:

hello-flask:0.1        # your tag — name + version
hello-flask:latest     # default tag if you do not specify one
sha256:7b8a9c…         # the immutable underlying ID

When you push to a registry, the full address looks like:

us-west1-docker.pkg.dev/srl-shared-services/research/hello-flask:0.1
└──── registry host ────┘ └─ project ─┘ └─ repo ─┘ └─── image ──┘ └tag┘

Useful local commands

docker images                # list local images
docker ps                    # list running containers
docker ps -a                 # include stopped containers
docker logs <container_id>   # container stdout/stderr
docker exec -it <id> bash    # shell into a running container
docker image inspect hello-flask:0.1   # layers, env, metadata
docker image history hello-flask:0.1   # show layers + sizes
docker system prune          # reclaim disk: stopped containers + dangling images
Cloud Shell users: Docker is already installed. Build and run exactly the same way; just keep in mind Cloud Shell's home directory is small, so docker system prune is your friend.
What costs you a week

Five mistakes you only make once

1. The 3 GB image for a 200-line service
You started FROM python:3.12 instead of python:3.12-slim. Or you forgot --no-install-recommends on apt-get. Or you left /var/lib/apt/lists in the image. Result: slow pushes, slow pulls, expensive Cloud Run cold starts. Fix: start slim, clean up in the same RUN line.
2. COPY . . with no .dockerignore
Your build context is the whole directory. Without a .dockerignore file, .git/, node_modules/, your local venv/, training checkpoints, and that 4 GB of test fixtures all get sucked into the build. Fix: always ship a .dockerignore (template below).
3. COPY . . before pip install
Now every code edit invalidates the dependency layer. A two-second fix triggers a five-minute rebuild. Fix: the pattern from Section 3 — copy the lockfile, install, then copy code.
4. Secrets baked into the image
COPY .env /app/.env, ENV API_KEY=..., hard-coded service account JSON. Anyone who can pull the image can read every layer's contents. Fix: never. Use Secret Manager at runtime, mounted by Cloud Run or GKE. We do this in M2L3.
5. Unpinned versions everywhere
FROM python:3, RUN pip install flask, RUN apt-get install curl. Your image is reproducible until Tuesday morning when the upstream changes. Fix: pin the base image tag (python:3.12.3-slim-bookworm), pin Python deps in requirements.txt (or use uv pip compile / pip-tools), pin apt versions if a regression bit you.

A .dockerignore that covers 90% of cases

.git
.gitignore
.venv
venv/
__pycache__/
*.pyc
.pytest_cache/
.mypy_cache/
.idea/
.vscode/
node_modules/
.DS_Store
*.log
*.env
secrets/
data/
checkpoints/
*.ckpt
*.pt
*.bin

Drop this in alongside your Dockerfile on day one. Tune as needed.

Hands-on exercise 2.1

Containerise your first Flask service

🐳 Build, run, hit it with curl
  1. Create a new directory and cd into it. Either on your laptop (Docker Desktop running) or in Cloud Shell.
  2. Save this as app.py: 
    from flask import Flask
app = Flask(__name__)

@app.get("/")
def root():
    return {"service": "hello-flask", "status": "ok"}

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=8080)
  3. Save this as requirements.txt: 
    flask==3.0.3
  4. Save the Dockerfile from Section 3 as Dockerfile (no extension).
  5. Save the .dockerignore from Section 5 as .dockerignore.
  6. Build:
    docker build -t hello-flask:0.1 .
    Watch the layers. Note which step took longest.
  7. Run, mapping the port:
    docker run --rm -p 8080:8080 hello-flask:0.1
  8. In another terminal, hit it:
    curl http://localhost:8080/
    You should see {"service":"hello-flask","status":"ok"}.
  9. Force a cache hit. Edit only app.py (change the status string). Rebuild and time it:
    time docker build -t hello-flask:0.2 .
    The dependency layer should be reused — rebuild in <5 seconds.
  10. Force a cache miss. Edit requirements.txt (add requests==2.32.3). Rebuild. The pip install layer reruns. Now you've seen both sides of the cache.
  11. Inspect the image:
    docker image history hello-flask:0.2
    Each line is a layer + its size. This is your bill of materials.
  12. Stop the running container with Ctrl-C. Clean up:
    docker rmi hello-flask:0.1 hello-flask:0.2

Knowledge check

Questions to sit with

  • You edit one line in app.py. Your rebuild takes 8 minutes because pip reinstalls every package. What's wrong with the Dockerfile? Hint
  • Your colleague says "just put the API key in an ENV directive — the image is private." Why is this still wrong? Hint
  • What's the practical difference between CMD ["python", "app.py"] and CMD python app.py? Hint
  • Your hello-flask:0.1 image is 1.2 GB. The actual app is 200 lines of Python. Name three places the bloat likely came from. Hint
