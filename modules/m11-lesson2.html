ML pipelines as code

L1 was the platform as code. This lesson is the ML workflow as code — Vertex AI Pipelines as compiled artifacts, Cloud Build CI/CD that takes a git push to a deployed model, the multi-environment dev/staging/prod pattern that lets you ship safely.

⏱ ~50 min reading + exercise 🔧 Vertex AI Pipelines, Cloud Build, Kubeflow KFP SDK 📍 Module 11 of 12
The artifact mindset

A Vertex AI Pipeline is a compiled artifact

M6L2 wrote pipelines as Python (KFP SDK) + submitted them via aiplatform.PipelineJob. For production: compile the pipeline to a JSON spec, version it like an image, register it as the deployment artifact. The Python is the source; the JSON is what runs.

The compile step

from kfp.compiler import Compiler
from pipelines.makawalu_cbm import makawalu_cbm_pipeline

Compiler().compile(
    pipeline_func=makawalu_cbm_pipeline,
    package_path="out/makawalu-cbm-v3.json",
)

The output JSON is portable, hashable, version-controllable. Treat it the same as a container image: build it, tag it, push it (to GCS or Artifact Registry), reference it by version in deployments.

The pipeline-as-artifact workflow

[1]

Author

Engineer edits Python pipeline code in the lab repo. Runs unit tests locally. Compiles to JSON via kfp.compiler.Compiler().compile().

[2]

CI: compile + validate + push

Cloud Build trigger on push to main: re-compile, run pipeline-shape validation, copy the resulting JSON to gs://srl-pipelines/makawalu-cbm/v3-{commit-sha}.json. Artifact is dated and traceable.

[3]

Deploy: submit to Vertex

Submission to Vertex AI Pipelines is a separate step — usually triggered by Cloud Scheduler (nightly) or by an event (M4L2). The submission references the JSON artifact by URI, not the source code.

[4]

Run + register

The pipeline executes, produces metrics + model artifact, conditionally registers in Vertex Model Registry (M6L2 patterns).

The discipline: the pipeline that ran last Tuesday and the pipeline that runs this Tuesday must both have a stable, versioned JSON in srl-pipelines/. The Python file in the repo can change; the compiled JSON is the immutable record. Reproducing last Tuesday\'s run means re-submitting that exact JSON, not re-compiling the Python.
CI for ML

Cloud Build pipelines for training images + ML pipelines

M2L2\'s Cloud Build setup pushes container images on git push. For ML, the same machinery handles two artifact streams: training images (Docker images that run the actual training code) and pipeline definitions (compiled KFP JSON).

A complete cloudbuild.yaml for ML workflows

# cloudbuild.yaml — runs on push to main
substitutions:
  _REGION: us-west1
  _REPO: research
  _PIPELINE_BUCKET: srl-pipelines

steps:
  # 1. Run pipeline + component unit tests
  - name: 'python:3.12'
    entrypoint: bash
    args:
      - -c
      - "pip install -r requirements-dev.txt && pytest tests/"

  # 2. Build the training container image (M2L2 pattern)
  - name: 'gcr.io/cloud-builders/docker'
    args:
      - build
      - --tag=${_REGION}-docker.pkg.dev/$PROJECT_ID/${_REPO}/makawalu-train:$SHORT_SHA
      - --tag=${_REGION}-docker.pkg.dev/$PROJECT_ID/${_REPO}/makawalu-train:latest
      - ./containers/train

  - name: 'gcr.io/cloud-builders/docker'
    args: [push, --all-tags, '${_REGION}-docker.pkg.dev/$PROJECT_ID/${_REPO}/makawalu-train']

  # 3. Compile the KFP pipeline (Python → JSON)
  - name: 'python:3.12'
    entrypoint: bash
    args:
      - -c
      - |
        pip install kfp google-cloud-aiplatform
        python pipelines/compile.py --image-tag=$SHORT_SHA --output=out/cbm-$SHORT_SHA.json

  # 4. Upload pipeline JSON to GCS
  - name: 'gcr.io/cloud-builders/gsutil'
    args:
      - -m
      - cp
      - out/cbm-$SHORT_SHA.json
      - gs://${_PIPELINE_BUCKET}/makawalu-cbm/cbm-$SHORT_SHA.json

  # 5. (Optional) submit a dev-tier pipeline run for smoke test
  - name: 'gcr.io/cloud-builders/gcloud'
    entrypoint: bash
    args:
      - -c
      - |
        gcloud ai custom-jobs create \
          --region=${_REGION} \
          --display-name="smoke-test-${SHORT_SHA}" \
          ...

options:
  logging: CLOUD_LOGGING_ONLY
  machineType: E2_HIGHCPU_8

The trigger setup

# Push trigger on main: build image + compile pipeline + smoke test
gcloud builds triggers create github \
  --repo-name=srl-makawalu-pipelines \
  --repo-owner=jshepherd46 \
  --branch-pattern="^main$" \
  --build-config=cloudbuild.yaml \
  --service-account=projects/srl-shared-services/serviceAccounts/cicd-sa@srl-shared-services.iam.gserviceaccount.com
The image-tag-as-pipeline-arg pattern. The compiled KFP JSON references the training image by tag. CI builds image makawalu-train:$SHORT_SHA AND compiles pipeline JSON that uses that exact tag. The two artifacts are paired — pipeline cbm-$SHA.json always uses image makawalu-train:$SHA. Same git commit produces a coherent set of artifacts.
The promote workflow

From a CI artifact to a deployed model

CI produced a versioned pipeline + image pair. The next question: how does a new model get safely from "compiled" to "serving production traffic"? The same canary discipline from M4L1 + M6L3, made explicit as a workflow.

Step 1 — run pipeline against dev data

Submit the pipeline against dev OMOP / training mirror

Automatically on CI merge to main. Smoke test that the compiled artifact actually runs end-to-end on a small dev dataset. Catches integration bugs cheap.

Step 2 — run pipeline against staging

Scheduled or manually triggered against staging data

Full training run on staging cohort. Output: model registered as candidate in Vertex Model Registry. Metrics in Vertex Experiments. The slice-fairness gate (M6L2 + M7L2) is evaluated here.

Step 3 — canary in prod

Deploy candidate to 10% canary on prod endpoint

Triggered by Pub/Sub event (M4L2) when a candidate registers + passes the gate. Cloud Function moves traffic split to candidate:10, production:90. Notification to the model owner.

Step 4 — human promotion

After observation window, human moves the production alias

Manual gcloud command (or Slack-bot approval flow) updates the Vertex Model Registry production alias to point at the new version. Cloud Run / GKE serving picks up the new version on next instance restart.

Step 5 — rollback if needed

One command restores the previous version

If issues surface post-promotion: update the alias back, traffic shifts to previous version within seconds (M4L1 instant rollback). The audit trail shows the timeline.

The whole flow as code

# scripts/promote.sh — the standard model-promote workflow
#!/usr/bin/env bash
set -euo pipefail

MODEL=$1       # e.g. makawalu-cbm
VERSION=$2     # e.g. v3.7
REGION=us-west1

# 1. Promote candidate → production alias
gcloud ai models update $MODEL --region=$REGION \
  --version=$VERSION \
  --alias=production

# 2. Trigger the serving redeploy (depends on infra)
#    For Cloud Run: kick a Cloud Build trigger that redeploys
#    For GKE: kubectl rollout restart deployment/xai-serve

# 3. Slack notification
curl -X POST -H "Content-Type: application/json" \
  -d "{\"text\": \"Promoted $MODEL $VERSION to production\"}" \
  $SLACK_WEBHOOK

# 4. Log the promotion for the audit trail
gcloud logging write model-promotions "$(whoami) promoted $MODEL $VERSION at $(date)"

Same script runs whether you\'re promoting CBM, the density CNN, or the ShepBERT adapter — different model IDs, identical flow.

Dev / staging / prod for ML

The three-environment pattern

From M1L2: project-per-study-per-environment. The same pattern applies to ML — every model lifecycle uses three Vertex AI environments mirroring the project topology.

srl-makawalu-dev

Pipeline development

Small synthetic data. Fast iteration. Failed pipelines OK. Vertex Pipelines + Experiments enabled; Endpoints rarely.

srl-makawalu-staging

Pre-production validation

De-id\'d staging cohort. Full pipeline runs. Model registry with candidate alias. Smoke-test Endpoint.

srl-makawalu-prod

Production serving

Real (or fully de-id\'d production) cohort. Model registry with production alias. Serving Endpoint with full audit + monitoring.

The same pipeline JSON runs in all three

The compiled pipeline takes parameters: data source, output bucket, model registry, image tag. Different parameter sets push the same pipeline JSON against different environments:

# Submit the same compiled pipeline to dev
aiplatform.PipelineJob(
    display_name=f"cbm-{SHA}-dev",
    template_path=f"gs://srl-pipelines/makawalu-cbm/cbm-{SHA}.json",
    parameter_values={
        "deid_store": "projects/srl-makawalu-dev/.../dicomStores/deid-store",
        "output_bucket": "gs://srl-makawalu-dev-pipelines",
        "image_tag": SHA,
    },
).submit()

# Same JSON, different params, in staging
aiplatform.PipelineJob(
    display_name=f"cbm-{SHA}-staging",
    template_path=f"gs://srl-pipelines/makawalu-cbm/cbm-{SHA}.json",
    parameter_values={
        "deid_store": "projects/srl-makawalu-staging/.../dicomStores/deid-store",
        "output_bucket": "gs://srl-makawalu-staging-pipelines",
        "image_tag": SHA,
    },
).submit()

The pipeline code never changes between environments. Only the parameters differ. That\'s the contract.

The promotion path. Pipeline runs in dev → passes smoke; same artifact runs in staging → produces candidate; same artifact (via promote.sh) updates prod model alias. The CI-built pipeline JSON is the durable artifact tying all three together. A model in production was produced by exactly the pipeline that ran in staging, which was the same pipeline that smoke-tested in dev. No drift.
Hands-on exercise 11.2

Build the full CI pipeline for ML

🔁 git push → pipeline JSON → smoke test
  1. Take the demo training pipeline from M6L2\'s exercise. Put it in a small git repo with the directory shape: 
    demo-ml-pipeline/
├── cloudbuild.yaml
├── containers/
│   └── train/
│       ├── Dockerfile
│       └── train.py
├── pipelines/
│   ├── compile.py
│   └── demo_pipeline.py
├── tests/
│   └── test_pipeline_shape.py
└── requirements-dev.txt
  2. Write a compile script pipelines/compile.py that takes an image-tag arg + an output path: 
    import argparse
from kfp.compiler import Compiler
from demo_pipeline import demo_pipeline

p = argparse.ArgumentParser()
p.add_argument("--image-tag", required=True)
p.add_argument("--output", required=True)
args = p.parse_args()
# The pipeline\'s compile step substitutes image_tag
Compiler().compile(pipeline_func=demo_pipeline, package_path=args.output)
print(f"Compiled to {args.output} with image tag {args.image_tag}")
  3. Write a minimal shape test in tests/test_pipeline_shape.py: 
    from pipelines.demo_pipeline import demo_pipeline
from kfp.compiler import Compiler
import tempfile, json

def test_pipeline_compiles():
    with tempfile.NamedTemporaryFile(suffix=".json") as f:
        Compiler().compile(pipeline_func=demo_pipeline, package_path=f.name)
        spec = json.load(open(f.name))
        assert "pipelineSpec" in spec
        assert "deploymentSpec" in spec["pipelineSpec"]
  4. Write the cloudbuild.yaml with the five steps from Section 2 (tests + image build + image push + pipeline compile + pipeline upload).
  5. Set up the trigger:
    gcloud builds triggers create github --repo-name=demo-ml-pipeline --repo-owner=$YOU --branch-pattern="^main$" --build-config=cloudbuild.yaml
  6. Push a commit to main. Watch the Cloud Build run in the console. Confirm: tests pass; image lands in Artifact Registry; pipeline JSON lands in GCS at the expected path.
  7. Submit the pipeline manually against your sandbox:
    python -c "
from google.cloud import aiplatform
aiplatform.init(project='$PROJECT', location='us-west1')
job = aiplatform.PipelineJob(
    display_name='demo-run-1',
    template_path='gs://$PROJECT-pipelines/demo-pipeline-$SHA.json',
    parameter_values={'image_tag': '$SHA'},
)
job.submit()
"
  8. Bonus — wire smoke-test submission into CI. Add a final cloudbuild step that submits the pipeline against a tiny dev dataset and waits for completion. CI now goes "git push → working pipeline run in dev" automatically.
  9. Tear down: cancel any running pipeline jobs; delete artefacts older than a few days. Leave the trigger; it\'s useful.
