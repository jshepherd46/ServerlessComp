Lesson 1 of 3

ML platform decisions
and the DICOM data plane

Phase 1 built the platform. Phase 2 puts research workloads on it. M6 starts with the two questions every imaging study faces: where does training run? and where does the imaging actually live? Decision trees that pay back over years, and the DICOM data plane for HIPIMR and Makawalu.

⏱ ~55 min reading + exercise 🔧 Vertex AI, Healthcare API, GCS, Workbench 📍 Module 6 of 12
The decision tree

Where does the ML run?

You have four real options on GCP for ML compute. Each fits a specific phase of the work. The decision is rarely "one platform for everything" — most studies use two or three over a model's life.

Are you exploring / prototyping in a notebook?
Vertex AI Workbench. Managed JupyterLab inside the perimeter, with mounted GCS, BigQuery, and DICOM-store access via the service account attached to the instance. Section 4 covers setup.
Are you training a single, well-defined model from a known recipe?
Vertex AI Custom Training. Hand it a container + a dataset URI; it allocates GPUs / TPUs, runs the job, captures logs and artifacts, tears down. Right shape for 90% of training jobs. M6L2 covers it.
Are you running a multi-step pipeline (preprocess → train → eval → register → deploy)?
Vertex AI Pipelines (Kubeflow). Each step is a containerised component with artifacts flowing between them. Reproducible, parameterised, scheduled. The right home for the Makawalu CBM training pipeline. M6L2 + M11.
Are you serving an HTTPS inference endpoint?
Three good answers — and M6L3 makes the matrix concrete. Cloud Run for simple containerised inference (Makawalu XAI demo). Vertex AI Endpoints for the managed-serving path with traffic splitting + monitoring + explainability built in. GKE for multi-GPU serving, custom networking, or in-cluster co-location with other services.
Are you doing long-running training with custom infrastructure needs (multi-GPU, special networking, persistent state)?
GKE. From M5: GPU node pools, full Pod control, multi-container patterns. PacificFed's FL aggregator and large multi-GPU training fit here.
Pragmatic default for SRL. Prototype in Workbench. Train via Vertex Custom Training (or Pipelines for multi-step). Serve from Cloud Run or Vertex Endpoints — pick GKE only when the M2L3 / M5 escalation triggers apply. Almost no study should be running on raw VMs.
The data layer

DICOM on GCP — Healthcare API or GCS?

For imaging studies, the choice is between two valid homes for DICOM data. Knowing when to use each — and how to flow data between them — is the foundation of every imaging pipeline you'll build.

DICOM in the Cloud Healthcare API

A DICOM store inside the Cloud Healthcare API exposes DICOM via DICOMweb (QIDO-RS, WADO-RS, STOW-RS). Studies and series are queryable by DICOM tag; instances are streamable by reference. Native BigQuery export of metadata, native de-identification, native FHIR linkage.

📦 srl-hipimr-prod (project) └── 🏥 dataset: hipimr-dataset # in Cloud Healthcare API, region us-west1 └── 🗂 dicomStore: hipimr-mammo # the DICOM store, CMEK encrypted ├── study: 1.2.840.113619.2.55.3.604688.2.0.42... │ ├── series: 1.2.840.113619.2...01 # MLO views │ └── series: 1.2.840.113619.2...02 # CC views └── study: ...

DICOM in GCS

DICOM as raw .dcm files in a GCS bucket. No DICOMweb, no SOP-aware indexing — just objects you read directly. PyDicom or DCMTK on the consumer side does the parsing.

When to use each

Healthcare API DICOM store
GCS bucket of .dcm files
Production PHI store. DICOMweb. Native de-id, metadata-to-BigQuery export, FHIR linkage. Audit Data Access logs by study UID.
Bulk training data, especially after de-identification. Easy random access via object path. Lower cost per GB at large scale.
Use for: the canonical PHI store - HIPIMR, Makawalu staging, anything that hits the IRB definition of "system of record."
Use for: training-time mirrors. De-id'd cohorts. Working datasets your training loop reads in random order.
The canonical pattern. Raw PHI DICOM lands in the Healthcare API DICOM store (system of record, audited, perimeter-protected). For training, a Dataflow or Cloud Run Job exports a de-identified subset as .dcm files (or numpy arrays) to a GCS bucket; training reads from there with random access. The DICOM store keeps the audit trail; GCS holds the cheaper, faster training mirror.
The worked example

The Makawalu data path — POCUS → CBM training tier

Makawalu's concept bottleneck model (CBM) for breast cancer XAI gets POCUS images from clinical sites and trains a model that predicts intermediate human-readable concepts (lesion shape, margins, posterior features) plus a final malignancy score. The data plane has five concrete steps from clinical site to GPU.

Step 1 — Ingest

Authenticated upload from clinical sites to GCS landing

Sites authenticate as a per-site service account with roles/storage.objectCreator only on a per-site landing prefix. From M3L2: Cloud Armor at the LB + IP allow-list + mTLS where available.

Step 2 — Validate + load to DICOM store

Cloud Run Job picks up GCS landing, STOW-RS into Healthcare API

An Eventarc trigger fires per object; the Job validates DICOM headers, redacts known-bad tags client-side, and posts via storeInstances to the Makawalu DICOM store. Each upload is per-study so STOW-RS calls are bounded.

Step 3 — De-identification + metadata export

Healthcare API de-id operation produces a de-id'd store + BigQuery metadata

The DICOM store's deidentify operation runs with a configured rule set (HIPAA Safe Harbor + image-pixel redaction for burned-in identifiers). Output: a sibling de-id'd DICOM store, plus a parallel metadata table in BigQuery that the training pipeline queries.

Step 4 — Training mirror in GCS

Vertex AI Pipeline exports de-id'd pixel arrays to GCS as training data

A Vertex AI Pipeline (M6L2) reads from the de-id'd store, runs preprocessing (resize, normalisation, augmentation prep), and writes resulting numpy arrays + tabular concept labels to gs://srl-makawalu-training/cbm-v3/. This is what the training loop reads with random access.

Step 5 — Training + registry

Vertex AI Custom Training reads from GCS, writes to Vertex Model Registry

The training container reads from the GCS mirror with random access (no DICOMweb at training time — too slow per-batch). On completion: model artifact uploaded to Vertex Model Registry, metrics logged to TensorBoard, the pipeline emits a Pub/Sub event for the deploy-canary step (M4L2).

The same shape applies to HIPIMR with one shift: HIPIMR is a longitudinal registry, so the DICOM store is bigger and grows monthly; training mirrors are versioned snapshots of the de-id'd subset for a particular cohort criterion (e.g. "screening mammograms with prior films available, 2018-2024").

Where prototyping lives

Vertex AI Workbench — managed JupyterLab inside the perimeter

Before you write a training pipeline, you're in a notebook trying things. Vertex AI Workbench gives you a JupyterLab instance hosted by Google, with a GCP service account attached, mounted GCS, and direct access to BigQuery and Healthcare API. Crucially, when configured correctly, the instance lives inside the VPC SC perimeter — so PHI never leaves the cluster of trust.

Create an Instance

gcloud workbench instances create makawalu-explore \
  --location=us-west1-a \
  --machine-type=n1-standard-4 \
  --accelerator-type=NVIDIA_TESLA_T4 \
  --accelerator-count=1 \
  --service-account=workbench-sa@srl-makawalu-prod.iam.gserviceaccount.com \
  --network=srl-prod-vpc \
  --subnet=srl-prod-subnet \
  --disable-public-ip \
  --metadata=enable-oslogin=true

What the service account needs

Per M1L3: a dedicated workbench-sa bound only with what the work needs. For Makawalu exploration: roles/healthcare.dicomViewer on the de-id store, roles/storage.objectViewer on the training mirror bucket, roles/bigquery.dataViewer on the metadata dataset. Never grant access to the raw PHI DICOM store.

A working Workbench session

# In a Workbench notebook
from google.cloud import bigquery, storage
from google.api_core import client_options
import requests

# BigQuery - read DICOM metadata for the cohort
bq = bigquery.Client()
df = bq.query("""
    SELECT study_uid, patient_age_at_imaging, view_position, image_laterality
    FROM `srl-makawalu-prod.imaging_meta.deid_studies`
    WHERE acquired_at BETWEEN '2024-01-01' AND '2024-06-30'
    LIMIT 1000
""").to_dataframe()

# DICOM store - fetch one instance's pixel data via WADO-RS
from google.auth.transport.requests import AuthorizedSession
from google.auth import default
session = AuthorizedSession(default()[0])
study = df.iloc[0].study_uid
url = f"https://healthcare.googleapis.com/v1/projects/srl-makawalu-prod/locations/us-west1/datasets/makawalu/dicomStores/deid-store/dicomWeb/studies/{study}"
r = session.get(url, headers={"Accept": "multipart/related; type=application/dicom"})

# GCS - read a preprocessed training array
storage_client = storage.Client()
blob = storage_client.bucket("srl-makawalu-training").blob("cbm-v3/sample_001.npy")
arr = np.load(io.BytesIO(blob.download_as_bytes()))
The Workbench-vs-laptop trade. A Workbench instance with no public IP, inside the perimeter, attached to a workload SA is the only safe place to look at de-identified PHI. Laptops never qualify — they're outside the perimeter and not subject to the audit posture. The notebook lives in the cloud; you connect over a tunnel.
Hands-on exercise 6.1

Stand up a DICOM store + Workbench instance

🏥 Imaging plumbing, end to end
  1. Enable the APIs you'll need:
    gcloud services enable healthcare.googleapis.com notebooks.googleapis.com aiplatform.googleapis.com
  2. Create a Healthcare dataset + DICOM store in your sandbox:
    gcloud healthcare datasets create demo-imaging --location=us-west1
    gcloud healthcare dicom-stores create demo-dicom --dataset=demo-imaging --location=us-west1
  3. Use a public sample DICOM. (Many datasets are available; one option:)
    curl -L -o sample.dcm "https://github.com/datacite/datacite-mds/raw/master/example.dcm"
    (If that link breaks, use any small valid .dcm you have, or generate one with pydicom.)
  4. STOW-RS the file into your DICOM store:
    TOKEN=$(gcloud auth print-access-token)
curl -X POST \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/dicom" \
  --data-binary @sample.dcm \
  "https://healthcare.googleapis.com/v1/projects/$PROJECT/locations/us-west1/datasets/demo-imaging/dicomStores/demo-dicom/dicomWeb/studies"
    A 200/201 response means it landed.
  5. Query for studies via QIDO-RS:
    curl -H "Authorization: Bearer $TOKEN" \
  "https://healthcare.googleapis.com/v1/projects/$PROJECT/locations/us-west1/datasets/demo-imaging/dicomStores/demo-dicom/dicomWeb/studies"
    You should get back a JSON array of study metadata.
  6. Export DICOM metadata to BigQuery (the canonical pattern):
    gcloud healthcare dicom-stores export bq demo-dicom \
  --dataset=demo-imaging --location=us-west1 \
  --bq-table=bq://$PROJECT.demo_imaging.dicom_meta \
  --overwrite-table
    Then query the metadata:
    bq query --use_legacy_sql=false 'SELECT * FROM `demo_imaging.dicom_meta` LIMIT 10'
  7. Create a Workbench instance (small, CPU-only for the exercise):
    gcloud workbench instances create demo-workbench \
  --location=us-west1-a \
  --machine-type=e2-standard-2
    Wait a few minutes, then open it from the Cloud Console.
  8. In the notebook, query your DICOM store with WADO-RS and visualise an instance using pydicom + matplotlib. This is what you'd do during real exploration.
  9. Tear down when done — Workbench instances cost money when running:
    gcloud workbench instances delete demo-workbench --location=us-west1-a
    gcloud healthcare dicom-stores delete demo-dicom --dataset=demo-imaging --location=us-west1
    gcloud healthcare datasets delete demo-imaging --location=us-west1
