Why federated analytics works

Same shape, same vocabulary, same cohort

The promise of OMOP is that "patients with HCC + immunotherapy in 2020-2024" is the same query at UH, at Queen\'s, at the Guam CCPI node, at Stanford, at Mayo. Each site stores its own data; each site runs the same compiled SQL; each site returns the same shape of summary results. Only those summary results — never patient-level data — leave the institution.

What "same shape" actually requires

  • → Same CDM version across all sites (typically OMOP CDM v5.4).
  • → Same vocabulary release (synchronised OHDSI Athena snapshot). A site on v2024-Q1 and a site on v2024-Q3 may produce subtly different counts.
  • → Same cohort definition, distributed as JSON or as Capr R code from M9L2.
  • → Same data-quality baseline — ACHILLES runs at each site, surfaces issues before federation.
  • → Agreed result schema — what counts (cohort size, person-time, demographics breakdown) is returned, and at what aggregation level.
The crucial property. Patient-level data never moves. Each site runs the cohort definition locally against its own OMOP database; computes the agreed summary statistics; ships those statistics to an aggregator. The aggregator combines them into a meta-analysis. PHI never crosses an institutional boundary; only aggregated counts and means do.
The worked example

PacificFed analytics — seven sites, one query

PacificFed is SRL\'s federated network across CCPI and USAPI nodes — UH, Queen\'s, Guam, American Samoa, CNMI, Palau, Marshall Islands. The analytics layer (separate from the ML training layer in M10) uses OMOP at each site so that the same cohort question runs everywhere without data movement.

The shared assets across sites

  • → OHDSI vocabulary snapshot — one versioned download per quarter; every site loads the same version on the same week.
  • → Cohort definitions — versioned JSON (or Capr R) in a shared git repo; each site pulls and runs.
  • → Result schema spec — a documented contract for what each site returns (CSV layout, summary statistic definitions). Drift here breaks meta-analysis.
  • → Data-quality baseline — ACHILLES results from each site visible in a shared dashboard before federation runs.
What gets centralised in srl-pacificfed-infra: the shared OMOP vocabulary cache; the cohort definitions repo; the aggregator service; the meta-analysis pipeline; the federated dashboard. What does not get centralised: any per-patient data from any site. The aggregator only ever sees site-level summary CSVs.
From ATLAS to research tier

Cohort exports — how cohorts flow into your pipeline

M8L3 described the cohort-instantiation pattern. M9 fills in how the cohort definition lands in the research tier as a usable dataset. Three steps from ATLAS to ML pipeline.

Step 1 — Define and generate in ATLAS

Per M9L2 — define the cohort, generate against the PHI-tier OMOP source. The result lives in halo_results.cohort as (cohort_definition_id, subject_id, cohort_start_date, cohort_end_date) rows.

Step 2 — Materialise the cohort into research tier

A Vertex Pipeline (M6L2 pattern) reads the cohort row set, joins it against the relevant OMOP tables (person + visit + observation + measurement, scoped to cohort dates), runs the M8L1 de-identification operation, and writes a per-cohort dataset to the research-tier project. Surrogates are consistent with prior cohorts via the shared surrogate map (M8L1).

# Conceptual pipeline (M6L2 KFP shape)
@dsl.pipeline(name="halo-cohort-export")
def cohort_export_pipeline(
    cohort_definition_id: int,
    target_dataset: str,
    irb_protocol_id: str,
):
    rows = pull_cohort_rows(definition_id=cohort_definition_id)
    enriched = enrich_with_omop_features(rows=rows.outputs["rows"])
    deid = deidentify_for_research(data=enriched.outputs["data"])
    materialised = write_to_research_tier(data=deid.outputs["data"], dataset=target_dataset)
    register_cohort_metadata(
        materialised=materialised.outputs["path"],
        irb=irb_protocol_id,
        cohort_def_id=cohort_definition_id,
    )

Step 3 — Grant access; researchers consume

Per M1L3 patterns — bind a group like halo-cohort-2026-hcc-io@ to the new dataset with roles/bigquery.dataViewer; add the IRB-approved researchers to the group. Conditional bindings limit access by date (matching the IRB end date). Researchers query, train models, build figures.

The cohort as ML training input

For studies where the cohort drives ML training (HALO outcome prediction, ShepBERT clinical-text fine-tuning anchored on a cohort), the materialised cohort feeds directly into M6L2\'s training pipelines:

# Trigger a model training pipeline when a new cohort materialises (M4L2 pattern)
#   Pub/Sub: cohort.materialised → Cloud Function triggers Vertex Pipeline
#   Pipeline reads gs://srl-halo-research/cohorts/hcc-io-2026/ as training input
#   Trains an outcome predictor, evaluates, conditionally registers (M6L2 gate)

The full chain — IRB protocol → ATLAS cohort definition → research-tier materialisation → Vertex Pipeline training run → registered model — is one continuous flow with audit logs at every step. From cohort idea to deployed model is a process, not a project.

Hands-on exercise 9.3

Run a cohort across two synthetic sites

🌐 Federated analytics in miniature
  1. From M9L1 + L2 you have an OMOP Synthea dataset + ATLAS running. Use a copy as the "second site" for the exercise.
  2. Make a second OMOP dataset. Generate a second Synthea population (or copy + perturb the existing data) into a second BigQuery dataset:
    bq mk --location=us-west1 demo_omop_site2
    bq cp -a demo_omop.person demo_omop_site2.person (and same for the other tables; perturb with a sample query to vary counts)
  3. Define one cohort in ATLAS. Use the T2DM cohort from M9L2\'s exercise. Export the cohort definition as JSON (in ATLAS: Cohort Definitions → Export → JSON). Save to cohorts/t2dm-cohort.json.
  4. Register both datasets as WebAPI sources. POST a second source-creation JSON pointing at demo_omop_site2 (same shape as the first, different tableQualifier).
  5. Generate the same cohort against both sources. Use the ATLAS UI: select the cohort, click Generate, pick source. Repeat for each source. Inspect the cohort counts — they should differ between sites (different synthetic populations).
  6. Export per-site summary CSVs. Run a SQL query against each source\'s cohort table: 
    SELECT
  "site_X" AS site_id,
  count(*) AS cohort_size,
  count(DISTINCT subject_id) AS unique_patients,
  min(cohort_start_date) AS first_entry,
  max(cohort_start_date) AS last_entry
FROM `demo_omop.cohort`
WHERE cohort_definition_id = 1234;
    Save the result CSV per site.
  7. Aggregate. Load both CSVs into a small aggregator BigQuery table; produce a meta-analysis result with SUM(cohort_size), AVG(unique_patients_per_site), .... That\'s the federated-analytics output without ever moving patient data.
  8. Bonus — write a tiny Vertex Pipeline that automates: pull cohort row set from a configured WebAPI source, summary-stat it, write to GCS. Run for each "site." This is the production shape of the workflow.
  9. Tear down:
    bq rm -r -d demo_omop demo_omop_site2
    (Plus the ATLAS / WebAPI infrastructure from M9L2 if not already torn down.)

Closing M9

You have the analytics layer

The arc of M9. L1 — OMOP CDM as a portable schema, on BigQuery with the partitioning patterns that matter. L2 — ATLAS and Broadsea as the cohort UI on top of OMOP, deployable on Cloud Run or GKE.
