Event-driven
architecture

Real research pipelines are not single services — they're chains. A file lands, a job processes it, a model trains, results publish. This lesson is the wiring: Cloud Functions, Eventarc, Pub/Sub, and Workflows, applied to the HIPIMR ingest pipeline end to end.

The lay of the land

Cloud Functions vs Cloud Run

Cloud Functions v2 is built on Cloud Run — same runtime, same scaling, same revision model. What differs is the deployment unit and the developer surface: Functions take a single function in your language; Cloud Run takes a container.

Cloud Functions (v2)
Cloud Run
Deployment unit
A single function: def handler(event, context):. GCP builds the container.
An OCI image you build yourself. Full control over deps, base image, layers.
Languages
Python, Node, Go, Java, .NET, Ruby, PHP — the supported runtimes only.
Any language. Anything that can be a container.
Triggering
First-class Eventarc and HTTP triggers wired by gcloud flags.
HTTP by default; Eventarc and Pub/Sub triggers wired separately.
Build pipeline
Cloud Build runs invisibly; you provide source.
You own the Dockerfile and the cloudbuild.yaml (M2).
Best for
Glue code, event handlers, "when X happens do Y" — under ~50 LOC.
Stateful APIs, custom dependencies, anything beyond a single handler.
Rule of thumb for SRL. Use Cloud Functions for the "when an object lands, kick off the pipeline" handlers — short, single-purpose, event-shaped. Use Cloud Run for everything else. They share a runtime, share a registry (Artifact Registry), share IAM patterns. Mix freely.
The universal router

Eventarc — one trigger model for everything

Eventarc turns GCP into one big event bus. Over 100 event sources — GCS object created, BigQuery job finished, Vertex training completed, Pub/Sub message, a generic Cloud Audit Log — get normalised into the CloudEvents format and routed to a Cloud Run service, Cloud Function, or Workflow.

The mental model

# A trigger says: "when THIS happens, call THIS, as THAT identity"

  Event source       →   Eventarc trigger    →   Target
  ──────────────         ─────────────────       ─────────
  GCS bucket             "new DICOM in              Cloud Run service or
  BigQuery job            srl-hipimr-landing"        Cloud Function
  Pub/Sub topic                                      Cloud Workflow
  Audit log filter                                   (as the trigger's SA)

Three trigger types you'll use

cloud.storage.object.v1.finalized

GCS object created → Cloud Run / Function

The classic "file landed in a bucket, do something with it." Filter by bucket + (optionally) object prefix or content type.

gcloud eventarc triggers create dicom-ingest-trigger \
  --location=us-west1 \
  --destination-run-service=dicom-ingest \
  --destination-run-region=us-west1 \
  --event-filters="type=google.cloud.storage.object.v1.finalized" \
  --event-filters="bucket=srl-hipimr-landing" \
  --service-account=eventarc-trigger-sa@$PROJECT.iam.gserviceaccount.com
type=google.cloud.audit.log.v1.written + filters

Audit log event → Cloud Run / Function

Any GCP API call that gets logged can become a trigger. Filter by serviceName + methodName. Powerful for things without first-class triggers: "when someone updates an IAM policy on a PHI project, run the policy-drift checker."

gcloud eventarc triggers create iam-drift-check \
  --location=global \
  --destination-run-service=iam-checker \
  --destination-run-region=us-west1 \
  --event-filters="type=google.cloud.audit.log.v1.written" \
  --event-filters="serviceName=cloudresourcemanager.googleapis.com" \
  --event-filters="methodName=SetIamPolicy" \
  --service-account=eventarc-trigger-sa@$PROJECT.iam.gserviceaccount.com
type=google.cloud.pubsub.topic.v1.messagePublished

Pub/Sub message → Cloud Run / Function

For your own custom events. Publish to a topic; subscribers (the trigger) receive each message as a CloudEvent. This is the pattern under most decoupled architectures.

Reading event payloads in your handler: CloudEvents come in as HTTP POSTs with the event in the JSON body. Use the official cloudevents Python / JS library to parse — don't roll your own parser. The metadata fields (eventTime, source, subject) are standardised across all 100+ source types.
The bus underneath

Pub/Sub patterns at depth

Pub/Sub is the plumbing under most of Eventarc and the right primitive for any custom event flow you design. Four patterns cover almost everything you'll build in a research lab.

Push vs pull subscriptions

PUSH subscription

Pub/Sub calls you

Configure the subscription with an HTTPS endpoint (a Cloud Run service or Function). Pub/Sub POSTs each message and retries on non-2xx.

Best for: serverless targets, low-volume events, when the consumer scales with traffic.

PULL subscription

Your code asks Pub/Sub

The consumer calls subscriber.pull() in a loop or stream. Acknowledges or nacks. More control; needs a long-lived runtime.

Best for: batch consumers, GKE pods, high-throughput streams, anywhere you need explicit flow control.

Four patterns you'll reach for

1. Fan-out — one event, many consumers

Decoupling downstream pipelines

Ingest publishes dicom.received to a topic. Three subscriptions on that topic: de-identification handler, indexing handler, audit notification handler. Each runs independently; one failing doesn't block the others.

2. Ordering keys — when sequence matters

Per-patient or per-study ordering

Pub/Sub default is at-least-once with no ordering. Set messageOrderingEnabled on the subscription and publish with an orderingKey (e.g., the patient MRN); messages with the same key arrive in order to the same consumer. Use sparingly — ordering reduces throughput.

3. Dead-letter queue — handling failures

When the handler keeps failing

Configure the subscription with --dead-letter-topic and --max-delivery-attempts=5. After 5 failed deliveries, the message moves to a separate topic for human investigation. Without a DLQ, a poison message can block a subscription forever.

gcloud pubsub subscriptions create dicom-ingest-sub \
  --topic=dicom-received \
  --push-endpoint=$CLOUD_RUN_URL \
  --push-auth-service-account=ingest-sa@$PROJECT.iam.gserviceaccount.com \
  --dead-letter-topic=dicom-received-dlq \
  --max-delivery-attempts=5 \
  --message-retention-duration=7d
4. Exactly-once (with the new flag)

When duplicates would corrupt downstream data

Pub/Sub now supports exactly-once delivery (--enable-exactly-once-delivery) on regional pull subscriptions. Useful when downstream is non-idempotent. Trade-off: lower throughput. Most well-designed pipelines should be idempotent anyway.

Orchestration without infrastructure

Cloud Workflows — chains of API calls

Cloud Workflows is a serverless state-machine engine. You declare a sequence of steps in YAML; each step calls a GCP API or an HTTP service. Workflows handles retries, conditional branching, parallelism, and waiting on long-running operations — without you running any infrastructure.

When to reach for Workflows instead of a Cloud Function chain: when the pipeline has more than two or three steps, or when you need waits, parallelism, or error recovery that's awkward to write in handler code.

A real example — post-ingest fan-out

# workflow.yaml — chain after DICOM ingest
main:
  params: [event]
  steps:
    - log_received:
        call: sys.log
        args:
          text: ${"Processing " + event.subject}
          severity: INFO

    - branch_parallel:
        parallel:
          shared: [event]
          branches:
            - deid_branch:
                steps:
                  - run_deid:
                      call: googleapis.run.v1.namespaces.jobs.run
                      args:
                        name: ${"namespaces/" + sys.get_env("PROJECT_ID") + "/jobs/deid-job"}
                        body:
                          overrides:
                            containerOverrides:
                              - env:
                                  - name: SOURCE_OBJECT
                                    value: ${event.subject}
            - index_branch:
                steps:
                  - bq_load:
                      call: googleapis.bigquery.v2.jobs.insert
                      args:
                        projectId: ${sys.get_env("PROJECT_ID")}
                        body:
                          configuration:
                            load:
                              sourceUris: [${"gs://srl-hipimr-metadata/" + event.subject}]
                              destinationTable:
                                projectId: ${sys.get_env("PROJECT_ID")}
                                datasetId: "hipimr_index"
                                tableId: "dicom_metadata"

    - notify_complete:
        call: googleapis.pubsub.v1.projects.topics.publish
        args:
          topic: ${"projects/" + sys.get_env("PROJECT_ID") + "/topics/dicom-processed"}
          body:
            messages:
              - data: ${base64.encode(text.encode(json.encode(event)))}

    - done:
        return: ${"Completed processing of " + event.subject}

Deploy and trigger:

gcloud workflows deploy dicom-post-ingest \
  --source=workflow.yaml \
  --location=us-west1 \
  --service-account=workflow-sa@$PROJECT.iam.gserviceaccount.com

# Trigger from Eventarc on the same GCS event that fired the Cloud Function
gcloud eventarc triggers create post-ingest-trigger \
  --location=us-west1 \
  --destination-workflow=dicom-post-ingest \
  --destination-workflow-location=us-west1 \
  --event-filters="type=google.cloud.storage.object.v1.finalized" \
  --event-filters="bucket=srl-hipimr-landing" \
  --service-account=eventarc-trigger-sa@$PROJECT.iam.gserviceaccount.com
Workflows vs Vertex AI Pipelines: Workflows is for chaining API calls and HTTP services; Vertex Pipelines is for chaining containerised ML steps with artifacts between them (M11). Use Workflows for ingest/notification/orchestration; use Vertex Pipelines for training, evaluation, model deployment.
All together

The HIPIMR ingest pipeline, fully event-driven

M3L2 showed the network path for HIPIMR ingest. Here is the same pipeline expressed as events and handlers — the actual sequence of triggers, services, and topics that move a Queen's DICOM batch from the landing bucket to the trained model.

[1] GCS landing bucket srl-hipimr-landing │ ▼ object created [2] Eventarc trigger dicom-landed │ ▼ POST CloudEvent [3] Cloud Function dicom-validate - Validates the upload (size, format, study UID metadata) - Rejects malformed: moves to /quarantine - On success: publishes to Pub/Sub │ ▼ message: dicom.validated [4] Pub/Sub topic dicom-validated ├──────────────────┬──────────────────┬──────────────────┐ ▼ sub: deid ▼ sub: index ▼ sub: audit ▼ sub: notify [5a] Cloud Run Job [5b] Cloud Func [5c] Cloud Func [5d] Cloud Func deid-job bq-index audit-log slack-notify │ │ │ ▼ publishes ▼ writes ▼ writes [6] Pub/Sub dicom-deid'd BigQuery hipimr_index BigQuery audit_extra │ ▼ sub: train-trigger [7] Cloud Run Job density-cnn-train - Pulls latest weights from GCS, runs Vertex training - Stores new model in Vertex Model Registry - Publishes to Pub/Sub │ ▼ message: model.updated [8] Cloud Function deploy-canary - Deploys the new model to xai-serve as a 10% canary - Notifies the team to validate, then promotes after 24h

Why this shape works

🧩
Decoupled
Any one step can fail or be rewritten without rebuilding the pipeline. The audit branch failing doesn't block training.
🔁
Idempotent
Every step is safe to retry. Re-running deid on the same object produces the same output. Pub/Sub's at-least-once delivery is fine.
🔍
Fully audited
Every event in every topic, every function invocation, every Workflow run lands in the audit sink from M1L4. End-to-end traceability for one DICOM is a single SQL query.
🚪
Boundary-respecting
Every step runs inside the VPC SC perimeter (M3L1); every SA is workload-scoped (M1L3); every secret in Secret Manager (M3L3). Architecture all the way down.
Hands-on exercise 4.2

Wire a GCS-triggered Function and a Pub/Sub fan-out

⚡ Event in, two consumers out
  1. Enable the relevant APIs in your sandbox:
    gcloud services enable eventarc.googleapis.com pubsub.googleapis.com cloudfunctions.googleapis.com workflows.googleapis.com
  2. Create a topic for your custom event:
    gcloud pubsub topics create demo-validated
  3. Create a landing bucket:
    gcloud storage buckets create gs://$PROJECT-demo-landing --location=us-west1
  4. Write the validator function. Save as main.py: 
    import os, json
from google.cloud import pubsub_v1
import functions_framework
from cloudevents.http import CloudEvent

publisher = pubsub_v1.PublisherClient()
TOPIC = f"projects/{os.environ['PROJECT_ID']}/topics/demo-validated"

@functions_framework.cloud_event
def validate(event: CloudEvent):
    data = event.data
    obj = f"gs://{data['bucket']}/{data['name']}"
    if data['size'] < "10":
        print(f"Rejecting {obj}: too small")
        return
    payload = json.dumps({"object": obj, "size": data['size']}).encode()
    publisher.publish(TOPIC, payload).result()
    print(f"Validated {obj}, published to demo-validated")
    And a requirements.txt: 
    functions-framework==3.*
google-cloud-pubsub==2.*
cloudevents==1.*
  5. Deploy it as a Cloud Function with a GCS trigger:
    gcloud functions deploy demo-validate --gen2 --region=us-west1 --runtime=python312 --source=. --entry-point=validate --trigger-bucket=$PROJECT-demo-landing --set-env-vars=PROJECT_ID=$PROJECT --service-account=eventarc-trigger-sa@$PROJECT.iam.gserviceaccount.com
  6. Create two subscriptions on demo-validated — one pulled by a tiny logger, one as a push to a Cloud Run service:
    gcloud pubsub subscriptions create demo-validated-logger --topic=demo-validated
    gcloud pubsub subscriptions create demo-validated-push --topic=demo-validated --push-endpoint=https://hello-flask-XXX.run.app/ --push-auth-service-account=push-sa@$PROJECT.iam.gserviceaccount.com
  7. Trigger the pipeline by uploading a file:
    echo "hello" > demo.txt && gcloud storage cp demo.txt gs://$PROJECT-demo-landing/
  8. Inspect each step:
    gcloud functions logs read demo-validate --gen2 --region=us-west1 --limit=10
    gcloud pubsub subscriptions pull demo-validated-logger --auto-ack --limit=5
    gcloud run services logs read hello-flask --region=us-west1 --limit=10
    You should see the validator log the upload, the logger pull the message, and your Cloud Run service log the inbound push.
  9. Bonus: deploy the Workflow from Section 4 with two parallel branches (one logs, one publishes to a third topic) and trigger it from another Eventarc trigger on the same bucket. Watch both pipelines run side-by-side per upload.
  10. Tear down when finished: delete the function, subscriptions, topic, bucket.

Knowledge check

Questions to sit with

  • You're writing a "when a new DICOM lands, kick off de-id" handler. Cloud Function or Cloud Run? Justify in one sentence. Hint
  • Your de-id pipeline keeps reprocessing the same DICOM occasionally. The audit log shows duplicates. Where does the duplication come from and what fixes it cheaply? Hint
  • The validator function rejects a malformed DICOM. Without a dead-letter topic, what happens after 5 retries? Hint
  • You have a four-step pipeline: validate → de-id → train → deploy-canary. Should this be one Cloud Function with sequential calls, four Cloud Functions chained via Pub/Sub, or a Cloud Workflow? Hint
