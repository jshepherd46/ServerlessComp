Lesson 4 of 4

Operational
architecture

Topology and identity tell you what to build and who can touch it. Operations tell you whether it's working, what it cost, and what to do when it isn't. This is the layer that turns a working setup into a sustainable lab.

⏱ ~45 min reading + exercise 🔧 Cloud Billing, Logging, Monitoring, Cloud Functions 📍 Module 1 of 12
The framing

Four pillars hold up a working lab

Researchers who stand up a beautiful GCP topology and watch it collapse three months later usually skipped the same four things. Treat these as non-negotiable from day one — and decide deliberately for each pillar what "good enough" looks like in your context.

💸 Cost

You know what each grant spends, what each study spends, and what is about to spend more than it should.

🪵 Audit

Every administrative action and every PHI read is recorded somewhere you can query — and that you actually keep.

📡 Observability

You see latency, errors, and throughput for the things you serve, and you get paged when they break.

📖 Runbook

The lab has a single written document that says: how to onboard, how to launch a study, how to respond to an incident, how to close out.

The honest framing: most research labs do pillar 1 and 4 badly, pillar 3 never, and pillar 2 only after an incident. The point of this lesson is to do all four lightly so none of them becomes a crisis.
Pillar 1 — Cost

Cost discipline: labels, exports, kill-switches

Three layers, ordered from cheapest to most defensive. Do at least the first two on every project; add the third anywhere the spend ceiling matters.

Layer 1 — Labels (you already did this in L2)

Every project carries lab, study, env, grant, phi. Most resources inherit project-level cost labels automatically. The ones that don't (BigQuery, Cloud Run, GKE workloads) take resource-level labels of their own. Use the same keys.

gcloud run services update xai-serve \
  --update-labels=study=makawalu,env=prod,workload=inference \
  --region=us-west1

Layer 2 — BigQuery billing export

Enable the standard billing export from your billing account to a BigQuery dataset in srl-shared-services. Within hours you have a queryable, labelled cost stream:

-- Last month's spend by study, in your shared services BQ
SELECT
  labels.value AS study,
  ROUND(SUM(cost), 2) AS usd
FROM `srl-shared-services.billing.gcp_billing_export_v1_XXXX`,
     UNNEST(labels) AS labels
WHERE labels.key = 'study'
  AND usage_start_time BETWEEN '2026-04-01' AND '2026-05-01'
GROUP BY study
ORDER BY usd DESC;

This is the query you run on the first of every month, screenshot, and paste into the lab's monthly update. Five minutes of work; complete clarity on where the money went.

Layer 3 — Budget alerts that actually do something

A budget alert on its own only emails. To make it enforce, wire the alert through Pub/Sub into a Cloud Function that disables billing on the project. The full pattern:

Cloud Billing ─[hits 100% of budget]─▶ Pub/Sub topic │ ▼ Cloud Function (kill-switch) │ ▼ billing.projects.updateBillingInfo # unlinks billing

A sketch of the Cloud Function (Python, <25 lines):

import base64, json
from google.cloud import billing_v1

PROJECT_ID = "srl-makawalu-dev"
client = billing_v1.CloudBillingClient()

def kill_switch(event, context):
    msg = json.loads(base64.b64decode(event["data"]))
    if msg.get("costAmount", 0) < msg.get("budgetAmount", 1):
        return  # not over budget yet
    name = f"projects/{PROJECT_ID}"
    info = billing_v1.ProjectBillingInfo(billing_account_name="")
    client.update_project_billing_info(name=name, project_billing_info=info)
    print(f"Disabled billing on {PROJECT_ID}")
Where to use this: always on sandboxes and student-accessible dev projects. Probably not on a prod ingest pipeline — disabling billing there could break PHI workflows mid-run. The right answer is project-specific.
Pillar 2 — Audit

A single audit sink for the whole lab

Cloud Audit Logs answer the only question that matters when something goes wrong: who did what, when, on which resource? Three kinds:

You don't enable these — they're on for every project by default. You just have to make sure you keep them somewhere you can query, because the 400-day default may not satisfy your IRB or grant retention.

For any project labelled phi=yes, turn on Data Access logging for the relevant services (Cloud Storage, BigQuery, Healthcare API). This is what auditors and the IRB will ask for. Without it, you cannot answer "who read patient 12345's mammogram?"

Rarely the log you start with, but include in the sink so you have it when you need it.

The sink architecture

Per-project log retention is a trap — you cannot query across studies, you lose logs when a project is deleted, and quotas surprise you mid-investigation. The architecture: a single aggregated log sink at the SRL folder that exports every project's audit logs into a BigQuery dataset in srl-shared-services.

Every SRL project ─▶ Folder-level aggregated sink ─▶ srl-shared-services.audit.logs # BigQuery dataset, partitioned by day 
gcloud logging sinks create srl-audit-sink \
  bigquery.googleapis.com/projects/srl-shared-services/datasets/audit \
  --folder=[SRL_FOLDER_ID] \
  --include-children \
  --log-filter='logName:"cloudaudit.googleapis.com"'

From here, every "who touched that bucket?" question is one SQL query. The dataset can hold years. Cost is trivial for a research lab.

Pillar 3 — Observability

Just enough monitoring

Full SRE-grade observability is overkill for most research labs. The minimum that pays for itself: logs you can search, golden signals on what you serve, and a notification channel that reaches you.

🔍
Cloud Logging Explorer
Already collecting everything from Cloud Run, GKE, Functions, Vertex. Learn one query pattern: resource.type="cloud_run_revision" AND severity>=ERROR. Bookmark it per service.
📈
Four golden signals per serving workload
Latency, traffic, error rate, saturation. Cloud Run and GKE surface them automatically; just create a dashboard per service so you can glance.
📟
Alerting policies that route somewhere real
Use Cloud Monitoring alerting → notification channel = email group + Slack webhook. Don't page individuals; page the team's channel and the on-call group.
🧪
Synthetic checks for PHI-facing APIs
Uptime Checks ping your XAI endpoint every minute with a non-PHI test payload. If it fails twice in five minutes, the alert fires.
What to skip until you need it: custom Prometheus stacks, OpenTelemetry collectors, traces across every span, full Grafana setups. They are great. They are also a research project of their own. Start with the four bullets above and add the rest only when a specific gap forces you to.
Pillar 4 — The runbook

The single document that ties it all together

Every lab needs one canonical document — Markdown in the lab's private repo is fine — that lives outside any one person's head. New people read it on day one. Returning people read it when they've forgotten. Auditors read it when they ask.

SRL operations runbook — minimum table of contents

  1. The topology. The diagram from M1L2. Update whenever you add a study.
  2. The IAM matrix. The table from M1L3. Update whenever a persona changes.
  3. Onboarding a new lab member. Which Google Groups to add, which dev project to assign, which docs to read (this course's M0 + the lab's data dictionary).
  4. Offboarding. Which groups to remove, which service accounts to rotate ownership of, which projects to leave alone.
  5. Launching a new study. The recipe: create folder structure, three projects, link billing, apply labels, enable APIs, request BAA enrollment.
  6. Closing a study. Archive bucket to lab archival GCS, export final OMOP/BigQuery snapshot, delete projects (30-day undo window).
  7. Incident response. "I think we leaked PHI." Who to call, which audit-log queries to run, what UH ITS expects within 24 hours.
  8. Monthly hygiene. Run the billing query, review IAM Recommender, scan for service accounts with key files, confirm budget alerts fired correctly.
  9. Pointers, not duplication. Every section links to authoritative docs (this course's relevant module, UH ITS's BAA process, GCP's own guides). The runbook is an index, not a textbook.

If your lab does not have this document, write a thirty-line stub of it tonight. Empty headings count. The act of having the document creates the obligation to fill it in — and the place to put answers when they come up. Without it, the same questions get answered from scratch every six months.

Hands-on exercise 1.4

Stand up the four pillars in your sandbox

📐 One pass through all four pillars
  1. Cost — billing export. In your sandbox project (or a shared-services project if you have one), create a BigQuery dataset called billing. Then in Billing → Billing export, enable the Standard usage and Detailed usage exports to that dataset. Wait a few hours for data to arrive.
  2. Cost — first query. When data appears, run the by-study query from Section 2 (adjust the table name to match your dataset). Even with only sandbox data, you'll see the pattern.
  3. Cost — kill-switch (optional, ~30 min). Wire up the kill-switch pattern on your sandbox: create a Pub/Sub topic, attach it to the budget alert you set in M0L1, deploy the Cloud Function above. Test by lowering the budget to $1.
  4. Audit — enable Data Access logs. IAM & Admin → Audit Logs, find Cloud Storage and BigQuery, enable Data Read + Data Write. (In a real PHI project this is mandatory; here it's practice.)
  5. Audit — folder sink. If you have a folder (sandbox doesn't, but follow the syntax), create an aggregated sink:
    gcloud logging sinks create srl-audit-sink bigquery.googleapis.com/projects/$PROJECT/datasets/audit --folder=[FOLDER_ID] --include-children --log-filter='logName:"cloudaudit.googleapis.com"'
  6. Observability — golden signals. Open your hello Cloud Run service from M0L1. Click Metrics. Note the four panels: request count, latency, container CPU, container memory. That's the dashboard.
  7. Observability — alerting policy. Monitoring → Alerting → Create Policy. Trigger when hello's 5xx error rate > 5% for 5 minutes. Notification channel = your email. Save.
  8. Runbook — write the stub. In your favourite editor, create srl-runbook.md with the nine headings from Section 5. Add a one-line description under each. Commit it to your lab's private repo.
