Fine-tuning
at lab scale

L1 set up the data plane. This lesson is the training itself — choose between zero-shot, LoRA, full fine-tune; the ShepBERT pipeline end to end; the evaluation patterns that actually catch problems; and the runbook section for fine-tuned NLP models.

⏱ ~55 min reading + exercise 🔧 Hugging Face Transformers + PEFT, Vertex Custom Training, Accelerate 📍 Module 7 of 12
The decision

How much fine-tuning do you actually need?

Four options in order of compute cost. Try them from lightest to heaviest; stop when one works. Most clinical NLP tasks land at LoRA — the others are real but rarer.

Option 1 — Zero compute

Zero/few-shot with a strong base

Use Bio_ClinicalBERT or Gemini directly with smart prompting. No training. Works for tasks where the base model already has the knowledge — common with general clinical summaries, simple entity extraction.

Try first. If F1 / clinical-evaluator approval clears the bar, ship it.
Option 2 — Prompt tuning / soft prompts

Train ~thousands of "prompt vector" parameters

Adds learned prefix tokens to the input; the model itself stays frozen. Tiny artifact, fastest training, often works surprisingly well.

Try when zero-shot is close but not enough. Cheap to experiment with.
Option 3 — LoRA / PEFT

Train rank-decomposition matrices on top of frozen weights

~0.5-2% of base parameters trained; the rest frozen. Adapter checkpoint is ~50-150 MB. Catches most of the benefit of full fine-tuning at a fraction of the compute. The SRL default for ShepBERT.

Where most clinical adapter tasks land.
Option 4 — Full fine-tune

Update every parameter

Highest cost (compute + memory), biggest artifact, risk of catastrophic forgetting. Justified when LoRA hits a clear ceiling on the metric you care about — usually when the target domain is very different from pretraining.

When LoRA has been tried at multiple ranks and the metric still misses.

Why LoRA wins for lab-scale clinical NLP

  • → Tiny artifacts. LoRA adapters are 50-150 MB. Many adapters per base — one per study, one per task, one per UH site if needed.
  • → Tiny compute. A100 single-GPU for hours, not days. Spot-friendly. The whole pipeline is cheap enough to iterate weekly.
  • → Adapter composition. Multiple adapters can be loaded into the same base; the base stays in GPU memory and you swap adapters at inference time. One server, many studies.
  • → Lower forgetting risk. Base weights frozen → behaviour on unrelated tasks doesn't degrade.
  • → Easier compliance story. Reproducing a result means re-running the LoRA training with the same data + hyperparameters — small data, small compute, fast.
LoRA in practice

The hyperparameters that actually matter

LoRA has a small number of knobs. Most of them have sensible defaults; two of them are worth understanding deeply because they drive everything else.

r (rank)
The single most important knob. Higher rank = more parameters = more capacity = better fit, but more compute and bigger artifact. Start at r=8 for most clinical encoder tasks; bump to 16 or 32 if validation metrics plateau. ShepBERT uses r=16 for its UH adapter.
alpha (LoRA scaling)
Effectively learning-rate scaling for the LoRA matrices. Conventional choice: alpha = 2 × r (so r=16 → alpha=32). Lets you change r without re-tuning the learning rate. Don't over-think; the default works.
target_modules
Which transformer layers get LoRA matrices. Default ["q_proj", "v_proj"] (query and value projections). For tasks where the base model needs more adaptation, expand to all attention matrices and the feed-forward layers. Bigger surface = more capacity = bigger artifact.
lora_dropout
Standard dropout regularisation on LoRA outputs. 0.05-0.1 typical. Increase if you see overfitting on the validation set.
learning rate
LoRA learning rates are typically 5-10× higher than full fine-tune rates. Start at 1e-4 for clinical encoder tasks; sweep 5e-5 to 5e-4 with Vizier if metrics need lifting.
epochs
3-10 typical for clinical adapters; LoRA converges fast. Watch validation loss; early-stop on plateau. Going long doesn't help and risks overfitting.
batch size
As large as one GPU memory allows for stable training. Use gradient accumulation if you need effective batch sizes larger than physical memory permits.

The minimal LoRA training script

from transformers import AutoModelForTokenClassification, AutoTokenizer, TrainingArguments, Trainer
from peft import LoraConfig, get_peft_model, TaskType
from datasets import load_dataset

base = AutoModelForTokenClassification.from_pretrained(
    "emilyalsentzer/Bio_ClinicalBERT",
    num_labels=len(LABELS),
)

config = LoraConfig(
    task_type=TaskType.TOKEN_CLS,
    r=16, lora_alpha=32, lora_dropout=0.1,
    target_modules=["query", "value"],   # BERT module names
    bias="none",
)
model = get_peft_model(base, config)
model.print_trainable_parameters()         # roughly 1% of base

dataset = load_dataset("json", data_files={
    "train": "gs://srl-shepbert-training/v3/train-*.jsonl",
    "eval":  "gs://srl-shepbert-training/v3/eval-*.jsonl",
})

args = TrainingArguments(
    output_dir="/tmp/shepbert-adapter",
    num_train_epochs=5,
    learning_rate=1e-4,
    per_device_train_batch_size=16,
    gradient_accumulation_steps=2,
    fp16=True,                            # half-precision for memory
    eval_strategy="epoch",
    save_strategy="epoch",
    load_best_model_at_end=True,
    metric_for_best_model="f1",
    logging_dir=os.environ["AIP_TENSORBOARD_LOG_DIR"],
)

Trainer(model=model, args=args, train_dataset=dataset["train"], eval_dataset=dataset["eval"]).train()
model.save_pretrained("gs://srl-shepbert-training/adapters/uh-ner-v3-7/")
The result. A ~100 MB adapter file (adapter_model.safetensors + adapter_config.json) in GCS, ready to register in Vertex Model Registry per M6L3. At serve time, you load Bio_ClinicalBERT once + load the adapter — the inference container is small, the base model is shared across multiple adapters (one per task), and a new task = new adapter, not new full model.
The pipeline

ShepBERT fine-tuning, end to end

Built with the M6L2 Pipelines patterns — same shape as Makawalu's CBM workflow, different content. Five components, run by Vertex AI Pipelines on a schedule (weekly) and on-demand when new label sets land.

# Conceptual pipeline shape - actual KFP code from M6L2 template
@dsl.pipeline(name="shepbert-finetune", pipeline_root="gs://srl-shepbert-training/pipelines/")
def shepbert_pipeline(
    note_table: str = "srl-halo-prod.omop.note",
    labels_table: str = "srl-shepbert-prod.labels.entities_v3",
    base_model: str = "emilyalsentzer/Bio_ClinicalBERT",
    lora_r: int = 16,
    epochs: int = 5,
):
    # 1. Build dataset from BigQuery
    dataset = build_corpus(note_table=note_table, labels_table=labels_table)

    # 2. Fine-tune with LoRA on a single A100
    adapter = lora_finetune(
        dataset=dataset.outputs["corpus"],
        base_model=base_model,
        r=lora_r, epochs=epochs,
    )

    # 3. Evaluate on held-out + by clinical slice
    eval_results = evaluate_adapter(
        adapter=adapter.outputs["model_out"],
        dataset=dataset.outputs["corpus"],
    )

    # 4. Conditional register + canary notify
    with dsl.If(eval_results.outputs["f1"] > 0.80):
        register_adapter(adapter=adapter.outputs["model_out"], metrics=eval_results.outputs["metrics"])
        notify_pubsub(message="shepbert-adapter-updated")

The five components, made concrete

build_corpus

Queries the BigQuery note + labels tables; chunks notes section-aware (HPI / PE / A&P); writes JSONL shards to GCS. Output artifact: corpus path + manifest.

lora_finetune

Vertex Custom Training, single A100, spot. Runs the LoRA script from Section 2. Saves checkpoint per epoch. Logs to Vertex Experiments + TensorBoard. Output: adapter artifact in GCS.

evaluate_adapter

Loads base + adapter; runs on the held-out split; computes entity-level F1, slice metrics by service line + race/ethnicity (fairness), and per-section accuracy (HPI vs A&P vs PE). Output: structured metrics JSON.

register_adapter (conditional)

If F1 passes the gate AND the worst slice doesn't regress vs the current production adapter, register a new version in Vertex Model Registry. Aliases: candidate, latest; production stays where it is until a human promotes.

notify_pubsub

Publishes to shepbert.adapter_updated. Downstream: M4L2's canary deployer subscribes; the new adapter goes to 10% canary on the GKE inference service; a Slack message hits Pavel + the on-call channel.

Evaluation patterns

Metrics that actually catch problems

A single headline F1 is necessary and insufficient. For clinical NLP, the failure modes that matter most are slice-specific — and aggregate metrics hide them. Four metric layers, in order.

1. Aggregate metric (the headline)

For ShepBERT entity extraction: micro-F1 across all entities. The number on the dashboard. Useful for "is this version better than the previous one in expectation" — but never sufficient for "is this version safe to deploy."

2. Per-entity-class breakdown

F1 per entity type (drug, symptom, anatomy, diagnosis). A new version with the same overall F1 may have improved diagnosis recall at the cost of drug precision. Almost always more revealing than the aggregate.

3. Slice metrics — the fairness check

F1 per: service line (oncology / cardiology / primary care), patient subgroup (race/ethnicity, age band), document type (admission / discharge / progress). A regression on Pacific Islander patients invisible in aggregate metrics is exactly the failure mode the OMOP & HIPAA tracks of this course exist to prevent.

4. Clinical-expert spot review

For each new adapter version, pull 50-100 disagreement cases (current vs candidate model) and have a clinician adjudicate. Slow; necessary. The disagreement set is where you find that the new model is "right" but in a brittle way, or "wrong" but in a learn-from-it way. M6L3's runbook step for human-in-loop promotion lives here.

The gate test in register_adapter uses two clauses — aggregate F1 passes the threshold AND worst_slice_delta > -0.02. A new version that gains 0.02 in aggregate but regresses 0.05 on a clinical slice fails the gate. The runbook documents the slices that matter; the pipeline enforces them.
Hands-on exercise 7.2

LoRA fine-tune a clinical encoder, evaluate slice metrics

🔧 The full loop on a small task
  1. In Vertex Workbench (or anywhere with a GPU), install dependencies:
    pip install transformers peft datasets accelerate seqeval
  2. Use a public clinical-NLP dataset as a stand-in. The i2b2-2010 or n2c2 datasets require access, but tner/bionlp2004 on Hugging Face is open and biological-NLP shaped:
    from datasets import load_dataset
ds = load_dataset("tner/bionlp2004")
print(ds["train"][0])
  3. Load Bio_ClinicalBERT + tokenizer:
    from transformers import AutoTokenizer, AutoModelForTokenClassification
tok = AutoTokenizer.from_pretrained("emilyalsentzer/Bio_ClinicalBERT")
LABELS = ds["train"].features["tags"].feature.names
base = AutoModelForTokenClassification.from_pretrained(
    "emilyalsentzer/Bio_ClinicalBERT",
    num_labels=len(LABELS),
    id2label={i: l for i, l in enumerate(LABELS)},
    label2id={l: i for i, l in enumerate(LABELS)},
)
  4. Wrap with LoRA via PEFT:
    from peft import LoraConfig, get_peft_model, TaskType
config = LoraConfig(
    task_type=TaskType.TOKEN_CLS,
    r=16, lora_alpha=32, lora_dropout=0.1,
    target_modules=["query", "value"], bias="none",
)
model = get_peft_model(base, config)
model.print_trainable_parameters()
  5. Tokenize + align labels (token classification trick — labels are per-word, tokenizer splits subwords):
    def align(examples):
    enc = tok(examples["tokens"], is_split_into_words=True, truncation=True)
    labels = []
    for i, lbl in enumerate(examples["tags"]):
        word_ids = enc.word_ids(batch_index=i)
        labels.append([lbl[wid] if wid is not None else -100 for wid in word_ids])
    enc["labels"] = labels
    return enc

ds_tok = ds.map(align, batched=True, remove_columns=ds["train"].column_names)
  6. Set up training and run (a few minutes on a T4):
    from transformers import TrainingArguments, Trainer, DataCollatorForTokenClassification
args = TrainingArguments(
    output_dir="/tmp/demo-adapter",
    num_train_epochs=3,
    learning_rate=1e-4,
    per_device_train_batch_size=16,
    eval_strategy="epoch",
    save_strategy="no",
    fp16=True,
)
trainer = Trainer(
    model=model, args=args,
    train_dataset=ds_tok["train"], eval_dataset=ds_tok["validation"],
    data_collator=DataCollatorForTokenClassification(tok),
)
trainer.train()
  7. Evaluate with seqeval entity-level F1:
    from seqeval.metrics import classification_report
preds = trainer.predict(ds_tok["test"])
# Decode predictions back to label strings, then call classification_report
print(classification_report(true_labels, pred_labels, digits=3))
    Note the per-entity F1 alongside the aggregate. That's the M7L2 evaluation pattern in miniature.
  8. Save the adapter:
    model.save_pretrained("/tmp/demo-adapter")
    (Files: adapter_config.json, adapter_model.safetensors. Total size ~100 MB.)
  9. Push the adapter directory to GCS, register as a Vertex Model (mirrors M6L3):
    gcloud storage cp -r /tmp/demo-adapter gs://$PROJECT-demo/adapters/v1/
  10. Bonus: wrap the whole training in a Vertex Custom Training job (build a small Dockerfile, push to Artifact Registry, submit). That's the production shape.
