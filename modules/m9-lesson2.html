The OHDSI cohort UI

ATLAS, WebAPI, and the Broadsea stack

ATLAS is a browser-based UI for OHDSI workflows — concept-set design, cohort definition, characterization, incidence analysis, data quality. It does not talk to your database directly; it talks to WebAPI, a Java REST service that holds the cohort logic, generates SQL via SqlRender, and executes against whichever OMOP database is configured.

Broadsea packages the whole OHDSI stack as containers — ATLAS frontend, WebAPI backend, a Postgres for WebAPI\'s own metadata, R serving for analytic packages, optional HADES tools. Two layers separate concerns cleanly: deployable units (Broadsea containers) vs analytic data (your OMOP dataset in BigQuery).

The stack

┌────────────────────────────┐ │ ATLAS (browser UI) │ # Node + Angular SPA └────────────┬───────────────┘ │ REST ┌────────────▼───────────────┐ │ WebAPI (Java) │ # cohort logic, SqlRender → SQL └──────┬─────────┬───────────┘ │ JDBC │ JDBC ▼ ▼ ┌───────────┐ ┌──────────────────┐ │ webapi DB │ │ OMOP analytical │ # your study\'s data │ (Postgres)│ │ (BigQuery) │ └───────────┘ └──────────────────┘   ┌────────────────────────────┐ │ R serving / HADES (Plumber)│ # optional: CohortMethod, PLP, etc. └────────────────────────────┘

Which Broadsea services SRL actually runs

  • → ATLAS — yes (cohort UI for HALO + collaborators).
  • → WebAPI — yes (one instance per OMOP source).
  • → WebAPI Postgres — yes (Cloud SQL for managed Postgres).
  • → R serving — sometimes (when running PLP or CohortMethod studies; can be a separate Vertex Custom Training job instead).
  • → Solr for concept search — yes (Broadsea includes it).
Where to put it

Cloud Run or GKE — same architecture, different fit

Recommended for SRL prod

Cloud Run + Cloud SQL

Each Broadsea container as a Cloud Run service with internal ingress (M3L2). WebAPI Postgres in Cloud SQL with private IP. Solr as a small Cloud Run service or in a managed search.

Scales to zero between sessions; HALO usage is bursty so this matters.

Use for: most labs. Lower operational overhead, predictable cost.
When multi-source

GKE

Broadsea ships a Docker Compose / Helm chart that maps cleanly to a single Kubernetes namespace. One Deployment per service.

Justified when you run multiple WebAPI instances (per OMOP source), need in-cluster R serving with state, or want service-mesh routing.

Use for: institutions with multiple OMOP sites or many simultaneous studies.

The BigQuery JDBC bridge

WebAPI talks to SQL databases via JDBC. For BigQuery, you use Google\'s Simba/Magnitude BigQuery JDBC driver — Broadsea has built-in support for the bigquery dialect. Configure the source via WebAPI\'s admin endpoint.

# POST a new "source" to WebAPI pointing at your OMOP BigQuery dataset
{
  "name": "halo-bq-2026",
  "key": "HALO",
  "dialect": "bigquery",
  "connectionString": "jdbc:bigquery://https://www.googleapis.com/bigquery/v2:443;ProjectId=srl-halo-prod;OAuthType=2",
  "daimons": [
    {"daimonType": "CDM", "tableQualifier": "omop"},
    {"daimonType": "VOCABULARY", "tableQualifier": "shared_vocab"},
    {"daimonType": "RESULTS", "tableQualifier": "halo_results"}
  ]
}

Now ATLAS connects to WebAPI which connects to your BigQuery OMOP dataset (via the workload SA, audit logged) and the shared vocabulary in srl-shared-services. Cohort definitions compile to BigQuery SQL via SqlRender and execute server-side.

Data quality

ACHILLES — what your data actually looks like

ACHILLES is an R package that runs ~200 pre-canned data-quality queries against an OMOP dataset and produces summary statistics: row counts per table, concept distributions, temporal patterns, demographic breakdowns, anomalies. The output feeds an ATLAS dashboard that surfaces "your data is fine" or "your data is broken in these specific ways."

What ACHILLES catches

  • → Coverage gaps — months with no condition_occurrence rows (ETL failure for that period).
  • → Concept drift — most-frequent diagnosis codes that shift abruptly between years.
  • → Implausible values — patient ages above 130, lab values outside biological ranges.
  • → Reference integrity — condition rows with condition_concept_id = 0 (unmapped source codes).
  • → Vocabulary mapping — large fraction of conditions still using source concepts rather than standard.

Running ACHILLES on a schedule

# Cloud Run Job (M2L3) running the ACHILLES R container quarterly
gcloud run jobs deploy halo-achilles \
  --image=docker.io/ohdsi/broadsea-hades:latest \
  --region=us-west1 \
  --service-account=achilles-sa@srl-halo-prod.iam.gserviceaccount.com \
  --task-timeout=4h --memory=8Gi \
  --command=Rscript --args=/scripts/run-achilles.R

# Schedule via Cloud Scheduler (M4L2 pattern)
gcloud scheduler jobs create http halo-achilles-quarterly \
  --location=us-west1 \
  --schedule="0 2 1 1,4,7,10 *" \
  --uri="https://us-west1-run.googleapis.com/apis/run.googleapis.com/v1/namespaces/$PROJECT/jobs/halo-achilles:run" \
  --http-method=POST --oauth-service-account-email=scheduler-sa@$PROJECT.iam.gserviceaccount.com

The R script connects to BigQuery as the workload SA, runs ACHILLES queries, writes results to halo_results.achilles_results + achilles_results_dist tables. ATLAS surfaces those tables as the Data Sources dashboard.

Run ACHILLES the day you set up the dataset, and every quarter after. The first run is your baseline. Subsequent runs catch drift, ETL regressions, and vocabulary update effects. Make it part of the M1L4 runbook\'s "monthly hygiene" section.
The workflow

From cohort idea to executable definition

A cohort in ATLAS is a precisely-defined population: who qualifies (inclusion criteria), who doesn\'t (exclusion criteria), how the start and end are anchored, what observation window applies. The mental model: a cohort is a function that takes a person and returns 0 or more cohort entry events for that person.

Step 1 — Define concept sets

Reusable named groups of concepts. "T2DM concepts" might include the standard T2DM concept + its descendants (M9L1\'s concept_ancestor trick). "Immunotherapy drugs" — a list of RxNorm ingredient ancestors. Define once; reuse across cohorts.

Step 2 — Pick the cohort entry event

The clinical event that marks "now this person is in the cohort." For HALO\'s HCC + immunotherapy cohort: first occurrence of an immunotherapy drug exposure for patients with prior HCC. Anchored by the drug_exposure row\'s start date.

Step 3 — Inclusion criteria

Additional filters applied at the cohort-entry date. "Prior HCC diagnosis within 2 years before index date." "Age >= 18 at index date." Each criterion is a sub-query that must hold.

Step 4 — Exclusion criteria + end strategy

"Excluded if prior other immunotherapy treatment." End strategy defines when the patient leaves the cohort — fixed offset (365 days), persistent exposure, or end of observation period.

Step 5 — Generate against a data source

Click "Generate" in ATLAS pointing at your OMOP source. WebAPI compiles the cohort definition to dialect-specific SQL via SqlRender, executes against BigQuery, writes the resulting cohort to halo_results.cohort.

Step 6 — Characterize and analyze

ATLAS runs descriptive analytics on the generated cohort: demographics, co-occurring conditions, drug history, lab values. Export the cohort definition as JSON; share with collaborators or run on another OMOP site for replication.

The same cohort, written programmatically (Capr)

For reproducibility and version control, define cohorts in R using OHDSI\'s Capr package; the R definition compiles to the same JSON ATLAS produces.

library(Capr)
hcc_concepts <- cs(descendants(4154027), name = "HCC")        # standard HCC concept
io_concepts  <- cs(descendants(21601783), name = "Immunotherapy")

halo_cohort <- cohort(
  entry = entry(
    drugExposure(io_concepts, firstOccurrence()),
    primaryCriteriaLimit = "First"
  ),
  attrition = attrition(
    "Has prior HCC" = withAll(
      atLeast(1, conditionOccurrence(hcc_concepts), duringInterval(eventStarts(-730, 0)))
    ),
    "Adult" = withAll(atLeast(18, age()))
  ),
  exit = exit(endStrategy = fixedExit("startDate", 365))
)

# Compile to ATLAS-compatible JSON
writeCohort(halo_cohort, "halo-hcc-io-v1.json")
The advantage of Capr. Cohort definitions live as code in your lab repo, version-controlled with git, peer-reviewable in pull requests, runnable from a Vertex Pipeline rather than from a browser session. The ATLAS UI is great for exploration; production cohorts should be code.
Hands-on exercise 9.2

Stand up Broadsea, define a cohort

📊 The OHDSI stack on your platform
  1. From M9L1, you have a Synthea OMOP dataset in BigQuery. Verify it\'s still there.
  2. Create a small Cloud SQL Postgres for WebAPI:
    gcloud sql instances create demo-webapi-db --database-version=POSTGRES_15 --region=us-west1 --tier=db-f1-micro
    (Public IP for the demo — in prod use private IP per Section 2.)
  3. Create the WebAPI schema:
    gcloud sql databases create webapi --instance=demo-webapi-db
    Set a password: gcloud sql users set-password postgres --instance=demo-webapi-db --password=$(openssl rand -base64 24)
  4. Deploy WebAPI to Cloud Run from the official Broadsea image:
    gcloud run deploy demo-webapi \
  --image=docker.io/ohdsi/broadsea-webapi:2.14.0 \
  --region=us-west1 \
  --memory=2Gi \
  --no-allow-unauthenticated \
  --add-cloudsql-instances=$PROJECT:us-west1:demo-webapi-db \
  --set-env-vars=DATASOURCE_DRIVERCLASSNAME=org.postgresql.Driver \
  --set-env-vars=DATASOURCE_URL=jdbc:postgresql:///webapi?cloudSqlInstance=$PROJECT:us-west1:demo-webapi-db\&socketFactory=com.google.cloud.sql.postgres.SocketFactory \
  --set-env-vars=DATASOURCE_USERNAME=postgres \
  --update-secrets=DATASOURCE_PASSWORD=projects/$PROJECT/secrets/webapi-pwd:latest
    Wait a few minutes; check the readiness endpoint at /WebAPI/info.
  5. Deploy ATLAS pointing at the WebAPI:
    WEBAPI_URL=$(gcloud run services describe demo-webapi --region=us-west1 --format='value(status.url)')
gcloud run deploy demo-atlas \
  --image=docker.io/ohdsi/broadsea-atlas:2.14.0 \
  --region=us-west1 \
  --memory=512Mi \
  --no-allow-unauthenticated \
  --set-env-vars=WEBAPI_URL=$WEBAPI_URL/WebAPI/
    Open the ATLAS URL in a browser (you\'ll need an identity token in the auth header, or temporarily allow unauthenticated for the demo).
  6. Configure a BigQuery source in WebAPI. POST the source-creation JSON from Section 2 to $WEBAPI_URL/WebAPI/source/sources. The WebAPI Cloud Run SA needs roles/bigquery.dataViewer on your demo OMOP dataset.
  7. Define a cohort in ATLAS. Navigate to Cohort Definitions → New Cohort. Build a simple T2DM cohort using the concept-set + entry-event pattern from Section 4. Generate against your demo OMOP source. Watch the SQL produced.
  8. Bonus — run ACHILLES. Deploy the HADES R container as a Cloud Run Job; mount config pointing at your demo OMOP source; run achilles() from R. View the resulting data-quality dashboard in ATLAS.
  9. Tear down aggressively — Cloud SQL accrues cost even idle:
    gcloud run services delete demo-webapi demo-atlas --region=us-west1
    gcloud sql instances delete demo-webapi-db
