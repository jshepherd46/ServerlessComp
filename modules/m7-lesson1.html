Clinical NLP on GCP
— the data plane

M6 was imaging — pixels in, scores out. M7 is text — clinical notes in, structured concepts or generation out. Different data plane, same architectural skeleton. This lesson is what makes clinical text hard, the BERT family, where models live, and the data path Pavel's ShepBERT pipeline runs on.

⏱ ~50 min reading + exercise 🔧 BigQuery, Healthcare API FHIR/HL7, OMOP NOTE_NLP, Hugging Face 📍 Module 7 of 12
The lay of the land

What makes clinical text different

A model that achieves 95% F1 on the IMDB review benchmark performs poorly on discharge summaries. The reasons are specific, and they shape every architectural choice from here on.

PHI everywhere — in the input, in the output

A clinical note contains names, MRNs, dates, providers, locations. Any model that reads a note is reading PHI. Any model that generates may surface training-set names verbatim. Both are HIPAA concerns from the first byte. Implication: compute lives inside the perimeter; training data is de-identified before training; outputs are post-filtered.

Abbreviations are domain-specific and ambiguous

"MS" is multiple sclerosis, mitral stenosis, or morphine sulfate depending on context. A clinical-domain model has seen these contexts during pretraining; a general-purpose model has not. Implication: start from a clinical-domain base (ClinicalBERT, BioMedLM, etc), not vanilla BERT.

Notes are templated, repetitive, and structured-within-unstructured

Most "free text" notes follow templates (HPI / PE / Assessment / Plan). The same boilerplate appears thousands of times. Implication: tokenisation and section parsing matter enormously; throwing a whole note at a 512-token window without section awareness wastes the budget on boilerplate.

Outcomes are sparse and confounded

"Did this patient have a stroke in the next 90 days?" is a label you can derive from billing codes — but billing codes are themselves a downstream artifact of clinical documentation. The label and the input are not independent. Implication: evaluate carefully; many clinical NLP benchmarks reward learning shortcuts that don't generalize.

Site shift is real and unforgiving

A model fine-tuned on MIMIC notes (Beth Israel) drops accuracy on UH Hawai'i notes. Demographics, documentation conventions, EHR templates, abbreviation conventions all shift between institutions. Implication: federated fine-tuning (M10) or domain adaptation on local notes; never deploy a foreign-site-only model as your primary inference path.

Where this leads. The architectural moves are mostly the same as M6 — perimeter, CMEK, audited workloads, Vertex training, registry-backed serving. The differences are: more de-identification work upstream, tighter constraints on what data leaves the perimeter, and more evaluation discipline. M7 takes the patterns from M6 and adds the NLP-specific layers.
The model zoo

Encoder, decoder, and which to reach for

Transformer models split into three families by architecture. Each is good at different tasks. Knowing the split keeps you from picking a chainsaw to peel an apple.

Encoder-only
BERT, RoBERTa, ClinicalBERT, BioBERT, BlueBERT. Reads input, produces a representation. Best for classification, entity extraction, similarity, retrieval embeddings. The right base for ShepBERT and most clinical NLP "label a note" / "extract entities" tasks.
Decoder-only
Gemini, GPT-4, Claude, Llama, Med-PaLM-2. Generates output autoregressively. Best for summarization, question-answering, chart-grounded generation. The right base for "summarize this discharge note" or "draft a referral letter" tasks. M7L3 covers their deployment.
Encoder-decoder
T5, BART, Flan-T5. Reads input, generates output. Best when the task has a structured "in/out" shape — translation, code generation, structured extraction with templated outputs. Less common than encoder-only or decoder-only in clinical NLP today.

Clinical-domain base models worth knowing

ClinicalBERT
BERT-base pretrained on MIMIC clinical notes (the Beth Israel ICU dataset). Strong starting point for most clinical encoder tasks. emilyalsentzer/Bio_ClinicalBERT on Hugging Face.
BioBERT
BERT-base further pretrained on PubMed abstracts + PMC full-text. Better for biomedical literature, slightly less specific to bedside notes. dmis-lab/biobert-base-cased-v1.1.
BlueBERT (Bio+Clinical)
Pretrained on both PubMed and MIMIC. Hybrid — broadly useful when your downstream task is mixed clinical/biomedical text. bionlp/bluebert_pubmed_mimic_uncased_L-12_H-768_A-12.
BioMedLM / PMC-LLaMA / Med-PaLM-2
Larger clinical/biomedical LLMs in the decoder family. For generative tasks. Some require licensing or specific deployment paths (Med-PaLM-2 is via Vertex AI Model Garden, not download).
SRL's default. Start from Bio_ClinicalBERT for encoder tasks (classification, entity extraction). Start from Med-PaLM-2 via Vertex Model Garden for generation tasks where its license fits, or Gemini via the Vertex API for de-identified inputs only. ShepBERT extends Bio_ClinicalBERT with LoRA fine-tuning on UH OMOP NOTE_NLP — M7L2 covers the pipeline.
Where text lives

Three data layers for clinical NLP on GCP

Layer 1 — Raw clinical notes (Cloud Healthcare API FHIR store)

Notes arrive from the EHR as FHIR DocumentReference resources or HL7v2 messages, ingested into a Cloud Healthcare API FHIR store. Same M3L1 perimeter discipline as DICOM: BAA-covered, CMEK-encrypted, Data Access logged. This is the system of record.

Layer 2 — Structured + de-identified text (BigQuery, OMOP NOTE / NOTE_NLP)

For HALO and other OMOP-based studies, notes are transformed into the OMOP NOTE table; downstream NLP results (entities, codes, attributes) populate NOTE_NLP. Both are queryable from BigQuery. De-identification happens at this layer — names, dates, providers all replaced with surrogates per HIPAA Safe Harbor or expert-determination rules.

Layer 3 — Training corpus (GCS objects, sharded)

For training and large-scale embedding extraction, BigQuery is the wrong shape — per-row queries are slow at training time. Materialise the de-id'd corpus as sharded GCS objects (JSONL or Arrow) that training pipelines read in random order.

When to use each

# Layer 1 - production ingestion and source of truth
gcloud healthcare fhir-stores import gcs $FHIR_STORE \
  --gcs-uri=gs://srl-shepbert-landing/ \
  --content-structure=BUNDLE

# Layer 2 - SQL-friendly analysis and cohort queries
SELECT note_id, person_id, note_date, note_text
FROM `srl-halo-prod.omop.note`
WHERE concept_id IN (
  SELECT note_concept_id FROM `srl-halo-prod.omop.note_nlp`
  WHERE concept_id = 4324326  # "discharge summary"
)
LIMIT 1000;

# Layer 3 - training-time random access
import tensorflow as tf
ds = tf.data.Dataset.list_files("gs://srl-shepbert-training/v3/train-*.jsonl")
ds = ds.interleave(lambda f: tf.data.TextLineDataset(f), num_parallel_calls=tf.data.AUTOTUNE)
The canonical flow. EHR → FHIR store (audit/perimeter/SoR) → Vertex Pipeline (de-identify + OMOP-shape) → BigQuery NOTE/NOTE_NLP (analytics + cohort) → GCS sharded corpus (training input). Three layers, each with a different consumer; each layer's de-identification stricter than the one above it.
The worked example

ShepBERT — UH clinical NLP, end to end

ShepBERT (Pavel Bushuyev) fine-tunes Bio_ClinicalBERT with LoRA on UH OMOP NLP data. The aim: clinical entity extraction adapted to UH documentation conventions, supporting downstream HALO and HIPIMR cohort definitions. Six concrete steps from EHR to served API.

Step 1 — Ingest

EHR → FHIR store

Nightly extract from UH EHR posts FHIR DocumentReference resources to srl-shepbert-prod's FHIR store. Same perimeter + CMEK + Data Access logging as HIPIMR DICOM (M3).

Step 2 — De-identification

Healthcare API de-id → de-id'd FHIR store

The Healthcare API deidentify operation runs on the FHIR store with HIPAA Safe Harbor rules plus custom rules for UH-specific patterns (provider initials, internal site codes). Output: a sibling de-id'd FHIR store. M8 covers the de-id rules in depth.

Step 3 — OMOP transformation

De-id'd FHIR → BigQuery OMOP NOTE table

A Cloud Run Job (or Dataflow for large batches) reads de-id'd FHIR, maps to OMOP NOTE schema, writes to BigQuery. Concept-mapping uses ATLAS-shared vocabulary (M9) so terms align with HALO.

Step 4 — Corpus materialisation

BigQuery → sharded GCS for training

Vertex Pipeline step exports a snapshot of NOTE (with section-aware chunking + tokenisation prep) as JSONL shards to gs://srl-shepbert-training/v3/. Versioned by date and pipeline run — training reads from a specific snapshot, not the live BigQuery table.

Step 5 — LoRA fine-tune

Vertex Custom Training: Bio_ClinicalBERT + LoRA adapters

Single-GPU A100. Base model frozen; LoRA matrices (~1% of parameter count) trained on the UH corpus. Checkpoints every epoch to GCS. M7L2 covers the script.

Step 6 — Register + serve

Adapter artifact in Vertex Model Registry; served from GKE

LoRA adapters are small (~100 MB); the full model is the base + adapters at load time. Served from GKE alongside other ShepBERT pipeline pods (the choice from M6L3's serving matrix). Inference: load base + adapter at startup; serve via FastAPI; per-request audit log per M3L3.

Same architectural skeleton as Makawalu's pipeline from M6 — different model family, different data layer, identical platform shape.

Hands-on exercise 7.1

Stand up a FHIR store, ingest, query notes

📝 The clinical-text plumbing
  1. Enable APIs:
    gcloud services enable healthcare.googleapis.com bigquery.googleapis.com
  2. Create a Healthcare dataset (or reuse from M6) and a FHIR store:
    gcloud healthcare datasets create demo-clinical --location=us-west1
    gcloud healthcare fhir-stores create demo-fhir --dataset=demo-clinical --location=us-west1 --version=R4
  3. Ingest a synthetic note. Create a tiny FHIR DocumentReference JSON with a fake note (no PHI - synthetic only for the exercise). Save as note.json: 
    {
  "resourceType": "DocumentReference",
  "status": "current",
  "type": {"coding": [{"system": "http://loinc.org", "code": "11506-3", "display": "Progress note"}]},
  "subject": {"reference": "Patient/synthetic-001"},
  "content": [{
    "attachment": {
      "contentType": "text/plain",
      "data": "UGF0aWVudCByZXBvcnRzIHN0YWJsZSBwYWluLiBQRSB1bnJlbWFya2FibGUu"
    }
  }]
}
    (The base64 decodes to "Patient reports stable pain. PE unremarkable." Synthetic, no PHI.)
  4. POST it to the FHIR store:
    TOKEN=$(gcloud auth print-access-token)
curl -X POST \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/fhir+json" \
  -d @note.json \
  "https://healthcare.googleapis.com/v1/projects/$PROJECT/locations/us-west1/datasets/demo-clinical/fhirStores/demo-fhir/fhir/DocumentReference"
    You should get back the resource with an assigned ID.
  5. Stream FHIR to BigQuery for analysis:
    bq mk --location=us-west1 demo_clinical

gcloud healthcare fhir-stores update demo-fhir \
  --dataset=demo-clinical --location=us-west1 \
  --enable-update-create \
  --pubsub-topic=projects/$PROJECT/topics/fhir-events
    (Or use the export-to-BQ operation for a snapshot.)
  6. Run a query against the OMOP NOTE pattern (using a synthetic table for the exercise — in real life this comes from your OMOP ETL):
    bq query --use_legacy_sql=false '
  SELECT note_id, note_text, note_class_concept_id
  FROM `demo_clinical.note`
  WHERE LENGTH(note_text) > 100
  LIMIT 10
'
  7. Load a tokenizer for ClinicalBERT in a Workbench notebook (or locally if installed):
    from transformers import AutoTokenizer
tok = AutoTokenizer.from_pretrained("emilyalsentzer/Bio_ClinicalBERT")
text = "Patient reports stable pain. PE unremarkable."
print(tok.tokenize(text))
# Look at how clinical terms tokenize - especially abbreviations like "PE"
    You\'ll see that PE tokenises as a single subword - one piece of why a clinical-domain base model exists.
  8. Tear down: delete the FHIR store, BigQuery dataset.
    gcloud healthcare fhir-stores delete demo-fhir --dataset=demo-clinical --location=us-west1
    gcloud healthcare datasets delete demo-clinical --location=us-west1
    bq rm -r -d demo_clinical
