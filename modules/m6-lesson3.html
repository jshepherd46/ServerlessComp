Serving and
the model lifecycle

L1 + L2 built a trained model. This lesson is everything between "trained" and "in production" — where it serves, how it's versioned, how you deploy a new version safely, and how you know when it's getting stale. The runbook section that ties it together.

⏱ ~55 min reading + exercise 🔧 Vertex Endpoints, Model Registry, Cloud Run, Batch Prediction 📍 Module 6 of 12
Where the model runs

Cloud Run vs Vertex Endpoints vs GKE

Three reasonable homes for an inference service. Each has its slot. The right choice depends on traffic shape, model size, and how much managed plumbing you want around it.

Capability Cloud Run Vertex Endpoints GKE
Setup overhead Lowest — deploy a container. Low — upload model + deploy. Highest — Deployment + HPA + Gateway + WI.
Scale to zero Yes — default behaviour. No — at least one node always. Yes via HPA min=0; cluster baseline cost persists.
Traffic splitting Built in (M4L1). Built in. Available via Gateway/mesh; more setup.
Explainability hooks DIY in the container. Native — Vertex Explainable AI. DIY.
Model monitoring DIY (Cloud Monitoring metrics). Native — Vertex Model Monitoring. DIY.
Multi-GPU serving Single-GPU at most. Native machine types. Full control.
Best fit Custom inference logic + bursty traffic. Standard ML predictions + explainability. Complex topology / serving + non-ML pods.

SRL's actual choices

  • → Makawalu XAI inference (CBM): Cloud Run. Custom output schema (concept scores + final + image overlays); bursty clinician demo traffic; scale-to-zero matters between demos.
  • → HIPIMR density prediction (registry workflow): Vertex Endpoint with batch prediction for backfills. Standard ResNet output; benefits from Vertex's monitoring.
  • → ShepBERT clinical NLP API (M7): GKE. Co-located with other ShepBERT pipeline pods; multi-GPU LoRA serving.
The default decision tree. If the model fits a single GPU and you don't need Vertex's native explainability/monitoring: Cloud Run. If you want Vertex Model Monitoring and the model is a standard predict-input-get-output shape: Vertex Endpoint. If you have multiple co-located services or multi-GPU serving: GKE. Most studies use Cloud Run for serving and Vertex for batch predictions.
The canonical artifact store

Vertex AI Model Registry

The Model Registry is your system of record for trained models. Every artifact registered there has a version, a lineage back to its training run, an evaluation record, and a status (deployed / staging / archived). Whatever serves traffic — Cloud Run, Vertex Endpoint, GKE — should pull its model from a registry version, not a freshly trained GCS blob.

Register a model from training output

# In the training pipeline\'s final step
from google.cloud import aiplatform

aiplatform.init(project="srl-makawalu-prod", location="us-west1")

model = aiplatform.Model.upload(
    display_name="makawalu-cbm",                  # the model name (reused across versions)
    artifact_uri="gs://srl-makawalu-training/cbm-v3/runs/v3-7/model/",
    serving_container_image_uri="us-west1-docker.pkg.dev/.../cbm-serve:0.4",
    description="CBM v3.7 - 4-channel concept heads + final classifier",
    labels={"study": "makawalu", "trained_by": "pipeline-v3-7", "auroc": "0.88"},
    version_aliases=["latest", "candidate"],     # human-friendly pointers to this version
)

Versions, aliases, and references

A Model has a stable resource name; versions accumulate under it. Aliases (latest, production, candidate) are mutable pointers at specific versions. The serving infrastructure references aliases, not version numbers — so promoting a candidate to production is one API call, not a redeploy.

# Promote the candidate version to production
gcloud ai models update $MODEL_ID --region=us-west1 \
  --version=v3.7 \
  --alias=production

# Or via the SDK
model_reg = aiplatform.Model(model_id=$MODEL_ID)
model_reg.update(version_aliases_to_add=["production"], version="v3.7")
The discipline this enables. Every version in the registry has metrics, lineage (which pipeline run produced it), and a labels dictionary you can use to filter. A year from now, "which CBM version was serving on March 15?" is a one-API-call answer. Without a registry, you're reconstructing it from GCS object timestamps and Slack messages.
Two serving paths in depth

Deploy: Vertex Endpoint or Cloud Run

Vertex Endpoint — managed

# Create the endpoint (one-time per service name)
gcloud ai endpoints create --region=us-west1 --display-name="makawalu-cbm-endpoint"

# Deploy a registered model version to it, with traffic split
gcloud ai endpoints deploy-model $ENDPOINT_ID \
  --region=us-west1 \
  --model=$MODEL_ID \
  --display-name=cbm-v3-7 \
  --machine-type=n1-standard-4 \
  --accelerator=type=nvidia-tesla-t4,count=1 \
  --min-replica-count=1 \
  --max-replica-count=4 \
  --traffic-split=0=100   # this version takes 100% (existing version 0 = none here)

# Predict
gcloud ai endpoints predict $ENDPOINT_ID --region=us-west1 \
  --json-request=test-input.json

Deploying a new version with a canary works the same way as M4L1 — deploy with a small traffic split, observe, promote.

gcloud ai endpoints deploy-model $ENDPOINT_ID --region=us-west1 \
  --model=$MODEL_ID --version=v3.8 \
  --traffic-split=v3-8=10,v3-7=90   # canary 10% to new version

Cloud Run — your own container, your own logic

For Makawalu's XAI output (concept scores + final + visualisation overlays), Cloud Run is cleaner — the inference container loads the model from the registry on startup and serves a custom JSON response.

# Inside the serving container - load model at startup
from google.cloud import aiplatform
import os

aiplatform.init(project=os.environ["PROJECT"], location="us-west1")
model = aiplatform.Model(model_name="makawalu-cbm", version="production")
model.download_to_local("/app/model/")            # pulls from registry to local disk

# Load + serve
import torch
weights = torch.load("/app/model/model.pt")
# ... standard inference handler ...

The Cloud Run service uses the M4L1 traffic-splitting model for canaries; the model promotion happens by updating the alias and redeploying. Both layers (registry alias + Cloud Run revision) have their own audit trail.

When real-time isn't the goal

Batch prediction — Vertex AI does the right thing

Many imaging studies need to score a backlog: "run the density model over all HIPIMR studies acquired before 2023." This is what Vertex Batch Prediction is for. You point it at the registered model + an input dataset (GCS URIs or BigQuery rows); Vertex parallelises across workers, writes results to GCS or BigQuery.

gcloud ai batch-predictions create \
  --region=us-west1 \
  --display-name=hipimr-density-backfill-2026 \
  --model=$MODEL_ID \
  --instances-format=jsonl \
  --gcs-source-uri=gs://srl-hipimr-training/backfill-inputs/*.jsonl \
  --predictions-format=jsonl \
  --gcs-destination=gs://srl-hipimr-training/backfill-outputs/2026-05/ \
  --machine-type=n1-standard-4 \
  --accelerator-type=NVIDIA_TESLA_T4 \
  --accelerator-count=1 \
  --starting-replica-count=4 \
  --max-replica-count=16

Vertex parallelises across the requested replica range, dynamically scaling to throughput. Output JSON includes the input + prediction; downstream BigQuery loading is trivial.

When to use which. Endpoints / Cloud Run for low-latency per-request prediction (real-time clinical demo, on-demand classifier). Batch Prediction for "score everything before the analysis next month." Most studies use both — Endpoints for the live API, Batch for the longitudinal backfills.
The model isn't fire-and-forget

Drift detection and the retraining loop

Clinical imaging shifts. New scanner models, new protocols, gradual population changes. A model trained on 2020 data degrades by 2026 — sometimes obviously, sometimes silently. The discipline: monitor for drift on every served model, and have a retraining trigger that does not depend on someone noticing the degradation.

Three kinds of drift to watch

1. Input drift (covariate shift)

The distribution of inputs to the model shifts over time. New scanner adds higher-resolution images; the model trained on lower-res inputs sees a feature distribution it didn't see in training. Detect by computing feature-statistics deltas between training set and recent live inputs.

2. Prediction drift

The distribution of predictions changes. The model used to predict 8% malignant on the cohort; it now predicts 14%. Could be a real population change — could be a problem with the model. Either way, you need to know.

3. Outcome drift (concept shift)

The mapping from inputs to outcomes shifts. Same image features now correspond to different ground-truth outcomes (new screening protocols, new biopsy guidelines). The hardest to detect; needs ground-truth labels on recent predictions.

Vertex Model Monitoring — the managed path for Endpoints

gcloud ai model-monitoring-jobs create \
  --region=us-west1 \
  --display-name=cbm-monitoring \
  --endpoint=$ENDPOINT_ID \
  --feature-thresholds="image_mean_intensity=0.3,age_at_imaging=0.2" \
  --emails=jshepherd@hawaii.edu \
  --sample-rate=0.5

Vertex Monitoring samples requests + predictions, computes feature/prediction stats over rolling windows, alerts when distribution-distance metrics (PSI, KL-divergence) exceed thresholds. The thresholds need tuning per study; start permissive, tighten as you understand the baseline noise.

DIY for Cloud Run / GKE serving

Without Vertex Endpoint, you build the same loop with: every prediction request logged with key features to BigQuery; a scheduled Cloud Run Job comparing recent distributions to a reference baseline; an alert published to Pub/Sub when a threshold breaches. The pieces are all things you've built before.

The retraining trigger

When drift is detected — automatically or manually — the trigger fires the Vertex AI Pipeline from M6L2 with the latest de-id'd dataset. If the new model passes the AUROC gate (the conditional in M6L2), it auto-registers and the canary deploy (M4L2) kicks off. The full loop:

drift detected → Pub/Sub: model.drift_detected
            → Cloud Function triggers Vertex Pipeline
            → Pipeline retrains, evaluates, conditional-registers new version
            → Pub/Sub: model.updated
            → Cloud Function deploys 10% canary to Cloud Run / Endpoint
            → 24h observation
            → human-in-loop promotion
            → alias "production" moves to new version
The human-in-loop step matters. For clinical-facing models, auto-promotion of a drift-triggered retrain is too risky — the new model might pass automated gates but have subtle issues (fairness regression on a subgroup; concept drift in the wrong direction). The pipeline goes all the way to "10% canary running"; a human promotes to 100% after explicit review. That's the right boundary between automation and oversight.
Putting it in the runbook

The model lifecycle, per-study

M1L4 introduced the lab runbook. Every served model deserves its own section, kept up to date as the model evolves. The template:

Section 1
What this model does. One paragraph: study, task, inputs, outputs, intended use. For Makawalu: "POCUS-based concept bottleneck model for breast cancer probability; inputs B-mode ultrasound clip frames; outputs four concept scores + final malignancy probability + visualization overlays. Not for stand-alone clinical decision-making."
Section 2
Where it lives. Registry path, current production version, serving endpoint URL, the SA that runs it, the secrets it consumes.
Section 3
Training lineage. Pipeline name + version. Where training data came from. Cohort definition. Last training date. Link to the experiment in Vertex AI.
Section 4
Evaluation envelope. Headline metrics (AUROC, accuracy, fairness slices). The data the metrics were computed on. Known limitations and out-of-distribution cases.
Section 5
Monitoring + retraining. Active monitoring jobs, drift thresholds, retraining schedule, the runbook entry for "drift alert fires."
Section 6
Deployment + rollback. How to deploy a new version (canary script). How to roll back if something goes wrong. Who must approve a production promotion.
Section 7
Deprecation. Conditions that would retire the model. The successor model's location, if known. The plan for sunsetting endpoints + archiving registry versions.
This section lives in Markdown, not in a slide deck. Versioned with the lab repo. Linked from M1L4's runbook table of contents. The next person who inherits Makawalu reads this on day one and knows: where the model lives, how it was trained, how to deploy a new version, what to do when monitoring fires. That's the difference between a research model and a sustainable research asset.
Hands-on exercise 6.3

Register a model, deploy to an Endpoint, canary

🎯 Registry → endpoint → traffic split
  1. Use the demo training output from M6L2's exercise (or any small model artifact you have in a GCS path). For simplicity, you can also use a Vertex pre-built classifier:
    MODEL_GCS="gs://cloud-samples-data/ai-platform/iris/iris_iris_v3_iris-mlp-model-with-explanation/" (or your own).
  2. Register the model in Vertex Model Registry:
    gcloud ai models upload --region=us-west1 \
  --display-name=demo-cbm \
  --container-image-uri=us-docker.pkg.dev/vertex-ai/prediction/sklearn-cpu.1-3:latest \
  --artifact-uri=$MODEL_GCS \
  --version-aliases=v1,latest
  3. List the model and its versions:
    gcloud ai models list --region=us-west1
    Save the model ID as MODEL_ID.
  4. Create an Endpoint:
    gcloud ai endpoints create --region=us-west1 --display-name=demo-endpoint
    Note the endpoint ID.
  5. Deploy the model at 100% traffic, CPU-only for the demo:
    gcloud ai endpoints deploy-model $ENDPOINT_ID --region=us-west1 \
  --model=$MODEL_ID \
  --display-name=demo-cbm-v1 \
  --machine-type=n1-standard-2 \
  --min-replica-count=1 \
  --max-replica-count=2 \
  --traffic-split=0=100
    Wait a few minutes for the deployment to come up.
  6. Send a test prediction:
    cat > instance.json <<EOF
{"instances": [[6.7, 3.0, 5.2, 2.3]]}
EOF

gcloud ai endpoints predict $ENDPOINT_ID --region=us-west1 \
  --json-request=instance.json
  7. Simulate a canary: upload the same model as version v2 (just for the exercise), deploy alongside v1, and split traffic 90/10:
    gcloud ai models upload --region=us-west1 --display-name=demo-cbm \
  --container-image-uri=us-docker.pkg.dev/vertex-ai/prediction/sklearn-cpu.1-3:latest \
  --artifact-uri=$MODEL_GCS --version-aliases=v2 \
  --parent-model=$MODEL_ID

gcloud ai endpoints deploy-model $ENDPOINT_ID --region=us-west1 \
  --model=$MODEL_ID --version=v2 \
  --display-name=demo-cbm-v2 \
  --machine-type=n1-standard-2 --min-replica-count=1 --max-replica-count=1 \
  --traffic-split=DEPLOYED_MODEL_v1=90,DEPLOYED_MODEL_v2=10
    (You'll need to replace DEPLOYED_MODEL_v1 / _v2 with the actual deployed-model IDs from gcloud ai endpoints describe.)
  8. Promote v2 to 100%:
    gcloud ai endpoints update-traffic $ENDPOINT_ID --region=us-west1 --traffic-split=DEPLOYED_MODEL_v2=100
  9. Stand up Model Monitoring on the endpoint (optional, costs accrue):
    gcloud ai model-monitoring-jobs create --region=us-west1 --display-name=demo-monitor --endpoint=$ENDPOINT_ID --emails=$EMAIL --sample-rate=1.0
  10. Tear down — endpoints accrue per-hour cost:
    gcloud ai endpoints undeploy-model $ENDPOINT_ID --region=us-west1 --deployed-model-id=
    (Repeat for v2.)
    gcloud ai endpoints delete $ENDPOINT_ID --region=us-west1
    Delete the monitoring job if created.
