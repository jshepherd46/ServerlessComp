Lesson 3 of 4

Identity & access
architecture

Topology decides where things live. Identity decides who can touch them. This lesson is about designing the access matrix on day one so you spend the rest of the year doing research, not chasing IAM tickets. HIPAA-specific controls land in M3 — here we build the patterns those controls bolt onto.

⏱ ~45 min reading + exercise 🔧 IAM, Google Groups, gcloud iam 📍 Module 1 of 12
The model

IAM is a decision system, not a checklist

IAM answers one question every time you call any GCP API: does this caller have permission to do this thing on this resource? Three primitives produce that answer:

👤
Principal
Who is calling. A Google user, a Google Group, a service account, or a federated identity. Every API call runs as exactly one principal.
🎚
Role
A named bundle of permissions. Predefined (per-service, like roles/run.invoker), basic (don't), or custom (write your own).
📍
Scope (the binding)
Where the role is granted: organization, folder, project, or single resource. Grants cascade down the hierarchy — never sideways or up.

The architecture question is not "which permissions does this person need" but where do I place the binding so the answer to that question is durable as people, studies, and grants come and go?

The cascade rule: a role granted at the folder level is inherited by every project below it. So roles/viewer on the SRL folder gives every postdoc read access to every SRL project automatically. Organization-level grants are even broader — use them only for org admins.
The design tool

The role-grant matrix

Don't grant roles ad hoc. Before you touch the console, write the matrix: rows are personas (the kinds of people and workloads that need access), columns are scopes (where the binding lives), cells are the role. Here is SRL's actual matrix, abridged:

Persona Scope Role Why
PI (J. Shepherd) folder: srl roles/owner Folder-level owner; cascades to every SRL project. Use sparingly — log every action.
Lab postdoc folder: srl roles/viewer Read-only across all SRL projects. Plus per-project elevated roles below.
Lab postdoc project: srl-makawalu-dev roles/editor Active developer rights only in their study's dev environment.
Research coordinator project: srl-hipimr-prod roles/storage.objectViewer Surgical: read DICOM objects only. No project metadata, no IAM visibility.
External collaborator (Queen's) project: srl-halo-prod roles/bigquery.dataViewer + roles/bigquery.jobUser Query OMOP tables but not create datasets. Conditional binding with time-bound expiry.
CI/CD service account project: srl-makawalu-prod roles/run.developer + roles/iam.serviceAccountUser Deploys Cloud Run revisions only. Cannot read user data.
Cloud Run workload SA project: srl-makawalu-prod roles/secretmanager.secretAccessor + roles/storage.objectViewer Reads its own secrets and the inference bucket. Nothing else.
UH ITS auditor folder: srl roles/iam.securityReviewer Sees every IAM binding, never the data. Read-only governance role.

Two things to notice. First, almost every human persona has a broad read role at the folder plus narrow elevated roles at the project. Second, every workload (CI/CD, Cloud Run) is a service account — not a person — with a sharply scoped role on exactly the project it runs in.

Delegation discipline

Bind to groups, not to people

Every human persona in the matrix above should be a Google Group, not a list of email addresses. The role binding targets the group; group membership is the only thing that changes as people join and leave.

Add them to srl-postdocs@hawaii.edu in Google Workspace. That single group is bound at the SRL folder with roles/viewer and at every *-dev project with roles/editor. The new postdoc immediately has the same access as every other postdoc — no IAM changes touched.

Their hawaii.edu account is deactivated by UH ITS, which removes them from every group automatically. Even if you forget, every binding evaluates membership at call time — no stale grants.

When the next PI joins HALO, you add them to the group. When you go on sabbatical, the lab does not need IAM gymnastics to keep running.

The only IAM bindings that should ever target individual email addresses are emergency break-glass admins (and even those are better as a group with two members). If you find yourself adding a person directly, ask: what is the group this belongs to, and does that group exist?
Workload identities

One service account per workload

Service accounts are not a convenience — they are the architectural primitive for everything your code does on GCP. The default discipline: one service account per workload, scoped to what that workload actually needs, and nothing else.

Naming pattern

<workload>-sa@<project-id>.iam.gserviceaccount.com

Makawalu-prod's actual service accounts

xai-serve-sa@srl-makawalu-prod.iam.gserviceaccount.com
Cloud Run service running the XAI inference API
Roles: roles/secretmanager.secretAccessor (for the model auth token), roles/storage.objectViewer (on the inference bucket only). No BigQuery, no PHI, no project metadata.
ingest-job-sa@srl-makawalu-prod.iam.gserviceaccount.com
Cloud Run Job that pulls Queen's POCUS uploads nightly
Roles: roles/healthcare.dicomEditor on the DICOM store, roles/pubsub.publisher on the ingest topic. Can write to one store, publish one event. Cannot read.
train-job-sa@srl-makawalu-prod.iam.gserviceaccount.com
Vertex AI Training jobs (CBM retraining)
Roles: roles/aiplatform.user, roles/storage.objectAdmin on the training bucket, roles/healthcare.dicomViewer on the source store. Reads source, writes outputs to training bucket only.
cicd-deploy-sa@srl-shared-services.iam.gserviceaccount.com
Cloud Build pipeline that deploys to Makawalu-prod
Roles in srl-makawalu-prod: roles/run.developer, roles/iam.serviceAccountUser on xai-serve-sa. Can deploy code; cannot touch data, cannot grant itself more.

Three rules that are not optional

🚫
No service account key files
Long-lived JSON keys are the #1 cause of GCP credential leaks. Use Workload Identity on GKE, attached service accounts on Cloud Run/Functions, and short-lived impersonation tokens for local dev.
🎭
Impersonation instead of keys
gcloud auth print-access-token --impersonate-service-account=xai-serve-sa@… gives you a one-hour token to act as the SA, fully audited, no key file on disk.
🪞
SA = workload, not person
Never share a service account between two workloads "just to save a few clicks." Every shared SA is a future blast-radius problem when one workload misbehaves.
🧨
Disable the default compute SA
Every project ships with [PROJECT_NUMBER]-compute@developer.gserviceaccount.com as Editor. Remove its role binding the first day. Workloads use SAs you create explicitly.
The day-to-day discipline

Granting the smallest role that works

Least privilege is not a one-time audit; it's the way you grant access every single time. The default workflow:

🎚 The least-privilege grant workflow
  1. Start at Viewer. The first binding for any new persona is roles/viewer at the most appropriate scope. If Viewer is enough, you're done.
  2. Add the narrowest predefined role that unblocks them. Need to deploy Cloud Run? roles/run.developer, not roles/editor. Need to read a bucket? roles/storage.objectViewer, not roles/storage.admin.
  3. Avoid the basic roles. roles/owner, roles/editor, roles/viewer are blunt instruments. Prefer the per-service predefined roles every time.
  4. Use IAM Conditions for time-bound access. External collaborator needs HALO access for the 3-month manuscript revision? Grant the role with request.time < timestamp("2026-09-01T00:00:00Z"). The grant expires automatically.
  5. Audit quarterly with the IAM Recommender. GCP analyses 90 days of usage and tells you which granted permissions a principal hasn't touched. Remove anything dormant.

Useful audit queries

# Every binding that touches a specific user
gcloud asset search-all-iam-policies \
  --scope=folders/[SRL_FOLDER_ID] \
  --query="policy:jshepherd@hawaii.edu"

# Every binding of a wide role across the lab
gcloud asset search-all-iam-policies \
  --scope=folders/[SRL_FOLDER_ID] \
  --query="policy.role.permissions:roles/owner"

# Recommender — surfaces over-granted roles
gcloud recommender recommendations list \
  --project=srl-makawalu-prod \
  --recommender=google.iam.policy.Recommender \
  --location=global
Hands-on exercise 1.3

Build the IAM matrix for one of your studies

📐 Design first, then grant
  1. On paper, draw the matrix for one study you actually run. Rows: PI, postdoc, RA, external collaborator, CI/CD, every workload. Columns: folder + each environment project. Fill cells with the narrowest role that works.
  2. Identify which rows should be groups, not individuals. Name the groups (srl-postdocs@, srl-halo-leads@, etc.).
  3. List every workload as its own service account row. Name them <workload>-sa.
  4. Switch to your sandbox and create three service accounts for a hypothetical pipeline:
    gcloud iam service-accounts create xai-serve-sa --display-name="XAI inference"
    gcloud iam service-accounts create ingest-job-sa --display-name="Nightly ingest"
    gcloud iam service-accounts create train-job-sa --display-name="Training runs"
  5. Grant each one a narrow predefined role (use whatever resources exist in your sandbox):
    gcloud projects add-iam-policy-binding $PROJECT --member="serviceAccount:xai-serve-sa@$PROJECT.iam.gserviceaccount.com" --role="roles/secretmanager.secretAccessor"
  6. Try impersonating one of them — no key file:
    gcloud auth print-access-token --impersonate-service-account=xai-serve-sa@$PROJECT.iam.gserviceaccount.com
  7. Find the default compute service account in your sandbox project's IAM page. Note its bindings. In M3 you'll remove those — but identify what would break if you did it today.
  8. Run the IAM Recommender against your sandbox:
    gcloud recommender recommendations list --project=$PROJECT --recommender=google.iam.policy.Recommender --location=global
