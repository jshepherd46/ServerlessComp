Lesson 3 of 3

Keys, secrets,
and audit

Perimeter and network keep PHI from leaving. This lesson is what happens to the data itself: who holds the keys, where the secrets live, what gets logged, and what to do in the first four hours when something goes wrong.

⏱ ~55 min reading + exercise 🔧 Cloud KMS, Secret Manager, Cloud Audit Logs 📍 Module 3 of 12
The framing

Three layers of data-side defense

M3L1 + L2 built the wall. This lesson is what's inside the wall — and what's behind it. Three independent layers, each protecting against a different failure mode.

Layer A · keys

Encryption you control

CMEK means GCP encrypts PHI with keys you own in Cloud KMS. Disable the key → data is unreadable, instantly, lab-wide. Without CMEK, GCP holds the keys; with CMEK, you do.

Layer B · secrets

Credentials at runtime

Secret Manager stores API keys, DB passwords, signing keys — versioned, IAM-scoped, audit-logged. No secret ever lives in code, in env vars stored in plain text, or in container images.

Layer C · audit

What happened, when

Audit logs answer the auditor's question: who read patient 12345's mammogram, when, from where. Always on for admin actions; you must turn on Data Access for PHI reads.

The defense-in-depth promise: if the perimeter fails, encryption keeps PHI unreadable to anyone without key access. If the perimeter and CMEK both fail, audit logs let you reconstruct what happened. Each layer is independent of the others — that's the point.
Layer A — keys

Cloud KMS keyring design

A keyring in Cloud KMS is a regional grouping of keys. A key is the cryptographic material (versioned, rotated). For SRL, the right structure is one keyring per region inside srl-shared-services, with one key per study + per purpose.

📦 srl-shared-services └── 🔑 keyring: srl-keys-uswest1 # one per region you operate in ├── 🗝 hipimr-gcs-uswest1 # encrypts HIPIMR GCS objects ├── 🗝 hipimr-bq-uswest1 # encrypts HIPIMR BigQuery tables ├── 🗝 makawalu-dicom-uswest1 # Makawalu DICOM store ├── 🗝 halo-omop-uswest1 # HALO BigQuery datasets └── 🗝 shared-secrets-uswest1 # Secret Manager encryption

Why keyrings live in shared-services

If a study project gets deleted, its CMEK keys go with it — and any data encrypted with those keys (including in OTHER projects, in your audit sink, in archives) becomes unrecoverable. Keys live in srl-shared-services precisely so their lifecycle is decoupled from any single study.

Create a keyring + key

gcloud config set project srl-shared-services
gcloud services enable cloudkms.googleapis.com

# 1. One keyring per region
gcloud kms keyrings create srl-keys-uswest1 --location=us-west1

# 2. One symmetric key per study/purpose, with automatic rotation
gcloud kms keys create hipimr-gcs-uswest1 \
  --keyring=srl-keys-uswest1 \
  --location=us-west1 \
  --purpose=encryption \
  --rotation-period=90d \
  --next-rotation-time=2026-08-22T00:00:00Z

Key IAM is separate from data IAM

This is the architectural lever that makes CMEK powerful. The roles on the KMS key are separate from the roles on the GCS bucket. Granting roles/storage.objectViewer on the bucket is not enough — the principal also needs roles/cloudkms.cryptoKeyEncrypterDecrypter on the key.

# Let HIPIMR's GCS service agent use the key (required for the bucket itself)
GCS_AGENT=$(gcloud projects describe srl-hipimr-prod --format="value(projectNumber)")
gcloud kms keys add-iam-policy-binding hipimr-gcs-uswest1 \
  --keyring=srl-keys-uswest1 --location=us-west1 \
  --member="serviceAccount:service-${GCS_AGENT}@gs-project-accounts.iam.gserviceaccount.com" \
  --role=roles/cloudkms.cryptoKeyEncrypterDecrypter

# Workload SA also needs the role to read objects (decryption happens in GCS, but the SA needs decrypt permission too)
gcloud kms keys add-iam-policy-binding hipimr-gcs-uswest1 \
  --keyring=srl-keys-uswest1 --location=us-west1 \
  --member="serviceAccount:ingest-job-sa@srl-hipimr-prod.iam.gserviceaccount.com" \
  --role=roles/cloudkms.cryptoKeyEncrypterDecrypter
The kill-switch property: disable the key with gcloud kms keys versions disable and within minutes every read of every object encrypted with that key fails. Re-enable to restore. There is no comparable lever without CMEK.
CMEK in practice

Applying keys to GCS, BigQuery, and disks

GCS bucket with CMEK at creation

KEY=projects/srl-shared-services/locations/us-west1/keyRings/srl-keys-uswest1/cryptoKeys/hipimr-gcs-uswest1

gcloud storage buckets create gs://srl-hipimr-dicom-prod \
  --project=srl-hipimr-prod \
  --location=us-west1 \
  --uniform-bucket-level-access \
  --default-encryption-key=$KEY

All new objects are encrypted with this key. Objects existing before the change keep their old encryption — you must rewrite them to re-encrypt.

BigQuery dataset with CMEK

bq --location=us-west1 mk \
  --dataset \
  --default_kms_key=$KEY \
  srl-halo-prod:omop_cdm

Tables inherit the default; you can override per-table for cross-key reads. Be aware: BigQuery jobs that materialise to a destination table need encrypt-decrypt rights on both the source and destination keys.

Persistent Disks with CMEK

gcloud compute disks create makawalu-training-pd \
  --zone=us-west1-a --size=200GB \
  --kms-key=$KEY

Enforcing CMEK at the org level

From M3L1: constraints/gcp.restrictNonCmekServices forces every new resource of the listed services to use CMEK. Apply at the SRL folder so all studies inherit the requirement:

cat > require-cmek.yaml <<EOF
constraint: constraints/gcp.restrictNonCmekServices
listPolicy:
  allowedValues:
    - storage.googleapis.com
    - bigquery.googleapis.com
    - compute.googleapis.com
    - healthcare.googleapis.com
    - secretmanager.googleapis.com
EOF
gcloud resource-manager org-policies set-policy require-cmek.yaml \
  --folder=$SRL_FOLDER_ID
Rotation behaviour: when a key version rotates, new writes use the new version. Old data stays encrypted with the old version, which remains accessible until you explicitly destroy it. So rotation is non-disruptive — and previous versions are recoverable for 30 days even after destroy if you ever rotate too aggressively.
Layer B — secrets

Secret Manager — the discipline

From M2L3 you already know how to mount a secret into Cloud Run. The HIPAA-grade discipline tightens four things:

🎯
One secret per workload per environment
Never share a secret across workloads "for convenience." Naming pattern: <workload>-<purpose>-<env> — e.g., xai-serve-api-key-prod.
🔄
Rotation built in
Configure a Pub/Sub topic to fire when a secret is due for rotation. A Cloud Function (or a human, depending on the secret) generates the new value and adds a new version.
📜
CMEK on the secret itself
Pass --kms-key-name when creating the secret. Now even the secret's stored bytes are encrypted with your key — disabling the key kills access to the secret too.
📊
Audited access on every read
Data Access logs on Secret Manager are off by default. Turn them on for every PHI project — see Section 5.

Create a hardened secret

# Secret encrypted with your CMEK key, in the right region, with a rotation period
gcloud secrets create xai-serve-api-key-prod \
  --project=srl-makawalu-prod \
  --replication-policy=user-managed \
  --locations=us-west1 \
  --kms-key-name=$KEY \
  --topics=projects/srl-makawalu-prod/topics/secret-rotations \
  --next-rotation-time=2026-08-22T00:00:00Z \
  --rotation-period=2592000s   # 30 days

# Add the first version
printf 'shhh-real-secret-value' | gcloud secrets versions add xai-serve-api-key-prod \
  --project=srl-makawalu-prod --data-file=-

Mount it into Cloud Run with least privilege

# Grant the workload SA accessor only on this specific secret
gcloud secrets add-iam-policy-binding xai-serve-api-key-prod \
  --project=srl-makawalu-prod \
  --member="serviceAccount:xai-serve-sa@srl-makawalu-prod.iam.gserviceaccount.com" \
  --role=roles/secretmanager.secretAccessor

# Mount as env var, latest version
gcloud run services update xai-serve \
  --region=us-west1 \
  --update-secrets=API_KEY=xai-serve-api-key-prod:latest

The mount reads the current version on each cold start. Add a new version, redeploy (or let the next cold start pick it up), retire the old version — never delete it until you know nothing is still reading it.

Layer C — audit

Data Access logs — what to enable, per service

M1L4 set up the folder-level aggregated sink. M3 adds the HIPAA-specific question: which services need Data Access logs on, for which projects?

The PHI-project checklist

storage.googleapis.com
Every GCS read (Data Read) and write (Data Write) on PHI buckets. Mandatory if you store DICOM, exports, or anything with PHI.
MUST
bigquery.googleapis.com
Every query, table read, job. Required for HALO OMOP, ShepBERT EHR work.
MUST
healthcare.googleapis.com
DICOM and FHIR reads. The most sensitive log — answers "who looked at this study?" Always on for PHI projects.
MUST
secretmanager.googleapis.com
Every secret access. Lets you correlate "this SA pulled this secret at this time" against the workload's audit trail.
MUST
cloudkms.googleapis.com
Every encryption/decryption call. Confirms which key versions were used and by whom.
MUST
run.googleapis.com
Cloud Run service invocations. Useful for tracing request paths through a HIPAA workflow.
SHOULD
pubsub.googleapis.com
Message publishes and pulls. Helps when reconstructing event-driven pipelines.
SHOULD
aiplatform.googleapis.com
Vertex AI Endpoint predictions, Training runs. Critical for tracing what data a model touched.
MUST

Enabling Data Access logs

# Per-project policy file. Apply to every PHI project.
cat > data-access-policy.yaml <<EOF
auditConfigs:
- service: storage.googleapis.com
  auditLogConfigs:
    - logType: DATA_READ
    - logType: DATA_WRITE
- service: bigquery.googleapis.com
  auditLogConfigs:
    - logType: DATA_READ
    - logType: DATA_WRITE
- service: healthcare.googleapis.com
  auditLogConfigs:
    - logType: DATA_READ
    - logType: DATA_WRITE
- service: secretmanager.googleapis.com
  auditLogConfigs:
    - logType: DATA_READ
- service: cloudkms.googleapis.com
  auditLogConfigs:
    - logType: DATA_READ
    - logType: DATA_WRITE
EOF

gcloud projects set-iam-policy srl-hipimr-prod data-access-policy.yaml

The aggregated sink from M1L4 already exists

All these new Data Access events flow through the folder-level sink into srl-shared-services.audit.logs. The query for the IRB question becomes:

SELECT
  timestamp,
  protoPayload.authenticationInfo.principalEmail AS who,
  protoPayload.methodName AS what,
  protoPayload.resourceName AS resource
FROM `srl-shared-services.audit.cloudaudit_googleapis_com_data_access`
WHERE protoPayload.serviceName = "healthcare.googleapis.com"
  AND protoPayload.resourceName LIKE "%studies/1.2.840.113619.2.55.3.604688.2.0.42%"
  AND timestamp BETWEEN '2026-02-01' AND '2026-05-01'
ORDER BY timestamp;

That's the answer to "show me every person who read DICOM study 0042 in the past 90 days." Without Data Access logs enabled on the day that study landed, this query returns nothing.

When something goes wrong

The first four hours of an incident

A postdoc emails: "I think I accidentally shared a HIPIMR bucket with my personal email last night, and the link works." What you do in the first four hours is what determines whether this is a contained mistake or a reportable breach. The runbook section from M1L4 has nine headings; this is the contents of "Incident response."

0–15 min · Stop the leak
Revoke the wrong binding right now
Find and remove the bad IAM grant. gcloud projects remove-iam-policy-binding srl-hipimr-prod --member="user:postdoc@gmail.com" --role="roles/storage.objectViewer". Disable the SA if the leak was via a key. If unsure of scope, disable the CMEK key with gcloud kms keys versions disable — every read fails instantly, lab-wide. You can re-enable in 15 minutes once you've contained.
15–60 min · Scope the damage
Query the audit logs
From the aggregated sink — was the bucket actually accessed by the unintended principal during the window of exposure? What objects were read? SELECT timestamp, protoPayload.resourceName FROM `…audit.cloudaudit_googleapis_com_data_access` WHERE protoPayload.authenticationInfo.principalEmail = "postdoc@gmail.com" AND timestamp > "[exposure_start]" ORDER BY timestamp. Save the query result. Take screenshots of the IAM history.
60–120 min · Notify
Page UH ITS Security and the PI
By policy, suspected PHI exposure must be reported to UH ITS Security and the IRB within a defined window (institution-specific — typically same-day for any unauthorised access). Have the audit-log evidence ready. Be precise: what was exposed, what was actually accessed, what was contained, when. The audit trail does the talking.
2–4 hours · Document & preserve
Snapshot, freeze, write the timeline
Export the relevant audit logs to a new BigQuery dataset with restricted access so the evidence is not subject to retention rotation. Don't delete anything related to the incident. Write a chronological timeline: time exposed, time discovered, time contained, time reported. This is what ends up attached to the breach review.
Day 1–3 · Root cause & close
Why was this possible, and what changes so it isn't next time
A single mistaken IAM grant should not be able to leak PHI. Common after-action items: tighten constraints/iam.allowedPolicyMemberDomains to disallow personal emails, add IAM Conditions for time-bound grants, enable Sensitive Action protection on the bucket, audit every existing binding for similar risk. Update the runbook.
The pre-condition for all of this: Data Access logs on, aggregated sink writing to BigQuery, IAM policy history retained, runbook in the lab's private repo with phone numbers. If those are not in place before the incident, the first four hours are a scramble instead of a procedure.
Hands-on exercise 3.3

CMEK-encrypted bucket + audited secret

🔐 Stand up the data layer in your sandbox
  1. In your sandbox project, enable KMS + Secret Manager APIs:
    gcloud services enable cloudkms.googleapis.com secretmanager.googleapis.com
  2. Create a keyring and a 90-day-rotating key:
    gcloud kms keyrings create demo-keys --location=us-west1
    gcloud kms keys create demo-gcs --keyring=demo-keys --location=us-west1 --purpose=encryption --rotation-period=90d --next-rotation-time=2026-08-22T00:00:00Z
  3. Grant the GCS service agent decrypt rights on the key:
    PNUM=$(gcloud projects describe $PROJECT --format='value(projectNumber)')
    gcloud kms keys add-iam-policy-binding demo-gcs --keyring=demo-keys --location=us-west1 --member="serviceAccount:service-${PNUM}@gs-project-accounts.iam.gserviceaccount.com" --role=roles/cloudkms.cryptoKeyEncrypterDecrypter
  4. Create a CMEK-encrypted bucket:
    KEY=projects/$PROJECT/locations/us-west1/keyRings/demo-keys/cryptoKeys/demo-gcs
    gcloud storage buckets create gs://$PROJECT-demo-cmek-$(date +%s) --location=us-west1 --uniform-bucket-level-access --default-encryption-key=$KEY
    Upload any file to it. Inspect the object: gcloud storage objects describe gs://<bucket>/<file> — the metadata shows your KMS key.
  5. Create a Secret Manager secret encrypted with the same key:
    gcloud secrets create demo-key --replication-policy=user-managed --locations=us-west1 --kms-key-name=$KEY
    printf 'test-value' | gcloud secrets versions add demo-key --data-file=-
    Add a second version with a different value to see versioning.
  6. Enable Data Access logs on the sandbox project for storage, secretmanager, and cloudkms — use the YAML policy from Section 5 (trim to those three services).
  7. Trigger some logged actions: gcloud storage ls gs://<your-bucket>, gcloud secrets versions access latest --secret=demo-key.
  8. Read the Data Access logs from Cloud Logging:
    gcloud logging read 'logName:"data_access" AND resource.type="audited_resource" AND protoPayload.serviceName="secretmanager.googleapis.com"' --project=$PROJECT --limit=5 --format=json
    Confirm you see your AccessSecretVersion call.
  9. Simulate the kill-switch. Disable the current key version:
    gcloud kms keys versions disable 1 --key=demo-gcs --keyring=demo-keys --location=us-west1
    Try to read the bucket object again — it fails immediately. Re-enable: gcloud kms keys versions enable 1 --key=demo-gcs --keyring=demo-keys --location=us-west1.
  10. Tear down: delete the bucket, the secret, the key (note: KMS keys cannot be hard-deleted, only "destroyed," and even destroyed they have a 30-day recovery window — by design).

Commands you used

Knowledge check

Questions to sit with

  • Why do CMEK keys live in srl-shared-services rather than in each study project? Hint
  • A workload SA has roles/storage.objectViewer on a CMEK-encrypted GCS bucket but reads are failing with KMS errors. What did you forget? Hint
  • You forgot to enable Data Access logs on healthcare.googleapis.com when HIPIMR-prod launched six months ago. An IRB audit asks "who read DICOM study 0042?" What do you tell them? Hint

Closing Module 3

You have a HIPAA-grade environment

The arc of M3: L1 — the boundary (BAA + perimeter + org policies). L2 — the network (private paths, no public IPs, internal-only services, edge protection). L3 — the data (CMEK, Secret Manager, Data Access logs, breach response). Together: an environment where PHI can land, be processed by ephemeral compute, and never escape — with an audit trail that answers any question an IRB or NIH auditor asks.

Module 4 takes you back to the compute layer. With M1's topology, M2's containers, and M3's HIPAA controls in place, M4 is where you actually build the serverless services that run inside all of it — Cloud Run and Cloud Functions, deeply.
