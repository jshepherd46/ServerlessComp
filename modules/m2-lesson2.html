The bar

What "production quality" actually means

Five properties. An image that has all five can ship to srl-makawalu-prod. An image missing any one of them is a near-miss waiting to be an incident.

🪶 Small

Sub-300 MB for pure Python; sub-2 GB even for CUDA workloads. Smaller = faster pushes, faster pulls, smaller cold starts, smaller attack surface.

🚀 Fast to start

Layered for cache reuse. Lazy imports. No heavy work in module top-level. Cold-start budget on Cloud Run is real money at scale.

🔒 Secure

Runs as a non-root user. No secrets baked in. No unused tools (bash, curl, ssh) shipped in prod. Minimal CVE surface.

♻️ Reproducible

Pin everything — base image digest, language deps with hashes, system packages. Same Dockerfile + same lockfile = bit-identical image, today and in 18 months.

🔍 Scanned

Every image scanned for CVEs on push. Findings triaged on a known cadence. Auditors don't ask "do you scan?" — they ask "what was your response time on CVE-2024-XXXX?"

The mindset shift from L1: in L1 you proved you could build an image. From here on, every Dockerfile change asks "which of the five did this just affect?" The exercise at the end of this lesson takes the hello-flask from L1 and makes it production-grade against all five.
Smaller, fewer moving parts

Multi-stage builds and bases that earn their bytes

A multi-stage build uses one image to compile things, then copies only the final artifacts into a clean, minimal image. The build-time tools (gcc, headers, cargo, npm) never make it into what you ship.

The pattern

# Stage 1 — build wheels with compilers available
FROM python:3.12-slim AS builder
WORKDIR /build
RUN apt-get update && apt-get install -y --no-install-recommends \
        build-essential gcc \
    && rm -rf /var/lib/apt/lists/*
COPY requirements.txt .
RUN pip wheel --no-cache-dir --wheel-dir /wheels -r requirements.txt

# Stage 2 — clean runtime, no compilers
FROM python:3.12-slim
WORKDIR /app
COPY --from=builder /wheels /wheels
COPY requirements.txt .
RUN pip install --no-cache-dir --no-index --find-links=/wheels -r requirements.txt \
    && rm -rf /wheels
COPY . .
RUN useradd --create-home --shell /bin/bash app && chown -R app:app /app
USER app
CMD ["python", "app.py"]

Stage 1 has gcc and build-essential — about 300 MB of extra tooling — used only to compile any C extensions in your dependencies. The final image is stage 2: slim base + wheels + your code. Nothing else.

Base image choices

Debian-derived, has apt, has a shell. Easy to debug. Slightly larger than alpine but no glibc/musl headaches with binary wheels (PyTorch, scipy, etc.). Pin the patch version + the Debian codename in prod: python:3.12.3-slim-bookworm.

Use for prod-prod images where someone exec'ing into the container is the wrong answer anyway. Pair with multi-stage: build in python:3.12-slim, copy artifacts into distroless/python3. Trade-off: docker exec -it … bash stops working — you debug via logs only.

Pick -runtime over -devel for serving — the devel tag adds compilers and headers you do not ship to prod. For Vertex AI Training where you need nvcc to JIT-compile kernels, use devel in the build stage and runtime in the final stage. For Makawalu CBM serving, runtime is enough.

Alpine uses musl libc; almost every ML wheel on PyPI is built for glibc. You end up compiling NumPy / PyTorch / scipy from source inside a multi-stage build — slow, fragile, and you save ~40 MB versus slim. Don't. Reach for distroless instead.

Reproducibility

Pin everything, with hashes

A reproducible image is one where today's build and next year's build are bit-identical. Three layers of pinning, applied in order:

1. Pin the base image to a digest, not just a tag

Tags are mutable. python:3.12-slim means whatever Python published most recently. Digests don't move.

# Fragile
FROM python:3.12-slim

# Tighter
FROM python:3.12.3-slim-bookworm

# Bulletproof
FROM python:3.12.3-slim-bookworm@sha256:e1cc7a3a8b9d1c…

Get the digest from docker inspect python:3.12.3-slim-bookworm --format='{{index .RepoDigests 0}}' after pulling. Update intentionally on a cadence — usually when you intentionally pull a newer patch version.

2. Pin Python deps with a lockfile (use uv)

# requirements.in — what you actually want
flask
gunicorn

# Generate requirements.txt with versions + hashes
uv pip compile requirements.in -o requirements.txt --generate-hashes

The generated requirements.txt pins every direct + transitive dep with a --hash=sha256:… line. Install with pip install --require-hashes in the Dockerfile so any tampering or registry compromise fails the build instead of poisoning the image.

If you're not using uv yet: pip-tools (older), poetry, or pdm all produce hashed lockfiles. Pick one per lab and stick with it.

3. Tag images with a meaningful version, not just latest

us-west1-docker.pkg.dev/srl-shared-services/research/hello-flask:0.2.0
us-west1-docker.pkg.dev/srl-shared-services/research/hello-flask:abc1234   # git short-sha
us-west1-docker.pkg.dev/srl-shared-services/research/hello-flask:2026-05-24 # date tag

In a Cloud Build pipeline, tag with the git SHA — every deploy is traceable to a commit. Avoid :latest in deploy targets; Cloud Run picks up the new image silently and you lose rollback.

The registry

Push to Artifact Registry

Artifact Registry is GCP's managed image store. One regional repository per use case, IAM-controlled, automatically scanned, lifecycle-managed. The convention from M1L2: a single srl-shared-services project holds the lab-wide registry; every study's compute pulls from it.

Create the repository once

gcloud config set project srl-shared-services
gcloud services enable artifactregistry.googleapis.com

gcloud artifacts repositories create research \
  --repository-format=docker \
  --location=us-west1 \
  --description="SRL research containers"

Authenticate Docker to the registry (per-machine, once)

gcloud auth configure-docker us-west1-docker.pkg.dev

Tag and push

IMG=us-west1-docker.pkg.dev/srl-shared-services/research/hello-flask:0.2.0

docker build -t $IMG .
docker push $IMG

Lifecycle — don't keep every image forever

Cloud Build pushes a new tag for every commit. Without a cleanup policy you accumulate thousands of images. Set a retention rule:

gcloud artifacts repositories set-cleanup-policies research \
  --location=us-west1 \
  --policy=cleanup-policy.json

A minimal cleanup-policy.json:

[
  {
    "name": "keep-tagged-releases",
    "action": {"type": "Keep"},
    "condition": {"tagState": "TAGGED", "tagPrefixes": ["v"]}
  },
  {
    "name": "delete-old-untagged",
    "action": {"type": "Delete"},
    "condition": {"tagState": "UNTAGGED", "olderThan": "30d"}
  }
]
Cross-project pulls: Cloud Run / GKE in srl-makawalu-prod needs roles/artifactregistry.reader on the srl-shared-services registry — granted to the runtime service account, not to the project. This is M1L3's one-SA-per-workload pattern in action.
Managed CI

Cloud Build — building, scanning, pushing on git push

Cloud Build is GCP's managed CI/CD. The minimum useful pipeline: on every push to main, build the image, tag with the git SHA, push to Artifact Registry. Scanning happens automatically on push.

The cloudbuild.yaml that ships with your repo

# cloudbuild.yaml
substitutions:
  _REGION: us-west1
  _REPO: research
  _IMAGE: hello-flask

steps:
  # 1. Build, tagging with both git SHA and a release tag
  - name: 'gcr.io/cloud-builders/docker'
    args:
      - build
      - --tag=${_REGION}-docker.pkg.dev/$PROJECT_ID/${_REPO}/${_IMAGE}:$SHORT_SHA
      - --tag=${_REGION}-docker.pkg.dev/$PROJECT_ID/${_REPO}/${_IMAGE}:latest
      - .

  # 2. Push both tags
  - name: 'gcr.io/cloud-builders/docker'
    args:
      - push
      - --all-tags
      - ${_REGION}-docker.pkg.dev/$PROJECT_ID/${_REPO}/${_IMAGE}

images:
  - ${_REGION}-docker.pkg.dev/$PROJECT_ID/${_REPO}/${_IMAGE}

options:
  logging: CLOUD_LOGGING_ONLY
  machineType: E2_HIGHCPU_8

Run it on demand

gcloud builds submit --config=cloudbuild.yaml .

Wire to git — the trigger

Connect your GitHub repo via Cloud Build → Triggers → Connect repository. Then create a push trigger:

1
Event
Push to branch — main
2
Source
Your GitHub repo (Cloud Build authenticates via the GitHub app)
3
Configuration
Cloud Build config file → cloudbuild.yaml
4
Service account
A dedicated cloud-build-sa with artifactregistry.writer on the registry repo. Per M1L3 — no shared identities.

Vulnerability scanning — included, must be read

Every image pushed to Artifact Registry is automatically scanned by Artifact Analysis. Findings appear in the registry UI and via:

gcloud artifacts docker images list-vulnerabilities \
  us-west1-docker.pkg.dev/srl-shared-services/research/hello-flask:0.2.0
The response policy: Critical findings — patch within 7 days or document why you can't. High — 30 days. Medium / Low — quarterly sweep. Findings that exist only in base-image packages your code never imports — note them and move on; don't burn engineering capacity rebuilding for a CVE in a library you don't call.
Hands-on exercise 2.2

Take hello-flask production

🏗 Convert, push, automate
  1. Start from the hello-flask repo from L1. If you didn't save it, recreate app.py, requirements.in (just flask + gunicorn), and .dockerignore.
  2. Lock dependencies with hashes. If you have uv: uv pip compile requirements.in -o requirements.txt --generate-hashes. Otherwise install pip-tools and use pip-compile --generate-hashes requirements.in.
  3. Rewrite the Dockerfile as multi-stage using the template from Section 2. Add --require-hashes to the pip install line. Switch CMD to use gunicorn:
    CMD ["gunicorn", "-b", "0.0.0.0:8080", "app:app"]
  4. Build and confirm size dropped.
    docker build -t hello-flask:0.2.0 .
    docker images hello-flask — compare to 0.1 from L1.
  5. Create your Artifact Registry repo (in your sandbox project for practice):
    gcloud services enable artifactregistry.googleapis.com
    gcloud artifacts repositories create practice --repository-format=docker --location=us-west1
  6. Auth Docker + push:
    gcloud auth configure-docker us-west1-docker.pkg.dev
    IMG=us-west1-docker.pkg.dev/$PROJECT/practice/hello-flask:0.2.0
    docker tag hello-flask:0.2.0 $IMG && docker push $IMG
  7. Check the scan results (wait 1–2 min after push):
    gcloud artifacts docker images list-vulnerabilities $IMG
    Note severity counts. With a slim base + Flask, there should be very few.
  8. Add a cloudbuild.yaml from Section 5 to your repo. Run it manually first:
    gcloud services enable cloudbuild.googleapis.com
    gcloud builds submit --config=cloudbuild.yaml --substitutions=_REPO=practice .
  9. Bonus — wire the trigger. Cloud Build → Triggers → Connect your hello-flask GitHub repo (or the ServerlessComp repo if you're using a subdirectory). Create a push-to-main trigger pointing at cloudbuild.yaml. Make any commit; watch the build fire automatically.
  10. Cleanup: the practice repo costs pennies but tidy up — gcloud artifacts repositories delete practice --location=us-west1 when you're done.

Questions to sit with
