Lesson 1 of 3

Federated learning
fundamentals

M9L3 was federated analytics — summary stats travel between sites. M10 is federated learning — model updates travel between sites. Both keep patient data local; FL trains a global model that benefits from every site\'s data. The PacificFed ML layer, made concrete.

⏱ ~50 min reading + exercise 🔧 Flower framework, FedAvg, PyTorch / TensorFlow 📍 Module 10 of 12
The framing

Federated analytics vs federated learning

Both keep data local. Both ship only derived quantities across sites. The difference is what kind of derived quantity:

M9L3 — federated analytics

Summary stats per site

Each site runs the same SQL cohort definition; ships cohort sizes, demographics breakdowns, prevalence rates. The aggregator combines them into a meta-analysis. Static cohort question.

"How many HCC patients on immunotherapy across the seven sites?"

M10 — federated learning

Model updates per site

Each site trains a model locally on its own data; ships gradient updates (not data, not predictions). The aggregator combines updates into a global model; ships the global model back. Repeat. Iterative training process.

"Train a 5-year HCC outcome predictor that learned from every site\'s data."

The core FL algorithm — FedAvg

Round t = 0: aggregator initialises a global model w_0   For each round t in [1..R]: aggregator selects K sites and broadcasts w_{t-1} to them │ ▼ each site site_k trains on its local data for E epochs, producing w_k │ ▼ each site uploads its update aggregator averages: w_t = Σ (n_k / N) * w_k # weighted by local dataset size │ ▼ next round, w_t becomes the broadcast model   After R rounds: global model w_R is the converged FL model

That\'s it. The whole algorithm — FedAvg, the foundation everything else builds on — fits in eight lines. The complexity is in (a) the systems that move weights between sites securely, (b) the privacy guarantees (M10L3), and (c) handling the realities of heterogeneous data and unreliable sites.

What FL is not. It is not encryption — data is in the clear at each site, just doesn\'t leave. It is not differential privacy by itself — model updates can leak information; you add DP-SGD on top (M10L3). It is not "send data to the cloud anonymously" — patient data never goes to the cloud at all. The benefit is institutional: each site\'s data contributes to one model without the data leaving the institution.
Why bother

Why federated learning matters for health data specifically

1. Legal: PHI cannot freely move between institutions

HIPAA, IRBs, and equivalent international rules make cross-institutional patient-data sharing slow and contentious. FL bypasses the problem — no data sharing means no data-sharing agreement.

2. Institutional: sites want their own data to stay theirs

Even when legal sharing is possible, institutions are reluctant to release raw EHR data. FL lets them contribute meaningfully without giving up data sovereignty — especially important for USAPI / NHPI institutions where data control is part of community trust.

3. Statistical: more sites = more diverse training

A model trained on one site\'s data over-fits that site\'s demographics, scanner models, documentation conventions. A federated model has seen the full diversity of all sites — it generalises better than any single-site model on out-of-distribution patients.

4. Operational: each site retains its own audit and compliance posture

Each site\'s training happens inside that site\'s perimeter, with that site\'s IAM, that site\'s audit logs. The institution\'s compliance officer can audit one site\'s FL training run the same way they\'d audit any other ML workload — no special cross-institutional review needed.

What FL costs you

  • → Slower training. Round-trip latency between sites + bandwidth for model updates. A 100-round FL training might take 24-48 hours where centralised training would take 4.
  • → Statistical heterogeneity. Sites have different patient distributions; vanilla FedAvg converges slower or to a worse model when distributions diverge. Mitigations: FedProx, FedAdam, personalisation layers.
  • → Systems complexity. M10L2\'s GKE multi-cluster + VPC peering setup is non-trivial. The first FL deployment takes weeks; subsequent ones reuse the infrastructure.
  • → Imperfect privacy. Vanilla FL does NOT guarantee privacy — model updates can be inverted to leak training data. M10L3\'s DP-SGD + secure aggregation closes the gap.
The framework

Flower — the FL framework SRL uses

Flower is an open-source FL framework that\'s framework-agnostic (PyTorch, TensorFlow, JAX, scikit-learn) and well-suited to the kind of cross-institutional FL PacificFed needs. The mental model: you write a client (your training code at one site) + a strategy (the aggregation logic at the aggregator); Flower handles the gRPC plumbing, round orchestration, and serialisation.

A minimal Flower client (PyTorch)

import flwr as fl
import torch
from torch import nn, optim

class HCCOutcomeClient(fl.client.NumPyClient):
    def __init__(self, model, train_loader, val_loader):
        self.model = model
        self.train_loader = train_loader
        self.val_loader = val_loader

    def get_parameters(self, config):
        return [p.detach().cpu().numpy() for p in self.model.parameters()]

    def set_parameters(self, parameters):
        for p, new_p in zip(self.model.parameters(), parameters):
            p.data = torch.tensor(new_p)

    def fit(self, parameters, config):
        self.set_parameters(parameters)
        opt = optim.SGD(self.model.parameters(), lr=config["lr"])
        for _ in range(config["local_epochs"]):
            for x, y in self.train_loader:
                opt.zero_grad()
                loss = nn.functional.binary_cross_entropy_with_logits(self.model(x), y)
                loss.backward()
                opt.step()
        return self.get_parameters({}), len(self.train_loader.dataset), {}

A minimal Flower aggregator

import flwr as fl

strategy = fl.server.strategy.FedAvg(
    fraction_fit=1.0,                # all sites participate each round
    min_fit_clients=5,               # at least 5 sites must check in
    min_available_clients=7,         # PacificFed: 7 sites
    on_fit_config_fn=lambda rnd: {"lr": 1e-3, "local_epochs": 2},
)

fl.server.start_server(
    server_address="0.0.0.0:8080",
    config=fl.server.ServerConfig(num_rounds=50),
    strategy=strategy,
)

Each side runs as a container. The aggregator runs in srl-pacificfed-infra on GKE. Clients run at each site (UH, Queen\'s, Guam, etc.) inside their own GKE clusters. M10L2 covers the deployment.

Beyond FedAvg — when you need more

  • → FedProx — adds a proximal term to local optimisation; helps when sites have very different data distributions (statistical heterogeneity).
  • → FedAdam / FedAdagrad — server-side adaptive optimisers; faster convergence on harder problems.
  • → FedAvgM (with momentum) — adds server-side momentum; modest improvement at no protocol cost.
  • → Personalisation (FedPer, Per-FedAvg) — train shared base layers + site-specific head layers; useful when sites need a globally-shared representation but locally-tuned predictions.

Start with FedAvg. Switch to FedProx if statistical heterogeneity is a problem (vanilla FedAvg fails to converge or converges to a poor model). The rest are refinements for specific failure modes.

The worked example

PacificFed — seven sites, one global model

PacificFed\'s ML layer trains shared models for cancer outcome prediction across the CCPI/USAPI network without any patient data leaving its institution. The architectural shape:

The roles

  • → Coordinator (UH, in srl-pacificfed-infra): hosts the Flower aggregator on GKE; owns the cohort definition + training script + study protocol; ships the initial model and orchestrates rounds.
  • → Sites (UH, Queen\'s, Guam, Am. Samoa, CNMI, Palau, Marshall Is.): each hosts a Flower client on its own GKE cluster, training against its local OMOP-shaped data. Each site has its own IRB approval, its own data steward, its own BAA situation.

The shared assets

  • → Cohort definition — Capr (M9L2) committed to the shared git repo.
  • → Training container — built once in srl-shared-services Artifact Registry; pulled by each site\'s GKE cluster.
  • → FL protocol contract — what each site\'s client sends each round (gradient update + dataset size), what it receives (broadcast model + round config). Specified once; durable across runs.
  • → Training script — runs at each site; reads from the site\'s local OMOP dataset; trains the local model; communicates with the central aggregator via Flower\'s gRPC.

What does NOT cross site boundaries

  • → Patient identifiers, records, demographics — anything PHI.
  • → Site-specific aggregated counts that would be re-identifiable (small cells).
  • → Raw gradients without DP noise (M10L3) for models trained on small cohorts.
What does cross. Model weights (gradient updates) and dataset-size scalars used in FedAvg weighting. With DP-SGD (M10L3), even the gradients are noised to provide formal privacy guarantees. With secure aggregation, the aggregator never sees individual sites\' updates in the clear — only the sum.
Hands-on exercise 10.1

Run a tiny FL simulation with Flower

🌐 The whole loop, locally
  1. Install Flower + dependencies in a Vertex Workbench (or any Python env):
    pip install flwr torch torchvision scikit-learn
  2. Write a tiny training script — single file with client + simulator. Save as fl_demo.py: 
    import flwr as fl
import torch, torch.nn as nn
from torch.utils.data import DataLoader, TensorDataset
from sklearn.datasets import make_classification

# Generate a synthetic binary classification dataset, split into 3 "sites"
X, y = make_classification(n_samples=3000, n_features=20, n_informative=10, random_state=0)
X = torch.tensor(X, dtype=torch.float32); y = torch.tensor(y, dtype=torch.float32)

site_datasets = [
    TensorDataset(X[0:1000], y[0:1000]),
    TensorDataset(X[1000:2000], y[1000:2000]),
    TensorDataset(X[2000:3000], y[2000:3000]),
]

def make_model():
    return nn.Sequential(nn.Linear(20, 32), nn.ReLU(), nn.Linear(32, 1))

class SiteClient(fl.client.NumPyClient):
    def __init__(self, dataset):
        self.model = make_model()
        self.loader = DataLoader(dataset, batch_size=32, shuffle=True)

    def get_parameters(self, config):
        return [p.detach().numpy() for p in self.model.parameters()]

    def set_parameters(self, params):
        for p, new in zip(self.model.parameters(), params):
            p.data = torch.tensor(new)

    def fit(self, params, config):
        self.set_parameters(params)
        opt = torch.optim.SGD(self.model.parameters(), lr=0.01)
        for _ in range(2):  # 2 local epochs
            for x, y in self.loader:
                opt.zero_grad()
                loss = nn.functional.binary_cross_entropy_with_logits(self.model(x).squeeze(), y)
                loss.backward(); opt.step()
        return self.get_parameters({}), len(self.loader.dataset), {}

    def evaluate(self, params, config):
        self.set_parameters(params)
        with torch.no_grad():
            preds = torch.cat([self.model(x).squeeze() for x, _ in self.loader])
            ys = torch.cat([y for _, y in self.loader])
            loss = nn.functional.binary_cross_entropy_with_logits(preds, ys)
            acc = ((torch.sigmoid(preds) > 0.5).float() == ys).float().mean()
        return float(loss), len(self.loader.dataset), {"acc": float(acc)}

def client_fn(cid):
    return SiteClient(site_datasets[int(cid)]).to_client()

# Run the FL simulation: 3 sites, 5 rounds, all participate each round
fl.simulation.start_simulation(
    client_fn=client_fn,
    num_clients=3,
    config=fl.server.ServerConfig(num_rounds=5),
    strategy=fl.server.strategy.FedAvg(
        fraction_fit=1.0, min_fit_clients=3, min_available_clients=3,
        evaluate_metrics_aggregation_fn=lambda ms: {"acc": sum(n*m["acc"] for n, m in ms) / sum(n for n, _ in ms)},
    ),
)
  3. Run it:
    python fl_demo.py
    You\'ll see 5 rounds; each round logs per-site fit + evaluate. Final accuracy should be a few percent better than any single site could achieve on 1000 examples alone.
  4. Perturb the site distributions. Make site 0\'s labels much rarer (skew the class balance). Re-run. Watch how the global model has to balance the diverging distributions; this is the statistical-heterogeneity problem in miniature.
  5. Switch the strategy from FedAvg to FedProx (one line change). Re-run with the same perturbed data. Compare convergence behaviour.
  6. Read the per-round logs. Note what gets reported: per-site dataset size, per-site fit loss, aggregate eval metric. That\'s exactly the audit surface you\'d see in a real PacificFed run.
  7. Tear down: nothing to tear down. It was all local.
