Training pipelines

L1 told you where to train. This lesson is how — Vertex AI Custom Training for a single recipe, Vertex AI Pipelines for the multi-step CBM workflow, plus the experiment tracking and hyperparameter tuning that make training reproducible.

⏱ ~55 min reading + exercise 🔧 Vertex AI Training, Pipelines, Experiments, Vizier, TensorBoard 📍 Module 6 of 12
Single-recipe training

Vertex AI Custom Training — submit a container, get a model

Vertex AI Custom Training is the cleanest way to run a training job that you don't need to orchestrate multiple steps for. You package training code as a container (M2 patterns hold), hand Vertex the image + a config, and it allocates GPUs, runs the job, captures logs and artifacts, then tears down.

Anatomy of a Custom Training job

# training-job.yaml
displayName: cbm-train-v3-7
jobSpec:
  workerPoolSpecs:
    - machineSpec:
        machineType: n1-standard-8
        acceleratorType: NVIDIA_TESLA_T4
        acceleratorCount: 1
      replicaCount: 1
      containerSpec:
        imageUri: us-west1-docker.pkg.dev/srl-shared-services/research/cbm-train:0.4.1
        command: ["python", "train.py"]
        args:
          - --train-data=gs://srl-makawalu-training/cbm-v3/train/
          - --eval-data=gs://srl-makawalu-training/cbm-v3/eval/
          - --output-dir=gs://srl-makawalu-training/cbm-v3/runs/v3-7/
          - --epochs=40
          - --batch-size=32
  serviceAccount: train-job-sa@srl-makawalu-prod.iam.gserviceaccount.com
  tensorboardId: projects/$PROJECT/locations/us-west1/tensorboards/srl-experiments

Submit and watch

gcloud ai custom-jobs create --region=us-west1 --config=training-job.yaml

# Track from CLI
gcloud ai custom-jobs list --region=us-west1 --limit=5

# Stream logs
gcloud ai custom-jobs stream-logs $JOB_ID --region=us-west1

What the training container should do

A Vertex Custom Training container is just a container that runs to completion. Inside, your code should:

  • → Read from GCS, not the DICOM store. Random-access training; per-batch network calls to DICOMweb are too slow.
  • → Authenticate via the attached SA. No keys. The standard google.cloud clients pick up credentials.
  • → Checkpoint to GCS every epoch. Vertex preempts spot training; checkpoints let you resume.
  • → Log to stdout (TensorBoard for metrics). Vertex auto-collects stdout into Cloud Logging.
  • → Save the final model to --output-dir. Vertex registers what's there as the job's output.

Spot training for cheap experiments

jobSpec:
  workerPoolSpecs:
    - machineSpec: {machineType: n1-standard-8, acceleratorType: NVIDIA_TESLA_T4, acceleratorCount: 1}
      replicaCount: 1
      scheduling:
        strategy: SPOT       # up to ~70% cheaper, can be preempted
        restartJobOnWorkerRestart: true      # auto-resume from latest checkpoint
      containerSpec: { ... }
The discipline that makes spot work. Checkpoint every epoch; resume cleanly from latest. With both, preemption costs minutes, not hours. Without either, preemption costs the whole run. Build the resume path into your training script on day one; spot then becomes a no-brainer for any non-critical run.
Scaling out

Distributed training when one GPU isn't enough

Vertex Custom Training supports multi-worker distributed training without you running a Kubernetes cluster. You declare the worker pool shape; Vertex provisions the topology and sets the environment variables your training framework expects.

The two patterns

Data parallelism
N workers, same model, each processes a different batch slice; gradients sync between workers. The default for "we just need it faster" — PyTorch DDP, TensorFlow MultiWorkerMirroredStrategy. Linear-ish speedup on well-tuned code.
Model parallelism
Model split across workers (different layers on different GPUs). Necessary when the model itself does not fit in one GPU's memory. Common for LLMs (M7); rarer for CNN imaging.

Multi-worker data parallelism on Vertex

jobSpec:
  workerPoolSpecs:
    - machineSpec:
        machineType: a2-highgpu-1g
        acceleratorType: NVIDIA_TESLA_A100
        acceleratorCount: 1
      replicaCount: 1        # chief worker
      containerSpec: {imageUri: $IMG, args: ["--world-size=4"]}
    - machineSpec:
        machineType: a2-highgpu-1g
        acceleratorType: NVIDIA_TESLA_A100
        acceleratorCount: 1
      replicaCount: 3        # 3 additional workers
      containerSpec: {imageUri: $IMG, args: ["--world-size=4"]}

Vertex sets CLUSTER_SPEC as a JSON env var with each worker's role and address. Your training code reads it and calls torch.distributed.init_process_group (or the TF equivalent) with the right backend.

Reduction Server — eliminating the gradient-sync bottleneck

For 8+ workers, gradient sync over network becomes the bottleneck. Vertex offers a managed Reduction Server — dedicated workers whose only job is to aggregate gradients. Add a third pool to the spec:

    - machineSpec: {machineType: n1-highcpu-16}
      replicaCount: 4
      containerSpec:
        imageUri: us-docker.pkg.dev/vertex-ai-restricted/training/reductionserver:latest
The pragmatic order. Most studies never need distributed training. Single-GPU on a T4 covers Makawalu CBM training; A100 single-GPU covers HIPIMR density CNN. Reach for multi-worker only when you have measured evidence that single-GPU is the limit and a multi-day single-GPU run is unacceptable. The complexity step is real; pay it deliberately.
Multi-step orchestration

Vertex AI Pipelines — Kubeflow on managed infrastructure

Vertex AI Pipelines is the right home for any ML workflow with more than one step. Each step is a containerised component; artifacts flow between components via GCS. The pipeline definition is Python (Kubeflow Pipelines SDK); Vertex executes it on managed infrastructure.

The Makawalu CBM pipeline, end to end

[1] preprocess # Component: pulls de-id'd DICOM via WADO-RS │ # → resizes, normalises, splits 80/10/10 ▼ artifact: processed_dataset (GCS path) [2] train_concept_heads # Vertex Custom Training, T4 GPU │ # → trains the concept-prediction heads ▼ artifact: concept_model [3] train_final_classifier # Vertex Custom Training, T4 GPU │ # → uses concept predictions → malignancy score ▼ artifact: final_model [4] evaluate # Computes metrics on the held-out test split │ # → AUROC, concept fidelity, fairness slices ▼ artifact: metrics (JSON) [5] conditional_register # If metrics.auroc > threshold: ├──▶ register_model # register in Vertex Model Registry └──▶ notify_pubsub # publish to dicom-model-updated # (kicks off M4L2 canary deploy)

A pipeline definition (Python KFP SDK)

from kfp import dsl
from kfp.dsl import Input, Output, Artifact, component
from google.cloud import aiplatform

@component(base_image="us-west1-docker.pkg.dev/.../cbm-preprocess:0.1")
def preprocess(
    deid_store_path: str,
    output_dataset: Output[Artifact],
):
    # Body of the component: download, preprocess, write to output_dataset.uri
    ...

@component(base_image="us-west1-docker.pkg.dev/.../cbm-train:0.4.1")
def train_concept_heads(
    dataset: Input[Artifact],
    epochs: int,
    model_out: Output[Artifact],
):
    ...

@dsl.pipeline(name="makawalu-cbm-v3", pipeline_root="gs://srl-makawalu-training/pipelines/")
def makawalu_cbm_pipeline(
    deid_store: str = "projects/.../dicomStores/deid-store",
    epochs: int = 40,
    auroc_threshold: float = 0.85,
):
    pp = preprocess(deid_store_path=deid_store)
    heads = train_concept_heads(dataset=pp.outputs["output_dataset"], epochs=epochs)
    final = train_final_classifier(dataset=pp.outputs["output_dataset"], concept_model=heads.outputs["model_out"])
    evald = evaluate(model=final.outputs["model_out"], dataset=pp.outputs["output_dataset"])

    with dsl.If(evald.outputs["auroc"] > auroc_threshold):
        register_model(model=final.outputs["model_out"])
        notify_pubsub(message="cbm-v3 retrained")

# Compile + submit
from kfp.compiler import Compiler
Compiler().compile(makawalu_cbm_pipeline, "cbm-pipeline.json")
aiplatform.PipelineJob(
    display_name="cbm-v3-7",
    template_path="cbm-pipeline.json",
    pipeline_root="gs://srl-makawalu-training/pipelines/",
).submit()
Why this beats a shell script. Every component is a versioned container. Artifacts between steps are tracked, lineage-traceable, queryable. Conditional logic (the AUROC gate) is native, not bolted on. Reruns from any step are one click; reruns with different hyperparameters are one parameter change. And the pipeline can be triggered by Eventarc (M4L2) — new de-id data → pipeline reruns → canary deploy.
Tracking and tuning

Experiments, TensorBoard, and Vizier

Three Vertex features that make training reproducible and tunable. Use all three from the first study; they pay back across years.

Vertex AI Experiments — runs you can compare

Every training run logs to an experiment with parameters, metrics, and a pointer to the model artifact. In the console: select N runs, compare metrics side by side, see which hyperparameter values won. The discipline: wrap every train.py with an experiment context manager that calls aiplatform.start_run() + aiplatform.log_metrics().

from google.cloud import aiplatform

aiplatform.init(project="srl-makawalu-prod", experiment="cbm-v3")
aiplatform.start_run("cbm-v3-7")
aiplatform.log_params({"lr": 1e-4, "batch_size": 32, "backbone": "resnet50"})

# ... training loop ...

aiplatform.log_metrics({"auroc": 0.88, "concept_fidelity": 0.74})
aiplatform.end_run()

TensorBoard on Vertex — managed metric visualisation

A managed TensorBoard instance with persistent storage, accessible per-experiment. Point your SummaryWriter at the Vertex TensorBoard URI; metrics appear in the dashboard in near-real-time during training. No more "let me ssh to the training node to see the loss curve."

# In training-job.yaml
jobSpec:
  tensorboardId: projects/$PROJECT/locations/us-west1/tensorboards/srl-experiments
  serviceAccount: train-job-sa@$PROJECT.iam.gserviceaccount.com

# In train.py - Vertex sets AIP_TENSORBOARD_LOG_DIR automatically
import os
from torch.utils.tensorboard import SummaryWriter
writer = SummaryWriter(log_dir=os.environ["AIP_TENSORBOARD_LOG_DIR"])
writer.add_scalar("loss/train", loss.item(), step)

Vizier — Bayesian hyperparameter tuning

Instead of grid search, declare a parameter study with ranges and a target metric; Vizier runs trials adaptively, learning from earlier results to pick the next configuration. Built into Vertex Custom Training as HyperparameterTuningJob.

# Submit a tuning job with 4 parallel trials, 20 max trials
gcloud ai hp-tuning-jobs create --region=us-west1 \
  --config=hp-tuning-config.yaml \
  --display-name=cbm-tune-v3 \
  --max-trial-count=20 \
  --parallel-trial-count=4

Config file declares the search space and the metric to optimise. Trials run in parallel; the next batch of trials picks parameter values informed by completed trials. Way more sample-efficient than grid search for >3 dimensions.

The boring but vital parts

Quotas, costs, and run hygiene

GPU quotas — request them ahead of time

Default project GPU quotas are low — often zero in some regions. Before running a training job that needs A100 or L4 capacity:

gcloud compute regions describe us-west1 \
  --format="value(quotas[].metric,quotas[].limit)" | grep -i gpu

If the relevant quota is 0, request an increase via the console: IAM & Admin → Quotas & System Limits. Allow 1–3 business days. For Vertex Training, the quota is on Custom Model Training GPU (not on Compute Engine GPUs directly).

Cost reality for typical SRL training

  • → Makawalu CBM, T4 GPU, 4 hours: ~$2–3 on demand; ~$0.80 spot.
  • → HIPIMR density CNN, A100, 6 hours: ~$20–25 on demand; ~$7 spot.
  • → 20-trial hyperparameter sweep (4 parallel, T4): ~$15–20.
  • → Same workloads run accidentally and left running over a weekend: Don't ask.

Run hygiene

  • → Always set --max-running-time on Vertex jobs as a safety net.
  • → Tag every run via Experiments (with a date, study, and short description) so the artifact in GCS is traceable to the parameters that produced it.
  • → Delete old TensorBoard logs and unsuccessful run artifacts quarterly. GCS retention adds up faster than people expect.
  • → Sanity-check every new pipeline against a known-good baseline before letting it write to the canonical model registry.
Hands-on exercise 6.2

Run a Vertex Custom Training job, log to Experiments

🧪 Submit, monitor, compare
  1. Enable the Vertex AI API and TensorBoard:
    gcloud services enable aiplatform.googleapis.com
  2. Create a TensorBoard instance in your sandbox:
    gcloud ai tensorboards create --display-name="demo-tb" --region=us-west1
    Note the resource name.
  3. Write a minimal training container. A tiny PyTorch (or scikit-learn) script that trains a model on synthetic data and logs metrics via the aiplatform SDK: 
    # train.py
import os, random, time
from google.cloud import aiplatform

PROJECT = os.environ["PROJECT"]
EXPERIMENT = "demo-exp"
RUN = os.environ.get("AIP_HYPERPARAMETER_TUNING_RUN_ID") or f"run-{int(time.time())}"
LR = float(os.environ.get("LR", "0.01"))

aiplatform.init(project=PROJECT, location="us-west1", experiment=EXPERIMENT)
aiplatform.start_run(RUN)
aiplatform.log_params({"lr": LR, "epochs": 5})

# Pretend training - simulate a loss curve
final_loss = 1.0
for epoch in range(5):
    final_loss *= 0.6 + LR + random.uniform(-0.05, 0.05)
    aiplatform.log_metrics({"loss": final_loss}, step=epoch)
    time.sleep(2)

aiplatform.log_metrics({"final_loss": final_loss})
aiplatform.end_run()
print(f"Done, final_loss={final_loss}")
    Dockerfile + requirements.txt the usual way. Push to your Artifact Registry.
  4. Submit a Custom Training job (no GPU needed for the demo):
    gcloud ai custom-jobs create --region=us-west1 \
  --display-name="demo-train" \
  --worker-pool-spec="machine-type=e2-standard-2,replica-count=1,container-image-uri=$IMG" \
  --args="--lr=0.01"
  5. Stream the logs:
    gcloud ai custom-jobs stream-logs $JOB_ID --region=us-west1
  6. When done, view the experiment in the Cloud Console: Vertex AI → Experiments → demo-exp. You should see the run with logged params and metrics.
  7. Submit two more runs with different LRs (--args="--lr=0.005" and --args="--lr=0.02"). Compare side-by-side in the Experiments UI.
  8. Bonus — wire up a Pipeline. Compile a 3-step Kubeflow Pipeline (preprocess → train → evaluate) and submit via PipelineJob(...).submit(). Watch the DAG in the Pipelines UI. Even with synthetic data, you'll see the artifact flow.
  9. Tear down: delete the TensorBoard instance, training jobs persist as completed records (cheap). Vertex storage accumulates - prune old artifacts periodically.
