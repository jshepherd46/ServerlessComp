Lesson 2 of 2

The GCP landscape

There are over 100 services on GCP. You need maybe 25 of them. This lesson sorts the ones that matter into seven families, defines the vocabulary, and gives you the serverless mental model. Every defined term is in the glossary for life.

⏱ ~45 min 🔧 Browser tour 📍 Prerequisite
The map

Seven families is all you need

Google publishes a poster with 100+ products. Ignore most of them. For health research and clinical AI, almost everything you'll touch sorts into seven families. Learn the families first; the services snap into place underneath.

Compute
Where your code runs. VMs, containers, functions, ML training.
🗄️

Mental model: when you read a GCP service name you've never heard, your first move is to ask which family. "Datastream" → eventing/glue. "AlloyDB" → storage. "Filestore" → storage. The family tells you what kind of decision it serves and whether you actually need to learn it now.
Family 1 / 6

Compute: where your code runs

Six services, ordered from most-control-most-burden to least-control-least-burden. The right answer for most research workloads sits in the middle.

Compute Engine (GCE)INFRA
Raw VMs. You manage the OS, patches, scaling. Maximum control. Used only when something else won't fit.
Rare here. Legacy ATLAS installs, special GPU configs.
Cloud RunINFRA
Fully managed serverless containers. Give it an image, get an HTTPS endpoint. Scales 0 → 1000 automatically. The default for APIs.
Makawalu XAI serving, ShepBERT inference API, any web service.
Cloud Run JobsINFRA
Run-to-completion containers on the same platform. No HTTP — just runs, finishes, stops. Pair with Cloud Scheduler for cron.
Nightly HIPIMR ingest, OMOP ETL passes, DICOM batch conversions.
Cloud FunctionsINFRA
A single function, wrapped in a container by GCP. Triggered by HTTP, Pub/Sub, GCS events. v2 is built on Cloud Run.
"When a new DICOM lands in GCS, kick off the de-id pipeline."
GKE (Autopilot)INFRA
Managed Kubernetes. Use when you need multi-container pods, GPU pools, in-cluster networking, or workloads Cloud Run can't host.
PacificFed aggregator, multi-tenant lab clusters, long-running training.
Vertex AI TrainingML
Managed ML training. You hand it a container + dataset; it allocates GPUs/TPUs, runs the job, captures logs and artifacts, then tears down.
Makawalu CNN, HIPIMR density models, ShepBERT LoRA fine-tunes.
The serverless instinct: when picking compute, start at Cloud Run and only step "up" (to GKE) or "sideways" (to Vertex) when Cloud Run genuinely can't do the job. Start at GCE only when the previous five have been ruled out.
Family 2 / 6

Storage & data: where your data lives

Pick by what you're storing. The shape of the data dictates the service — not the other way around.

Cloud Storage (GCS)INFRA
Object storage. Buckets hold objects (files). Strongly consistent, lifecycle-managed, cheap at scale.
DICOM, model weights, training images, exports, anything file-shaped.
BigQueryOMOP
Serverless data warehouse. Petabyte-scale SQL with no infrastructure. Datasets contain tables.
HALO OMOP CDM, federated analytics, billing exports.
Cloud SQLINFRA
Managed Postgres/MySQL/SQL Server. Real relational DB with transactions.
ATLAS WebAPI backend, app metadata, registry state.
FirestoreINFRA
Managed NoSQL document store. Cheap, fast, integrates cleanly with Cloud Run and Cloud Functions.
Audit trails, ingest manifests, session state, light metadata.
Cloud Healthcare APIHIPAA
HIPAA-eligible managed store for DICOM, FHIR, and HL7v2. Built-in de-id, BigQuery streaming, DICOMweb.
HIPIMR DICOM, HALO FHIR ingest, Queen's bulk exports.
Artifact RegistryINFRA
Managed registry for container images, Python wheels, Maven jars. Where Cloud Run and GKE pull from.
Every container you build in M2 onward gets pushed here.
The course paradigm — restated: data has gravity. PHI sits in GCS or BigQuery inside the perimeter; compute (Cloud Run, GKE pods, Vertex jobs) comes to the data, does the work, and exits. You never copy patient data to your laptop, and never to a project that isn't BAA-covered.
Family 3 + 4 / 6

Networking, identity, and the bones of HIPAA

For research workloads, networking and security blur into one conversation: how do I keep PHI inside the perimeter? Three layers — network, identity, secrets.

Network

VPCINFRA
Your project's private network. Global by default. Contains subnets per region.
Every project gets one (default or custom).
VPC Service ControlsHIPAA
The PHI perimeter. Wraps projects so their GCS / BigQuery / Healthcare API resources cannot be reached from outside the perimeter — even with valid credentials.
The single biggest HIPAA control on GCP. Every PHI project sits inside one.
Private Service ConnectHIPAA
Reach Google APIs over private IPs. Keep traffic off the public internet entirely.
Pair with VPC SC for closed-loop research projects.
Cloud NATINFRA
Managed outbound NAT for VMs and pods without public IPs.
Required when private pods need to reach PyPI, model hubs, etc.

Identity & access

IAMINFRA
Principals are granted roles that bundle permissions on resources. Every action runs as some IAM identity.
Default to predefined roles, never the basic Owner/Editor/Viewer trio.
Service accountsINFRA
Non-human identities for code. A Cloud Run service or GKE pod gets a service account with only the roles it needs.
Never put personal credentials in code. One service account per workload.
Workload IdentityINFRA
The bridge that lets a GKE pod assume a GCP service account without key files.
The right way to give pods access to GCP APIs. Always.

Secrets & encryption

Secret ManagerHIPAA
Versioned, IAM-controlled, audit-logged store for API keys, DB passwords, signing keys. Cloud Run and GKE mount secrets directly.
Anywhere you would otherwise hardcode a credential.
CMEKHIPAA
Customer-Managed Encryption Keys — keys you own in Cloud KMS, used to encrypt GCS, BigQuery, Persistent Disks. On top of Google's default encryption.
Often required by institutional HIPAA policy.
DLPHIPAA
Sensitive Data Protection — inspects text, images, structured data for identifiers; can redact, mask, tokenize. The backbone of de-identification.
Every PHI export path runs through DLP somewhere.
Audit LogsHIPAA
Immutable record of who did what, when. Admin Activity is always on; Data Access (every read of a PHI object) must be enabled.
Required for HIPAA evidence. Enable Data Access logs on every PHI project.
Family 5 / 6

Glue: how services trigger each other

Real research pipelines are choreography, not monoliths. These services are the wires.

Pub/SubINFRA
Managed message bus. Publishers write to a topic; subscribers pull or are pushed. The plumbing under most event-driven patterns.
Decoupling: ingest publishes "new DICOM" → de-id pipeline subscribes.
EventarcINFRA
Managed event router. Subscribes to ~100 GCP event sources (GCS object created, BigQuery job done) and delivers them to Cloud Run / Functions / Workflows.
"On every new file in this bucket, run this container."
Cloud SchedulerINFRA
Managed cron. Fires HTTP requests or Pub/Sub messages on a schedule.
Nightly Cloud Run Jobs, weekly OMOP refresh, monthly cost reports.
Cloud WorkflowsINFRA
Serverless orchestration — chain GCP APIs and HTTP services with retries, parallelism, conditionals. Lightweight alternative to Airflow.
Multi-step ingest pipelines that don't justify Vertex AI Pipelines.
Cloud BuildINFRA
Managed CI/CD. Builds images, runs tests, deploys. Triggered by git pushes or PRs.
Every container you ship from M2 onward.
DataflowOMOP
Managed Apache Beam — batch & streaming ETL. Used for large-scale PHI scans and de-id without standing up infrastructure.
HALO ingest, DLP-at-scale, federated aggregation.
Pattern you'll see repeatedly: Cloud Scheduler → triggers Cloud Run Job → reads GCS, writes BigQuery → emits Pub/Sub → Cloud Function on the other end fans out to Vertex AI for retraining. Every arrow is one of these services.
Family 6 / 6 — and the core vocabulary

The serverless vocabulary you'll hear constantly

"Serverless" doesn't mean there are no servers — it means you don't manage them. Five terms encode the entire mental shift. Internalize these and the rest of the course speaks plain English.

📦
Container / image
An image is your code + dependencies + start command, frozen into a portable bundle. A container is a running instance of an image. M2 is entirely about building good ones.
⚡
Scale to zero
When idle, the platform runs zero copies of your service and charges you nothing. The economic core of serverless. A Cloud Run service at 3 AM with no traffic costs $0.
🥶
Cold start
The latency cost of scale-from-zero — pulling the image, starting the container, loading your code. Mitigations: smaller images, lazy imports, min-instances > 0 for hot paths.
💨
Ephemeral compute
Compute that exists only for the duration of the work. State lives in storage (GCS, BigQuery, Firestore); the container is disposable. This is the paradigm the course is built on.
🎚
Concurrency
Max in-flight requests a single Cloud Run container handles. Default 80. Lower for heavy ML inference (1–4); higher for cheap I/O APIs (200+).
🔧
API enabling
Every GCP service is gated by a per-project API toggle. Before you use Cloud Run in a project you must gcloud services enable run.googleapis.com. Catches a lot of "but I have permission" confusion.
The single sentence to remember: sensitive data stays in a secure cloud boundary; curated containers and compute spin up beside it, do the work, and terminate. You never pay for idle capacity, and your data never moves. Every architectural choice in this course points back at that line.
Hands-on exercise 0.2

A guided tour of three services

🧭 Console tour — 20 minutes
  1. In your srl-sandbox project from Lesson 1, open Cloud Storage. Create a bucket: name srl-sandbox-tour-[yourname], location us-west1, standard class. Upload any small file. Note the gs:// URI.
  2. Open BigQuery. Run this query against a public dataset — no setup required:
    SELECT name, gender, SUM(number) AS total FROM `bigquery-public-data.usa_names.usa_1910_2013` GROUP BY name, gender ORDER BY total DESC LIMIT 10
    Notice the bytes-scanned indicator on the right. That's what you pay for.
  3. Open Cloud Run. Your hello service from Lesson 1 should be listed. Click it. Look at: Metrics (requests, latency), Logs (filtered to this service automatically), Revisions (every deploy is a revision), YAML (the full service spec — this is what you'll Terraform later).
  4. Open IAM & Admin → IAM. Find the entry that looks like [PROJECT_NUMBER]-compute@developer.gserviceaccount.com. That is the default service account your hello container runs as. Click the pencil — see the bundle of roles it has. In M3 we replace this with a tight per-workload identity.
  5. Quick classification drill. For each, name the family (compute / storage / network / security / eventing / ML / dev tooling) and a one-line use case in your own work:
    • → Pub/Sub
    • → Cloud SQL
    • → Cloud Healthcare API
    • → Vertex AI Pipelines
    • → Artifact Registry
    • → VPC Service Controls
