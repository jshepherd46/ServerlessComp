Core concept

The resource hierarchy

GCP organizes everything in a four-level tree — Organization → Folder → Project → Resource. Understanding this tree is not optional — it determines your compliance posture, your billing structure, and the blast radius of any mistake. Click each level to see how it maps to your world.

Design principles

Three rules that shape every decision

🔒
Separation is free — use it
A new GCP project costs nothing until you use resources. Separate HIPIMR from Makawalu from HALO. The blast radius of a misconfiguration stays contained. Per-grant billing just works.
🔑
Least privilege from day one
Service accounts are non-human identities for your code. A Cloud Run service gets a service account with only the roles it actually needs. Never put personal credentials in code — not even temporarily.
🗄️
Data has gravity — design around it
PHI sits in one place inside the VPC perimeter. Compute comes to the data. Cloud Run, GKE pods, and Vertex jobs spin up beside it and terminate. Data does not move.
⚡
Serverless = right-sized compute
You don't manage servers. Cloud Run scales to zero between runs. GKE Autopilot manages nodes. You pay for what fires — not for capacity sitting idle overnight waiting for a cron job.
The key insight: GCP's resource hierarchy is not an IT concern — it's a research design concern. How you structure projects determines your compliance boundaries, your grant accounting, and how much damage a misconfiguration can do. Get this right now and every module after this is cleaner.
Trade-off map

Managed service vs. serverless vs. self-hosted

You'll make this choice repeatedly. Here's the framework — most of your research workloads land in the middle column.

Dimension Self-hosted (GCE) Serverless (Cloud Run / GKE) Fully managed (Vertex AI)
Control Full — you own the OS Container boundary — you own the code Minimal — API surface only
Ops burden High — patches, scaling, HA Low — GCP handles nodes Near-zero
Cost model Always-on VM billing Per-request / per-second Per-training-hour + serving
HIPAA fit Possible — more surface to harden Good — smaller attack surface Best — Google manages infra security
Best for Legacy software, special hardware config Most research workloads — this course Standard ML pipelines, fast iteration
Hands-on exercise 1.1

Your first 20 minutes in Cloud Shell

📟 Guided exercise — Cloud Shell + gcloud CLI
  1. Open console.cloud.google.com — sign in with your hawaii.edu Google account.
  2. Open Cloud Shell (terminal icon, top right). Confirm your identity:
    gcloud auth list
  3. Create your sandbox project:
    gcloud projects create srl-sandbox-[yourname] --name="SRL Sandbox"
  4. Set it as your active project:
    gcloud config set project srl-sandbox-[yourname]
  5. List all your projects:
    gcloud projects list
  6. Explore IAM: go to IAM & Admin in the console. What roles does your account have? What can you do — and what's missing?
  7. Sketch on paper: Draw your UHCC project topology. Where does HIPIMR data live? Where does Makawalu compute run? Draw the boundary between them. This sketch is what we formalize in Lesson 1.2.
