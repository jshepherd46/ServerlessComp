Capstone — build
your own research cloud

Eleven modules in, you have the platform, the patterns, and the discipline. This capstone is the integration: pick one of four real SRL studies, apply every module end to end, ship a complete research cloud for that work.

⏱ ~6 hours guided work 🔧 Everything from M1–M11 📍 Course close
The setup

What you'll build

Pick one of the four tracks below. For your chosen study, you'll work through the complete architecture — from M1\'s topology to M11\'s automation — and produce a working research cloud for that study. The output: a Terraform repo, a runbook, a working data pipeline, a deployed model (or trained adapter), and a paper-grade reproducibility story.

Six hours of guided work. You won\'t finish everything in one sitting — that\'s fine. The capstone is a structure for the work; you ship it incrementally as your real research progresses.

Success criteria — universal across tracks

1
Terraform repo with at least two committed modules (PHI project + research project) and the study\'s project instantiation. Runs from a clean clone via terraform apply.
2
Per-study runbook in handbook/studies/<name>.md with all nine sections of the M11L3 template filled in.
3
Data pipeline — a Cloud Run Job or Vertex Pipeline that takes data from source to research tier, with audit + monitoring per M8L3.
4
Model or analytic deliverable — depends on track. A registered Vertex model, an ATLAS cohort, or a fine-tuned adapter. With a candidate alias + reproducible training pipeline.
5
Observability — at minimum: ingest freshness metric + per-service request-latency dashboard + alerting policy on the workload\'s SLO.
6
A reproducibility statement — one paragraph in the runbook that describes how a new lab member could re-derive your central result from the Terraform + pipeline JSON + cohort definition in the repo.

Track 1

Makawalu — POCUS XAI for breast cancer

Architecture

The shape

Queen\'s POCUS → GCS landing → DICOM store → de-id store → GCS training mirror → CBM training (Vertex) → Cloud Run XAI serving

Topology (M1L2)

  • srl-makawalu-dev — synthetic data, permissive IAM
  • srl-makawalu-staging — de-id\'d Queen\'s data, BAA-covered, perimeter
  • srl-makawalu-prod — live serving, real cohort, full perimeter + audit

Data path (M6L1 + M8)

  • POCUS DICOMs upload via authenticated GCS landing (per-site SA, IP allow-list)
  • Cloud Run Job validates + STOWs to srl-makawalu-prod Healthcare API DICOM store
  • Nightly Healthcare API deidentify → sibling de-id store + DLP for burned-in PHI
  • Weekly Vertex Pipeline materialises de-id\'d frames + concept labels into gs://srl-makawalu-training/cbm-v3/

Model (M6L2)

  • Concept bottleneck model — frozen ResNet50 backbone → concept heads (lesion shape, margins, posterior features) → final malignancy classifier
  • Vertex Custom Training on single A100, spot, with epoch checkpointing
  • Vertex Pipelines for the full preprocess → train_concept_heads → train_final_classifier → evaluate → conditional_register flow

Serving (M6L3)

  • Cloud Run — custom JSON response with concept scores + final probability + visualisation overlays
  • --concurrency=2 (heavy ML inference), --min-instances=1 during clinical-demo windows
  • Authenticated only; Workload Identity → xai-serve-sa with secret accessor on the model auth token

The capstone-specific decisions

  • Concept fidelity vs final accuracy. The CBM gates the final on concept predictions. Tune the trade.
  • Fairness across USAPI subgroups. Slice metrics per NHPI patient grouping (M6L3 + M7L2 patterns). The capstone deliverable must include this slice analysis.
  • Burned-in-PHI redaction. POCUS commonly has PHI in the pixels themselves. The de-id pipeline must run OCR + redaction (M8L1).
  • Demo SLO. Clinician demos cannot tolerate a 10-second cold start. Sustained min-instances or warmup pings during demo windows.

Deliverables

  • Terraform for the three projects + perimeter + Healthcare API stores
  • Vertex Pipeline (compiled JSON) for the CBM training workflow
  • Registered model makawalu-cbm with version aliases (latest / candidate / production)
  • Cloud Run service serving the latest candidate, with traffic-split scripts
  • Runbook section with the four capstone decisions documented + fairness analysis attached
Track 2

HIPIMR — Hawai‘i Pacific Islands Mammography Registry

Architecture

The shape

Queen\'s PACS nightly → HIPIMR DICOM store → de-id store → density CNN training tier → registry-integrated serving + longitudinal cohort tracking

The HIPIMR-specific challenge

HIPIMR is a longitudinal registry — DICOMs arrive monthly for years; cohort grows continuously; vocabulary refreshes interact with ETL; results from earlier years must remain reproducible as new data lands. The architectural moves all serve durability over time.

Topology + ingest

  • srl-hipimr-prod with CMEK keys that rotate annually (key versions retained for archive reads)
  • Nightly Cloud Run Job + Eventarc pull-from-Queen\'s pattern (M3L2 ingest path)
  • The DICOM store is the system of record; every read is Data Access logged for the registry-audit trail
  • Storage class lifecycle: hot for <2 years, nearline for 2-5 years, archive for >5 years

Density CNN model (M6L2)

  • ResNet50 → continuous density score (BI-RADS A-D as ordinal output)
  • Trained on de-id\'d backfill cohorts; retrained quarterly with new ingest
  • Each quarterly retrain registered as a new version with the cohort cutoff date as a label

Registry integration (M9 + M11)

  • Density score per study writes back to a registry-tier BigQuery table
  • OMOP-shaped registry data (M9) enables cross-cohort queries: "density > threshold for > 3 years"
  • ACHILLES re-runs quarterly to surface ingest gaps as the registry grows

The capstone-specific decisions

  • Multi-year cohort snapshotting. A paper from 2024 should reproduce against the 2024-cohort snapshot, not the 2026 live state. Versioned BigQuery snapshots + the training-mirror snapshots.
  • Vocabulary version skew. The OHDSI vocabulary used by the 2022 cohort may differ from the 2026 vocabulary. Snapshot the vocabulary at cohort-build time alongside the data.
  • Long-tail storage cost. Lifecycle policies + nearline / archive tiering. Set up day one or the bill grows quietly.
  • Per-cohort retention. IRB-approved retention varies; the runbook tracks each cohort\'s retention date.

Deliverables

  • Terraform for HIPIMR prod project + perimeter + DICOM store + lifecycle policies
  • Vertex Pipeline for quarterly density CNN retraining
  • Registered hipimr-density model with quarter-labelled version history
  • Quarterly ACHILLES report + the registry BigQuery integration
  • Runbook with multi-year retention plan + cohort snapshot inventory
Track 3

HALO — HCC + immunotherapy outcomes

Architecture

The shape

Queen\'s EHR nightly FHIR → de-id\'d FHIR store → OMOP transformation → BigQuery analytical + ATLAS cohorts → Vertex outcome predictor + federated extension via PacificFed

The HALO-specific shape

HALO is OMOP-native. The architectural emphasis is on the analytics layer (M9) and the federated analytics extension (M9L3) for cross-site validation of HCC outcome predictions in NHPI patients. The ML layer (a survival / outcome model in Vertex) is downstream of the analytics.

Data path (M8 + M9)

  • Nightly Queen\'s extract → FHIR store in srl-halo-prod
  • Healthcare API deidentify → sibling de-id FHIR store
  • OMOP ETL (Vertex Pipeline) → BigQuery OMOP tables in research tier
  • OHDSI vocabulary in srl-shared-services.omop_vocab (shared with PacificFed sites)

Analytics layer (M9)

  • Broadsea on Cloud Run — ATLAS UI for collaborators
  • Capr-defined cohorts in git for reproducibility
  • Quarterly ACHILLES + monthly cohort-build SLO tracking

Outcome predictor (M6)

  • Time-to-event Cox or DeepSurv model on the cohort\'s features
  • Vertex Pipeline for the cohort-export → train → register flow
  • Vertex Endpoint for low-latency outcome scoring on a new patient (with explainability)

Federated extension (M9L3 + M10)

  • Same cohort definition runs on Queen\'s\' own OMOP + UH\'s + future partner sites
  • Federated analytics for meta-analysis (M9L3)
  • Optional: federated learning for the predictor (M10) when 2+ sites have aligned data + protocol

The capstone-specific decisions

  • Cohort drift. HCC + immunotherapy patient counts grow as treatments shift. Cohort definition needs scheduled re-validation; ACHILLES dashboard helps.
  • Outcome-definition mapping. "Death from HCC" vs "all-cause mortality within 5 years" — the OMOP death + death_cause mapping is non-trivial; document it in the runbook.
  • Federated readiness. The cohort + predictor design must be portable to other OMOP sites without code changes. Test this with a synthetic second-site OMOP dataset.
  • IRB + DUA across sites. Each PacificFed site has its own protocol; the runbook tracks the joint study state.

Deliverables

  • Terraform for HALO PHI + research projects + Broadsea Cloud Run stack
  • OMOP ETL Vertex Pipeline + ACHILLES schedule
  • Capr-defined HCC + IO cohort (versioned in git)
  • Registered halo-outcome-predictor model with deployed Endpoint
  • Federated analytics PoC against a synthetic second-site OMOP dataset
  • Runbook with cohort drift + outcome mapping + federated readiness sections
Track 4

ShepBERT — clinical NLP at lab scale

Architecture

The shape

UH EHR FHIR DocumentReference → de-id → OMOP NOTE → sharded GCS corpus → LoRA fine-tune on Bio_ClinicalBERT → GKE multi-adapter serving

The ShepBERT-specific shape

Two characteristic moves vs the other tracks: LoRA fine-tuning at the model layer (compact adapters per task), and GKE multi-adapter serving at the inference layer (one base model + many task adapters loaded dynamically). Pavel\'s pipeline is the worked example throughout.

Data path (M7L1 + M8)

  • Nightly EHR FHIR DocumentReference extracts to srl-shepbert-prod FHIR store
  • Healthcare API de-id (rule set + DLP for free text) → sibling de-id store
  • OMOP NOTE materialisation via Cloud Run Job
  • Vertex Pipeline section-aware chunks notes → sharded JSONL training corpus in GCS

Fine-tuning (M7L2)

  • Base: emilyalsentzer/Bio_ClinicalBERT
  • LoRA with r=16, alpha=32, target_modules=[query, value], 5 epochs
  • One adapter per task (entity extraction, classification, etc.)
  • Vertex Pipeline: corpus → LoRA train → evaluate (per-entity + slice) → conditional register
  • Adapters under ~100 MB each — many adapters per base in the registry

Multi-adapter serving on GKE (M5 + M7L3)

  • vLLM with --enable-lora on a GKE GPU pool
  • Base Bio_ClinicalBERT loaded once into GPU memory; adapters loaded dynamically
  • Per-request: client specifies which adapter; vLLM swaps in milliseconds
  • One inference instance serves many ShepBERT tasks at marginal cost per added task

The capstone-specific decisions

  • Section-aware chunking. 512-token windows wasted on boilerplate kill F1. Implement section detection (HPI / PE / A&P) before chunking.
  • Adapter rank selection. Sweep r=8, 16, 32; pick where per-task F1 plateaus.
  • Fairness slices. Per service line + per NHPI grouping; the gate check uses worst-slice delta, not aggregate.
  • Multi-tenancy on the serving instance. One inference SA serves N adapters; per-request audit must capture which adapter ran on what request, not just "the inference instance was called."

Deliverables

  • Terraform for ShepBERT projects + FHIR stores + corpus bucket + GKE cluster
  • Vertex Pipeline for LoRA fine-tuning with the M7L2 evaluation gate
  • At least one registered adapter version with the production alias
  • vLLM Deployment on GKE serving the base + adapter
  • Runbook with section-aware chunking + rank choice + fairness slices documented
Implementation checklist

The universal walkthrough — module by module

Regardless of track, walk these steps in order. Each step references the canonical module + lesson; check off as you complete.

📋 The capstone walkthrough
  1. M0L1 — Setup. Confirm gcloud installed + authed; pick the GCP org + folder you\'ll work in; create a sandbox project for the capstone.
  2. M1L2 — Topology. Decide your three project names (srl-yourstudy-dev/staging/prod). Document them in the runbook.
  3. M1L3 — Identity. Define the IAM matrix for your track (personas + scopes + roles). Document in the runbook.
  4. M3L1 — Perimeter + BAA. Open the UH ITS ticket for BAA enrollment on prod. While waiting, set up the perimeter in dry-run mode with your three projects.
  5. M11L1 — Terraform. Use the SRL Terraform modules (or write a thin version) to instantiate your projects + perimeter membership + service accounts. Run plan + apply.
  6. M2 — Containers. Build a base training container + push to Artifact Registry. CI/CD via Cloud Build for the image (M2L2).
  7. M8L1 — Data plane. Stand up your Healthcare API store (DICOM for Makawalu/HIPIMR, FHIR for HALO/ShepBERT). Configure the de-id config (M8L2 DLP template).
  8. M6L2 / M7L2 — Pipeline. Write the training pipeline as KFP code; compile to JSON via CI (M11L2); submit smoke run against dev.
  9. M9 — Analytics (HALO + ShepBERT tracks). Deploy Broadsea Cloud Run + register WebAPI source against your OMOP dataset; load vocabulary.
  10. M6L3 / M7L3 — Serving. Pick Cloud Run vs Vertex Endpoint vs GKE per your track\'s requirements; deploy a candidate version.
  11. M4L3 — Operations. Configure an SLO + alert policy on the serving workload. Set up the billing-export query (M1L4) filtered to your study.
  12. M8L3 — End-to-end. Walk one cohort through your full ingest → de-id → training pipeline; document each step in the runbook.
  13. M11L3 — Runbook. Fill in the per-study runbook template; commit to the lab handbook repo.
  14. (Optional) M10 — Federated. If your track has a federation play (Makawalu/HALO/ShepBERT all do), prototype against a synthetic second site.
Realistic pacing. Most labs won\'t do all 14 steps in one six-hour session. The capstone is structured so each step\'s deliverables stand alone — a finished step is useful even if the next step is months away. The full walkthrough is what you accumulate over the first 6-12 months of running a study with this platform.
Closing the course

What you have now

Twelve modules. Roughly 30 dense lessons. A glossary that turns "VPC SC" and "DP-SGD" into actionable concepts. The patterns from M1 through M11 are the substrate; M12 is the assembly into a real research cloud.

The end state

→
A HIPAA-grade, audited, perimeter-protected research cloud that lives as Terraform in a git repo.
→
A data pipeline that ingests, de-identifies, and materialises cohorts on a schedule, with operational metrics and alerts.
→
Models trained, evaluated with fairness slices, registered with versioned lineage, and served with canary discipline.
→
A runbook the next member of the lab reads on day one and understands.
→
A reproducibility statement that lets the work survive turnover, IRB review, and the journal revision in three years.

Where to go from here

Read the glossary when you hit a term you don\'t remember. Revisit individual modules when you need the depth on a specific decision. Treat the lab handbook (M11L3) as the living document — every new study, every new model, every new collaborator adds an entry.

The course was built around five real SRL studies — HIPIMR, Makawalu, HALO, ShepBERT, PacificFed. Your next study\'s architecture probably looks like one of them with the details swapped. The patterns repeat; the names change.

The single sentence to remember from twelve modules. Sensitive data stays in a secure cloud boundary. Curated containers and compute spin up beside it, do the work, and terminate. Data never moves. You never pay for idle capacity. Every architectural choice in this course points back at that line — and now you can build it.
