Production operations
and the GKE escalation playbook

Deploying is the easy part. Keeping a serverless lab running is about deployment discipline, the SLOs you actually care about, the cost levers, and knowing the day you'd move a workload to GKE — and how. Closes Phase 1.

⏱ ~50 min reading + exercise 🔧 Cloud Run, Cloud Monitoring SLOs, billing exports 📍 Module 4 of 12
Deployment discipline

Three strategies, picked by stakes

M4L1 gave you the mechanics (traffic splitting, tags, no-traffic deploys). This lesson gives you the policy: which strategy you pick for which kind of change. Three patterns cover almost everything.

Strategy 1

Direct deploy

Push a new revision; 100% traffic instantly. The default.

Use for: internal dev services, demo apps, anything where a 5-second blip during a redeploy is fine.

Strategy 2

Canary rollout

Deploy with --no-traffic. Smoke-test via tag URL. Shift 10% → 50% → 100% with observation between each step.

Use for: prod APIs where a regression would affect users; any model-output change you want to watch in flight.

Strategy 3

Blue-green

Run two equivalent revisions, one taking traffic, one warm idle. Flip 100% to green; old blue stays available for instant rollback.

Use for: clinician-facing services, scheduled-event-window deploys, when a fast bidirectional flip is more important than gradual exposure.

A canary rollout script you can crib

#!/usr/bin/env bash
set -euo pipefail

SERVICE=xai-serve
REGION=us-west1
NEW_IMG=$1

# 1. Deploy with no traffic and a canary tag
gcloud run deploy $SERVICE --image=$NEW_IMG --region=$REGION --no-traffic --tag=canary

# 2. Smoke test on the tag URL
CANARY_URL=$(gcloud run services describe $SERVICE --region=$REGION \
  --format="value(status.traffic[?tag=='canary'].url)")
TOKEN=$(gcloud auth print-identity-token)
curl -sf -H "Authorization: Bearer $TOKEN" "$CANARY_URL/health" > /dev/null \
  || { echo "Canary smoke test failed"; exit 1; }

# 3. Step traffic and observe at each stage
NEW_REV=$(gcloud run services describe $SERVICE --region=$REGION \
  --format="value(status.traffic[?tag=='canary'].revisionName)")

for pct in 10 50 100; do
  gcloud run services update-traffic $SERVICE --region=$REGION \
    --to-revisions=$NEW_REV=$pct
  echo "Now $pct% on $NEW_REV. Observing for 5 minutes..."
  sleep 300
  # Check error rate; abort if it spiked
done

This is what an operations runbook (M1L4) section looks like in practice. Keep it in your lab repo; every prod service uses the same script with its own service name.

What "working" means

SLOs and error budgets, sized for research

A Service Level Objective is a measurable target for "is this service working." For a research lab, you do not need Google-grade five-nines reasoning. You need a couple of crisp numbers that let you say "yes, this is working" or "no, intervene now."

Reasonable SLO targets for SRL services

Clinician-facing inference API (Makawalu XAI)
Successful response within 2 seconds, 99% of requests, over rolling 30 days. The 1% error budget is ~7 hours/month — enough room for deploys, the occasional cold start, and a single bad rev.
99.0%
Internal-team APIs (ShepBERT serving for Pavel)
Successful response, 95% of requests, over rolling 7 days. Anything more strict is wasted effort here — the consumers are humans on the team who will tell you if it's broken.
95.0%
Nightly batch ingest (HIPIMR pipeline)
Job completes successfully, 99% of nights, over rolling 90 days. Failure budget: ~1 missed night per quarter, which a runbook entry covers.
99.0%
Demo / portfolio services
No SLO. Best effort. If it broke at 2am you will not be paged.

Configure an SLO in Cloud Monitoring

# Easiest path: console.
# Cloud Run service → SLOs → Create SLO → "Availability" or "Latency"
# Set target, set rolling window, set alert when budget burn rate > threshold
# The console writes a JSON spec you can also apply via gcloud monitoring slos create

# Programmatic shape:
{
  "displayName": "xai-serve availability",
  "serviceLevelIndicator": {
    "requestBased": {
      "goodTotalRatio": {
        "goodServiceFilter": "metric.type=\"run.googleapis.com/request_count\" resource.labels.service_name=\"xai-serve\" metric.labels.response_code_class=\"2xx\"",
        "totalServiceFilter": "metric.type=\"run.googleapis.com/request_count\" resource.labels.service_name=\"xai-serve\""
      }
    }
  },
  "goal": 0.99,
  "rollingPeriod": "30d"
}
The error budget rule of thumb: if you're burning the 30-day budget faster than 30% per week (i.e., trending toward exhaustion), pause new feature deploys and focus on stability. If you're consistently coming in well under budget, deploy more freely — you have margin. Treat it as a budget you spend deliberately.
Cost vs latency

Tuning the autoscaler for your traffic shape

The math from M4L1 told you how Cloud Run scales. The operational question is: for this service, with this traffic, what configuration minimises cost while staying inside the SLO?

Three traffic shapes you'll encounter, and the configuration that fits each:

Shape 1 — Sparse and bursty

Demo APIs, ad-hoc internal tools

A handful of requests per day. min-instances=0, concurrency=80, max-instances=10. Cold starts happen; you don't care. Cost: near zero between sessions.

Shape 2 — Steady low traffic with latency SLO

Internal team APIs

min-instances=1, concurrency=80, max-instances=20. One always-warm instance handles the steady-state. Cost: ~$30-40/month per always-on instance — the price of meeting the SLO.

Shape 3 — Heavy ML inference with SLO

Clinician-facing XAI / Makawalu

min-instances=1 (warm), concurrency=2 (model is the bottleneck), max-instances=20 (cost cap for sudden demos), CPU memory sized to model + buffer (--memory=4Gi, --cpu=2), --no-cpu-throttling if the model has post-request work.

The data that drives the tuning

Cloud Monitoring's pre-built metrics for Cloud Run answer the questions you need. Bookmark a dashboard with these four panels per service:

  • → Request count (per revision, with 2xx/4xx/5xx breakdown)
  • → Request latency p50/p95/p99 (where the SLO lives)
  • → Container instance count (vs your max-instances cap)
  • → Container CPU + memory utilisation (drives sizing decisions)

If p99 latency is high but CPU/memory utilisation is low, you're concurrency-limited — drop concurrency. If utilisation is high and you're hitting max-instances, raise max-instances (and reconsider memory/CPU sizing).

The bill

Cost levers, in order of impact

Cloud Run billing is per request + per vCPU-second + per GB-second of memory. Different levers move the bill in different ways — and most labs are paying for invisible inefficiencies they could fix in an afternoon.

--min-instances
Each always-warm instance bills ~$30-50/month at 1 vCPU / 2 GB. Audit every prod service: does it actually need warm? Demos and internal tools rarely do.
HIGH
CPU mode
--no-cpu-throttling costs ~2-3× per instance-second vs default throttled mode. Default to throttled; turn on always-allocated only when measured background work demands it.
HIGH
Concurrency
Higher concurrency = fewer instances for the same load. If your service is I/O-bound, raise it. Caveat: too high for CPU/GPU-bound services causes latency spikes — measure first.
MID
Region selection
Pricing varies modestly across regions. Bigger lever: keep services in the same region as their data to avoid egress charges. Cross-region calls add ~$0.08-0.12/GB on top.
MID
Image size
M2L2's multi-stage builds are not just about security — smaller images mean faster cold starts, less egress on pulls, less storage on Artifact Registry retention.
LOW
Log volume
Cloud Logging has a free tier (50 GiB/month) then $0.50/GB. Verbose request logging on a busy service can run hundreds of dollars/month. Filter what you ingest at the sink, not at the source.
MID

The monthly cost query

From M1L4's billing export, the query that surfaces lab-wide Cloud Run spend by service:

SELECT
  project.id AS project,
  resource.global_name AS service,
  ROUND(SUM(cost), 2) AS usd
FROM `srl-shared-services.billing.gcp_billing_export_v1_XXXX`
WHERE service.description = "Cloud Run"
  AND usage_start_time BETWEEN '2026-04-01' AND '2026-05-01'
GROUP BY project, service
ORDER BY usd DESC;

Run this on the 1st of every month. The top three services are where to look for tuning wins.

When to escalate

The playbook for moving to GKE

M2L3 listed the five conditions that mean Cloud Run is the wrong answer. When you hit one, the question becomes how do you move? The path is short and well-trodden.

The four conditions you'll actually hit

1. Long-running training / serving

You started with Cloud Run Jobs for nightly training; the model has grown to need 6-hour training runs and you're approaching the 24-hour Job ceiling. Or your inference service needs a process that lives between requests for hours. Move to GKE with a dedicated workload.

2. Multi-GPU / specialised GPU pools

Cloud Run GPUs (M4L1) cover single L4 workloads. PacificFed federated aggregation on multi-A100, Makawalu CBM training with mixed-precision multi-GPU — these need GKE with a dedicated GPU node pool. The GKE GPU support is the right floor.

3. Service mesh / complex networking

You've grown to 6+ services that talk to each other by name; you want mTLS between them, traffic-shifting at the mesh level, or per-namespace identity. Cloud Service Mesh on GKE is the answer.

4. Stateful runtime / Persistent Disks

You discover your ATLAS WebAPI install needs a writeable filesystem, or you want to run a self-managed Redis. GKE with PersistentVolumeClaim is the right shape; Cloud Run filesystems are ephemeral.

The migration shape

# 1. Same image, no rebuild. GKE pulls from the same Artifact Registry.
# 2. Same service account, with a Workload Identity binding instead of attached SA.
# 3. Same secrets, mounted via Workload Identity into the pod (M3L3 patterns hold).
# 4. Same network posture: GKE Autopilot in a private cluster inside the VPC SC perimeter.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: xai-serve
spec:
  replicas: 2
  selector: {matchLabels: {app: xai-serve}}
  template:
    metadata:
      labels: {app: xai-serve}
    spec:
      serviceAccountName: xai-serve-ksa   # Workload Identity → xai-serve-sa
      containers:
        - name: app
          image: us-west1-docker.pkg.dev/srl-shared-services/research/hello-flask:0.2.0
          ports: [{containerPort: 8080}]
          resources:
            requests: {memory: 2Gi, cpu: 1000m}
            limits:   {memory: 4Gi, cpu: 2000m}

The image, the SA, the secrets, the IAM are all the same. You're trading the Cloud Run service spec for a Kubernetes Deployment + Service. The operational cost goes up (cluster management, pod lifecycle, manifests in version control) but you gain the affordances GKE provides.

The rule: escalate only when one of the four conditions actively applies. "We might need GKE someday" is not a reason to migrate today. M5 covers GKE properly — that's where you'll learn the apparatus when you actually need it.
Hands-on exercise 4.3

Stand up an SLO and read the cost

📊 Observe what you've already built
  1. If your hello-flask service from M4L1 is gone, redeploy it.
  2. Generate some traffic — both successful and failed. From your laptop:
    URL=$(gcloud run services describe hello-flask --region=us-west1 --format='value(status.url)')
    for i in $(seq 1 50); do curl -sf -H "Authorization: Bearer $(gcloud auth print-identity-token)" $URL/ > /dev/null && echo ok || echo fail; done
    Then a few intentional 404s:
    for i in $(seq 1 5); do curl -sf -H "Authorization: Bearer $(gcloud auth print-identity-token)" $URL/nope > /dev/null || echo fail; done
  3. Create an SLO in the console. Cloud Run → hello-flask → SLOs → Create SLO. Pick "Availability" (request_count). Target 95%. Rolling window: 7 days. Save. Look at the SLO board: you should see your traffic and the budget remaining.
  4. Configure an error-rate alert. Cloud Monitoring → Alerting → Create Policy. Condition: run.googleapis.com/request_count{response_code_class="5xx"} > 0.05 sustained 5 min. Notification channel: your email. Save. Now if the error rate climbs, you get paged.
  5. Cost discipline check. In Cloud Run UI, set --min-instances=1 on hello-flask. Wait a few minutes. Check the billing console — your sandbox should accrue a few cents/day for the always-on instance. Confirm the effect; then set it back to 0.
  6. Query the billing export if you set it up in M1L4. Find your Cloud Run costs broken down by service:
    bq query --use_legacy_sql=false 'SELECT resource.global_name AS service, ROUND(SUM(cost),4) AS usd FROM `[YOUR_BILLING_DATASET].gcp_billing_export_v1_XXXX` WHERE service.description = "Cloud Run" GROUP BY service ORDER BY usd DESC'
    Note how small a number you're at — and how the bill scales if you set min-instances on five services at once.
  7. Walk through the GKE escalation playbook on paper. Pick one of your prod services (or one of the SRL studies). Ask: which of the four GKE escalation triggers applies? If none, what would have to change for one to apply? Write this down in your runbook.
