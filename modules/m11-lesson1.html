Lesson 1 of 3

Terraform the platform

Everything from M1-M10 stood up by hand becomes one Terraform apply. This lesson is the IaC layer for the SRL platform — why it matters for a research lab, the module structure that lets a new study launch in hours, state management that survives turnover.

⏱ ~50 min reading + exercise 🔧 Terraform, hashicorp/google provider, GCS state backend 📍 Module 11 of 12
Why IaC for research labs specifically

The four wins that pay for the discipline

1. Reproducibility

Every study\'s infrastructure (projects, perimeters, IAM bindings, buckets, registries, secrets) lives as version-controlled code. The infrastructure that ran in May 2026 can be reconstructed exactly in May 2028 from git checkout + terraform apply. Critical when papers get revised years after the work.

2. Speed-to-study

From "we got the grant" to "the study has projects, perimeters, vocabulary copies, ATLAS, training pipelines" goes from weeks to hours when standing up a new study is terraform apply -var="study_name=halo-v2" against a tested module. The next study isn\'t the same scale of effort as the first.

3. Drift detection

terraform plan against existing state surfaces any manual changes someone made via the console. In a HIPAA project, that\'s the difference between "we know what state we\'re in" and "we\'re finding out at audit time." Run weekly; investigate any unexpected diffs.

4. Onboarding + bus-factor

The Terraform repo IS the documentation. A new postdoc reads the modules to understand "what is the infrastructure for this study?" — same answer everywhere, no tribal knowledge required. When the original engineer leaves, the lab doesn\'t lose the infrastructure design.

What IaC isn\'t. It\'s not a magic bullet — terraform doesn\'t obviate the architectural skill from M1-M10; it codifies the result. The first Terraform write of the SRL platform takes weeks. The payoff is from study #2 onward; the first study often takes longer in Terraform than by hand. Commit to it knowing the curve.
The toolkit

Terraform basics applied to GCP

Terraform is a declarative IaC tool: you describe the desired state in .tf files; terraform plan shows the diff between current and desired; terraform apply reconciles. State is tracked in a backend (we use GCS) so subsequent applies know what already exists.

A minimal Terraform file

# main.tf — one project, one bucket, one service account
terraform {
  required_providers {
    google = {
      source  = "hashicorp/google"
      version = "~> 5.0"
    }
  }
}

provider "google" {
  project = "srl-makawalu-prod"
  region  = "us-west1"
}

resource "google_storage_bucket" "inference_models" {
  name                        = "srl-makawalu-inference-models"
  location                    = "US-WEST1"
  uniform_bucket_level_access = true
  encryption {
    default_kms_key_name = "projects/srl-shared-services/locations/us-west1/keyRings/srl-keys/cryptoKeys/makawalu"
  }
}

resource "google_service_account" "xai_serve" {
  account_id   = "xai-serve-sa"
  display_name = "XAI inference service"
}

resource "google_storage_bucket_iam_member" "xai_serve_read" {
  bucket = google_storage_bucket.inference_models.name
  role   = "roles/storage.objectViewer"
  member = "serviceAccount:${google_service_account.xai_serve.email}"
}

Three resources, declarative, idempotent. terraform apply creates the bucket, the SA, and the IAM binding. Re-running is a no-op (state matches). Editing any value and re-applying surgically updates that resource.

The workflow

terraform init       # installs providers, configures backend
terraform plan       # shows what would change vs current state
terraform apply      # makes the changes
terraform destroy    # tears down everything in this module
terraform state list # inspect tracked resources

The discipline: never apply without first reading plan. In a HIPAA project, an unexpected change in plan is a stop-and-investigate event.

The shape

The SRL Terraform repo structure

Two levels: reusable modules (the building blocks) + environments (the actual instantiations). One git repo per lab.

~30 lines bring up a complete HALO study with PHI-tier project + research-tier project + Healthcare API + the M3 perimeter membership + the M9 OMOP vocabulary linkage. The modules carry the architectural decisions; the study file carries the values.

Where Terraform keeps its memory

State backends, workspaces, locking

Terraform tracks the mapping from your declarative .tf files to the real resources in a state file. For any lab Terraform setup, the state file must live in a shared, locked, encrypted backend — never in local files. GCS is the right home.

The GCS backend

# studies/halo/backend.tf
terraform {
  backend "gcs" {
    bucket = "srl-terraform-state"      # one bucket per lab
    prefix = "studies/halo"             # one prefix per study/module
  }
}

GCS gives you encryption at rest (CMEK if needed), object versioning (rollback), and built-in locking via GCS\' atomic operations. Two engineers can\'t run terraform apply on the same study simultaneously without conflicting.

Workspaces — per-environment separation

# Use workspaces for dev / staging / prod within a single module
terraform workspace new dev
terraform workspace new staging
terraform workspace new prod

terraform workspace select prod
terraform apply -var-file=prod.tfvars

Each workspace gets its own state file under the same prefix. Studies that have meaningfully different dev/staging/prod often live in different directories instead (cleaner), but workspaces are useful for cases where the resources are nearly identical across environments.

State hygiene rules

  • → Never edit state files by hand. Use terraform state mv + rm when restructuring.
  • → Enable GCS object versioning on the state bucket. Rollback to a known-good state is sometimes necessary.
  • → CMEK the state bucket. State files often contain sensitive values (instance IDs, generated passwords).
  • → Restrict the state bucket\'s IAM to the Terraform-runner service account + a small admin group; no general access.
  • → Set up terraform plan in CI on every PR; require the plan output in review. Apply is restricted to merged-to-main + manual workflow.
Hands-on exercise 11.1

Terraform a minimal study scaffold

📜 IaC in miniature
  1. Install Terraform (brew install hashicorp/tap/terraform or via your package manager). Verify: terraform version.
  2. Authenticate Terraform with the same identity gcloud uses:
    gcloud auth application-default login
  3. Create a state bucket (one-time):
    gcloud storage buckets create gs://$PROJECT-tf-state --location=us-west1 --uniform-bucket-level-access
    gcloud storage buckets update gs://$PROJECT-tf-state --versioning
  4. Create a study scaffold module. In a new directory, save main.tf: 
    terraform {
  required_providers {
    google = { source = "hashicorp/google", version = "~> 5.0" }
  }
  backend "gcs" { bucket = "YOUR-PROJECT-tf-state", prefix = "demo-study" }
}

provider "google" {
  project = "YOUR-PROJECT"
  region  = "us-west1"
}

resource "google_storage_bucket" "demo_data" {
  name                        = "demo-study-data-${var.suffix}"
  location                    = "US-WEST1"
  uniform_bucket_level_access = true
  versioning { enabled = true }
}

resource "google_service_account" "demo_sa" {
  account_id   = "demo-study-sa"
  display_name = "Demo study workload"
}

resource "google_storage_bucket_iam_member" "demo_sa_reader" {
  bucket = google_storage_bucket.demo_data.name
  role   = "roles/storage.objectViewer"
  member = "serviceAccount:${google_service_account.demo_sa.email}"
}

variable "suffix" { type = string }
output "bucket_name" { value = google_storage_bucket.demo_data.name }
output "sa_email"    { value = google_service_account.demo_sa.email }
  5. Initialize + plan + apply:
    terraform init
    terraform plan -var="suffix=demo01"
    terraform apply -var="suffix=demo01"
  6. Inspect state:
    terraform state list
    terraform output
    You should see the bucket name and SA email.
  7. Introduce a drift. Manually delete the SA from the console (or change its display name via gcloud). Then run terraform plan again — it should detect the drift and propose to re-create the SA. terraform apply reconciles.
  8. Convert to a module. Move the resources into ../modules/demo-study/; have the root main.tf call the module. Re-apply. State refactor: terraform state mv google_storage_bucket.demo_data module.demo_study.google_storage_bucket.demo_data.
  9. Add a CI workflow. Create a GitHub Actions workflow that runs terraform fmt, terraform validate, terraform plan on every PR. Bonus: post the plan output as a PR comment.
  10. Tear down:
    terraform destroy -var="suffix=demo01"
    The state file remains in GCS (use it next time). The bucket + SA are gone.

Knowledge check

Questions to sit with

  • The first SRL Terraform iteration of an existing study takes longer than re-doing it by hand. When does the investment pay off? Hint
  • Terraform state contains sensitive values (generated passwords, internal resource IDs). What\'s the practical hardening for the state bucket? Hint
  • Someone manually changed an IAM binding via the console. Your terraform plan now wants to "fix" it back. What\'s the right response - just apply, or investigate? Hint
  • Why is the modules/ directory separate from the studies/ directory, and what would go wrong if you collapsed them? Hint
