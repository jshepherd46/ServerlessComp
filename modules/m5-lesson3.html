GKE at scale
— GPUs, networking, ops

L1 + L2 got you running. This lesson is the rest of what a research lab actually needs from GKE — GPU compute for training, the networking primitives, observability, autoscaling, and the upgrade discipline that keeps a cluster running for years.

Compute for ML

GPU node pools — getting training onto GKE

The single most common reason to escalate from Cloud Run to GKE: GPU training and inference that needs more than what Cloud Run GPUs cover. On GKE the pattern is dedicated GPU pools (Standard) or GPU-bearing Pod requests (Autopilot), with workloads scheduled onto GPUs via Pod resource requests.

Autopilot — just ask for GPUs

Autopilot provisions appropriate GPU nodes automatically when a Pod requests them. Simplest path; less control over machine families.

apiVersion: batch/v1
kind: Job
metadata: {name: density-cnn-train}
spec:
  template:
    spec:
      nodeSelector:
        cloud.google.com/gke-accelerator: nvidia-l4      # or nvidia-tesla-a100, etc.
      restartPolicy: OnFailure
      serviceAccountName: train-job-ksa
      containers:
        - name: trainer
          image: us-west1-docker.pkg.dev/srl-shared-services/research/cnn-train:0.4
          resources:
            requests:
              cpu: 4000m
              memory: 16Gi
              nvidia.com/gpu: 1
            limits:
              cpu: 4000m
              memory: 16Gi
              nvidia.com/gpu: 1

Autopilot sees the GPU request and provisions an L4-equipped node. When the Job completes, the node deprovisions. You pay only for the time the GPU node ran. No node-pool management.

Standard — explicit GPU node pool

When you need specific machine types, spot pricing, or stable pool sizing, use Standard with a dedicated GPU pool:

gcloud container node-pools create gpu-pool \
  --cluster=srl-ml-cluster --location=us-west1 \
  --machine-type=a2-highgpu-1g \
  --accelerator=type=nvidia-tesla-a100,count=1,gpu-driver-version=DEFAULT \
  --enable-autoscaling --min-nodes=0 --max-nodes=4 \
  --node-taints=workload=gpu:NoSchedule

The --node-taints ensures only Pods with a matching toleration land on GPU nodes — CPU workloads stay off the expensive pool. Match in the Pod spec:

      tolerations:
        - key: workload
          operator: Equal
          value: gpu
          effect: NoSchedule

The cost shape

A100 is roughly $3-4/GPU-hour on-demand; L4 is roughly $0.70/GPU-hour. Spot pricing cuts both by ~70% with the risk of preemption. For training that checkpoints, spot is almost always the right call. For low-latency inference where preemption ruins UX, on-demand.

Two patterns SRL actually uses. Makawalu CBM training — Job on Autopilot with one L4, checkpoint every epoch to GCS, restart on failure. PacificFed FL aggregator — long-running Deployment on Standard with a dedicated multi-A100 pool, never preempted, lifecycle managed manually because uptime is the whole point.
In and out of the cluster

Networking — Services, Ingress, Gateway

M5L1 introduced Service. There are four practical variants for exposing workloads — three on the Service object and one above it. Pick by who's talking to the Pod.

type: ClusterIP (default)

Internal-only

A stable DNS name and IP reachable only from inside the cluster. Cluster-local Pods call it by name.

Use for: service-to-service inside the cluster.
type: LoadBalancer (external)

Public regional LB

Provisions a public IP + a Google L4 load balancer in front of the Service. Reachable from the internet.

Use for: rare. Most public services should go through Gateway / Ingress for HTTPS + WAF.
annotations: cloud.google.com/load-balancer-type: Internal

Internal LB

Same as LoadBalancer but the IP is RFC1918, reachable only from inside the VPC.

Use for: cross-cluster services, on-prem hybrid, any in-VPC consumer outside the cluster.
kind: Gateway (recommended) or Ingress (older)

HTTPS routing with WAF

Layer 7 routing: path-based or host-based, TLS termination, Cloud Armor attached, IAP integration. The right answer for public web traffic.

Use for: clinician-facing UIs, public APIs, anything exposed externally to humans.

Gateway — a modern HTTPS front door

The Gateway API supersedes Ingress for new deployments. It's more expressive and shares the API across cloud and OSS.

apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata: {name: xai-gateway}
spec:
  gatewayClassName: gke-l7-global-external-managed
  listeners:
    - name: https
      protocol: HTTPS
      port: 443
      tls:
        mode: Terminate
        certificateRefs:
          - kind: SecretRef
            name: xai-tls-cert
---
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata: {name: xai-route}
spec:
  parentRefs:
    - name: xai-gateway
  hostnames: ["xai.makawalu.uhcancercenter.org"]
  rules:
    - matches:
        - path: {type: PathPrefix, value: /}
      backendRefs:
        - name: xai-serve
          port: 80

The same Cloud Armor policy from M3L2 can attach via GCPBackendPolicy. IAP attaches via GCPGatewayPolicy. The HTTP routing is portable; the GCP-specific behaviour (WAF, IAP) attaches through targeted policy objects.

Knowing what's happening

Observability — kubectl, Cloud Logging, Cloud Monitoring

GKE pipes everything to Cloud Logging and Cloud Monitoring automatically. The skill is knowing which tool answers which question — and where the gaps are.

kubectl — for live, ad-hoc investigation

# What's running, by namespace
kubectl get pods -n makawalu -o wide

# Why is this Pod not Ready?
kubectl describe pod xai-serve-abc -n makawalu

# Last 100 log lines, follow live
kubectl logs deployment/xai-serve -n makawalu --tail=100 -f

# Exec into a Pod (interactive shell)
kubectl exec -it xai-serve-abc -n makawalu -- /bin/sh

# Resource utilisation snapshot
kubectl top pods -n makawalu --sort-by=memory

Cloud Logging — for historical queries and cross-cluster views

Every container's stdout/stderr lands here automatically. The query for "errors from xai-serve in the last hour":

resource.type="k8s_container"
resource.labels.cluster_name="srl-ml-cluster"
resource.labels.namespace_name="makawalu"
resource.labels.container_name="app"
labels."k8s-pod/app"="xai-serve"
severity>=ERROR

Save as a saved query in Logs Explorer; reuse during incidents.

Cloud Monitoring — for SLOs and dashboards

GKE ships with a pre-built dashboard per cluster. The metrics that matter for an SLO:

  • → kubernetes.io/container/cpu/core_usage_time — CPU per container
  • → kubernetes.io/container/memory/used_bytes — memory per container
  • → kubernetes.io/container/restart_count — restart counter (alert on increases)
  • → Custom metrics from your app via OpenTelemetry → Cloud Monitoring

SLO setup mirrors M4L3 — create an SLO target on the metric of interest, set burn-rate alerts, wire to a notification channel.

The one piece GKE doesn't do for free: distributed tracing. If you want to see a request flow across multiple services in your cluster, instrument with OpenTelemetry. For most research labs, the per-service request-latency histogram in Monitoring is sufficient; tracing comes in when you have ≥3 services on the critical path.
Three layers

Autoscaling — Pods, nodes, vertical sizing

Cloud Run had one autoscaler (request-based). GKE has three working at different levels. In Autopilot they're all rolled together; in Standard you configure each.

Layer 1 — HorizontalPodAutoscaler (HPA)

Scale the Pod count up and down

Watches a metric (CPU, memory, custom); adds or removes Pod replicas to keep the metric within a target. The most useful autoscaler for serving workloads.

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata: {name: xai-serve, namespace: makawalu}
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: xai-serve
  minReplicas: 2
  maxReplicas: 20
  metrics:
    - type: Resource
      resource: {name: cpu, target: {type: Utilization, averageUtilization: 70}}
Layer 2 — Cluster Autoscaler (Standard) / Autopilot

Scale the node count up and down

When the HPA wants more Pods but no node has room, the cluster autoscaler adds a node. When Pods drain off a node, it removes the node. In Autopilot this is invisible; in Standard you configure min/max per pool.

Layer 3 — VerticalPodAutoscaler (VPA)

Right-size resource requests automatically

Watches actual resource usage over time and updates the Pod's requests to match. Useful for guessing the right starting point; controversial in prod because it churns deployments. Most labs use VPA in "off" mode (recommendation-only) to inform manual tuning.

Cost levers on GKE

  • → HPA minReplicas — Cloud Run's min-instances equivalent. Default to 1; raise only if you have a measured reason.
  • → Resource requests — request only what you actually use. Pods asking for 8 GB and using 2 GB pay for 8.
  • → Spot nodes for batch — training that checkpoints can run on spot for ~70% off. Add a spot pool with a clear toleration and put your Jobs there.
  • → Cluster lifecycle — dev clusters that you only need office hours can be created in the morning, deleted in the evening. Helm makes redeploy fast.

Run the M1L4 billing query filtered to service.description = "Kubernetes Engine" on the first of every month. Watch the per-service distribution; tune the outliers.

Keeping the cluster alive for years

Upgrades, release channels, maintenance windows

A GKE cluster has a Kubernetes version, and Kubernetes versions move quarterly. Auto-upgrade is on by default in Autopilot. The discipline is choosing the right channel and the right maintenance window so upgrades happen on your terms, not in the middle of a study run.

Release channels

RAPID
Newest minor + patches first. ~2-4 weeks behind upstream. Best for development clusters where you want to try new features early.
REGULAR
Default. ~2-3 months behind upstream. Recommended for most prod clusters — balances stability with not falling behind.
STABLE
Most conservative. ~6 months behind upstream. Use when an upgrade-related outage would be unacceptable and you can absorb the upstream lag.

SRL default: REGULAR for prod clusters, RAPID for sandbox.

Maintenance windows

Pin upgrades to a window when downtime risk is lowest:

gcloud container clusters update srl-ml-cluster --location=us-west1 \
  --maintenance-window-start="2026-06-01T11:00:00Z" \
  --maintenance-window-end="2026-06-01T15:00:00Z" \
  --maintenance-window-recurrence="FREQ=WEEKLY;BYDAY=SU"

Sunday 3–7 AM HST. No clinician demos, no training runs scheduled. The upgrade may take a few minutes; Pods drain to a temporary node, the old node upgrades, drained Pods reschedule. Zero-downtime if your Deployments use PodDisruptionBudget + rolling-update strategy.

PodDisruptionBudget — surviving evictions

apiVersion: policy/v1
kind: PodDisruptionBudget
metadata: {name: xai-serve-pdb, namespace: makawalu}
spec:
  minAvailable: 1
  selector: {matchLabels: {app: xai-serve}}

During an upgrade, the autoscaler honours the PDB and won't evict more Pods than the budget allows. Without it, all Pods can be evicted simultaneously during a node drain.

The minimum runbook entry for GKE. Cluster: name, location, channel, maintenance window. Per workload: Deployment + PDB + HPA + Service + (Gateway if public) + ServiceAccount with WI binding. When you start a new study, you copy these patterns from the runbook; the cluster you inherit a year from now is comprehensible because the patterns are consistent.
Hands-on exercise 5.3

HPA + Gateway + PodDisruptionBudget

⚙️ Operational shaping
  1. Restart your Autopilot cluster from earlier if needed (srl-demo-cluster). Re-deploy hello-flask via the Helm chart from M5L2.
  2. Add an HPA on the hello-flask Deployment: 
    apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata: {name: hello-flask, namespace: demo-study}
spec:
  scaleTargetRef: {apiVersion: apps/v1, kind: Deployment, name: demo-app}
  minReplicas: 2
  maxReplicas: 8
  metrics:
    - type: Resource
      resource: {name: cpu, target: {type: Utilization, averageUtilization: 60}}
    kubectl apply -f hpa.yaml. Confirm: kubectl get hpa -n demo-study.
  3. Generate load to see scaling. From inside the cluster:
    kubectl run loadgen --rm -it --image=busybox --restart=Never -n demo-study -- /bin/sh -c "while true; do wget -q -O- http://demo-app.demo-study.svc.cluster.local/; done"
    Wait a couple minutes; kubectl get hpa -n demo-study --watch — replicas should climb. Stop the loadgen (Ctrl-C); replicas scale back down after the cooldown.
  4. Add a PodDisruptionBudget: 
    apiVersion: policy/v1
kind: PodDisruptionBudget
metadata: {name: demo-app-pdb, namespace: demo-study}
spec:
  minAvailable: 1
  selector: {matchLabels: {app.kubernetes.io/instance: demo-app}}
    kubectl apply -f pdb.yaml. Confirm with kubectl get pdb -n demo-study.
  5. Simulate a drain (replicates what an upgrade does). Pick a node hosting demo-app Pods, then:
    kubectl get pods -n demo-study -o wide (note the node names)
    kubectl drain <node> --ignore-daemonsets --delete-emptydir-data (will respect your PDB; will not evict beyond it)
  6. Set up an internal LB in front of demo-app — useful pattern for cross-namespace service-to-service: 
    apiVersion: v1
kind: Service
metadata:
  name: demo-app-ilb
  namespace: demo-study
  annotations:
    networking.gke.io/load-balancer-type: "Internal"
spec:
  type: LoadBalancer
  selector: {app.kubernetes.io/instance: demo-app}
  ports: [{port: 80, targetPort: 8080}]
    kubectl apply -f ilb.yaml. Wait a couple of minutes — kubectl get svc demo-app-ilb -n demo-study should show a private IP. Reachable only from inside the VPC.
  7. Configure the maintenance window:
    gcloud container clusters update srl-demo-cluster --location=us-west1 --maintenance-window-start="2026-06-01T11:00:00Z" --maintenance-window-end="2026-06-01T15:00:00Z" --maintenance-window-recurrence="FREQ=WEEKLY;BYDAY=SU"
  8. Run the M1L4 billing query filtered to GKE:
    bq query --use_legacy_sql=false 'SELECT service.description AS svc, ROUND(SUM(cost),2) usd FROM \`[YOUR_BILLING_DATASET].gcp_billing_export_v1_XXXX\` WHERE service.description LIKE "Kubernetes%" GROUP BY svc'
  9. Tear down everything when you're done — the cluster itself accrues cost regardless of workload count:
    helm uninstall demo-app -n demo-study
    kubectl delete namespace demo-study
    gcloud container clusters delete srl-demo-cluster --location=us-west1

Knowledge check

Questions to sit with

  • You want to run Makawalu's CBM training on GKE Autopilot with an L4 GPU. What's the minimum Pod spec change from a CPU-only Pod? Hint
  • You have a clinician-facing XAI demo that needs to be public-https with WAF protection. Service type LoadBalancer or Gateway? Hint
  • HPA is at minReplicas: 1, maxReplicas: 20. Traffic spikes; your monitoring shows 4 Pods running but latency p99 is climbing. Three possible causes? Hint
  • The cluster auto-upgraded on a Wednesday afternoon and one Deployment briefly hit zero ready Pods. What did you forget to add? Hint
