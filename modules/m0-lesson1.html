Setup, part 1

Accounts, the BAA, and billing

Before you touch a single GCP service, four things have to be true. None of them are technical. All four block real work later if you skip them.

Why it matters
An Organization in GCP is keyed to a verified domain. Only identities under that domain can be granted roles inside it without extra friction.
What to check
Sign in at console.cloud.google.com with jshepherd@hawaii.edu. Top-right avatar should show the UH org.
Common mistake
Spinning up a personal-gmail GCP project for "just a quick test." It can never join the UH org or use UH's BAA — you'll have to migrate later.
Action
If you don't see the UH org in the project picker, email UH ITS or your cloud admin to get added to the shepherd-research-lab folder.

A BAA is the contract Google signs with UH that lets HIPAA-eligible GCP services process PHI on your behalf. It is signed once by UH ITS, not by you — but you must confirm your project is covered before you load any patient data.

Who signs
UH ITS / Cloud Compliance — never an individual researcher.
What it covers
Only HIPAA-eligible services (GCS, BigQuery, Cloud Run, GKE, Vertex AI, Healthcare API, KMS, Pub/Sub). Not every GCP service is on the list.
Per-project enrollment
Each project that will hold PHI must be enrolled under the BAA. UH ITS does this when you request a HIPAA-tier project.
Verification
Cloud Console → IAM & Admin → Compliance → Compliance reports → search "BAA." Your project should appear in the enrolled list.
HIPIMR / Makawalu / HALO → all need a BAA-covered project before real data lands. The course's M0 sandbox does not — you'll keep it PHI-free and use synthetic data for learning.

A billing account is the payment entity. A billing account can fund many projects; a project must be linked to exactly one. For grant accounting, this is your hook — link halo-prod to the HALO R01 billing account, makawalu-prod to the Makawalu R01 billing account, and the per-grant invoices come out clean every month.

Sandbox
Use UH's discretionary or your $300 free-tier credit for the M0 sandbox. No PHI, no grant funds.
Per-study
Each NIH grant should have its own billing account so its monthly invoice maps 1:1 to that award's cloud line.
Cost labels
Tag every resource with grant=R01CA999999 and study=halo. BigQuery billing export then slices cost by tag.
Set this up later
Cost labels and BQ billing export are an M3/M11 topic. For now, just confirm a billing account exists and is linked.

A budget alert emails you at 50/90/100% of a monthly threshold. It does not stop charges. If you want a hard kill switch (recommended for sandboxes), pair the 100% alert with a Pub/Sub topic that triggers a Cloud Function which disables the billing link.

For your sandbox, $50/month is plenty. For HIPIMR-prod, set it to your monthly NIH cloud line and pad 20%. We'll wire up the kill-switch pattern in M11.

Setup, part 2

Two paths to a working CLI

You can drive GCP from a browser tab or from your laptop terminal. Both work for this course. Pick one to start; you'll likely end up using both.

Path A · easiest

Cloud Shell

A free browser-based Linux VM with gcloud, kubectl, Docker, Python, and an editor pre-installed. Pre-authenticated to your identity. Click the terminal icon in the top-right of the Cloud Console.

Use when: learning, demos, exercises in M0–M4, any time you're on a borrowed machine.

Path B · what you'll grow into

Local gcloud + VS Code

Install the Cloud SDK on your laptop, auth it once, drive everything from your own terminal. Your editor, your shell, your dotfiles.

Use when: serious development, building containers (M2+), running long sessions, version control.

Recommendation: start with Cloud Shell today. Install the local CLI by the end of this lesson if you have 10 free minutes. Either way, the rest of the course works.
The CLI

Install, auth, and configure gcloud

The gcloud CLI is the primary tool you'll use across the course. Cloud Shell already has it. If you're going local:

macOS

# Homebrew
brew install --cask google-cloud-sdk

# Verify
gcloud version

Windows

# Download the installer
https://cloud.google.com/sdk/docs/install#windows

# Then in a new PowerShell:
gcloud version

Linux (Debian / Ubuntu)

curl https://sdk.cloud.google.com | bash
exec -l $SHELL
gcloud version

Auth — two commands you run once

There are two distinct auth flows. You need both.

# 1. Authenticates you (the human) for gcloud commands
gcloud auth login

# 2. Authenticates SDKs running on your machine (Python, Go, Node)
#    via Application Default Credentials
gcloud auth application-default login

The first command is what you use. The second is what your code uses — when you write a Python script that calls the BigQuery client, it reaches for ADC automatically. Run both, once, on every machine you work from.

On GCP, ADC just works. A Cloud Run service or GKE pod automatically picks up the credentials of the service account attached to it. You never set env vars or copy key files. Local dev is the only place you call application-default login.

Multi-project switching with configurations

You'll be flipping between HIPIMR, Makawalu, HALO, and your sandbox constantly. Don't edit gcloud config properties one at a time — use named configurations:

# Create a configuration per study
gcloud config configurations create hipimr-prod
gcloud config set project hipimr-prod
gcloud config set compute/region us-west1
gcloud config set account jshepherd@hawaii.edu

gcloud config configurations create makawalu-dev
gcloud config set project makawalu-dev
gcloud config set compute/region us-west1

# Switch between them with one command
gcloud config configurations activate makawalu-dev

# See what you have
gcloud config configurations list

Set this up once for every project you regularly touch. Saves you from the entire class of "wait, which project just got that command?" mistakes.

Editor & container runtime

VS Code + Cloud Code, and Docker for M2

For local work the recommended stack is VS Code with the Cloud Code extension. Cloud Code gives you:

  • → One-click "Run on Cloud Run" — builds, pushes, deploys from your editor
  • → Live GKE cluster browser, pod logs, exec-into-pod from a sidebar
  • → YAML schemas for Kubernetes and Cloud Run service manifests
  • → Secret Manager browser without leaving the editor
# Install VS Code, then add the extension:
# Marketplace → search "Cloud Code" → Google → Install
# Sign in with your hawaii.edu account when prompted.

Container runtime (needed from M2 onward)

M2 builds container images on your laptop. You need a local container runtime. Three real options:

Most common

Docker Desktop

The default. Free for personal and small academic use; check UH licensing for paid use. Mac + Windows + Linux.

License-friendly

Rancher Desktop

Open-source, Apache 2.0. Drop-in replacement for Docker Desktop's daemon. Ships with k3s for local Kubernetes.

If you're staying in Cloud Shell: skip this section. Cloud Shell has Docker built in. You'll only need a local runtime when you want to iterate on containers without round-tripping to the cloud.
Hands-on exercise 0.1

Your first 30 minutes — sandbox to "hello, Cloud Run"

📟 Guided exercise — Cloud Shell (no local install needed)
  1. Open console.cloud.google.com and sign in with your-name@hawaii.edu. Confirm the org switcher shows the UH organization.
  2. Open Cloud Shell (terminal icon, top-right of the console). Run gcloud auth list — your hawaii.edu account should show as ACTIVE.
  3. Create your sandbox project:
    gcloud projects create srl-sandbox-[yourname] --name="SRL Sandbox"
  4. Set it as your active project and pick a region:
    gcloud config set project srl-sandbox-[yourname]
    gcloud config set run/region us-west1
  5. Link a billing account (use UH discretionary or your free-tier credit):
    gcloud billing projects link srl-sandbox-[yourname] --billing-account=[BILLING_ACCOUNT_ID]
  6. Enable the APIs you'll need:
    gcloud services enable run.googleapis.com artifactregistry.googleapis.com
  7. Deploy the Cloud Run hello-world straight from a public image — no Dockerfile yet:
    gcloud run deploy hello --image=us-docker.pkg.dev/cloudrun/container/hello --allow-unauthenticated --region=us-west1
  8. Visit the URL gcloud prints. You should see the Cloud Run welcome page. Congratulations — you just deployed a serverless container.
  9. Set a budget alert: Billing → Budgets & alerts → Create budget. Scope to your sandbox project. $50/month. Alerts at 50% / 90% / 100%.
  10. Optional (10 min): install gcloud locally on your laptop, run gcloud auth login + gcloud auth application-default login, then gcloud config configurations create srl-sandbox and re-set the project. Confirm gcloud run services list --region=us-west1 shows your hello service.

Useful commands to keep on hand

gcloud config list gcloud config configurations list gcloud auth list gcloud projects list gcloud services list --enabled gcloud run services list gcloud help
