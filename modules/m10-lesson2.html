Lesson 2 of 3

FL on GCP — clusters,
networking, deployment

L1 was the algorithm. This lesson is the infrastructure — multi-cluster GKE across sites, VPC peering for the inter-site comms, deploying the Flower aggregator + per-site clients, and the model-versioning + audit posture that makes federated training reproducible.

⏱ ~55 min reading + exercise 🔧 GKE Autopilot, VPC peering, Flower, Vertex Model Registry 📍 Module 10 of 12
The infrastructure shape

Multi-cluster GKE with one aggregator

Each PacificFed site runs its own GKE cluster inside the site\'s own GCP project (or on-prem, but for SRL we assume GCP at each site). The aggregator lives in a separate project, srl-pacificfed-infra, with its own cluster. Inter-site communication happens over private IP via VPC peering.

┌─────────────────────────────────────┐ │ Aggregator (srl-pacificfed-infra) │ │ GKE cluster: pf-aggregator │ │ Flower server pod (port 8080) │ │ ILB: 10.0.0.10 │ # internal LoadBalancer └─────────────────────────────────────┘ ▲ VPC peering ┌────────────────┼────────────────┬────────────────┐ │ │ │ │ ┌──────▼──────┐ ┌──▼──────────┐ ┌─▼─────────┐ ... ┌──▼──────────┐ │ UH cluster │ │ Queen\'s │ │ Guam │ │ Marshall Is.│ │ pf-client │ │ pf-client │ │ pf-client │ │ pf-client │ │ Flower pod │ │ Flower pod │ │ ... │ │ ... │ │ local OMOP │ │ local OMOP │ │ │ │ │ └─────────────┘ └─────────────┘ └───────────┘ └─────────────┘

Why GKE for the FL clients

  • → Long-running. A FL training round can take hours; the client is a long-running process that waits for the aggregator\'s next round. Cloud Run\'s 60-min request ceiling rules it out (M4L3 escalation triggers apply).
  • → GPU access. Training the local model at each round usually wants a GPU. GKE Autopilot or Standard with a GPU node pool (M5L3) gives it cleanly.
  • → Site-local identity. The FL client at each site runs as a workload SA scoped to that site\'s OMOP data + the FL coordination network — same M1L3 patterns.
  • → Consistent operations. The same M5 patterns apply everywhere: same Helm chart, same upgrade story, same observability.

Why a separate aggregator project

  • → Isolation. The aggregator sees aggregate updates but no patient data. Different IAM, different audit posture, different IRB protocol context.
  • → Shared service. One aggregator project can host the aggregators for many concurrent FL studies, each a separate Deployment/Service inside the same cluster.
  • → Lifecycle clarity. When a particular FL study ends, you tear down the corresponding aggregator Deployment; sites can stay configured for the next study.
The wiring

VPC peering across sites

Each site\'s Flower client needs to reach the aggregator. Options: public internet over HTTPS (cheap, exposed); GCP VPC peering (private, secure, free between regions); GCP VPN tunnels (private, encrypted, has cost). For PacificFed: VPC peering between each site\'s VPC and the aggregator\'s VPC.

Setting up peering

# In the aggregator project: peer with UH\'s VPC
gcloud compute networks peerings create pf-uh-peering \
  --network=pf-aggregator-vpc \
  --peer-project=srl-pacificfed-uh \
  --peer-network=uh-fl-vpc \
  --export-custom-routes --import-custom-routes

# In UH\'s project: peer back
gcloud compute networks peerings create pf-aggregator-peering \
  --network=uh-fl-vpc \
  --peer-project=srl-pacificfed-infra \
  --peer-network=pf-aggregator-vpc \
  --export-custom-routes --import-custom-routes

Both directions of the peering must be created. Once both exist, private IPs in one VPC can reach private IPs in the other. The aggregator\'s internal LB (10.0.0.10) becomes addressable from UH\'s GKE cluster.

The connectivity matrix

Each site only needs to reach the aggregator — sites do not talk to each other directly. So PacificFed needs N pairs of peerings (one per site), not N² pairs:

aggregator ←→ UH         # 1 pair
aggregator ←→ Queen\'s    # 1 pair
aggregator ←→ Guam       # 1 pair
aggregator ←→ Am. Samoa  # ...
aggregator ←→ CNMI
aggregator ←→ Palau
aggregator ←→ Marshall Is.
# Total: 7 peering pairs for 7 sites + 1 aggregator

When peering is not possible

For sites that are NOT on GCP (or are on a different GCP org without policy approval for cross-org peering), the alternatives in order of preference:

  • → HA VPN tunnel. Encrypted, predictable cost, well-documented setup.
  • → Public endpoint with mTLS + IP allow-list. Aggregator exposes a public Cloud Armor-protected endpoint; site authenticates via client certificate.
  • → Cross-cloud Anthos or workload-identity federation. Complex; only for institutional FL networks crossing multiple cloud providers.
The thing peering buys you operationally. Communication between aggregator and clients shows up as same-VPC traffic from each side\'s perspective. VPC SC perimeter rules apply on each side; the aggregator never sees per-site PHI. The connection itself is implicit; no tunnel state to monitor.
Deploying the aggregator

Flower aggregator on GKE

The aggregator is a long-running gRPC server. One Deployment, one Service (internal LoadBalancer), one ServiceAccount with Workload Identity for the GCP service account that has registry write + Pub/Sub publish rights.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: pf-aggregator-halo-hcc
  namespace: pf-active-studies
spec:
  replicas: 1                  # Flower aggregator is single-instance per study
  selector: {matchLabels: {app: pf-aggregator-halo-hcc}}
  template:
    metadata: {labels: {app: pf-aggregator-halo-hcc, study: halo-hcc-fl-v3}}
    spec:
      serviceAccountName: pf-aggregator-ksa
      containers:
        - name: aggregator
          image: us-west1-docker.pkg.dev/srl-shared-services/research/pf-aggregator:0.4
          args:
            - --study=halo-hcc-fl-v3
            - --rounds=100
            - --strategy=fedprox
            - --min-clients=5      # at least 5 of 7 sites must check in per round
            - --port=8080
          ports: [{containerPort: 8080}]
          resources:
            requests: {memory: 4Gi, cpu: 2000m}
            limits:   {memory: 8Gi, cpu: 4000m}
---
apiVersion: v1
kind: Service
metadata:
  name: pf-aggregator-halo-hcc
  namespace: pf-active-studies
  annotations: {networking.gke.io/load-balancer-type: "Internal"}
spec:
  type: LoadBalancer
  selector: {app: pf-aggregator-halo-hcc}
  ports: [{port: 8080, targetPort: 8080, protocol: TCP}]

The internal LB gets a private IP reachable by peered VPCs. Sites configure their Flower clients to point at pf-aggregator-halo-hcc.pf-active-studies.svc or directly at the LB IP.

Multiple concurrent FL studies

PacificFed can host many simultaneous FL studies — HCC outcome predictor, breast cancer screening adherence, NLP entity extractor. Each gets its own Deployment + Service in the same aggregator cluster. Per-study namespaces give per-study RBAC.

pf-active-studies/
├── pf-aggregator-halo-hcc        # HCC outcomes
├── pf-aggregator-screening-bc    # breast cancer screening
└── pf-aggregator-shepbert-fl     # clinical NLP FL

One cluster, many studies, clean isolation. The M5L2 namespace strategy applies directly.

Per-site clients

Deploying the Flower client at each site

Each site\'s GKE cluster hosts one Flower client Pod per FL study it participates in. The Pod runs the same training container as every other site (built once in srl-shared-services); it differs in the workload SA (site-specific), the data path (site\'s own OMOP dataset), and the connection to the aggregator (via the peered VPC).

apiVersion: apps/v1
kind: Deployment
metadata:
  name: pf-client-halo-hcc
  namespace: pf-uh   # site-specific namespace at UH
spec:
  replicas: 1
  selector: {matchLabels: {app: pf-client-halo-hcc}}
  template:
    metadata: {labels: {app: pf-client-halo-hcc}}
    spec:
      serviceAccountName: pf-client-uh-ksa  # bound via WI to pf-client-uh-sa@srl-pacificfed-uh.iam
      nodeSelector:
        cloud.google.com/gke-accelerator: nvidia-l4
      containers:
        - name: client
          image: us-west1-docker.pkg.dev/srl-shared-services/research/pf-client:0.4
          args:
            - --study=halo-hcc-fl-v3
            - --aggregator-host=10.0.0.10:8080      # internal LB IP, reachable via peering
            - --site-id=uh
            - --data-source=bq://srl-pacificfed-uh.omop_deid.training_cohort_v3
            - --rounds-to-participate=100
          resources:
            requests: {memory: 16Gi, cpu: 4000m, nvidia.com/gpu: 1}
            limits:   {memory: 16Gi, cpu: 4000m, nvidia.com/gpu: 1}

Each site deploys this manifest into its own GKE cluster. The image is identical across sites; the arguments and the workload SA are per-site. Bring up the aggregator first; then each site\'s client starts and connects.

The per-site data path

The training container reads from the site\'s local de-identified OMOP cohort table (the M8L3 + M9L3 patterns produced these). The data never leaves the site. The site\'s workload SA reads BigQuery via Workload Identity — same M3L3 + M5L1 patterns.

Configuration via Helm

Per M5L2 — the per-site client Deployment is a Helm chart shipped from the shared lab repo. Each site customizes with a small values file:

# values-uh.yaml
site:
  id: uh
  dataSource: bq://srl-pacificfed-uh.omop_deid.training_cohort_v3
study:
  id: halo-hcc-fl-v3
  aggregatorHost: 10.0.0.10:8080
  rounds: 100
serviceAccount:
  gcpServiceAccount: pf-client-uh-sa@srl-pacificfed-uh.iam.gserviceaccount.com
helm install pf-client-halo-hcc charts/pf-client/ \
  --namespace=pf-uh --create-namespace \
  --values=values-uh.yaml

One chart; per-site values; clean reproduction at every site.

Per-round discipline

Model versioning and observability across rounds

A 100-round FL run produces 100 intermediate global models, each one a checkpoint that someone might want to roll back to or analyze later. Treat each round\'s output the same as a Vertex training job\'s output: register, version, log metrics.

The per-round flow

# Inside the aggregator: end of each round
def on_round_end(round_num, aggregated_weights, metrics):
    # 1. Save the global model snapshot
    save_weights_to_gcs(aggregated_weights, f"gs://srl-pacificfed-infra/runs/halo-hcc-fl-v3/round-{round_num}.pt")

    # 2. Log metrics to Vertex Experiments
    aiplatform.log_metrics({
        "global_loss": metrics["loss"],
        "global_auroc": metrics["auroc"],
        "min_site_loss": min(m["loss"] for m in per_site_metrics),
        "max_site_loss": max(m["loss"] for m in per_site_metrics),
        "participating_sites": len(per_site_metrics),
    }, step=round_num)

    # 3. Register every Nth round as a Vertex Model version
    if round_num % 10 == 0 or round_num == final_round:
        register_in_model_registry(
            display_name="halo-hcc-fl",
            artifact_uri=f"gs://srl-pacificfed-infra/runs/halo-hcc-fl-v3/round-{round_num}.pt",
            version_aliases=[f"v3-round-{round_num}"] + (["candidate"] if round_num == final_round else []),
        )

The per-site audit

Each site\'s client logs its own per-round events to Cloud Logging — round number received, local training start/end, gradient upload completed. Combined with the aggregator\'s round-level logs, the full federated run is reconstructable forensically.

# From any site\'s perspective, the audit trail is:
[round 12] received broadcast model from aggregator (sha256=$HASH)
[round 12] local training started, dataset_size=4127, GPU L4
[round 12] local training completed in 312s, local loss=0.42 → 0.38
[round 12] gradient update uploaded to aggregator (sha256=$HASH)
[round 13] received broadcast model from aggregator (sha256=$HASH)
...

The dashboard the lab actually reads

Per M4L3\'s SLO patterns — a Cloud Monitoring dashboard per active FL study, showing:

  • → Global metric trajectory (loss / AUROC across rounds) — convergence sanity check.
  • → Per-site participation rate (how often each site checked in vs how often it should have).
  • → Spread of per-site losses each round (large spread = statistical heterogeneity warning).
  • → Round duration (mean + worst) — flags when one slow site is dragging the whole run.
The discipline payoff. A converged FL run with the above instrumentation is rerunnable end-to-end, traceable per round, comparable across iterations, and registerable as a Vertex Model that downstream serving (M6L3) picks up automatically. The complexity of FL is contained by treating each round as a small, well-instrumented unit of work.
Hands-on exercise 10.2

Deploy a 2-cluster Flower setup on GKE

⚙️ Two clusters, one peering, one FL run
  1. Create two GKE Autopilot clusters — one as "aggregator," one as "site" (in the same project for the exercise; in real PacificFed they\'d be in different projects):
    gcloud container clusters create-auto demo-agg --location=us-west1 --network=demo-vpc --subnetwork=demo-subnet
    gcloud container clusters create-auto demo-site --location=us-west2 --network=demo-vpc --subnetwork=demo-subnet-us-west2
    (Same VPC for simplicity. Add a us-west2 subnet first if you don\'t have one.)
  2. Get credentials for both clusters; use kubectl contexts to switch:
    gcloud container clusters get-credentials demo-agg --location=us-west1
    kubectl config rename-context $(kubectl config current-context) demo-agg
    gcloud container clusters get-credentials demo-site --location=us-west2
    kubectl config rename-context $(kubectl config current-context) demo-site
  3. Build a tiny Flower image. Take the fl_demo.py from M10L1, split it into a server script and a client script, package each as a small Docker image, push to Artifact Registry.
  4. Deploy the aggregator on demo-agg:
    kubectl --context demo-agg create namespace fl-demo
    cat <<EOF | kubectl --context demo-agg apply -f -
apiVersion: apps/v1
kind: Deployment
metadata: {name: fl-server, namespace: fl-demo}
spec:
  replicas: 1
  selector: {matchLabels: {app: fl-server}}
  template:
    metadata: {labels: {app: fl-server}}
    spec:
      containers:
        - name: server
          image: $IMG_SERVER
          ports: [{containerPort: 8080}]
---
apiVersion: v1
kind: Service
metadata:
  name: fl-server
  namespace: fl-demo
  annotations: {networking.gke.io/load-balancer-type: "Internal"}
spec:
  type: LoadBalancer
  selector: {app: fl-server}
  ports: [{port: 8080, targetPort: 8080, protocol: TCP}]
EOF
    Get the assigned internal LB IP:
    kubectl --context demo-agg get svc fl-server -n fl-demo
    Note EXTERNAL-IP — that\'s the internal LB IP your site cluster can reach.
  5. Deploy the client on demo-site, pointing at the aggregator\'s LB IP:
    kubectl --context demo-site create namespace fl-demo
    cat <<EOF | kubectl --context demo-site apply -f -
apiVersion: apps/v1
kind: Deployment
metadata: {name: fl-client, namespace: fl-demo}
spec:
  replicas: 1
  selector: {matchLabels: {app: fl-client}}
  template:
    metadata: {labels: {app: fl-client}}
    spec:
      containers:
        - name: client
          image: $IMG_CLIENT
          env: [{name: SERVER, value: "AGGREGATOR_LB_IP:8080"}]
EOF
  6. Watch the FL run unfold:
    kubectl --context demo-agg logs deployment/fl-server -n fl-demo -f
    (In another shell) kubectl --context demo-site logs deployment/fl-client -n fl-demo -f
  7. Bonus — add a second client cluster (us-central1?), peer it in (if you split into separate VPCs), and watch the aggregator wait for both clients before each round.
  8. Tear down:
    gcloud container clusters delete demo-agg --location=us-west1
    gcloud container clusters delete demo-site --location=us-west2
