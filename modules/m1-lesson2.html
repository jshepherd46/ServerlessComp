Lesson 2 of 4

Designing your
project topology

Lesson 1 gave you the four-level tree. This lesson tells you how to actually fill it in for a real research lab. The blueprint we land on here is what HIPIMR, Makawalu, HALO, ShepBERT, and PacificFed actually run on.

⏱ ~40 min reading + exercise 🔧 gcloud CLI, IAM & Resource Manager 📍 Module 1 of 12
Core principle

One project per study, per environment

A project is the smallest unit on GCP where IAM, billing, quotas, APIs, and VPCs live independently. Sharing a project across two studies means sharing all of those — which is almost never what you want. The default rule is straightforward:

Default: one project per (study × environment). HIPIMR-prod, HIPIMR-dev, Makawalu-prod, Makawalu-dev. Separate everything you can separate, because separation is free.

Four things follow automatically when you do this:

💥
Blast radius is contained
A misconfigured IAM binding in Makawalu-dev cannot expose HIPIMR data. A runaway training job in HALO-staging cannot exhaust HIPIMR's quota. Failures stay where they happen.
💰
Billing maps to grants
Link each project to the right billing account. Monthly invoices come out per-grant with no spreadsheets — your post-award team will thank you.
🪪
Access stays scoped
A research coordinator on HALO doesn't accidentally see HIPIMR DICOMs. You grant roles on the project, not on individual resources, and the boundary holds.
⏳
Lifecycle is clean
When a grant ends, you delete the project. One command shuts down every resource, cancels every API, voids every key. There is no archaeology.
The exception worth knowing now: when something is genuinely shared across studies — your container registry, your audit log sink, your OMOP vocabulary — it gets its own dedicated "shared services" project. We come back to this pattern in Section 4.
Three axes of decomposition

Lab, study, environment

Every project in a research org sits at the intersection of three things: which lab owns it, which study it serves, and which environment it represents. Get those three right and the topology designs itself.

Why a folder, not just a project?
Folders carry org policies, lab-wide IAM (every SRL postdoc gets Viewer on the lab folder), and a billing-export sink. Projects can't host those at scale.
Who owns it
The PI and one or two delegated cloud admins. Folder admin is rare and powerful.
Sits under
The uhcc folder, which sits under the hawaii.edu organization.
Why not nest deeper?
You can — UHCC → SRL → Active Grants → R01s — but nest only when you have a concrete reason. Deep trees become hard to reason about during an audit.
What counts as a study
A funded grant, a defined cohort, or a deliverable with its own IRB protocol and PI. If the work has a unique compliance posture, it's a study.
Lifetime
As long as the grant. When the grant closes, the project closes — but a final archive can land in the lab's archival GCS bucket first.
Sub-projects
A large study can have multiple projects (data project + compute project) — but that's an optimisation, not the default.
Multi-PI studies
Pick the lead institution's lab folder. Other PIs get IAM as principals; the project still belongs to one folder.
-dev
Synthetic data only. No BAA needed. Permissive IAM. Where you break things.
-staging
De-identified subset of prod. BAA-covered. Mirror of prod's network and IAM. Where you rehearse releases.
-prod
Real PHI. Inside the VPC SC perimeter. Tightest IAM. Audit logs on everything.
When to skip an env
Small or short studies often skip -staging and run only -dev + -prod. Never skip -dev.
Naming is architecture

A scheme your scripts can parse

Project IDs and resource names are forever. You cannot rename a project. You cannot rename a bucket. Get the scheme right on day one and every script you write for the next five years becomes shorter.

The pattern

<lab>-<study>-<env>          # projects
<lab>-<study>-<purpose>-<env> # resources: buckets, datasets, secrets
srl-hipimr-prod
Project ID. Lab = SRL, study = HIPIMR, env = prod.
srl-makawalu-dev
Project ID. Iteration environment for Makawalu.
srl-hipimr-dicom-prod
GCS bucket for HIPIMR production DICOMs. Globally unique, recognisable at a glance.
srl-halo-omop-prod
BigQuery dataset (or project) for HALO production OMOP CDM.
srl-shared-registry
Shared Artifact Registry project — explicitly not tied to a single study.
The boring constraints to respect: project IDs are 6–30 characters, lowercase, digits and hyphens only, must start with a letter, and are globally unique across all of GCP. Pick the lab prefix wisely — it lives forever in every URL, every log line, every IAM binding.

Labels — the metadata layer

Project IDs encode the shape; labels encode the metadata. Apply a label set to every project so billing exports and tooling can slice cleanly:

gcloud projects update srl-halo-prod \
  --update-labels=lab=srl,study=halo,env=prod,grant=r01ca999999,phi=yes,pi=jshepherd

The phi=yes label is the one you'll lean on most — it's how the BigQuery billing export answers "show me my PHI footprint and what it cost last quarter."

The exception to the rule

The shared-services project

Some things genuinely belong to the lab, not a study. Put them in a single dedicated project so every study consumes them instead of re-creating them.

📦 Artifact Registry
One Artifact Registry for all your container images. Every study's Cloud Run / GKE pulls from it. Don't sprawl registries.
🪵 Audit log sink
A single BigQuery dataset that receives audit logs from every lab project. One query gives you a lab-wide answer.
🧬 OMOP vocabularies
The OHDSI standard vocabulary is multi-GB and identical across studies. Host it once; HALO, ShepBERT, and PacificFed all reference the same dataset.
🔐 KMS keyring
A lab-level Cloud KMS keyring holds CMEK keys you reuse across studies. Audit posture stays consistent.

Convention: name it srl-shared-services (or srl-shared-registry if you want a separate project per shared concern — fine, just don't go further than that).

The test: if removing this resource would break more than one study, it belongs in shared services. If removing it only affects one study, it lives in that study's project.
Putting it together

SRL's actual topology

Here is what every principle from this lesson, applied to the Shepherd Research Lab portfolio, looks like in one tree:

🏛 hawaii.edu # organization └── 📁 uhcc # UH Cancer Center folder └── 📁 shepherd-research-lab # SRL folder — lab-wide IAM & org policies ├── 📦 srl-shared-services # registry, audit sink, OMOP vocab, KMS │ ├── 📦 srl-hipimr-dev # synthetic DICOMs, no BAA ├── 📦 srl-hipimr-prod # real PHI, VPC SC perimeter │ ├── 📦 srl-makawalu-dev # synthetic CBM training data ├── 📦 srl-makawalu-staging # de-id Queen's data, BAA-covered ├── 📦 srl-makawalu-prod # live POCUS XAI serving │ ├── 📦 srl-halo-dev # synthetic OMOP ├── 📦 srl-halo-prod # Queen's HCC cohort, OMOP on BigQuery │ ├── 📦 srl-shepbert-dev # LoRA fine-tunes (Pavel) ├── 📦 srl-shepbert-prod # clinical NLP API on GKE │ └── 📦 srl-pacificfed-infra # FL aggregator, GKE multi-cluster
Why this works

Eleven projects, one lab folder, one shared services project

Folder-level IAM lets every SRL postdoc see all dev projects without explicit per-project bindings. PHI is contained in three specific projects, all inside a VPC SC perimeter you can audit in one place. When the HCC R01 ends, you delete two projects and the rest are unaffected.

What you avoided

The "one big lab project" anti-pattern

Mixing all five studies in srl-lab would mean: shared IAM (Pavel can see HIPIMR), shared quota (HALO trainings starve Makawalu), shared billing (no per-grant slice), single VPC SC perimeter (one mistake exposes everything). When NIH audits HALO, they see the whole lab.

Hands-on exercise 1.2

Design and create your own slice of the topology

📐 Design first, then build
  1. On paper, sketch the topology for one of your studies. Pick a real one (or use Makawalu as the worked example). Draw the folder, name every project, and label which holds PHI vs synthetic data.
  2. List the shared resources that would not belong to a single study — registry, OMOP vocab, key ring, audit sink. These go to srl-shared-services, not into the study projects.
  3. Switch to your active configuration from M0L1:
    gcloud config configurations activate srl-sandbox
  4. Create three projects that match the pattern — a fake study with dev, staging, and prod:
    gcloud projects create srl-demo-dev --name="SRL Demo (dev)"
    gcloud projects create srl-demo-staging --name="SRL Demo (staging)"
    gcloud projects create srl-demo-prod --name="SRL Demo (prod)"
  5. Apply labels to all three so billing exports later slice cleanly:
    gcloud projects update srl-demo-prod --update-labels=lab=srl,study=demo,env=prod,phi=no
  6. Link each to your billing account:
    gcloud billing projects link srl-demo-dev --billing-account=[BILLING_ACCOUNT_ID] (repeat for staging + prod)
  7. List your projects and confirm the three appear with the right labels:
    gcloud projects list --filter="labels.lab=srl"
  8. Cleanup discipline. When you're done practising, delete all three:
    gcloud projects delete srl-demo-dev (repeat — the 30-day grace period gives you a way out if you click too fast)
  9. Bonus, if you have a folder admin handy: create a real shepherd-research-lab folder under uhcc and move srl-demo-* into it. gcloud projects move srl-demo-dev --folder=[FOLDER_ID].

Commands you used or saw

gcloud projects create gcloud projects update --update-labels gcloud projects list --filter gcloud billing projects link gcloud projects move gcloud projects delete gcloud resource-manager folders list
