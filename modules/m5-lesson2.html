Deployments at production grade

Strategy, probes, requests

A bare Deployment works in dev. A production-grade Deployment specifies how rolling updates behave, how Kubernetes decides a Pod is ready, and what resources each Pod commits to. Three additions to every prod manifest.

1. Rolling update strategy

apiVersion: apps/v1
kind: Deployment
metadata: {name: xai-serve}
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1          # at most 1 extra Pod during rollout
      maxUnavailable: 0    # zero downtime — every old Pod stays up until its replacement is Ready
  selector: {matchLabels: {app: xai-serve}}
  template:
    ...

maxUnavailable: 0 is the safe default for SLO-bound services. maxSurge: 1 trades a small temporary capacity bump for zero-downtime rollouts. For Pods with large startup costs (model weights, JIT compile), the slow-but-safe rollout matters more than rollout speed.

2. Three probes, distinct purposes

startupProbe
"Is the Pod alive yet?" Runs only at startup. Buys time for slow initialisations — a 4 GB model load, a JIT warmup. While the startup probe is running, the other probes don't apply. Configure failureThreshold high enough to cover your worst-case startup.
readinessProbe
"Should the Service route traffic here?" Runs throughout the Pod's life. If it fails, the Pod is removed from the Service's endpoint list — no traffic, no restart. Use to drain in-flight work during graceful shutdown, or to back off when a dependency is down.
livenessProbe
"Is the Pod stuck and needs killing?" If it fails, Kubernetes restarts the container. Use sparingly — a misconfigured liveness probe causes restart loops. Most services need readiness + startup, no liveness.
    spec:
      containers:
        - name: app
          image: $IMG
          ports: [{containerPort: 8080}]
          startupProbe:
            httpGet: {path: /healthz/startup, port: 8080}
            periodSeconds: 5
            failureThreshold: 36     # 3 minutes max startup
          readinessProbe:
            httpGet: {path: /healthz/ready, port: 8080}
            periodSeconds: 5
            failureThreshold: 3

3. Resource requests and limits

          resources:
            requests:
              memory: 2Gi
              cpu: 1000m            # 1 vCPU
              ephemeral-storage: 1Gi
            limits:
              memory: 4Gi
              cpu: 2000m
              ephemeral-storage: 2Gi

Requests tell GKE what to schedule; in Autopilot, that's also what you're billed for. Limits are hard ceilings — a Pod exceeding memory limit gets OOM-killed. Set requests honestly (autoscaling decisions depend on them); set limits with headroom for spikes.

Batch on GKE

Jobs and CronJobs

Cloud Run Jobs (M2L3 + M4) handle most batch up to the 24-hour ceiling. On GKE, two analogous objects extend that further and add parallelism patterns Cloud Run doesn't have.

Job — run-to-completion

apiVersion: batch/v1
kind: Job
metadata: {name: density-cnn-train}
spec:
  backoffLimit: 2          # retry the Pod twice on failure before giving up
  activeDeadlineSeconds: 28800  # 8h hard timeout
  template:
    spec:
      serviceAccountName: train-job-ksa
      restartPolicy: OnFailure
      containers:
        - name: trainer
          image: $IMG
          command: ["python", "train.py"]
          resources:
            requests: {memory: 16Gi, cpu: 4000m}
            limits:   {memory: 16Gi, cpu: 4000m}

The Pod runs, exits 0 on success, exits non-zero on failure. Kubernetes retries up to backoffLimit times. activeDeadlineSeconds kills the whole Job if it exceeds the limit — protect against runaway training.

Parallel Job — fan-out batch

spec:
  parallelism: 8       # up to 8 Pods running concurrently
  completions: 100     # run 100 successful task instances total
  template: ...

Useful for embarrassingly parallel batch — process 100 DICOMs in 8 parallel workers. Each Pod picks up work via an external queue (Pub/Sub, Firestore, BigQuery). Kubernetes manages the fleet; your code handles work distribution.

CronJob — Job on a schedule

apiVersion: batch/v1
kind: CronJob
metadata: {name: nightly-ingest}
spec:
  schedule: "0 2 * * *"          # 02:00 daily, cluster's timezone
  timeZone: "Pacific/Honolulu"
  concurrencyPolicy: Forbid     # don't start a new run if the previous is still going
  successfulJobsHistoryLimit: 5
  failedJobsHistoryLimit: 5
  jobTemplate:
    spec:
      template:
        spec:
          serviceAccountName: ingest-job-ksa
          restartPolicy: OnFailure
          containers:
            - name: ingest
              image: $IMG
CronJob vs Cloud Scheduler → Cloud Run Job. Both work. CronJob is in-cluster, sees cluster identity, can mount Persistent Volumes. Cloud Scheduler → Cloud Run Job is serverless, scales to zero, no cluster cost. Default to Cloud Scheduler + Cloud Run Job; pick CronJob when the work needs cluster-local state (mounted volume, in-cluster networking to another Pod).
Configuration in the cluster

ConfigMaps, Secrets, and Workload-Identity wiring

M3L3 said: every secret lives in Secret Manager, never in code. That discipline holds on GKE — but Kubernetes has its own concept of Secrets, and you need to know how to bridge them without duplicating the secret material.

ConfigMaps — non-sensitive config

# Plain config — endpoint URLs, feature flags, model parameters
apiVersion: v1
kind: ConfigMap
metadata: {name: xai-config}
data:
  MODEL_PATH: "/models/v3"
  INFERENCE_BATCH_SIZE: "4"
  LOG_LEVEL: "INFO"
---
spec:
  template:
    spec:
      containers:
        - name: app
          envFrom:
            - configMapRef: {name: xai-config}

All key-value pairs appear as environment variables in the container. Or mount as files:

          volumeMounts:
            - name: config-vol
              mountPath: /etc/xai
      volumes:
        - name: config-vol
          configMap: {name: xai-config}

The Secret Manager pattern — don't duplicate secrets into the cluster

You could create Kubernetes Secrets manually with the same values as your Secret Manager secrets. Don't. Two stores of the same secret means two rotation paths, two audit logs, and inevitable drift. Instead: have your application read directly from Secret Manager at startup, using the Pod's Workload Identity bound to a GSA with roles/secretmanager.secretAccessor.

# In your app — Python example. No env vars, no mounted files.
from google.cloud import secretmanager
client = secretmanager.SecretManagerServiceClient()
resp = client.access_secret_version(
    request={"name": "projects/srl-makawalu-prod/secrets/xai-api-key/versions/latest"}
)
api_key = resp.payload.data.decode()

The KSA from M5L1's pattern maps to a GSA that holds the accessor role. The Python client picks up Workload Identity credentials with zero config.

When you DO need Kubernetes Secrets — External Secrets Operator

Some workloads can't easily call Secret Manager directly — third-party Helm charts, off-the-shelf services like ATLAS WebAPI, anything that expects a file at /etc/credentials. The clean answer: the External Secrets Operator (ESO) syncs Secret Manager into Kubernetes Secrets automatically.

apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata: {name: xai-api-key, namespace: makawalu}
spec:
  refreshInterval: 1h
  secretStoreRef: {name: gcp-secret-store, kind: ClusterSecretStore}
  target:
    name: xai-api-key       # the resulting Kubernetes Secret
  data:
    - secretKey: API_KEY
      remoteRef: {key: xai-api-key-prod}

ESO re-fetches every hour; you rotate in Secret Manager and the in-cluster Secret updates automatically. The Secret Manager remains the source of truth.

Templates and reuse

Helm — your lab's reusable chart

By the third study, you'll have copy-pasted the same Deployment + Service + HPA YAML five times. Helm is the package manager that templates these manifests so each study customises with a small values file instead of duplicating hundreds of lines.

Anatomy of a chart

srl-app/ ├── Chart.yaml # chart metadata + version ├── values.yaml # default values — overridable └── templates/ ├── deployment.yaml # Pod template, with {{ .Values.* }} substitutions ├── service.yaml ├── hpa.yaml # HorizontalPodAutoscaler ├── serviceaccount.yaml # KSA with Workload Identity annotation └── configmap.yaml

A minimal values.yaml for SRL apps

# srl-app/values.yaml — defaults; each install overrides what it needs
image:
  repository: us-west1-docker.pkg.dev/srl-shared-services/research
  name: ""        # required override per app
  tag: latest

replicas: 2

resources:
  requests: {memory: 512Mi, cpu: 250m}
  limits:   {memory: 1Gi, cpu: 500m}

serviceAccount:
  create: true
  gcpServiceAccount: ""  # required override

autoscaling:
  enabled: true
  minReplicas: 2
  maxReplicas: 10
  targetCPUUtilizationPercentage: 70

ingress:
  enabled: false

Installing the chart per study

# Install for Makawalu — override image and SA only
helm install xai-serve ./srl-app \
  --namespace=makawalu \
  --create-namespace \
  --set image.name=xai-serve \
  --set image.tag=0.3.0 \
  --set serviceAccount.gcpServiceAccount=xai-serve-sa@srl-makawalu-prod.iam.gserviceaccount.com \
  --set replicas=3

# Same chart, different study, different values
helm install shepbert-api ./srl-app \
  --namespace=shepbert \
  --create-namespace \
  --set image.name=shepbert-api \
  --set image.tag=0.5.1 \
  --set serviceAccount.gcpServiceAccount=shepbert-api-sa@srl-shepbert-prod.iam.gserviceaccount.com \
  --set resources.requests.memory=4Gi

# Upgrade — change image tag, helm re-templates and re-applies
helm upgrade xai-serve ./srl-app --namespace=makawalu --reuse-values \
  --set image.tag=0.3.1

# Rollback — same idea as kubectl rollout undo, at the chart level
helm rollback xai-serve 1 --namespace=makawalu
The discipline this enables. One srl-app chart in your lab repo, owned by you, reviewed by you, versioned with the rest of the runbook. Every study's deploy is a values file + one helm command. Drift between studies disappears; the next study setup is 15 minutes instead of half a day.
Multi-study isolation

The namespace strategy

From M1L2: one project per study. Inside a multi-tenant cluster (the case when you share a GKE cluster across studies — common for sandbox or pre-prod), the equivalent at the cluster level is one namespace per study. Namespaces carry RBAC, ResourceQuotas, and NetworkPolicies, giving each study its own slice of the cluster without sharing identity or capacity.

📦 GKE cluster: srl-research-cluster ├── 📁 namespace: makawalu │ ├── RoleBinding: srl-makawalu-leads@ → edit │ ├── ResourceQuota: cpu=20, memory=80Gi, gpu=2 │ ├── NetworkPolicy: deny-all + selective allow │ └── Deployment/xai-serve, Job/density-cnn-train ├── 📁 namespace: shepbert │ ├── RoleBinding: srl-shepbert-leads@ → edit │ ├── ResourceQuota: cpu=10, memory=40Gi │ └── Deployment/shepbert-api └── 📁 namespace: kube-system # GKE-managed; off-limits

ResourceQuota — caps a namespace's slice

apiVersion: v1
kind: ResourceQuota
metadata: {name: makawalu-quota, namespace: makawalu}
spec:
  hard:
    requests.cpu: "20"
    requests.memory: 80Gi
    requests.nvidia.com/gpu: "2"
    persistentvolumeclaims: "5"
    pods: "50"

Quotas prevent one study from starving another. In a multi-tenant cluster they're not optional — set them per namespace at creation.

NetworkPolicy — default-deny inside the cluster

# Inside makawalu namespace: deny all cross-namespace traffic by default
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: {name: deny-other-namespaces, namespace: makawalu}
spec:
  podSelector: {}
  policyTypes: [Ingress]
  ingress:
    - from:
        - podSelector: {}        # Pods in the same namespace

Then allow specific cross-namespace traffic only where needed (shared OMOP vocabulary in shared-services namespace, say). The default-deny posture matches the M3 perimeter discipline applied at the in-cluster level.

Project-per-study still wins for prod. Multi-namespace-in-one-cluster is a fine pattern for dev / pre-prod (shared cluster, contained namespaces). For prod PHI workloads, stay with project-per-study and one cluster per project — the VPC SC perimeter, billing, and audit story are cleaner than trying to do multi-tenancy at the namespace level.
Hands-on exercise 5.2

Deployment + Job + Helm + Workload Identity

🛠 Production-shaped workloads end-to-end
  1. Restart your Autopilot cluster from M5L1 if you tore it down:
    gcloud container clusters create-auto srl-demo-cluster --location=us-west1 --release-channel=regular --network=demo-vpc --subnetwork=demo-subnet
    gcloud container clusters get-credentials srl-demo-cluster --location=us-west1
  2. Create a namespace + quota for a hypothetical study:
    kubectl create namespace demo-study
    cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: ResourceQuota
metadata: {name: demo-quota, namespace: demo-study}
spec:
  hard: {requests.cpu: "4", requests.memory: 8Gi, pods: "10"}
EOF
  3. Workload Identity setup — create a GSA, KSA, and bind them. (You can skip granting GCP roles if the demo doesn't call GCP APIs.)
    gcloud iam service-accounts create demo-app-sa
    kubectl create serviceaccount demo-app-ksa -n demo-study
    kubectl annotate serviceaccount demo-app-ksa -n demo-study iam.gke.io/gcp-service-account=demo-app-sa@$PROJECT.iam.gserviceaccount.com
    gcloud iam service-accounts add-iam-policy-binding demo-app-sa@$PROJECT.iam.gserviceaccount.com --role=roles/iam.workloadIdentityUser --member="serviceAccount:$PROJECT.svc.id.goog[demo-study/demo-app-ksa]"
  4. Build a minimal Helm chart for hello-flask. From a fresh directory:
    helm create srl-app
    Edit srl-app/values.yaml to set image repo + name + tag for your hello-flask. Edit srl-app/templates/deployment.yaml to add serviceAccountName: {{ .Values.serviceAccount.name }} if not already there. (The default helm create output is close; one small tweak.)
  5. Install with the chart:
    helm install demo-app ./srl-app -n demo-study --set image.repository=us-west1-docker.pkg.dev/$PROJECT/practice/hello-flask --set image.tag=0.2.0 --set serviceAccount.create=false --set serviceAccount.name=demo-app-ksa
    Confirm: kubectl get pods -n demo-study.
  6. Create a CronJob that runs every minute and prints a hello (saves cronjob.yaml): 
    apiVersion: batch/v1
kind: CronJob
metadata: {name: hello-cron, namespace: demo-study}
spec:
  schedule: "*/1 * * * *"
  concurrencyPolicy: Forbid
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
            - name: hello
              image: busybox
              args: ["sh", "-c", "echo hello from $(date)"]
    kubectl apply -f cronjob.yaml. Wait a couple minutes; check executions: kubectl get jobs -n demo-study, then logs: kubectl logs job/<jobname> -n demo-study.
  7. Practice helm upgrade: change the replicas value:
    helm upgrade demo-app ./srl-app -n demo-study --reuse-values --set replicas=3
    Watch the rolling update happen: kubectl rollout status deployment/demo-app -n demo-study.
  8. Tear down at the end:
    helm uninstall demo-app -n demo-study
    kubectl delete cronjob hello-cron -n demo-study
    kubectl delete namespace demo-study
    Then delete the cluster if you're done for the session.

Knowledge check

Questions to sit with

  • You set a livenessProbe on a service that takes 90 seconds to load a model. Pods start restarting in a loop. What's wrong, and what's the right shape? Hint
  • Your team duplicated the Secret Manager value into a Kubernetes Secret "to make it easier to mount." A month later, the in-cluster Secret is stale because someone rotated only the Secret Manager copy. What's the durable fix? Hint
  • You\'re sharing a dev cluster across three studies. Why is one namespace per study with quotas + RBAC the right partition rather than three clusters? Hint
  • You install srl-app chart for Makawalu and ShepBERT. Both deploys go fine, but ShepBERT pods can\'t pull the image. Where do you check first? Hint
