GKE fundamentals
and your first cluster

M4L3 told you when to escalate from Cloud Run to GKE. This lesson is the escalation. Same image, same service account, same secrets — different runtime. By the end you'll have hello-flask running on an Autopilot cluster, and the Kubernetes vocabulary every later lesson assumes.

⏱ ~55 min reading + exercise 🔧 GKE Autopilot, kubectl, Workload Identity 📍 Module 5 of 12
The shift

From Cloud Run to GKE — what actually changes

The image is the same. The container contract from M2L3 (listen on the port, handle SIGTERM, be stateless, log to stdout) is the same. What changes is the surface area you control and the deploy interface.

Cloud Run
GKE
Unit of deploy
A service (one spec). gcloud run deploy.
A Deployment, plus a Service, plus a HorizontalPodAutoscaler, in YAML. kubectl apply.
Identity
Attached service account on the service.
Workload Identity binds a Kubernetes ServiceAccount to a GCP service account.
Networking
HTTPS endpoint provisioned for you.
You author Service + Ingress (or Gateway) objects. More flexible; more responsibility.
Scaling
Request-based autoscaler, built in.
HPA on CPU / memory / custom metrics. Cluster autoscaler manages nodes (Standard) or Google manages them (Autopilot).
Multi-container
Sidecars are recent and constrained.
First-class: Pods naturally hold multiple containers.
State
Ephemeral filesystem only.
PersistentVolumeClaim, ConfigMap, Secret, emptyDir, downwardAPI — many shapes.
Operational cost
Near-zero between requests with scale-to-zero.
Cluster has a baseline cost; pods can scale to 0 replicas but cluster nodes persist.
The trade you're making. Cloud Run hides Kubernetes from you for the 80% case. GKE exposes it for the 20% where you need it. The skills from M2 + M4 do not get thrown out — they get extended. Same image, same SA, same secrets discipline; new vocabulary for the runtime that surrounds them.
The vocabulary

Five Kubernetes primitives carry almost everything

You will hear about Kubernetes objects in the dozens. For a research lab you need to be fluent in five. The rest are variations.

apiVersion: v1 · kind: Pod

Pod — one or more containers sharing a network and storage

The smallest deployable unit. Pods are ephemeral — they die and get replaced. You almost never create Pods directly; controllers like Deployment make them for you.

apiVersion: apps/v1 · kind: Deployment

Deployment — a managed set of Pod replicas

Declares "I want N copies of this Pod template running." Handles rolling updates, rollback, and replacing failed Pods. The workhorse object for long-running services. Cloud Run's equivalent is a service.

apiVersion: v1 · kind: Service

Service — a stable network identity in front of Pods

Pods come and go; their IPs change. A Service has a stable DNS name (xai-serve.default.svc.cluster.local) and load-balances across the Pods that match its selector. Three types you'll use: ClusterIP (internal-only), LoadBalancer (external IP), and ExternalName (DNS alias).

apiVersion: v1 · kind: Namespace

Namespace — a logical partition inside the cluster

Used for multi-tenancy. Per M1L2: one namespace per study. RBAC, quotas, and network policies attach at the namespace level. kubectl get pods -n makawalu only shows that study's Pods.

apiVersion: v1 · kind: ConfigMap / Secret

ConfigMap and Secret — configuration mounted into Pods

ConfigMap holds plain config (env vars, YAML, config files); Secret holds sensitive values. Both get mounted as environment variables or files. M5L2 shows the pattern that wires Secret Manager (M3L3) into Kubernetes Secrets via Workload Identity.

A complete deployment in 30 lines

# hello-flask.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: hello-flask
  namespace: default
spec:
  replicas: 2
  selector:
    matchLabels: {app: hello-flask}
  template:
    metadata:
      labels: {app: hello-flask}
    spec:
      serviceAccountName: hello-flask-ksa     # bound via Workload Identity
      containers:
        - name: app
          image: us-west1-docker.pkg.dev/srl-shared-services/research/hello-flask:0.2.0
          ports: [{containerPort: 8080}]
          resources:
            requests: {memory: 256Mi, cpu: 250m}
            limits:   {memory: 512Mi, cpu: 500m}
---
apiVersion: v1
kind: Service
metadata:
  name: hello-flask
spec:
  type: ClusterIP
  selector: {app: hello-flask}
  ports:
    - port: 80
      targetPort: 8080

Apply once: kubectl apply -f hello-flask.yaml. The cluster reconciles toward the desired state — creates two Pods, exposes the Service, keeps them alive. Replace the image tag, re-apply, and it does a rolling update automatically.

The mode decision

Autopilot vs Standard

GKE has two operating modes. Pick once per cluster; you cannot convert later. The decision shapes ~80% of your cluster operational experience.

Default for SRL

GKE Autopilot

Google manages nodes. You declare Pods with resource requests; Autopilot provisions compute, handles upgrades, right-sizes nodes, deletes idle capacity. You only see Pods — never nodes.

Per-Pod pricing model (CPU-seconds + memory-seconds + ephemeral storage). No node-pool minimums.

Use for: 95% of research workloads — APIs, ingest jobs, training, ATLAS deployments.
Specialist mode

GKE Standard

You manage nodes. Multiple node pools (CPU pool, GPU pool, spot pool), per-pool configuration of machine types, autoscaling parameters, OS images. More control; more knobs.

Per-node pricing — you pay for nodes whether Pods fill them or not.

Use for: very specific hardware needs (Trillium TPUs, custom kernels), tight cost control on large stable workloads, multi-tenant cluster shared across many studies.

When Autopilot is genuinely wrong

  • → You need a node-level DaemonSet for something privileged (security agent, custom CNI) that Autopilot blocks.
  • → You want to mix Spot VMs and standard nodes with very specific ratios.
  • → Your workload uses node-local SSDs or specific machine families Autopilot doesn't currently support.
  • → You're running a multi-tenant shared cluster where per-node cost predictability matters more than the operational simplicity.

None of these apply to a typical research lab's first cluster. Start with Autopilot. Only move to Standard when a concrete constraint forces it.

What this means in practice. For the rest of this module, every cluster command targets Autopilot. The YAML is identical between modes — Pod specs are portable. Only the cluster-level creation differs, and Autopilot is two flags.
Identity inside the cluster

RBAC and Workload Identity — the bridge from M1L3

Two identity systems intersect in GKE. RBAC governs who can do what inside the cluster (kubectl operations, pod-level actions). GCP IAM governs what cluster workloads can do on GCP services outside the cluster. Workload Identity is the bridge between them.

RBAC — humans and tools inside the cluster

# A namespace-scoped role for postdocs in Makawalu's namespace
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: makawalu
  name: postdoc-edit
rules:
  - apiGroups: ["", "apps", "batch"]
    resources: ["pods", "pods/log", "deployments", "jobs", "configmaps"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  namespace: makawalu
  name: postdocs-bind
subjects:
  - kind: Group
    name: srl-postdocs@hawaii.edu          # Google Group from M1L3
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: postdoc-edit
  apiGroup: rbac.authorization.k8s.io

The Group binding works because GKE integrates with Google Groups out of the box — exactly the M1L3 pattern of binding to groups, not individuals.

Workload Identity — pods talking to GCP services

A Kubernetes ServiceAccount (KSA) binds to a Google Service Account (GSA). Pods that serviceAccountName: my-ksa automatically authenticate as the bound GSA when calling GCP APIs — no key files, audited per pod.

Pod spec: serviceAccountName: xai-serve-ksa # Kubernetes ServiceAccount │ │ Workload Identity binding (annotation on KSA │ + iam.workloadIdentityUser binding on GSA) ▼ xai-serve-sa@srl-makawalu-prod.iam.gserviceaccount.com # GCP SA │ ▼ roles/secretmanager.secretAccessor # M1L3 patterns roles/storage.objectViewer (inference bucket) roles/cloudkms.cryptoKeyEncrypterDecrypter (M3L3)

The patterns from M1L3 (one SA per workload) and M3L3 (CMEK + secret accessor scoped narrowly) hold without modification. The only addition is the KSA→GSA binding step.

Setting up the bridge

# 1. Create the GSA per workload (M1L3 pattern)
gcloud iam service-accounts create xai-serve-sa --project=srl-makawalu-prod

# 2. Create the KSA in the cluster
kubectl create serviceaccount xai-serve-ksa --namespace=makawalu

# 3. Annotate the KSA with the GSA email (tells GKE which GSA to assume)
kubectl annotate serviceaccount xai-serve-ksa \
  --namespace=makawalu \
  iam.gke.io/gcp-service-account=xai-serve-sa@srl-makawalu-prod.iam.gserviceaccount.com

# 4. Grant the KSA the right to impersonate the GSA
gcloud iam service-accounts add-iam-policy-binding \
  xai-serve-sa@srl-makawalu-prod.iam.gserviceaccount.com \
  --role=roles/iam.workloadIdentityUser \
  --member="serviceAccount:srl-makawalu-prod.svc.id.goog[makawalu/xai-serve-ksa]"
What this delivers. Inside the Pod, the Google Cloud SDK auto-detects Workload Identity. from google.cloud import storage; storage.Client() picks up the GSA's credentials with zero config. No key files, no env vars, no copy-pasted JSON. The same Python code that runs on Cloud Run runs unchanged on GKE.
Putting it together

Your first cluster: hello-flask on Autopilot

Create an Autopilot cluster (one command)

gcloud services enable container.googleapis.com

gcloud container clusters create-auto srl-demo-cluster \
  --location=us-west1 \
  --release-channel=regular \
  --enable-private-nodes \
  --enable-private-endpoint=false \
  --master-ipv4-cidr=172.16.0.0/28 \
  --network=demo-vpc \
  --subnetwork=demo-subnet

The flags worth noting: create-auto selects Autopilot. --enable-private-nodes gives every Pod no public IP (M3L2 discipline). --release-channel=regular opts in to auto-upgrades on a balanced cadence. Provisioning takes 3–5 minutes.

Wire kubectl to the cluster

gcloud container clusters get-credentials srl-demo-cluster \
  --location=us-west1

# Confirm kubectl can talk to the cluster
kubectl cluster-info
kubectl get nodes      # Autopilot: shows nodes but you don't manage them

Deploy hello-flask

# Save the YAML from Section 2 as hello-flask.yaml, then:
kubectl apply -f hello-flask.yaml

# Watch the rollout
kubectl get pods -l app=hello-flask -w

# Hit the service from inside the cluster
kubectl run curl-test --rm -it --image=curlimages/curl --restart=Never -- \
  curl -s http://hello-flask/

Two Pods come up; the Service load-balances between them. The Pods can pull the image from Artifact Registry because Autopilot nodes have the right IAM by default.

Expose it (carefully)

A ClusterIP Service is internal-only. To expose externally, you'd add an Ingress (or Gateway) — covered in M5L3. For now, port-forward for testing:

kubectl port-forward service/hello-flask 8080:80
# In another terminal:
curl http://localhost:8080/
What just happened. Same image as Cloud Run. Same identity model (KSA→GSA→narrow IAM). Same secrets approach. The cluster reconciles toward your declared state. Two Pods running; if one dies, GKE replaces it within seconds. This is the substrate every M5L2 + M5L3 capability builds on.
Hands-on exercise 5.1

Stand up Autopilot and migrate hello-flask

☸️ From Cloud Run to GKE in 30 minutes
  1. Use the VPC + subnet you built in M3L2's exercise (demo-vpc / demo-subnet). Recreate them if you tore them down.
  2. Enable the container API and create the Autopilot cluster:
    gcloud services enable container.googleapis.com
    gcloud container clusters create-auto srl-demo-cluster --location=us-west1 --release-channel=regular --network=demo-vpc --subnetwork=demo-subnet
    Wait 3-5 minutes.
  3. Wire kubectl:
    gcloud container clusters get-credentials srl-demo-cluster --location=us-west1
    kubectl cluster-info
  4. Save this as hello-flask.yaml (use your image from M2L2): 
    apiVersion: apps/v1
kind: Deployment
metadata: {name: hello-flask}
spec:
  replicas: 2
  selector: {matchLabels: {app: hello-flask}}
  template:
    metadata: {labels: {app: hello-flask}}
    spec:
      containers:
        - name: app
          image: us-west1-docker.pkg.dev/PROJECT_ID/practice/hello-flask:0.2.0
          ports: [{containerPort: 8080}]
          resources:
            requests: {memory: 256Mi, cpu: 250m}
---
apiVersion: v1
kind: Service
metadata: {name: hello-flask}
spec:
  type: ClusterIP
  selector: {app: hello-flask}
  ports: [{port: 80, targetPort: 8080}]
    Replace PROJECT_ID with yours. kubectl apply -f hello-flask.yaml.
  5. Watch Pods come up:
    kubectl get pods -l app=hello-flask -w
    Both should hit Running within ~30 seconds.
  6. Hit the service from inside the cluster:
    kubectl run curl-test --rm -it --image=curlimages/curl --restart=Never -- curl -s http://hello-flask/
  7. Set up Workload Identity for a per-workload KSA (skip the GSA binding for now — the demo service doesn't call GCP APIs):
    kubectl create serviceaccount hello-flask-ksa
    Edit the deployment YAML to add serviceAccountName: hello-flask-ksa under spec.template.spec. Re-apply. Confirm the new pods use the new KSA.
  8. Do a rolling update. Edit the deployment YAML to bump the image tag (or change replicas: 3). kubectl apply -f hello-flask.yaml. Watch:
    kubectl rollout status deployment/hello-flask
    Old pods are replaced one at a time, zero downtime.
  9. Practice rollback:
    kubectl rollout undo deployment/hello-flask
    Back to the previous spec. The revision system is conceptually like Cloud Run's.
  10. Tear down when you're done — clusters cost money even idle:
    gcloud container clusters delete srl-demo-cluster --location=us-west1
