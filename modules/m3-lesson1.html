Perimeter, BAA,
and org policy

M1 gave you a topology. M2 gave you containers. Neither has touched PHI yet. This lesson is the boundary you draw before any patient data lands: who signs what, where the wall is, and which org policies pin every project below it.

⏱ ~50 min reading + exercise 🔧 VPC SC, Org Policies, Access Context Manager 📍 Module 3 of 12
The framework

HIPAA on GCP is a three-way split

HIPAA compliance is not something Google sells you and it is not something you build from scratch. It is a shared responsibility across three parties — and each party's job is well-defined. Confusion about this is where most "are we compliant?" questions come from.

Google's responsibility

The infrastructure

  • Physical security of data centres
  • Default encryption at rest + in transit
  • Service availability + maintenance
  • HIPAA-eligible services list (the BAA covers these and only these)
  • Signed BAA with your institution
UH ITS / institution

The legal & org layer

  • Signs and maintains the BAA with Google
  • Enrols specific projects under the BAA
  • Approves IRB protocols and data-use agreements
  • Provides folder-level org policies for HIPAA tier
  • Breach notification process
You (the lab)

The configuration

  • Build the VPC SC perimeter around PHI projects
  • Apply CMEK + Secret Manager discipline
  • Enable Data Access logs on PHI services
  • Implement least-privilege IAM (M1L3)
  • Document everything in the runbook (M1L4)
  • Use only HIPAA-eligible services for PHI
Two things to internalise. First, a signed BAA without correctly configured projects gets you nothing — the BAA is the necessary condition for compliance, not the sufficient one. Second, the rest of this module is the right-hand column. We assume the BAA exists. We focus on what you control.
The institutional workflow

Enrolling a project under the BAA

Before any PHI lands in a project, that project must be enrolled under the institutional BAA. The mechanics vary by institution, but the workflow is roughly the same everywhere.

1
Create the project under the right folder
Per M1L2 — name it srl-<study>-prod, place it in the shepherd-research-lab folder, apply the phi=yes label.
You
2
Open a HIPAA-tier project request with UH ITS
Provide: project ID, study name, PI, IRB protocol number, expected data types (DICOM, FHIR, OMOP), data sources, retention requirements.
You → ITS
3
ITS enrols the project under the BAA
A small administrative action on Google's side. ITS confirms enrollment and may apply folder-level org policies you didn't put there.
ITS
4
Verify in the console
IAM & Admin → Compliance → Compliance reports → search "HIPAA." Your project should appear in the enrolled list. If not, do not load data.
You
5
Build the perimeter (rest of this lesson)
VPC SC, org policies, network isolation. Enrollment makes PHI allowed; your configuration makes it contained.
You
6
Apply CMEK + Data Access logs (M3L3)
The data-layer controls. Done before the first PHI write, not after.
You
7
Document in the lab runbook
Project ID, BAA enrollment date, IRB protocol, perimeter ID, the SA for ingest. M1L4's "launching a new study" section.
You
The timing matters. Steps 1–4 typically take 1–5 business days at UH. Build them into study timelines from the very start. The single biggest cause of "we couldn't process data this week" is forgetting that step 2 has a turnaround.
The perimeter

VPC Service Controls — the wall around PHI

VPC Service Controls draw a software-defined perimeter around a set of projects so their resources cannot be reached from outside the perimeter — even with valid IAM credentials. This is defence-in-depth on top of IAM. If a service account key leaks, the perimeter still holds.

A perimeter is a list of projects + a list of services that the perimeter applies to + rules for what can cross the wall.

What the perimeter looks like for SRL

SRL-PHI perimeter 📦 srl-hipimr-prod — DICOM + density CNN 📦 srl-makawalu-prod — POCUS XAI 📦 srl-halo-prod — OMOP on BigQuery 📦 srl-shared-services — registry, audit, KMS Restricted services: GCS, BigQuery, Healthcare API, AI Platform, KMS, Secret Manager, Pub/Sub, Cloud Run, Logging, Container Registry 🚫 Outside the perimeter — anything else, anywhere: a laptop with valid IAM, a personal project, another lab's perimeter, the public internet. A read from inside → inside: ALLOW A read from outside → inside: DENY (unless explicit ingress rule) A read from inside → outside: DENY (unless explicit egress rule)

Ingress and egress rules — the explicit holes

A perimeter that allows nothing through is also a perimeter that lets you do nothing. You poke holes with named rules, each scoped to a specific identity, resource, and operation.

Source = identity outside the perimeter (an SA in srl-shared-services-cicd). Destination = resource inside (Artifact Registry repo). Methods = the specific API calls. Without this rule, even legitimate CI pushes fail with VPC SC denied errors.

Source = identity inside the perimeter. Destination = resource outside. Methods = restricted. Egress rules are the most consequential — they are the only paths data can take out. Every egress rule should map to a documented IRB-approved data-use plan.

An access level is a named condition (IP range, device posture, identity, country) defined in Access Context Manager. Used in ingress rules to gate human access by context, not just identity. A leaked credential plus a non-UH IP fails the check.

Dry-run mode — the only sane way to start

Turning on a perimeter in enforced mode without testing it will break something. Dry-run mode logs what would have been denied without actually denying. Run dry-run for a full week of normal lab activity, read the deny logs, add the ingress/egress rules you need, then flip to enforced.

# Create an access policy (one per org — UH usually already has this)
ORG=$(gcloud organizations list --format="value(ID)")

# Create a service perimeter in dry-run mode
gcloud access-context-manager perimeters create srl_phi \
  --title="SRL PHI perimeter" \
  --perimeter-type=regular \
  --resources=projects/$HIPIMR_PROD_NUM,projects/$MAKAWALU_PROD_NUM,projects/$HALO_PROD_NUM,projects/$SHARED_NUM \
  --restricted-services=storage.googleapis.com,bigquery.googleapis.com,healthcare.googleapis.com,aiplatform.googleapis.com,cloudkms.googleapis.com,secretmanager.googleapis.com,pubsub.googleapis.com,run.googleapis.com,logging.googleapis.com \
  --policy=$POLICY_ID \
  --dry-run

# Read denied-attempt logs across the perimeter projects
gcloud logging read \
  'protoPayload.metadata.violationReason="vpcServiceControlsRestriction"' \
  --project=srl-shared-services --limit=20 --format=json

When the deny log is empty for a normal week of lab work, switch to enforced:

gcloud access-context-manager perimeters update srl_phi \
  --policy=$POLICY_ID \
  --no-dry-run
What the perimeter protects: the listed services in the listed projects. A perimeter on GCS in srl-hipimr-prod does not protect GCS in a different project, or any service not on the restricted list. Be explicit about both axes.
The guardrails

Org policies that pin every project below the folder

An org policy applied at the SRL folder cascades to every project in the lab — automatically, no per-project work required. These are the constraints UH ITS may apply for you, but every one of them is something you should understand and verify.

constraints/compute.vmExternalIpAccess
No external IPs on VMs
Denies public IPs on Compute Engine instances. Forces all egress through Cloud NAT (M3L2). A defence-in-depth measure on top of the perimeter.
constraints/iam.disableServiceAccountKeyCreation
No service account key files
Disables creating long-lived SA JSON keys. Forces Workload Identity on GKE, attached SAs on Cloud Run, and impersonation for local dev. Closes the #1 GCP credential-leak vector.
constraints/iam.allowedPolicyMemberDomains
Restrict identities to UH
Only principals from hawaii.edu (and a small allow-list of partner domains) can be granted IAM roles. Prevents accidentally adding a personal gmail to a PHI project.
constraints/gcp.resourceLocations
Restrict resource locations to approved regions
Limits resource creation to us-west1, us-west2, us-central1. Keeps data in the US so HIPAA + IRB obligations stay simple.
constraints/gcp.restrictNonCmekServices
Require CMEK on listed services
Forces new GCS buckets, BigQuery datasets, and Persistent Disks in PHI projects to be created with a customer-managed key — never with default encryption. M3L3 covers the KMS keyring design.
constraints/compute.requireOsLogin
Require OS Login on Compute Engine
If you ever use VMs, SSH access goes through IAM — not via local SSH keys you copy around. Audited, revocable, group-scoped.
constraints/storage.uniformBucketLevelAccess
Uniform bucket-level access
Disables per-object ACLs on Cloud Storage. Every bucket's access is governed by IAM bindings only. Auditable; predictable; no "but the object had a public-read ACL we forgot about" incidents.

Applying them — gcloud

# Enforce "no external IPs" at the SRL folder
gcloud resource-manager org-policies enable-enforce \
  constraints/compute.vmExternalIpAccess \
  --folder=$SRL_FOLDER_ID

# Restrict resource locations
cat > locations-policy.yaml <<EOF
constraint: constraints/gcp.resourceLocations
listPolicy:
  allowedValues:
    - in:us-west1-locations
    - in:us-west2-locations
    - in:us-central1-locations
EOF
gcloud resource-manager org-policies set-policy locations-policy.yaml \
  --folder=$SRL_FOLDER_ID
Test before enforcing. Some org policies will break workflows you didn't realise depended on the now-forbidden behaviour. Apply each policy at a single dev project first; watch for a week; then escalate to the folder. Especially true for disableServiceAccountKeyCreation — anything still using key files will fail immediately.
Operating the perimeter

Bringing a project into (and out of) the perimeter

A new study spins up; you create srl-newstudy-prod; that project needs to join the SRL-PHI perimeter before it holds PHI. The mechanics:

# 1. Find the new project's number
NEW=$(gcloud projects describe srl-newstudy-prod --format="value(projectNumber)")

# 2. Add it to the perimeter
gcloud access-context-manager perimeters update srl_phi \
  --policy=$POLICY_ID \
  --add-resources=projects/$NEW

Removing a project is just as easy — and is the right action when a study closes out:

gcloud access-context-manager perimeters update srl_phi \
  --policy=$POLICY_ID \
  --remove-resources=projects/$NEW

What to expect after enforcement

The first time a workflow breaks because of VPC SC, the error is unmistakable:

PERMISSION_DENIED: Request is prohibited by organization's policy.
vpcServiceControlsUniqueIdentifier: 1a2b3c4d5e
 
# Look up what was blocked
gcloud logging read \
  'protoPayload.metadata.violationReason="vpcServiceControlsRestriction"
   AND protoPayload.metadata.vpcServiceControlsUniqueIdentifier="1a2b3c4d5e"' \
  --project=srl-shared-services --limit=1 --format=json

The unique identifier in the error message is also in the audit log entry — that's how you correlate "this call failed" with "the perimeter said no." From there: either fix the workflow (run inside the perimeter), or add an ingress/egress rule with deliberate scope. Never widen the restricted-services list to "fix" a denial.

Hands-on exercise 3.1

Build a dry-run perimeter and try to break it

Prerequisites: you need an Access Policy at the org level (UH almost certainly has one — confirm with ITS). For this exercise, use two of the sandbox projects you created in M1L2 (srl-demo-prod and srl-demo-staging, or recreate them).

🧱 Stand it up, observe, switch on
  1. Find your org's access policy:
    gcloud access-context-manager policies list --organization=$ORG
    Save the policy ID as POLICY_ID.
  2. Get the project numbers for two sandbox projects:
    gcloud projects describe srl-demo-prod --format='value(projectNumber)'
    gcloud projects describe srl-demo-staging --format='value(projectNumber)'
  3. Enable the Access Context Manager API in any project (only needs to happen once per org):
    gcloud services enable accesscontextmanager.googleapis.com
  4. Create a dry-run perimeter around the two projects, restricting GCS and BigQuery:
    gcloud access-context-manager perimeters create demo_perimeter --title="Demo perimeter" --perimeter-type=regular --resources=projects/$PROD_NUM,projects/$STAGING_NUM --restricted-services=storage.googleapis.com,bigquery.googleapis.com --policy=$POLICY_ID --dry-run
  5. Try to break it. From your laptop (outside the perimeter), create a bucket in srl-demo-prod:
    gcloud storage buckets create gs://srl-demo-prod-test-$(date +%s) --project=srl-demo-prod --location=us-west1
    This will succeed — you're in dry-run.
  6. Read the dry-run deny log to see it would have blocked you in enforced mode:
    gcloud logging read 'protoPayload.metadata.violationReason="vpcServiceControlsRestriction"' --project=srl-demo-prod --limit=5 --format=json
    You should see your bucket-create call logged as a would-be-denied event.
  7. Flip to enforced:
    gcloud access-context-manager perimeters update demo_perimeter --policy=$POLICY_ID --no-dry-run
  8. Try the same bucket-create again. It will fail with PERMISSION_DENIED and a vpcServiceControlsUniqueIdentifier in the error. That's the perimeter working.
  9. Apply one org policy at the folder level (if you have a folder; otherwise skip):
    gcloud resource-manager org-policies enable-enforce constraints/iam.disableServiceAccountKeyCreation --folder=$FOLDER_ID
    Then try to create an SA key — it should fail.
  10. Tear down when finished:
    gcloud access-context-manager perimeters delete demo_perimeter --policy=$POLICY_ID
    (Plus disable the org policy if you applied it for testing.)
