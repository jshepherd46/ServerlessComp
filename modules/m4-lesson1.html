The runtime model

Revisions — every deploy is immutable

A Cloud Run service is a stable name with a stable URL. A revision is a snapshot of everything that defines an instance of that service: container image, env vars, secrets, memory, CPU, concurrency, service account, network settings. Every gcloud run deploy creates a new revision and points the service's traffic at it.

xai-serve-00006-abc
100%
Current. Image: hello-flask:0.2.3, mem 2Gi, concurrency 4, --min-instances 1.
xai-serve-00005-xyz
0%
Previous good. Image 0.2.2. Kept warm by Cloud Run's history for instant rollback.
xai-serve-00004-def
0%
Older. Same as 5 but with a smaller concurrency. Retained indefinitely until pruned.

Three properties make this model powerful:

🧊
Immutable
A revision, once created, cannot be modified. Want different memory? That's a new revision. The full configuration is captured at deploy time and stays captured.
↩️
Instantly reversible
Rollback is a traffic-split command, not a redeploy. The old revision is still there. gcloud run services update-traffic --to-revisions=<rev>=100.
🌗
Splittable
Send 10% of traffic to a new revision, 90% to the current. Canary deploys are a built-in primitive, not something you script.
🔍
Auditable
Every revision has a manifest stored in the service's revision history. gcloud run revisions describe shows the exact configuration that ever served traffic.

Inspect:

gcloud run revisions list --service=xai-serve --region=us-west1
gcloud run revisions describe xai-serve-00005-xyz --region=us-west1
gcloud run services describe xai-serve --region=us-west1 --format="value(status.traffic)"
Operating on revisions

Traffic splitting — canary, blue-green, rollback

By default every new deploy gets 100% of traffic. That's fine for low-stakes services. For a HIPAA-tier inference API, you want gradual rollouts.

Deploy a new revision without shifting traffic

gcloud run deploy xai-serve \
  --image=$NEW_IMG \
  --region=us-west1 \
  --no-traffic      # revision is created and ready but receives 0%

The new revision exists, can serve requests via a tag-specific URL, but the production URL still routes to the previous revision.

Address a specific revision for smoke tests

# Tag the new revision
gcloud run services update-traffic xai-serve \
  --region=us-west1 \
  --update-tags=canary=xai-serve-00007-ghi

# Now this URL hits ONLY the tagged revision
gcloud run services describe xai-serve --region=us-west1 \
  --format="value(status.traffic[?tag=='canary'].url)"
# https://canary---xai-serve-abc.us-west1.run.app

Canary — 10% / 90%

gcloud run services update-traffic xai-serve \
  --region=us-west1 \
  --to-revisions=xai-serve-00007-ghi=10,xai-serve-00006-abc=90

Promote — 100% to the new revision

gcloud run services update-traffic xai-serve \
  --region=us-west1 \
  --to-revisions=xai-serve-00007-ghi=100

Rollback — back to the previous good revision

gcloud run services update-traffic xai-serve \
  --region=us-west1 \
  --to-revisions=xai-serve-00006-abc=100
Where the canary pattern matters most: any time a regression in model output, latency, or error rate could affect care or a publication. Roll out at 10% for an hour, watch metrics + log-based alerts, then promote. If anything goes wrong, the rollback is one command and ~2 seconds — faster than any redeploy.
The autoscaler

Scaling internals — the math you actually need

Cloud Run's autoscaler is request-based. It adds container instances when in-flight requests exceed what existing instances can handle, removes them when traffic falls. The shape of that behaviour is set by three numbers.

instances ≈ ⌈ in-flight requests ÷ concurrency ⌉, bounded by [min, max]
Light traffic API
2 req/s, latency 100 ms → ~0.2 in-flight requests on average. With concurrency=80, one instance handles it. --min-instances=0 means most of the day, zero instances run.
Heavy ML inference
2 req/s, latency 4 s → ~8 in-flight on average. With concurrency=2 (model is the bottleneck), you need 4 instances. Set --min-instances=1 to hide the cold start from the first request.
Bursty traffic
A presentation hits the demo: 100 req/s, latency 800 ms → ~80 in-flight. With concurrency=4, the autoscaler wants 20 instances. --max-instances=10 caps you and the remaining requests queue (slower) or 429 (depending on your settings).

The three numbers, in order of impact

A model that uses 1.5 GB of GPU memory and saturates a single inference at a time should have concurrency=1. Raising it would cause overlapping requests to thrash. A JSON-API webhook with no heavy state can run concurrency=250, serving thousands of requests on a handful of instances.

For clinical-facing services where the first request matters (you don't want a clinician demo to spin up the model on stage), --min-instances=1. For internal demos or nightly admin APIs, 0 is right. The cost: a min-instance Cloud Run instance running 24/7 at 1 vCPU + 2 GB memory is roughly $30–50/month per instance — order of magnitude awareness, not a budget.

Max-instances is not a target; it's a safety net. A misconfigured trigger, a load-test runaway, or an upstream attack can push request volume far beyond what the service was sized for. The autoscaler honours max-instances strictly. Set it for prod services where the value is "if traffic exceeds this, queue (or 429) rather than spend."

Two configurations that matter

CPU allocation and startup probes

CPU allocation — request-time vs always-allocated

--cpu-throttling

CPU only during requests (default)

The instance gets full CPU when handling a request, very little between requests. You pay only for the CPU you used during request handling.

Use for: stateless APIs, anything that doesn't run background work between requests.
--no-cpu-throttling

Always-allocated CPU

The instance has full CPU even between requests. You pay for the whole instance-second the container is alive.

Use for: background processing (a goroutine consuming Pub/Sub between requests), services that need to handle SIGTERM gracefully under load, anything with periodic work.

Per Cloud Run pricing, always-allocated costs roughly 2–3× per-instance vs throttled mode. Default to throttled; switch only when you observe a problem.

gcloud run services update xai-serve \
  --region=us-west1 \
  --no-cpu-throttling     # switch to always-allocated

Startup probes — telling Cloud Run when you're really ready

Default: Cloud Run considers an instance "ready" as soon as it starts listening on the port. For services with heavy initialisation (loading a 4 GB model from GCS), you want Cloud Run to wait until the model is in memory before routing real traffic.

gcloud run services update xai-serve \
  --region=us-west1 \
  --startup-probe=httpGet.path=/healthz/ready,initialDelaySeconds=10,periodSeconds=5,failureThreshold=24,timeoutSeconds=2

Now Cloud Run polls /healthz/ready every 5 seconds, gives the instance up to 2 minutes (24×5s) to come back 200, and only then sends real traffic. Your app returns 503 from /healthz/ready until the model load completes.

The pattern this enables: deploy a new revision with --no-traffic; canary to 10% only after the startup probe has confirmed the new revision is actually ready to serve. Failed probes never receive production traffic.
Beyond *.run.app

Custom domains and the global load balancer

The default *.run.app URL is fine for internal services and demos. For anything public-facing — a clinician portal, a study landing page — you want your own domain plus the global HTTPS load balancer for Cloud Armor, Identity-Aware Proxy, and multi-region failover.

Domain mapping (simple but limited)

For a single region with no LB features:

gcloud beta run domain-mappings create \
  --service=xai-serve \
  --region=us-west1 \
  --domain=xai.makawalu.uhcancercenter.org

Global HTTPS Load Balancer (production-grade)

The full stack — required for Cloud Armor, IAP, multi-region:

# 1. Serverless NEG pointing at the Cloud Run service
gcloud compute network-endpoint-groups create xai-serve-neg \
  --region=us-west1 --network-endpoint-type=serverless \
  --cloud-run-service=xai-serve

# 2. Backend service consuming the NEG, with security policy
gcloud compute backend-services create xai-serve-backend --global
gcloud compute backend-services add-backend xai-serve-backend \
  --global --network-endpoint-group=xai-serve-neg --network-endpoint-group-region=us-west1
gcloud compute backend-services update xai-serve-backend --global \
  --security-policy=srl-public-policy

# 3. URL map + HTTPS proxy + forwarding rule (omitting cert + IP setup for brevity)
gcloud compute url-maps create xai-serve-lb --default-service=xai-serve-backend
gcloud compute target-https-proxies create xai-serve-https \
  --url-map=xai-serve-lb --ssl-certificates=xai-makawalu-cert
gcloud compute forwarding-rules create xai-serve-fwd --global \
  --target-https-proxy=xai-serve-https --address=$LB_IP --ports=443

After this, traffic flows: client → global LB IP → Cloud Armor → backend → serverless NEG → Cloud Run service. Cloud Armor from M3L2 attaches at the backend service. IAP attaches as a load balancer feature. Failover to a secondary region is one extra backend.

When to skip the LB: internal-only services, ad-hoc demos, anything where the *.run.app URL is enough. The LB has real cost (~$25/month minimum) — match it to actual production exposure.
When Cloud Run grew up

GPUs and sidecars

Cloud Run GPUs

Cloud Run now supports NVIDIA L4 GPUs attached to instances. The use case: serverless model serving without provisioning a GKE GPU node pool. Good fit for Makawalu CBM inference; not a fit for multi-GPU training.

gcloud run deploy makawalu-xai \
  --image=$IMG \
  --region=us-west1 \
  --gpu=1 --gpu-type=nvidia-l4 \
  --memory=16Gi --cpu=4 \
  --no-cpu-throttling \
  --max-instances=4

Caveats: limited regions, requires --no-cpu-throttling, max-instances capped tightly per project, cost is dramatically higher than CPU-only. Verify the regional availability and quota before designing around it.

Sidecar containers

A Cloud Run revision can run up to nine containers in a single instance, sharing local network and an emptyDir-like volume. The main use case: deploying a managed proxy alongside your app (e.g., the Cloud SQL Auth Proxy, or a custom telemetry forwarder).

gcloud run services replace service.yaml --region=us-west1

With a YAML manifest defining multiple containers::

apiVersion: serving.knative.dev/v1
kind: Service
metadata:
  name: xai-serve
spec:
  template:
    spec:
      containers:
        - name: app
          image: $IMG
          ports: [{containerPort: 8080}]
        - name: sql-proxy
          image: gcr.io/cloud-sql-connectors/cloud-sql-proxy:latest
          args: ["--port=5432", "$PROJECT:us-west1:srl-omop"]
Sidecars are a recent addition and worth knowing about, but the bar for using them is high — most needs are still met by single-container services with managed-service clients (e.g., the BigQuery client library replaces a sidecar proxy). Reach for sidecars when you genuinely have a separate process responsibility, not for code organisation.
Hands-on exercise 4.1

Canary a new revision and watch it

🐤 Two revisions, 10/90 split, promote or roll back
  1. Start from your hello-flask service from M2L3 in your sandbox project. If you tore it down, redeploy quickly:
    IMG=us-west1-docker.pkg.dev/$PROJECT/practice/hello-flask:0.2.0
    gcloud run deploy hello-flask --image=$IMG --region=us-west1 --no-allow-unauthenticated
  2. Edit app.py from M2L1 — change the response payload (e.g., add a "version": "0.3" field). Build a new image with a different tag:
    docker build -t hello-flask:0.3.0 .
    IMG2=us-west1-docker.pkg.dev/$PROJECT/practice/hello-flask:0.3.0
    docker tag hello-flask:0.3.0 $IMG2 && docker push $IMG2
  3. Deploy with no traffic:
    gcloud run deploy hello-flask --image=$IMG2 --region=us-west1 --no-allow-unauthenticated --no-traffic
  4. Inspect revisions:
    gcloud run revisions list --service=hello-flask --region=us-west1
    Note both revision names — call the old one OLD and the new one NEW.
  5. Tag the new revision for direct addressability:
    gcloud run services update-traffic hello-flask --region=us-west1 --update-tags=canary=$NEW
  6. Hit the canary URL directly (smoke test):
    URL=$(gcloud run services describe hello-flask --region=us-west1 --format="value(status.traffic[?tag=='canary'].url)")
    curl -H "Authorization: Bearer $(gcloud auth print-identity-token)" $URL/
    You should see your 0.3 payload — only via this URL.
  7. Verify the main service URL still returns 0.2:
    MAIN=$(gcloud run services describe hello-flask --region=us-west1 --format='value(status.url)')
    curl -H "Authorization: Bearer $(gcloud auth print-identity-token)" $MAIN/
  8. Canary 10%:
    gcloud run services update-traffic hello-flask --region=us-west1 --to-revisions=$NEW=10,$OLD=90
    Hit the main URL 20 times in a loop; ~2 of the responses should be 0.3.
  9. Watch metrics: Cloud Run UI → service → Metrics. Filter by revision. The new revision should show a small request count; latency and error rate should match the old.
  10. Promote to 100%:
    gcloud run services update-traffic hello-flask --region=us-west1 --to-revisions=$NEW=100
  11. Practise rollback: simulate a regression by rolling back to OLD:
    gcloud run services update-traffic hello-flask --region=us-west1 --to-revisions=$OLD=100
    Confirm the main URL returns 0.2 again. That's your rollback drill.
  12. Tune for fun: set min-instances=1 on the service and observe latency on the next cold request — it should be near-instant. Reset to 0 to scale back to zero between sessions.

Knowledge check

Questions to sit with

  • Your latest deploy of the XAI service has a regression — output values are subtly off. Your rollback time is <5 seconds. Why? Hint
  • Your CNN inference service has concurrency=80 by default. Latency spikes to 30 seconds under any real traffic. What's the lever? Hint
  • You set --min-instances=1 on a low-traffic API to avoid cold starts. A month later your bill is $40 higher than expected. Where did it go? Hint
  • You deploy a revision with a startup probe and it never reaches Ready. What happens to traffic and to the previous revision? Hint
