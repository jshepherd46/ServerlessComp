Lesson 2 of 3

Network hardening

The VPC SC perimeter is the wall. This lesson is the plumbing inside it: keeping traffic off the public internet, denying public IPs by default, fronting the few public services with edge protection, and applying all of it to the real HIPIMR ingest path.

⏱ ~50 min reading + exercise 🔧 VPC, Private Service Connect, Cloud NAT, Cloud Armor, VPC connectors 📍 Module 3 of 12
Why this layer exists

Network hardening sits inside the perimeter

M3L1's perimeter denies any cross-boundary call without an explicit rule. Network hardening makes sure that even inside the perimeter, traffic stays where it should — and the few necessary public endpoints aren't bare to the open internet. Five layers, stacked.

L1 — VPC SC perimeter
Cross-boundary control (M3L1). The only layer that operates on identity and resource regardless of network path.
L2 — Private connectivity
Traffic from workloads inside the perimeter to GCP APIs never touches the public internet. Section 2.
L3 — No public IPs
Compute and GKE workloads have no external IPs. Outbound (when needed) goes through Cloud NAT. Section 3.
L4 — Internal-only services
Cloud Run / Functions reachable only from inside the VPC, fronted by VPC connectors. Section 4.
L5 — Edge protection on public endpoints
The few services that must face the internet sit behind Cloud Armor with WAF rules and rate limits. Section 5.
The framing: any one of these layers can fail. The point of all five is that an adversary or a misconfiguration has to defeat several at once before they reach PHI. None of them is optional in a prod HIPAA project.
Layer 2

Private connectivity — never touch the internet

A Cloud Run service inside srl-makawalu-prod needs to read from GCS and write to BigQuery. By default those calls go through Google's public API endpoints — which are technically over HTTPS, but the request leaves the VPC and traverses the internet on the way back to Google. The hardening: keep that traffic on Google's backbone.

Private Google Access — the simple case

Enable Private Google Access on each subnet. VMs and pods in that subnet without public IPs can reach *.googleapis.com via private IPs, routed entirely on Google's network.

gcloud compute networks subnets update srl-prod-subnet \
  --region=us-west1 \
  --enable-private-ip-google-access

Private Service Connect — the rigorous case

Private Service Connect (PSC) goes further: instead of routing public endpoint traffic privately, you create an endpoint inside your VPC that is the address for the Google API. DNS for storage.googleapis.com resolves to your private IP. The service literally cannot be reached over the public endpoint from inside the VPC.

# Create a PSC endpoint for Google APIs (one-time per VPC)
gcloud compute addresses create psc-googleapis \
  --global \
  --purpose=PRIVATE_SERVICE_CONNECT \
  --addresses=10.0.0.100 \
  --network=srl-prod-vpc

gcloud compute forwarding-rules create psc-googleapis-rule \
  --global \
  --network=srl-prod-vpc \
  --address=psc-googleapis \
  --target-google-apis-bundle=all-apis \
  --service-directory-registration=projects/$PROJECT/locations/us-west1

After this, your workloads' DNS resolves storage.googleapis.com → 10.0.0.100. The packet never leaves the VPC. Pair with the VPC SC perimeter for the airtight closed-loop.

The pragmatic choice: Private Google Access for nearly all cases. Add PSC when you specifically need DNS-level enforcement (regulators ask, or you want to be certain a misconfigured workload cannot reach the public endpoint). Both together is a common HIPAA-tier baseline.
Layer 3

No public IPs, ever — Cloud NAT for outbound

From M3L1's org policy: constraints/compute.vmExternalIpAccess denies public IPs on Compute Engine VMs. The same discipline applies to GKE nodes (use private clusters), Cloud SQL (use private IP), and Vertex AI Workbench (use Private Service Connect).

The problem: workloads sometimes legitimately need to reach the public internet — pulling Python packages, fetching a model from HuggingFace, calling an external Healthcare API. Without a public IP, how?

Cloud NAT — outbound only, fully managed

# Create a Cloud Router (NAT requires one)
gcloud compute routers create srl-prod-router \
  --network=srl-prod-vpc \
  --region=us-west1

# Attach a Cloud NAT to it
gcloud compute routers nats create srl-prod-nat \
  --router=srl-prod-router \
  --region=us-west1 \
  --auto-allocate-nat-external-ips \
  --nat-all-subnet-ip-ranges \
  --enable-logging

Now any workload in srl-prod-vpc without a public IP can still reach external services — but only outbound. Nothing can reach in from the internet. The NAT logs show every outbound flow, which is exactly what you want when an auditor asks "where did this workload talk to?"

🚪
One-way doors
NAT is outbound only. Nothing initiates a connection inbound through it. The asymmetry is the security.
📝
Logged by default
Every NAT flow appears in Cloud Logging. Filterable by source IP, destination, protocol. Audit-grade.
🌐
Stable egress IPs
Allocate static external IPs to the NAT if partners (Queen's, Vertex external endpoints) need to allow-list your egress.
⛔
Per-subnet scoping
You can deny NAT to specific subnets — e.g., the high-sensitivity ones that should never reach anything external.
Layer 4

Internal-only Cloud Run and Functions

By default, a Cloud Run service has a public *.run.app URL. For internal-only services — the ones other lab workloads call, but no human should reach — close that door at deploy time.

gcloud run deploy makawalu-internal-api \
  --image=$IMG \
  --region=us-west1 \
  --service-account=internal-api-sa@srl-makawalu-prod.iam.gserviceaccount.com \
  --ingress=internal \
  --no-allow-unauthenticated \
  --vpc-connector=srl-prod-connector \
  --vpc-egress=all-traffic

Three flags doing the work:

The public *.run.app URL still exists but rejects requests from anywhere outside. Pair with --no-allow-unauthenticated so that even internal callers must present a valid identity. Use --ingress=internal-and-cloud-load-balancing when fronted by an external load balancer + Cloud Armor.

Create one connector per region per VPC; reuse across services. gcloud compute networks vpc-access connectors create srl-prod-connector --region=us-west1 --network=srl-prod-vpc --range=10.8.0.0/28. Egress through the connector means outbound traffic from the service flows through your VPC's firewall rules and Cloud NAT — same path as a VM.

Default is private-ranges-only — only private IPs go through the connector, public traffic uses Cloud Run's default egress. For HIPAA workloads in a PHI project, force everything through the VPC so all egress hits your audit logs and (when relevant) Cloud NAT logs.

The same pattern applies to Cloud Functions (v2 is Cloud Run under the hood, so the flags map directly).

Layer 5

Cloud Armor — protecting the few public endpoints

Some services do need to face the internet — the Makawalu XAI demo, a public-facing FHIR portal, a clinician-facing tool. Cloud Armor sits at the global load balancer layer in front of those services with WAF rules, rate limits, geo restrictions, and DDoS protection.

The minimum useful Cloud Armor policy

# Create a security policy
gcloud compute security-policies create srl-public-policy \
  --description="Baseline protection for public services"

# 1. Rate-limit: 100 requests / 60s per source IP
gcloud compute security-policies rules create 1000 \
  --security-policy=srl-public-policy \
  --action=rate-based-ban \
  --rate-limit-threshold-count=100 \
  --rate-limit-threshold-interval-sec=60 \
  --ban-duration-sec=600 \
  --src-ip-ranges="*" \
  --conform-action=allow \
  --exceed-action=deny-429

# 2. Block known-bad SQLi patterns (OWASP rule)
gcloud compute security-policies rules create 2000 \
  --security-policy=srl-public-policy \
  --expression="evaluatePreconfiguredExpr('sqli-v33-stable')" \
  --action=deny-403

# 3. Block known-bad XSS patterns
gcloud compute security-policies rules create 2010 \
  --security-policy=srl-public-policy \
  --expression="evaluatePreconfiguredExpr('xss-v33-stable')" \
  --action=deny-403

# 4. Restrict to US clients (optional, depends on your audience)
gcloud compute security-policies rules create 3000 \
  --security-policy=srl-public-policy \
  --expression="origin.region_code != 'US'" \
  --action=deny-403

# Attach to the backend service in front of your Cloud Run service
gcloud compute backend-services update makawalu-xai-backend \
  --global \
  --security-policy=srl-public-policy
What Cloud Armor is not: a substitute for authentication, IAM, or VPC SC. It is an extra filter at the edge — useful, cheap, and audit-friendly — but a public service should never depend on it as the only line of defense. Auth and least-privilege still apply.
Putting all five layers together

The HIPIMR ingest path, hardened end to end

Nightly: Queen's PACS exports a batch of mammography DICOMs to a transfer GCS bucket; SRL ingests, de-identifies (where possible for research), and loads into the HIPIMR Healthcare API DICOM store. Every arrow in this picture is hardened.

Queen's PACS ─[mTLS, IP allow-list to NAT egress IP]─▶ VPC SC perimeter (srl_phi) srl-hipimr-prod / VPC (no public IPs, PGA + PSC enabled) ├ GCS landing bucket # CMEK, uniform IAM, audit logs on ├ Cloud Run Job (ingest-sa) # --ingress=internal --vpc-egress=all-traffic ├ Healthcare API DICOM store # CMEK, perimeter-restricted service └ Pub/Sub: ingest-events # perimeter-restricted ─X─ No reach inbound from public internet ───▶ Downstream consumers (de-id, model training) subscribe via internal Pub/Sub

What each layer is doing

Queen's → GCS landing bucket
mTLS + IP allow-list
Queen's writes via authenticated gcloud storage cp with their SA (signed under the BAA). Cloud Armor (or signed URLs in front of the bucket) restricts the source IP. The SA's role is roles/storage.objectCreator only — write, never read.
GCS landing bucket → Cloud Run Job
Eventarc + internal ingress
Eventarc trigger on GCS object created invokes the Cloud Run Job. The Job's URL is --ingress=internal — Eventarc reaches it via a Google-internal path, and no human URL works.
Cloud Run Job → Healthcare API
Private Service Connect + perimeter ingress allow
The Job uses Workload Identity → ingest-sa. Its API call to the Healthcare API DICOM store resolves to the PSC endpoint (private IP). The VPC SC perimeter has an ingress allow for the Healthcare API for SAs from the same project.
Healthcare API → downstream consumers
Pub/Sub, perimeter-restricted, internal subs
The DICOM store emits notifications to a perimeter-protected Pub/Sub topic. The de-id pipeline (running in the same perimeter) subscribes. Nothing crosses the boundary; nothing touches a public network.

Every hop has logs in the audit sink from M1L4. If anything ever breaks, you can trace the request from Queen's all the way to the DICOM store using the audit logs alone.

Hands-on exercise 3.2

Stand up the network layer

🛡 Private path, no public IPs, internal-only Cloud Run
  1. In your sandbox project, create a custom VPC + subnet (don't use the default):
    gcloud compute networks create demo-vpc --subnet-mode=custom
    gcloud compute networks subnets create demo-subnet --network=demo-vpc --region=us-west1 --range=10.10.0.0/24 --enable-private-ip-google-access
  2. Create a Cloud Router + Cloud NAT for outbound:
    gcloud compute routers create demo-router --network=demo-vpc --region=us-west1
    gcloud compute routers nats create demo-nat --router=demo-router --region=us-west1 --auto-allocate-nat-external-ips --nat-all-subnet-ip-ranges --enable-logging
  3. Create a Serverless VPC Access connector:
    gcloud compute networks vpc-access connectors create demo-connector --region=us-west1 --network=demo-vpc --range=10.8.0.0/28
  4. Deploy your hello-flask from M2 as internal-only and force egress through the connector:
    IMG=us-west1-docker.pkg.dev/$PROJECT/practice/hello-flask:0.2.0
    gcloud run deploy demo-internal --image=$IMG --region=us-west1 --ingress=internal --no-allow-unauthenticated --vpc-connector=demo-connector --vpc-egress=all-traffic
  5. Try to hit it from your laptop with a valid auth token:
    URL=$(gcloud run services describe demo-internal --region=us-west1 --format='value(status.url)')
    curl -H "Authorization: Bearer $(gcloud auth print-identity-token)" $URL/
    Expect 404 or 403 — internal ingress blocks you. Auth is valid, network path is not.
  6. Spin up a tiny VM inside the VPC and hit it from there:
    gcloud compute instances create demo-vm --zone=us-west1-a --subnet=demo-subnet --no-address --image-family=debian-12 --image-project=debian-cloud --service-account=$PROJECT-compute@developer.gserviceaccount.com --scopes=cloud-platform
    gcloud compute ssh demo-vm --zone=us-west1-a --tunnel-through-iap
    On the VM: TOKEN=$(curl -s -H "Metadata-Flavor: Google" http://metadata/computeMetadata/v1/instance/service-accounts/default/identity?audience=$URL) then curl -H "Authorization: Bearer $TOKEN" $URL/
    Expect a successful response. The internal path works.
  7. Inspect the NAT logs in Cloud Logging:
    Filter: jsonPayload.connection.nat_ip:* AND resource.labels.gateway_name="demo-nat". If the VM has done any outbound (apt update, etc.), you'll see flows here.
  8. Tear down everything when finished — Cloud NAT, router, connector, VM, Cloud Run service, VPC. Each command is the verb-reverse of what you used to create them.
