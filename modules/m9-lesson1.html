Lesson 1 of 3

OMOP CDM
on BigQuery

Phase 3 opens with the analytics layer. OMOP is the canonical schema for "clinical events that happened to patients"; ATLAS is its query UI. This lesson is the model: what OMOP CDM is, its core tables, the vocabulary backbone that ties everything together, and how it lives on BigQuery for HALO.

⏱ ~55 min reading + exercise 🔧 OMOP CDM v5.4, BigQuery, OHDSI vocabularies 📍 Module 9 of 12
The framing

OMOP CDM — one schema, many institutions

OMOP CDM (Observational Medical Outcomes Partnership Common Data Model) is the schema OHDSI (Observational Health Data Sciences and Informatics) maintains for clinical observational data. The promise: any institution that transforms its EHR data into OMOP can run any OHDSI analytic package — ATLAS, ACHILLES, PatientLevelPrediction, CohortMethod — without changing a line. Studies run identically at Queen's, at Mayo, at UPenn.

For SRL, OMOP is the substrate for HALO (Hawai'i Liver Cancer IO Outcomes) and the foundation that makes PacificFed federated analytics possible — every site's data fits the same shape, so the same cohort definition runs across all of them.

What you get from adopting OMOP

  • → Portable cohort definitions. A cohort defined as "patients with HCC + immunotherapy in 2020-2024" runs unchanged on HALO and on any other OMOP site.
  • → Reusable analytic tooling. ATLAS cohort builder, ACHILLES data-quality dashboards, R packages (CohortMethod, PatientLevelPrediction, FeatureExtraction).
  • → Federated study capability. Each site keeps its data; only summary results travel — see M10 for the FL extension.
  • → Concept-level consistency. "Diabetes" doesn't depend on which ICD code your institution happens to use; OMOP\'s standardised vocabulary resolves equivalences.
The trade. Adopting OMOP costs the ETL work to map your source data into the CDM. Done once per data source; pays off forever in tool reuse and federated capability. For HALO this happens nightly: Queen\'s clinical extracts → ETL → OMOP-shaped BigQuery dataset → ATLAS + research tier.
The schema

The core clinical tables

OMOP CDM v5.4 has ~25 tables. Eight cover the vast majority of queries. Learn these; the rest fall into place when you need them.

person

Each patient is one row

Demographics — gender, year of birth, race, ethnicity. person_id is the patient surrogate; every other table references it.

PK: person_id · ~10M rows for a 5-year regional EHR extract
visit_occurrence

Each encounter is one row

Inpatient admission, outpatient visit, ED visit, etc. Has start/end dates and a visit type concept. Most clinical events reference back to a visit.

PK: visit_occurrence_id · FK: person_id · ~50M rows typical
condition_occurrence

Each diagnosis is one row

Mapped from ICD-10-CM, SNOMED, etc. condition_concept_id points at the standard vocabulary entry.

PK: condition_occurrence_id · ~hundreds of millions of rows
drug_exposure

Each medication event

Prescriptions, inpatient administrations, dispense records. Mapped to RxNorm via drug_concept_id.

PK: drug_exposure_id
procedure_occurrence

Each procedure event

Surgeries, labs ordered, imaging studies, etc. Mapped to SNOMED procedure codes via procedure_concept_id.

PK: procedure_occurrence_id
measurement

Each numeric or categorical measurement

Lab values, vitals, scoring results. Has value_as_number, value_as_concept_id, and a unit concept. Mapped to LOINC for labs.

PK: measurement_id · the largest table at most sites
observation

Catch-all clinical facts

Family history, smoking status, allergy, anything that doesn\'t fit elsewhere. Less standardised than the others; varies by ETL.

PK: observation_id
death

Each patient\'s death (if applicable)

One row per person; ties to a death-cause condition. Critical for outcome-tracking studies like HALO.

PK: person_id (1:1 with person)

The relationship graph

person 1 ──┬── visit_occurrence N
            ├── condition_occurrence N ── (FK: visit_occurrence)
            ├── drug_exposure N
            ├── procedure_occurrence N
            ├── measurement N
            ├── observation N
            └── death 0..1

Most cohort queries are joins across these tables filtered by date ranges + concept IDs. The relationship is wide (one person → many events) but shallow (events rarely reference other events directly).

The backbone

Vocabularies — the OMOP magic

The reason OMOP works across institutions is the standardised vocabulary. Every clinical fact references a concept_id — an integer that points at a row in the concept table. The vocabulary tables ship as part of OHDSI; you load them once into BigQuery and your ETL maps source codes (ICD, NDC, CPT, custom local codes) onto OMOP concept IDs.

The vocabulary tables

concept

The master table. Each row: concept_id, name, domain, vocabulary, code, standard concept flag, valid date range. ~10M rows.

concept_relationship

How concepts relate. "ICD-10-CM E11.9 → standard SNOMED 'Type 2 diabetes mellitus'", "RxNorm ingredient → brand name". The mapping graph.

concept_ancestor

Hierarchical "is-a" closure. Quickly find all descendants of "Heart disease" without recursing through concept_relationship.

vocabulary

Metadata about source vocabularies (SNOMED, RxNorm, LOINC, ICD-10-CM, NDC, CPT4, etc.) — version, source URL, validity.

The vocabulary tree, conceptually

Standardised vocabulary (the destination) └── SNOMED CT # conditions, procedures └── concept: "Type 2 diabetes mellitus" → concept_id 201826 (standard) ↑ mapped from source codes: ├── ICD-10-CM E11.9 → via concept_relationship "Maps to" ├── ICD-9-CM 250.00 → via concept_relationship "Maps to" └── SNOMED CT source code → via "Maps to"

Standard vs source concept IDs

Every clinical table has BOTH condition_concept_id (standard concept, used in queries) AND condition_source_concept_id (what the EHR sent). Always query against the standard — that\'s what makes the same cohort definition portable across institutions.

# Find all "Type 2 diabetes" diagnoses regardless of source coding
SELECT DISTINCT person_id
FROM `srl-halo-prod.omop.condition_occurrence`
WHERE condition_concept_id IN (
  SELECT descendant_concept_id
  FROM `srl-halo-prod.omop.concept_ancestor`
  WHERE ancestor_concept_id = 201826    -- T2DM
);

The concept_ancestor trick is key: matching the descendants of a high-level concept rather than enumerating every code that maps to it. Always use ancestor instead of listing standard-concept IDs by hand.

The vocabulary loads quarterly. OHDSI publishes a new vocabulary release ~4× per year — new codes, refined relationships, deprecated terms. Refresh your BigQuery vocabulary tables on the same cadence; ATLAS cohort definitions are tied to the vocabulary version current at definition time, so be deliberate about when you upgrade.
The warehouse

OMOP on BigQuery — schema, partitioning, performance

OHDSI publishes a reference DDL for OMOP on BigQuery. Load it once per study. Two design choices matter most for performance: partitioning by date and clustering by person_id.

The partition + cluster pattern

CREATE TABLE `srl-halo-prod.omop.condition_occurrence` (
  condition_occurrence_id INT64 NOT NULL,
  person_id INT64 NOT NULL,
  condition_concept_id INT64 NOT NULL,
  condition_start_date DATE NOT NULL,
  condition_start_datetime TIMESTAMP,
  condition_end_date DATE,
  visit_occurrence_id INT64,
  condition_source_concept_id INT64,
  condition_source_value STRING
  -- ... more cols
)
PARTITION BY condition_start_date
CLUSTER BY person_id, condition_concept_id;

A cohort query like "T2DM diagnoses in 2023 for patients seen at the cancer center" hits a single year of partitions (50% data reduction vs scanning full table) and benefits from clustering by person_id when joining to person. Cost goes from $10/query to $0.50/query at HALO scale.

Per-table partition strategy

  • → person — small (~10M rows); no partitioning needed.
  • → visit_occurrence, condition_occurrence, drug_exposure, procedure_occurrence, measurement, observation, death — partition by the start date; cluster by person_id.
  • → concept, concept_relationship, concept_ancestor — vocabulary, ~10M rows; no partitioning; consider clustering by domain_id.

Loading from your EHR ETL

The OHDSI community maintains ETL packages for major EHR systems (Epic Clarity, Cerner, etc.). For UH-specific extractions, the pattern is the same: Cloud Run Job (or Dataflow for large batches) reads source extracts from GCS → applies the OMOP mapping → writes _staging tables → SQL merge into the OMOP tables.

# Nightly ETL: Queen's extract → OMOP
gcloud run jobs deploy halo-omop-etl \

  --service-account=halo-etl-sa@srl-halo-prod.iam.gserviceaccount.com \
  --region=us-west1 \

# Wired by Cloud Scheduler nightly (M4L2 pattern)

The HALO project shape, summarised

srl-halo-prod (PHI tier)

OMOP dataset with full identifiers (date of birth, names attached via person). CMEK encrypted. Data Access logging on. Strict IAM. ETL workload SA writes; ATLAS WebAPI service account reads.

srl-halo-research (de-id tier)

Sibling OMOP dataset, populated by the M8L3 cohort-instantiation pipeline. Surrogate person_ids, date-shifted timestamps. Read access for cohort groups; the analytics layer.

srl-shared-services (vocabulary)

The OMOP standardised vocabulary tables, shared across studies. One copy; HALO, ShepBERT, future OMOP studies all reference it. Refreshed quarterly with vocabulary releases.

Vertex Pipelines

ETL, ACHILLES data-quality runs (M9L2), cohort exports — all orchestrated as Vertex Pipelines (M6L2 pattern). Each pipeline run is logged, reproducible, traceable.

Hands-on exercise 9.1

Load a tiny OMOP dataset; query the concept hierarchy

🗄 The OMOP query muscle
  1. Use the OHDSI Synthea sample dataset for the exercise — public, OMOP-shaped synthetic data with all the tables populated:
    https://github.com/OHDSI/Eunomia (R package) or https://github.com/synthetichealth/synthea (Java generator with OMOP output).
  2. Create a BigQuery dataset + load Synthea OMOP CSVs:
    bq mk --location=us-west1 demo_omop
    gcloud storage cp synthea/omop/*.csv gs://$PROJECT-demo-omop/
    for f in person visit_occurrence condition_occurrence drug_exposure measurement; do bq load --source_format=CSV --skip_leading_rows=1 --autodetect demo_omop.$f gs://$PROJECT-demo-omop/${f}.csv; done
  3. Load the OMOP vocabulary (downloadable from athena.ohdsi.org after registration). Smallest useful version: SNOMED + RxNorm + ICD-10-CM + LOINC. Load concept, concept_relationship, concept_ancestor tables.
  4. Query for "all patients with Type 2 diabetes." Use the concept_ancestor trick: 
    SELECT DISTINCT co.person_id
FROM `demo_omop.condition_occurrence` co
WHERE co.condition_concept_id IN (
  SELECT descendant_concept_id
  FROM `demo_omop.concept_ancestor`
  WHERE ancestor_concept_id = 201826  -- T2DM
);
  5. Combine with demographics. Find female T2DM patients aged 50-70: 
    SELECT p.person_id, p.year_of_birth, p.gender_concept_id
FROM `demo_omop.person` p
JOIN (
  SELECT DISTINCT co.person_id
  FROM `demo_omop.condition_occurrence` co
  WHERE co.condition_concept_id IN (
    SELECT descendant_concept_id FROM `demo_omop.concept_ancestor`
    WHERE ancestor_concept_id = 201826
  )
) c USING(person_id)
WHERE p.gender_concept_id = 8532  -- female
  AND p.year_of_birth BETWEEN 1955 AND 1975;
  6. Find diabetes medications in the same cohort. Practise joining drug_exposure with concept_ancestor on RxNorm ingredient ancestors.
  7. Apply partitioning to a table. Re-create condition_occurrence partitioned by condition_start_date and clustered by person_id. Run the same query against both versions; compare bytes scanned:
    CREATE TABLE demo_omop.condition_occurrence_part PARTITION BY condition_start_date CLUSTER BY person_id AS SELECT * FROM demo_omop.condition_occurrence;
  8. Tear down when done:
    bq rm -r -d demo_omop
    gcloud storage rm -r gs://$PROJECT-demo-omop/
