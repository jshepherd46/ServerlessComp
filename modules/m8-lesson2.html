DLP / Sensitive
Data Protection at scale

L1 used DLP as "the thing the Healthcare API de-id calls under the hood." This lesson is DLP standalone — InfoTypes, templates, the inspect/de-id/redact operations, and how to run DLP across petabytes of unstructured text with Dataflow.

⏱ ~50 min reading + exercise 🔧 Cloud DLP / Sensitive Data Protection, Dataflow, BigQuery 📍 Module 8 of 12
The framing

DLP is the pattern-matching layer, not the data store

Cloud DLP (also called Sensitive Data Protection) is a managed service that scans text, images, and structured data for InfoTypes — patterns like phone numbers, SSNs, MRNs — and can mask, redact, tokenize, or hash matches. It does not store your data; it consumes it, returns findings or transformed output, and forgets.

Three things DLP is good at, ranked by frequency of use:

  • → De-identifying free text. The clinical-note de-id step in M8L1's Healthcare API de-id config calls DLP under the hood. Direct API access for custom pipelines.
  • → Inspecting a corpus for PHI leaks. Audit a GCS bucket or BigQuery table for accidental PHI; report findings; remediate.
  • → Discovering sensitive columns automatically. Run DLP against a new dataset to flag which columns hold what kinds of sensitive data — useful for IRB review of new data sources.
What DLP is not. It's not a database for sensitive data. It's not an encryption service (that's Cloud KMS / CMEK from M3L3). It's not a real-time PHI detector at the network layer (that's a different problem). It's pattern matching at scale, applied to content you hand it.
What DLP detects

InfoTypes — built-in and custom

DLP comes with hundreds of built-in InfoTypes — named patterns plus the detector logic for each. They split into categories. For clinical research, three categories matter most.

Medical / health InfoTypes

MEDICAL_RECORD_NUMBER US_HEALTHCARE_NPI US_DEA_NUMBER ICD9_CODE ICD10_CODE MEDICAL_TERM FDA_CODE

US PII InfoTypes (most overlap with HIPAA's 18 identifiers)

PERSON_NAME DATE_OF_BIRTH US_SOCIAL_SECURITY_NUMBER PHONE_NUMBER EMAIL_ADDRESS STREET_ADDRESS US_DRIVERS_LICENSE_NUMBER US_PASSPORT CREDIT_CARD_NUMBER IP_ADDRESS

Custom InfoTypes — for UH-specific patterns

UH_INTERNAL_PATIENT_ID QUEENS_MRN_PATTERN UH_PROVIDER_INITIALS

Custom InfoTypes are defined as either regex, dictionary (a list of known terms), or stored (compiled and reused). Useful when your institution has identifier patterns that don't match the built-ins.

# A regex-based custom InfoType for UH internal patient IDs
{
  "infoType": {"name": "UH_INTERNAL_PATIENT_ID"},
  "regex": {"pattern": "UHID-[0-9]{8}"},
  "likelihood": "VERY_LIKELY"
}

# A dictionary-based custom InfoType for UH provider initials
{
  "infoType": {"name": "UH_PROVIDER_INITIALS"},
  "dictionary": {
    "wordList": {"words": ["JS-MD", "TS-MD", "RG-MD", "PB-PhD"]}
  }
}
The likelihood field. DLP returns a likelihood per finding (POSSIBLE, LIKELY, VERY_LIKELY). You configure inspection to filter at a threshold — usually LIKELY for high-recall PHI removal, VERY_LIKELY for lower-noise reporting. Setting it to POSSIBLE produces lots of false positives; setting it to VERY_LIKELY may miss real PHI. Tune per workload.
Three operations

Inspect, de-identify, redact

inspect

"What sensitive content is here?"

Scans the content, returns the locations and types of every match. Does not modify the content. The diagnostic operation — use to audit a corpus or to discover the InfoType distribution before designing a de-id config.

gcloud dlp jobs create inspect-gcs \
  --project=$PROJECT \
  --gcs-uri=gs://srl-shepbert-landing/notes/ \
  --info-types=PERSON_NAME,MEDICAL_RECORD_NUMBER,PHONE_NUMBER \
  --output-table=bq://$PROJECT.dlp_findings.scan_results
deidentify

"Replace matches with surrogates."

Transforms content according to a config — replace each PERSON_NAME with [NAME-1], replace each PHONE_NUMBER with [REDACTED-PHONE], etc. Returns the transformed content. The operation you call from production de-id pipelines.

{
  "deidentifyConfig": {
    "infoTypeTransformations": {
      "transformations": [
        {
          "infoTypes": [{"name": "PERSON_NAME"}],
          "primitiveTransformation": {
            "replaceConfig": {"newValue": {"stringValue": "[NAME]"}}
          }
        },
        {
          "infoTypes": [{"name": "PHONE_NUMBER"}],
          "primitiveTransformation": {"redactConfig": {}}
        },
        {
          "infoTypes": [{"name": "DATE"}],
          "primitiveTransformation": {
            "dateShiftConfig": {"upperBoundDays": 30, "lowerBoundDays": -30}
          }
        }
      ]
    }
  }
}
redact

"Remove matches from images."

Specifically for image content — DLP runs OCR + redacts identified bounding boxes. Used in the Healthcare API DICOM de-id for burned-in PHI; rarely called standalone.

Transformation types worth knowing

  • → replaceConfig — substitute a constant string
  • → replaceWithInfoTypeConfig — substitute the InfoType name (e.g., "John" → "[PERSON_NAME]")
  • → redactConfig — remove the match entirely
  • → characterMaskConfig — partial mask (e.g., XXXX-XXXX-4321)
  • → cryptoHashConfig — keyed hash; consistent across occurrences with the same key, can be linked but not reversed
  • → cryptoDeterministicConfig — encrypted with a CMEK key; reversible by anyone with the key
  • → dateShiftConfig — shift by a random offset (preserves intervals)

The right combination depends on the use case: cryptoHash for linkability without reversibility (research surrogate IDs), dateShift for temporal preservation, redact for "this should not exist," replaceWithInfoType for readability when humans review.

Reuse

DLP templates — configurations as code

Configuring 15 transformations inline on every API call is tedious and error-prone. DLP templates let you save a configuration once and reference it by ID. Two kinds:

  • → InspectTemplate — a named inspect configuration (InfoTypes + likelihood threshold + custom InfoTypes).
  • → DeidentifyTemplate — a named de-identify configuration (transformations + surrogate keys).

Creating reusable templates

# A shared DeidentifyTemplate for all SRL clinical-text de-id
cat > srl-deid-template.json <<EOF
{
  "displayName": "SRL clinical text de-id v2",
  "deidentifyConfig": {
    "infoTypeTransformations": {
      "transformations": [
        {"infoTypes":[{"name":"PERSON_NAME"}],
         "primitiveTransformation":{"replaceWithInfoTypeConfig":{}}},
        {"infoTypes":[{"name":"MEDICAL_RECORD_NUMBER"},{"name":"UH_INTERNAL_PATIENT_ID"}],
         "primitiveTransformation":{
           "cryptoHashConfig":{
             "cryptoKey":{"kmsWrapped":{"cryptoKeyName":"projects/srl-shared-services/locations/us-west1/keyRings/dlp-keys/cryptoKeys/srl-surrogate"}}}}},
        {"infoTypes":[{"name":"PHONE_NUMBER"}],"primitiveTransformation":{"redactConfig":{}}},
        {"infoTypes":[{"name":"DATE_OF_BIRTH"},{"name":"DATE"}],
         "primitiveTransformation":{"dateShiftConfig":{"upperBoundDays":30,"lowerBoundDays":-30,
           "cryptoKey":{"kmsWrapped":{"cryptoKeyName":"projects/srl-shared-services/locations/us-west1/keyRings/dlp-keys/cryptoKeys/srl-dateshift"}}}}}
      ]
    }
  }
}
EOF

gcloud dlp deidentify-templates create --location=us-west1 \
  --project=$PROJECT \
  --template-file=srl-deid-template.json \
  --template-id=srl-clinical-text-v2

Now every pipeline references projects/$PROJECT/locations/us-west1/deidentifyTemplates/srl-clinical-text-v2 instead of duplicating the config. Update the template once; every pipeline picks up the new behaviour.

The shared-services play. Put the canonical SRL de-id template in srl-shared-services (same pattern as the Artifact Registry, KMS keyring, OMOP vocabulary from M1L2). Per-study de-id pipelines reference the shared template. The lab gets one place to evolve the de-id rule set; every study inherits improvements automatically.
Scaling out

DLP + Dataflow for petabyte-scale work

DLP's per-request API is fine for thousands of items. For tens of millions of records — a multi-year EHR backfill, a full historical DICOM pixel scan, a corpus migration — you wrap DLP in a Dataflow pipeline that parallelises calls and writes results back to BigQuery or GCS.

The pattern

BigQuery / GCS source # 100M clinical notes ▼ Dataflow pipeline # auto-scaling workers ├─ Read source ├─ Batch into chunks (~500 items per request) ├─ Call DLP deidentify with template "srl-clinical-text-v2" ├─ Receive transformed content └─ Write to sink ▼ BigQuery / GCS sink # de-id\'d note corpus

Dataflow templates — Google ships pre-built ones

Google maintains pre-built Dataflow templates for the most common DLP shapes — you don't have to write the Apache Beam code yourself:

  • → Data Masking/Tokenization from Cloud Storage to BigQuery — flat-file in, BQ out.
  • → BigQuery to BigQuery (DLP) — table in, transformed table out.
  • → Pub/Sub to BigQuery (DLP) — streaming de-id on ingest.

Launching one of these is a single gcloud command:

gcloud dataflow flex-template run srl-deid-backfill \
  --template-file-gcs-location=gs://dataflow-templates/latest/flex/Stream_DLP_GCS_Text_to_BigQuery \
  --region=us-west1 \
  --parameters=^~^inputFilePattern=gs://srl-shepbert-landing/notes/*.txt~outputTableSpec=$PROJECT:demo_dlp.deid_notes~deidentifyTemplateName=projects/$PROJECT/locations/us-west1/deidentifyTemplates/srl-clinical-text-v2~datasetName=demo_dlp \
  --service-account-email=dataflow-sa@$PROJECT.iam.gserviceaccount.com \
  --subnetwork=https://www.googleapis.com/compute/v1/projects/$PROJECT/regions/us-west1/subnetworks/srl-prod-subnet

Cost reality at scale

DLP per-request pricing is by content size + InfoType complexity. Rough order-of-magnitude:

  • → 100K clinical notes with standard InfoTypes: ~$5-10. Fits in a single Dataflow run; turnaround minutes to an hour.
  • → 10M notes: ~$500-1000. Plan it; run during off-hours; checkpoint.
  • → Full 5-year UH backfill: ~$5K-15K depending on note volume. Budget it like a major data-engineering effort; design + dry-run a 1% sample first.
The discipline: always dry-run a 0.1–1% sample, inspect the de-id output by eye for the first run of any new template, validate the cost projection before authorising the full scan. Custom InfoTypes can be expensive; the validation step catches regex backreferences that scale poorly.
Hands-on exercise 8.2

DLP template + inspect + deidentify

🛡 The de-id pipeline in miniature
  1. Enable DLP:
    gcloud services enable dlp.googleapis.com
  2. Inspect a synthetic note with a built-in DLP request (no PHI in the test text): 
    cat > inspect-req.json <<EOF
{
  "item": {"value": "Patient John Doe was seen by Dr. Smith. Phone: 808-555-1234. MRN: UHID-12345678."},
  "inspectConfig": {
    "infoTypes": [
      {"name": "PERSON_NAME"},
      {"name": "PHONE_NUMBER"},
      {"name": "MEDICAL_RECORD_NUMBER"}
    ],
    "minLikelihood": "LIKELY",
    "includeQuote": true
  }
}
EOF

curl -X POST -H "Authorization: Bearer $(gcloud auth print-access-token)" \
  -H "Content-Type: application/json" \
  -d @inspect-req.json \
  "https://dlp.googleapis.com/v2/projects/$PROJECT/locations/us-west1/content:inspect"
    The response shows each finding with type, likelihood, and the matched substring.
  3. De-identify the same text with a one-shot config: 
    cat > deid-req.json <<EOF
{
  "item": {"value": "Patient John Doe was seen by Dr. Smith. Phone: 808-555-1234. MRN: UHID-12345678."},
  "deidentifyConfig": {
    "infoTypeTransformations": {
      "transformations": [
        {"infoTypes":[{"name":"PERSON_NAME"}],"primitiveTransformation":{"replaceWithInfoTypeConfig":{}}},
        {"infoTypes":[{"name":"PHONE_NUMBER"}],"primitiveTransformation":{"redactConfig":{}}},
        {"infoTypes":[{"name":"MEDICAL_RECORD_NUMBER"}],"primitiveTransformation":{"characterMaskConfig":{"maskingCharacter":"*","numberToMask":-1}}}
      ]
    }
  },
  "inspectConfig": {"infoTypes":[{"name":"PERSON_NAME"},{"name":"PHONE_NUMBER"},{"name":"MEDICAL_RECORD_NUMBER"}],"minLikelihood":"LIKELY"}
}
EOF

curl -X POST -H "Authorization: Bearer $(gcloud auth print-access-token)" \
  -H "Content-Type: application/json" \
  -d @deid-req.json \
  "https://dlp.googleapis.com/v2/projects/$PROJECT/locations/us-west1/content:deidentify"
    The response includes the de-id'd text: Patient [PERSON_NAME] was seen by Dr. [PERSON_NAME]. Phone:. MRN: ****************.
  4. Add a custom InfoType for UHID and rerun: 
    cat > deid-with-custom.json <<EOF
{
  "item": {"value": "Patient ID is UHID-12345678 visited on 2025-04-15."},
  "deidentifyConfig": {
    "infoTypeTransformations": {
      "transformations": [
        {"infoTypes":[{"name":"UH_INTERNAL_PATIENT_ID"}],"primitiveTransformation":{"replaceWithInfoTypeConfig":{}}}
      ]
    }
  },
  "inspectConfig": {
    "customInfoTypes": [{
      "infoType":{"name":"UH_INTERNAL_PATIENT_ID"},
      "regex":{"pattern":"UHID-[0-9]{8}"},
      "likelihood":"VERY_LIKELY"
    }]
  }
}
EOF

curl -X POST -H "Authorization: Bearer $(gcloud auth print-access-token)" \
  -H "Content-Type: application/json" \
  -d @deid-with-custom.json \
  "https://dlp.googleapis.com/v2/projects/$PROJECT/locations/us-west1/content:deidentify"
    The UHID is now masked as [UH_INTERNAL_PATIENT_ID].
  5. Save a reusable DeidentifyTemplate using the gcloud CLI (or the API), then call the deidentify endpoint with deidentifyTemplateName instead of inline config.
  6. Bonus — launch a Dataflow DLP job. Drop a few synthetic .txt files into a GCS bucket; run the Stream_DLP_GCS_Text_to_BigQuery Dataflow template; verify a BigQuery output table with de-identified content.
  7. Tear down: delete the DLP templates, any Dataflow jobs, the test bucket/dataset.

Questions to sit with
